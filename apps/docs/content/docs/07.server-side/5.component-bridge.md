::caution
This page is for packaged integrations and advanced host-app seams. **Most app teams can ignore it.** If you're not building a reusable Trellis-aware package for other apps to consume, skip to [Server routes](/docs/server-side/server-routes).
::

## What it is

`createComponentBridge(...)` forwards an explicit [principal](/docs/concepts/glossary#principal) into Convex component refs, and exposes stable root-level refs from the host app. It's the seam for packaged components that expect to be mounted into a Trellis runtime.

```ts [convex/bridge.ts]
const bridge = createComponentBridge(builders, { principal })
```

It forwards identity. It does not replace business authorization — guards still run on every call.

This is separate from the normal app-local feature boundary. In a regular Trellis app, a top-level feature component already acts like a mini-Trellis boundary: thin route shell outside, one backend-facing seam inside.

## When it's worth using

Reach for it when:

- You're shipping a Trellis-aware package (component-style integration) that host apps install.
- The host app needs a durable root-level surface — stable across internal component refactors.
- You want a contract that doesn't require consumers to call internal component refs directly.

For everything else — normal routes, webhooks, your own business logic — stay on `serverConvex*` helpers and regular root handlers. They're simpler.

## Host bridge workflow

Bridge manifests are package infrastructure consumed by bridge-owned tooling.
A bridge-aware component owns the manifest; `@lupinum/trellis-bridge` owns
installation and drift checks.

Packages can declare two kinds of host ownership through that manifest:

- generated host bridge files such as `convex/auth.ts` or `convex/<package>/*`
- managed edits inside host-owned files such as `convex/convex.config.ts`

That means a packaged integration can keep the app's root config file app-owned while still asking Trellis to maintain one explicit managed block inside it.

This remains the stable automation seam for packaged Trellis component integrations — not a general-purpose codegen utility.

## Authoring a bridge-aware component

Expose a default bridge manifest at `./convex/manifest` from your package:

```ts [src/module/bridge-manifest.ts]
import { defineComponentBridgeManifest } from '@lupinum/trellis-bridge'

export default defineComponentBridgeManifest({
  packageName: '@acme/cms',
  version: '0.1.0',
  renderFiles: [
    {
      relativePath: 'convex/acmeCms/public.ts',
      content: 'export const list = query({ args: {}, handler: async () => [] })\n',
    },
  ],
  managedEdits: [
    {
      relativePath: 'convex/convex.config.ts',
      apply(current) {
        return (
          current ?? "import { defineApp } from 'convex/server'\n\nexport default defineApp()\n"
        )
      },
    },
  ],
})
```

Publish that file through `package.json`:

```json [package.json]
{
  "exports": {
    "./convex/manifest": {
      "types": "./convex/manifest.d.ts",
      "import": "./convex/manifest.js"
    }
  }
}
```

Consumers install and check the bridge explicitly through bridge-owned tooling:

```bash
pnpm exec trellis-bridge install @acme/cms
pnpm exec trellis-bridge check @acme/cms
```

If your Nuxt module needs startup validation, call the Node-only bridge helper
from app/module runtime code, not from Convex functions:

```ts
import { checkBridgeDrift, loadManifestFromPackage } from '@lupinum/trellis-bridge'

const manifest = await loadManifestFromPackage('@acme/cms', rootDir)
const drift = await checkBridgeDrift(manifest, rootDir)
if (drift.length > 0) {
  throw new Error('@acme/cms bridge files are missing or stale.')
}
```

## What it isn't

- Not a general authorization bypass.
- Not a replacement for guards or app-owned actor logic.
- Not a reason to stop modeling principals and actors cleanly.

::warning
If you're tempted to reach for the bridge because "it's easier than wiring an actor properly," stop. The bridge forwards identity; it doesn't invent one. A handler without a real actor still fails its guards.
::

## Common pitfalls

- Using the bridge to skip actor resolution. It doesn't skip anything — it just forwards a principal.
- Reaching for it in a regular app. Host-app flows work fine on `serverConvex*` helpers.
- Generating bridge files and then calling internal component refs anyway. Pick one path.
- Importing bridge package helpers from normal app code. They are for packaged
  integrations and host setup, not day-to-day feature work.
- Treating generated `convex/manifest.js` as source. Keep an authored TypeScript source and emit the package-facing manifest during build.
