Handle webhooks and automation traffic without pretending they're browser sessions. Trusted forwarding verifies the request came from a system you trust, then forwards who the real caller is and, optionally, which app [actor](/docs/concepts/glossary#actor) it is acting for — guards and authorization still run like any other call.

```ts [server/api/webhooks/task-create.post.ts]
import { api } from '#trellis/api'

export default defineEventHandler(async (event) => {
  await verifyWebhookSignature(event) // your webhook verification
  const body = await readBody(event)
  const actorUserId = process.env.WEBHOOK_BOT_AUTH_ID

  return await serverConvexMutation(
    event,
    api.tasks.create,
    { projectId: body.projectId, title: body.title },
    {
      auth: 'trusted',
      principal: {
        kind: 'service',
        serviceId: 'task-webhook',
        subject: 'service:task-webhook',
      },
      delegation: {
        subject: `user:${actorUserId!}`,
        reason: 'verified task webhook',
      },
    },
  )
})
```

Two checks, one flow: Nitro verifies the request is real, then the Convex handler runs on the trusted path with an explicit server-owned caller identity and optional delegation target.

## The rule that matters

::caution
Do not treat external server traffic like a browser session. "It's a webhook" is not a permission grant. The webhook is a caller identity; business authorization still applies.
::

A webhook with a valid signature can reach your endpoint. It still has to act _as_ someone — some user, some system bot — and guards still decide what that someone can do. Skipping this step is how teams ship webhook handlers that can do more than any human user.

## What Trellis verifies server-side

Trusted-forwarding payloads have to carry two fields or Trellis denies the call:

- `_trustedForwardingKey` — must match the expected secret.
- `_trustedForwarding.principalSubject` — the canonical caller identity the call is forwarding.

The secret lives in `CONVEX_TRUSTED_FORWARDING_KEY`, set both in your local env **and** in the Convex deployment handling the traffic.

If you also forward a `principal` for a richer caller shape, that forwarded principal only belongs on the same trusted path. Treat `principal` as verified transport metadata, not as a public request argument.

::warning
Missing `CONVEX_TRUSTED_FORWARDING_KEY` on the Convex side is the most common trusted-forwarding failure. Browser auth will work fine; trusted traffic will silently reject. Doctor catches this if you run it.
::

## Resolve a real app actor — don't fork policy

The right pattern for webhooks isn't "webhook can do anything." It's:

1. Verify the trusted forwarding key.
2. Resolve (or ensure) a real user row for the webhook — often a bot user like `webhook@yourapp.internal`.
3. Return an app actor (role, tenant) for that user.

That way webhook traffic goes through the same guards, the same permission checks, and the same tenant scoping as browser traffic. There's no second "integration bypass" policy model to maintain.

If your webhook needs different permissions from any human user, express that in the actor — a bot with a specific role — not by skipping the actor entirely.

## Browser auth vs. trusted forwarding auth

They solve different problems:

|                     | Browser auth                   | Trusted forwarding auth                                |
| ------------------- | ------------------------------ | ------------------------------------------------------ |
| Answers             | Who is the signed-in end user? | Is this server call allowed to inject a user identity? |
| Token source        | Session cookie → Convex JWT    | `CONVEX_TRUSTED_FORWARDING_KEY`                        |
| Replaces the other? | No                             | No                                                     |

A valid browser cookie doesn't substitute for trusted-forwarding verification on a webhook. A trusted forwarding key doesn't substitute for business authorization in the handler.

## `auth: 'none'` is narrower than it sounds

`auth: 'none'` is right for calling a genuinely public handler, or for a route that owns its whole security model. It's wrong as a way to dodge a missing trusted-forwarding setup.

If the Convex handler has a guard, and you're calling it with `auth: 'none'`, something's wrong — either the guard shouldn't be there, or the call shouldn't be `none`.

## Common pitfalls

- Calling protected Convex handlers from a webhook without trusted verification — the handler will reject, you'll debug "why is my webhook breaking" for an hour.
- Treating the trusted key as authorization instead of identity forwarding.
- Forgetting `CONVEX_TRUSTED_FORWARDING_KEY` on the Convex deployment (only setting it locally).
- Building a parallel "integration permissions" model instead of resolving a real actor with a role.
- Using `auth: 'none'` to silence a guard you should have configured properly.
