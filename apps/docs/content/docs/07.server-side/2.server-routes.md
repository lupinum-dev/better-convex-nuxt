Call a protected Convex function from a Nitro `/api/*` route — same function, same guards, same actor, different transport. The per-call helpers make the auth mode explicit so the route can't accidentally forge permissions.

```ts [server/api/tasks/export.get.ts]
import { api } from '#trellis/api'

export default defineEventHandler(async (event) => {
  const projectId = getQuery(event).projectId as string

  return await serverConvexQuery(
    event,
    api.tasks.listForExport,
    { projectId },
    { auth: 'required' },
  )
})
```

Four helpers, one shape: `serverConvexQuery`, `serverConvexMutation`, `serverConvexAction`, and `createServerConvexCaller` for multi-call routes.

## Pick the auth mode deliberately

The per-call helpers support an `auth` option. If you omit it, they default to `auto`. In practice, you should still choose the trust boundary deliberately:

| Mode | Who's the caller |
| --- | --- |
| `auto` | Use request auth when available, fall through if not |
| `required` | Fail if request auth can't be resolved |
| `none` | Never attach auth (use for public data) |
| `trusted` | Inject [trusted-caller](/apps/docs/server-side/webhooks-and-trusted-callers) identity, optionally with an explicit actor |

::warning
Don't reach for `trusted` to "just make it work." Trusted mode bypasses normal user identity — use it for server-to-server flows where the signal is real (signed webhook, trusted backend). For user-initiated routes, use `required`.
::

## Mutation from a Nitro route

Writes look the same. This example is a trusted-caller flow (webhook, admin tool — not a user request):

```ts [server/api/internal/tasks.post.ts]
import { api } from '#trellis/api'

export default defineEventHandler(async (event) => {
  const body = await readBody(event)

  return await serverConvexMutation(
    event,
    api.tasks.create,
    { projectId: body.projectId, title: body.title },
    {
      auth: 'trusted',
      actor: { userId: body.createdBy },
    },
  )
})
```

The `actor` field tells Trellis which app-level identity the trusted call should act as. Guards and authorization still run — you're not skipping business policy, you're declaring whose policy to run.

## Multi-call routes: one caller, many calls

When a single Nitro request needs several Convex calls, create a request-scoped caller instead of passing `event` and auth every time:

```ts [server/api/dashboard.get.ts]
import { api } from '#trellis/api'

export default defineEventHandler(async (event) => {
  const convex = createServerConvexCaller(event, { auth: 'required' })

  const [summary, activity, notifications] = await Promise.all([
    convex.query(api.dashboard.summary, {}),
    convex.query(api.dashboard.recentActivity, {}),
    convex.query(api.notifications.recent, {}),
  ])

  return { summary, activity, notifications }
})
```

::note
The caller is bound to this request. Don't cache it across requests — the event and auth are per-request, and a leaked caller would serve the wrong user.
::

## Passing `event` matters

The helpers need the H3 event to read auth from the request. Pass it explicitly when you're inside a normal route. If you try to call them somewhere Nitro request context isn't available, they throw a clear error — which is the right behavior. A helper that invented request state silently would be worse than one that fails.

## Errors include helper context

If a call fails, Trellis reports which helper was used, the function path, and the auth mode. Debugging a generic `fetch('/api/query')` wrapper means guessing; these errors tell you the exact shape of the failing call.

## What goes in the Nitro route vs. the Convex handler

Nitro routes should stay thin. They exist to:

- validate HTTP input (body shape, query params);
- pick the right auth mode;
- call the existing protected Convex function;
- shape the HTTP response.

They **should not** duplicate business authorization. That lives in the Convex handler's guard and authorization, where every transport path enforces it. If you find yourself re-checking permissions in the Nitro route, move the check into the Convex handler instead.

## Common pitfalls

- Relying on the default `auth: 'auto'` when the route really needs `required` or `trusted`.
- Using `trusted` for user-initiated routes. That's a way to ship auth bypasses.
- Duplicating permission checks in the Nitro layer. One source of truth (the Convex handler).
- Caching a `createServerConvexCaller` across requests.
