Reference for the public Trellis MCP runtime surface. Each tool definition projects the same backend authorization path as a browser call â€” transport changes, business rules don't.

## `defineMcpApp(options)`

Creates the MCP runtime and the tool factory bound to your Trellis backend model.

### Options

| Option                         | Purpose                                                          |
| ------------------------------ | ---------------------------------------------------------------- |
| `callConvex(event, principal)` | Convex caller the MCP runtime uses for every tool invocation     |
| `resolvePrincipal(event)`      | Map the transport request to a Trellis principal                 |
| `resolveCapabilities?(...)`    | Optional tool-visibility / capability snapshot layer             |
| `runtime?(...)`                | Optional per-request runtime data passed to tool handlers        |
| `principalKey?(principal)`     | Stable identity key for destructive confirmation + observability |
| `observability?`               | MCP runtime observability options (transport, correlation IDs)   |

### Returns

| Export                                   | Purpose                                                        |
| ---------------------------------------- | -------------------------------------------------------------- |
| `tool(options)`                          | Regular tool factory â€” use for read or non-destructive tools   |
| `tool.fromOperation(operation, options)` | Operation-backed tool factory â€” required for destructive tools |

## `defineTool(options)`

Standalone tool definition. Use inside `tool(...)` from `defineMcpApp`, or import directly for non-projected tools.

### Options

| Option          | Type                                   | Purpose                                                                 |
| --------------- | -------------------------------------- | ----------------------------------------------------------------------- |
| `schema`        | `ConvexSchema`                         | Shared input schema â€” validates args and feeds MCP metadata             |
| `call`          | `FunctionReference`                    | The Convex query, mutation, or action the tool invokes                  |
| `operation?`    | `'query' \| 'mutation' \| 'action'`    | Call kind; defaults to `'mutation'`                                     |
| `preview?`      | `FunctionReference \| PreviewResolver` | Read-only projection used to power confirmation UX                      |
| `capability?`   | `string`                               | Capability key used to gate tool visibility                             |
| `enabled?`      | `(ctx) => boolean \| Promise<boolean>` | Dynamic visibility check â€” principal and capabilities available         |
| `middleware?`   | `(args, ctx, next) => Result`          | Wrap the call with auth checks, rate limiting, or post-processing       |
| `mapResult?`    | `(ctx) => unknown`                     | Transform the raw Convex result before the envelope is built            |
| `summary?`      | `(ctx) => string`                      | Short human-readable line attached to successful results                |
| `respond?`      | `(ctx) => unknown`                     | Fully custom envelope shape â€” replaces the default `ok`/`error` wrapper |
| `outputSchema?` | `ZodRawShape`                          | Optional structured output schema for the result envelope               |
| `group?`        | `string`                               | Grouping label for tool listings                                        |
| `tags?`         | `string[]`                             | Tags used by capability snapshots and tool discovery                    |

::caution
Destructive generic tools aren't supported through `defineTool`. For destructive tools, use `defineMcpApp(...).tool.fromOperation(...)` â€” that path binds preview, confirmation, and execute to one operation identity.
::

::note
`capability` and `enabled` affect _tool visibility_ or _availability_, not backend business authorization. Guards on the handler still run on every call.
::

## `tool.fromOperation(operation, options)`

Preferred path for operation-backed tools, especially destructive ones. Accepts the same options as `defineTool` (except `schema`, `call`, `preview`, `operation`, `previewOperation` â€” those come from the operation itself).

### Options

| Option                                                                                                  | Type                                | Purpose                                                     |
| ------------------------------------------------------------------------------------------------------- | ----------------------------------- | ----------------------------------------------------------- |
| `execute`                                                                                               | `FunctionReference`                 | The mutation or action that runs the operation              |
| `preview?`                                                                                              | `FunctionReference`                 | Preview function reference (for destructive operations)     |
| `executeOperation?`                                                                                     | `'query' \| 'mutation' \| 'action'` | Call kind for `execute`                                     |
| `previewOperation?`                                                                                     | `'query' \| 'mutation' \| 'action'` | Call kind for `preview`                                     |
| `schema?`                                                                                               | `ConvexSchema`                      | Optional schema override â€” defaults to the operation's args |
| _(plus everything in `defineTool` except `schema`, `call`, `preview`, `operation`, `previewOperation`)_ |                                     |                                                             |

### Semantics

- Destructive operations **require** a preview projection â€” binding fails without one.
- Trellis binds confirmation tokens to operation identity **and** previewed state.
- Mismatched preview or execute refs are rejected at bind time, not at call time.

## Session and server helpers

| Helper            | Purpose                                                        |
| ----------------- | -------------------------------------------------------------- |
| `useMcpServer()`  | Access the active MCP server instance inside a request         |
| `useMcpSession()` | Per-session storage namespace (when `mcp.sessions` is enabled) |

## Result helpers

Normalize tool results into the expected structured MCP envelope:

| Helper                                  | Purpose                                  |
| --------------------------------------- | ---------------------------------------- |
| `wrapSuccess(data, summary?)`           | Success envelope                         |
| `wrapError(category, message, issues?)` | Error envelope                           |
| `wrapPreview(preview)`                  | Preview envelope (awaiting confirmation) |
| `withSummary(result, summary)`          | Add a summary line to a result           |

## Toolkit re-exports

Trellis re-exports MCP toolkit surfaces for composing the surrounding app:

- `defineMcpHandler`
- `defineMcpPrompt`
- `defineMcpResource`
- `defineMcpTool`

Trellis-specific business alignment still lives in `defineMcpApp(...)` and the Trellis tool factory. When the tool layer needs a stable automation surface, project root handlers or bridge refs into the MCP runtime instead of duplicating business behavior in tool files.
