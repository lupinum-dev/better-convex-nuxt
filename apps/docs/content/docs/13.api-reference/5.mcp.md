Reference for the public Trellis MCP runtime surface. Each tool definition projects the same backend authorization path as a browser call: transport changes, business rules do not.

## `defineMcpApp(options)`

Creates the MCP runtime and the tool factory bound to your Trellis backend model.

### Options

| Option                         | Purpose                                                          |
| ------------------------------ | ---------------------------------------------------------------- |
| `callConvex(event, principal)` | Convex caller the MCP runtime uses for every tool invocation     |
| `resolvePrincipal(event)`      | Map the transport request to a Trellis principal                 |
| `resolveDelegation?(...)`      | Optional represented-user mapping for trusted forwarded calls    |
| `resolveCapabilities?(...)`    | Optional tool-visibility / capability snapshot layer             |
| `rateLimitStore?`              | Optional distributed MCP rate-limit store for tool `rateLimit`   |
| `runtime?(...)`                | Optional per-request runtime data passed to tool handlers        |
| `principalKey?(principal)`     | Stable identity key for destructive confirmation + observability |
| `observability?`               | MCP runtime observability options (transport, correlation IDs)   |

### Returns

| Export                                   | Purpose                                                        |
| ---------------------------------------- | -------------------------------------------------------------- |
| `tool(options)`                          | Regular tool factory — use for read or non-destructive tools   |
| `tool.fromOperation(operation, options)` | Operation-backed tool factory — required for destructive tools |

## `defineTool(options)`

Standalone tool definition for custom MCP handlers.

Use this when you want to implement the tool body directly in the MCP layer. If you want Trellis to project a protected Convex ref with permission metadata, prefer `tool(...)` from `defineMcpApp(...)`.

### Core options

| Option               | Type                                       | Purpose                                                                        |
| -------------------- | ------------------------------------------ | ------------------------------------------------------------------------------ |
| `schema`             | `ConvexSchema`                             | Shared input schema for validation and MCP metadata                            |
| `handler`            | `(args, ctx) => unknown`                   | Tool implementation body                                                       |
| `operation?`         | `'query' \| 'mutation' \| 'action'`        | MCP annotation baseline; defaults to `'mutation'`                              |
| `auth?`              | `'required' \| 'optional' \| 'none'`       | Transport auth requirement                                                     |
| `scoped?`            | `boolean`                                  | Require authenticated actor with `tenantId`; only valid with `auth:'required'` |
| `check?`             | `(actor) => boolean \| Promise<boolean>`   | Extra actor check re-run at discovery and execution                            |
| `resolveAuth?`       | `(event) => actor \| null`                 | Override default auth extraction                                               |
| `resolvePrincipal?`  | `({ event, actor }) => unknown`            | Optional richer principal mapping for trusted forwarded Convex calls           |
| `resolveDelegation?` | `({ event, actor }) => Delegation \| null` | Optional explicit represented-user mapping for trusted forwarded Convex calls  |
| `rateLimitStore?`    | `McpRateLimitStore`                        | Optional distributed store for tool `rateLimit`                                |
| `middleware?`        | `(args, ctx, next) => Result`              | Wrap the handler with validation or policy logic                               |
| `enabled?`           | `(event) => boolean \| Promise<boolean>`   | Hide or expose the tool before auth/check evaluation                           |

::caution
Destructive generic tools aren't supported through `defineTool`. For destructive tools, use `defineMcpApp(...).tool.fromOperation(...)` — that path binds preview, confirmation, and execute to one operation identity.
::

::note
`capability` and `enabled` affect _tool visibility_ or _availability_, not backend business authorization. Guards on the handler still run on every call.
::

::note
`resolveDelegation` stays explicit. If you omit it, Trellis forwards no represented user.
::

## `tool.fromOperation(operation, options)`

Preferred path for operation-backed tools, especially destructive ones. Accepts the same options as `defineTool` (except `schema`, `call`, `preview`, `operation`, `previewOperation` — those come from the operation itself).

### Options

| Option                                                                                                  | Type                                | Purpose                                                     |
| ------------------------------------------------------------------------------------------------------- | ----------------------------------- | ----------------------------------------------------------- |
| `execute`                                                                                               | `FunctionReference`                 | The mutation or action that runs the operation              |
| `preview?`                                                                                              | `FunctionReference`                 | Preview function reference (for destructive operations)     |
| `executeOperation?`                                                                                     | `'query' \| 'mutation' \| 'action'` | Call kind for `execute`                                     |
| `previewOperation?`                                                                                     | `'query' \| 'mutation' \| 'action'` | Call kind for `preview`                                     |
| `schema?`                                                                                               | `ConvexSchema`                      | Optional schema override — defaults to the operation's args |
| _(plus everything in `defineTool` except `schema`, `call`, `preview`, `operation`, `previewOperation`)_ |                                     |                                                             |

### Semantics

- Destructive operations **require** a preview projection — binding fails without one.
- Trellis binds confirmation tokens to operation identity **and** previewed state.
- Mismatched preview or execute refs are rejected at bind time, not at call time.
- `execute` and `preview` must be the real exported projections of the same operation.

## Session and server helpers

| Helper            | Purpose                                                        |
| ----------------- | -------------------------------------------------------------- |
| `useMcpServer()`  | Access the active MCP server instance inside a request         |
| `useMcpSession()` | Per-session storage namespace (when `mcp.sessions` is enabled) |

## Result helpers

Normalize tool results into the expected structured MCP envelope:

| Helper                                  | Purpose                                  |
| --------------------------------------- | ---------------------------------------- |
| `wrapSuccess(data, summary?)`           | Success envelope                         |
| `wrapError(category, message, issues?)` | Error envelope                           |
| `wrapPreview(preview)`                  | Preview envelope (awaiting confirmation) |
| `withSummary(result, summary)`          | Add a summary line to a result           |

## Toolkit re-exports

Trellis re-exports MCP toolkit surfaces for composing the surrounding app:

- `defineMcpHandler`
- `defineMcpPrompt`
- `defineMcpResource`
- `defineMcpTool`

Trellis-specific business alignment still lives in `defineMcpApp(...)` and the Trellis tool factory. When the tool layer needs a stable automation surface, project root handlers or bridge refs into the MCP runtime instead of duplicating business behavior in tool files.

`@nuxtjs/mcp-toolkit` owns MCP handler/session/runtime behavior. Trellis layers Convex caller construction, business-safe projection, structured result envelopes, and destructive confirmation semantics on top of those toolkit primitives.
