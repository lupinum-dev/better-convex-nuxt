Reference for `trellis.permissions`. One key points `usePermissions()` and `useAuthGuard()` at your backend-owned permission-context query. The optional codegen flag adds tighter types and doctor metadata without changing runtime behavior.

## Two shapes

```ts [nuxt.config.ts]
trellis: {
  permissions: 'permissions/context.getPermissionContext',
}
```

```ts [nuxt.config.ts]
trellis: {
  permissions: {
    query: 'permissions/context.getPermissionContext',
    codegen: true,
  },
}
```

Both work. Use the shorthand for the common case; use the object when you need codegen or future options.

## What it wires

| Composable | Reads from |
| --- | --- |
| `usePermissions()` | The configured query's result |
| `useAuthGuard({ permission })` | The `can` map in that result |

## `codegen`

Optional and additive:

```ts [nuxt.config.ts]
trellis: {
  permissions: {
    query: 'permissions/context.getPermissionContext',
    codegen: true,
  },
}
```

You can also pass an object when you need a non-canonical include list:

```ts [nuxt.config.ts]
trellis: {
  permissions: {
    query: 'permissions/context.getPermissionContext',
    codegen: {
      include: ['convex/auth/permissions.ts'],
    },
  },
}
```

When enabled, Trellis emits:

- `.nuxt/types/trellis-permissions.d.ts` for typed permission-key/module augmentation
- `.nuxt/trellis/permissions.json` for `trellis doctor` and devtools inspection

This does not generate runtime behavior and it does not try to analyze record-bound checks.

## Important constraint

::warning
The string must point at a real Convex query export path. If it doesn't, Trellis raises a *configured-function error* at setup — it won't silently do nothing. Fix the path; the error tells you exactly which export it couldn't find.
::
