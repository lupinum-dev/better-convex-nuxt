Model destructive work — delete, archive, revoke — as one [operation](/docs/concepts/glossary#operation). The guard, the load, the authorization check, the preview shown to the user, and the actual delete all come from one definition instead of three functions that have to agree with each other.

```ts [convex/features/todos/domain.ts]
export const removeTodoOp = defineOperation({
  id: 'todos.remove',
  name: 'removeTodo',
  kind: 'destructive',
  args: deleteTodo.args,
  guard: canReadTodo,
  load: async (ctx, args) => {
    const todo = await ctx.db.get(args.id)
    requireRecord(todo, 'Todo')
    return { todo }
  },
  authorize: {
    check: (_actor, { todo }) => canDeleteTodo(todo),
  },
  preview: async (_ctx, _args, { todo }) =>
    operationPreview({
      summary: `Will permanently delete "${todo.title}"`,
      warnings: [
        operationIssue({
          code: 'permanent-delete',
          message: 'This cannot be undone.',
        }),
      ],
      effects: [
        operationEffect({
          type: 'delete',
          target: 'todo',
          count: 1,
        }),
      ],
      confirm: {
        operation: 'todos.remove',
        targetId: todo._id,
        affectedCounts: { todos: 1 },
      },
    }),
  handler: async (ctx, args) => {
    await ctx.db.delete(args.id)
    return null
  },
})

export const remove = mutation.protected(removeTodoOp)
export const previewRemove = query.protected(previewOf(removeTodoOp))
```

Two projections, one definition. The confirm dialog reads the preview query; the
delete button calls the mutation. They can't drift apart because they both trace
back to `removeTodoOp`.

Preview describes the work. Execute does the work. A destructive preview may
read current state and return a neutral operation preview envelope plus durable
confirmation material, but it must not mutate app data, call external systems,
enqueue jobs, send notifications, or perform irreversible work.

## Runtime setup is mandatory

Destructive operations aren't opt-in. You have to configure `destructiveOperations` in `defineTrellis(...)` before registering one — otherwise Trellis rejects the registration at boot.

```ts [convex/functions.ts]
export const { mutation, query } = defineTrellis(
  { query: generatedQuery, mutation: generatedMutation },
  {
    appIdentity: getAppIdentity,
    isolation: { tables: ['todos'] },
    destructiveOperations: {
      confirmationTable: 'destructiveConfirmations' as never,
      auditTable: 'destructiveAuditLog' as never,
    },
  },
)
```

::note
Those tables are used by the MCP destructive-tool path (token confirmation, drift detection, audit). They're required at config time so the runtime can't silently go without them when an agent binds a destructive tool later.
::

## Browser flow

The common shape: select the record, fetch the preview, show it in a confirm dialog, execute if the user confirms.

```vue [app/components/DeleteTodoDialog.vue]
<script setup lang="ts">
import { api } from '#trellis/api'

const selectedId = ref<string | null>(null)

const previewArgs = computed(() =>
  selectedId.value ? { id: selectedId.value as never } : undefined,
)

const { data: preview, pending: previewPending } = await useConvexQuery(
  api.todos.previewRemove,
  previewArgs,
  { server: false, subscribe: false },
)

const removeTodo = useConvexMutation(api.todos.remove, {
  onSuccess: () => {
    selectedId.value = null
  },
})

async function confirmRemove() {
  if (!selectedId.value) return
  await removeTodo({ id: selectedId.value as never })
}
</script>
```

The important property isn't the dialog — it's that preview and execute are
projected from the same operation. A preview result does not authorize the later
mutation. Execute still runs the operation's guard, load, authorize, tenant
binding, and drift checks before the handler deletes anything.

## What you own, what Trellis enforces

Destructive operations split cleanly between UI choices and structural guarantees.

| Trellis enforces                                                     | You own                                                        |
| -------------------------------------------------------------------- | -------------------------------------------------------------- |
| `kind: 'destructive'` required                                       | Whether to open a modal                                        |
| `id` required and stable                                             | How the preview is rendered                                    |
| `preview(...)` must be defined                                       | Whether to re-fetch preview before confirm                     |
| Preview is side-effect-free                                          | Modal copy and product-specific friction                       |
| Execute re-runs guard, load, authorize, tenant, and drift checks     | Typed confirmation, extra clicks, or other browser friction    |
| `destructiveOperations` config required at boot                      | Whether browser flows opt into stricter token-backed execution |
| MCP tool surface: token-backed confirm, confirmation, drift checking | —                                                              |

::warning
Browser mutations **don't** automatically go through the token-backed
confirmation that MCP destructive tools require. If you want that stricter path
for browser flows too, opt into it explicitly; do not assume a confirm dialog is
the same thing as MCP confirmation confirmation.
::

## When a plain mutation is clearer

Don't wrap every scary-sounding write in an operation. Keep it a plain mutation when:

- the action isn't actually destructive (toggling a flag, renaming a field);
- there's no preview worth showing;
- the flow is local, one-shot, and won't be exposed to agents or admin UI;
- the team doesn't need the operation as a reusable contract.

Operations pay off when the destructive path has to work the same way across browser UI, admin surfaces, server callers, and agents. For a one-off button on a settings page, a mutation is fine.

## Common pitfalls

- Writing preview and execute as two separate functions and hoping they stay in sync. They won't.
- Putting side effects in preview. Preview may read and compute; execute mutates.
- Treating a preview result as permission to skip execute-time checks.
- Assuming browser calls enforce the MCP confirmation-token path. They don't — that's an MCP-surface guarantee.
- Turning every write into an operation because the primitive exists.
- Forgetting `destructiveOperations` in `defineTrellis(...)` and getting a confusing boot error.
