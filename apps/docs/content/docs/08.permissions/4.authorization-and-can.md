When a permission decision depends on a loaded record, don't stretch the entry guard past its job. Load first, authorize second. That keeps the coarse gate and the resource-bound rule in their own phases.

This is the main Trellis authorization path for application policy. If the rule is about the business meaning of a record, keep it here, not in a DB policy layer.

```ts [convex/features/todos/domain.ts]
export const setCompleted = mutation.protected({
  args: setTodoCompleted.args,
  guard: todoRead,
  load: async (ctx, args) => {
    const todo = await ctx.db.get(args.id)
    requireRecord(todo, 'Todo')
    return { todo }
  },
  authorize: {
    check: (_actor, { todo }) => canUpdateTodo(todo),
  },
  handler: async (ctx, args) => {
    await ctx.db.patch(args.id, { completed: args.completed })
  },
})
```

Four phases, each one readable on its own: guard, load, authorize, handler.

For the common case, `authorize` can stay terse:

```ts
authorize: (actor, { todo }) => actor.userId === todo.ownerId
authorize: {
  label: 'todo.update',
  check: (_actor, { todo }) => canUpdateTodo(todo)
}
```

Use the `(actor, loaded)` form for inline one-offs. Use the object form when the decision is naturally loaded-resource specific or when you want an explicit label. Trellis 1.0 does not infer one-argument `authorize` callbacks from function arity; old loaded-resource factory callbacks should move to object form.

## The four-phase shape

| Phase       | Job                                    | Has loaded data? |
| ----------- | -------------------------------------- | ---------------- |
| `guard`     | Reject obviously invalid callers early | No               |
| `load`      | Fetch the records the decision needs   | —                |
| `authorize` | Decide against loaded data             | Yes              |
| `handler`   | Do the work                            | Yes              |

`guard` is cheap and coarse. `authorize` is specific and resource-bound. Splitting them means a handler can say "you need read access to see this todo _and_ owner-or-admin to modify it" without one enormous predicate.

Keep `guard` synchronous and DB-free. If the decision needs `ctx.db`, it belongs in `load` plus `authorize`, not in the entry gate.

::note
Tenant isolation is a separate runtime guardrail. It answers "is this row in the active tenant?" It does not answer "may this actor perform this business action?"
::

## Backend `can(...)` evaluates checks safely

Trellis exports a backend helper:

```ts
import { can } from '@lupinum/trellis'

if (can(actor, hasRole('owner', 'member'))) {
  // ...
}
```

`can(actor, check)` returns `false` when the check fails — including when it throws a `ConvexError`. That's why Trellis examples use it inside permission-context projection instead of wrapping every decision in `try/catch`.

## Frontend `allows(...)` reads a projection

`usePermissions().allows(...)` is intentionally not the backend `can(...)` helper:

```ts
const { allows } = usePermissions()

const canCreateProject = allows(projectCreate)
```

::caution
The browser's `allows(...)` is a projection of backend policy, not a policy engine. It reads booleans computed on the server. Don't import backend guards into frontend code and re-run them — that creates two sources of truth.
::

| Helper                                | Runs where | What it does                                      |
| ------------------------------------- | ---------- | ------------------------------------------------- |
| `can(actor, check)`                   | Backend    | Evaluates a backend check safely                  |
| `usePermissions().allows(permission)` | Browser    | Reads a projected boolean from permission context |

## Permission context should expose stable UI decisions

The normal projection path:

```ts [convex/permissions/context.ts]
import { projectCreate } from '../auth/permissions'

export const getPermissionContext = query.protected(
  definePermissionContext({
    resolve: getActor,
    permissions: [projectCreate],
    extend: async (ctx, actor) => {
      const memberships = await getMemberships(
        ctx.db.escapeTenantIsolation({
          reason: 'Agency dashboard context spans memberships across workspaces.',
        }),
        actor.userId,
      )

      return {
        agencyDashboard: memberships.some((m) =>
          ['agency_admin', 'agency_manager'].includes(m.role),
        ),
      }
    },
  }),
)
```

The browser receives decisions from the same backend model that runs on handlers. Static named permissions live in `can`. Extra UI state that is not a projected static permission belongs in `extend`.

## What belongs in permission context

| Good fit                                                        | Bad fit                                                          |
| --------------------------------------------------------------- | ---------------------------------------------------------------- |
| Page-level actions and navigation states                        | Ephemeral one-record ownership for a row the page already loaded |
| Button visibility and disabled states                           | Giant per-record permission maps for large lists                 |
| Stable app capabilities (`project.create`, `workspace.members`) | Duplicated business logic copied from handlers                   |
| Plan and usage fields that affect UI affordances                | Anything that needs per-request args to decide                   |

Rule of thumb: if the question is "what can this user do in general," put it in permission context. If the question is "can this user edit this specific row," use `authorize` in the handler and read the resulting data in the UI.

## Common pitfalls

- Using `guard` alone when the decision depends on a loaded record. Move it to `authorize`.
- Trying to encode business authorization as DB policy. Keep app authorization in handler code so the action and the rule stay reviewable together.
- Recomputing backend policy in the browser. The browser projects; it doesn't decide.
- Confusing backend `can(...)` with `usePermissions().allows(...)`. They answer related questions on different sides of the boundary.
- Stuffing every permission detail into the permission context query so the UI "always has it." That bloats the hot path — scope it to stable capabilities.
