Read this page when the app has crossed from "signed-in" into "capability and workspace policy."

If you are still on the public lane or the first personal auth lane, skip this page for now. The first protected app does not need a permission context query yet.

Put the policy on the backend once. Project it to the UI with one query and one composable. You don't recompute permissions in the browser; you read what the server decided.

```ts [convex/permissions/context.ts]
import { todoCreate, todoRead } from '../auth/permissions'

export const getPermissionContext = query(
  definePermissionContext({
    resolve: getActor,
    permissions: [todoRead, todoCreate],
  }),
)
```

```vue [pages/index.vue]
<script setup lang="ts">
const { allows, ready } = usePermissions()
</script>
```

One source of truth. The UI is a projection of it.

## The shape of a permissions setup

Five steps, each in a different layer:

::steps{level="3"}

### Resolve an [actor](/apps/docs/concepts/glossary#actor) in Convex

From the principal (who called) to the app identity (who they are in your business model). See [Principal and actor](/apps/docs/permissions/principal-and-actor).

### Declare static permissions in app-owned backend code

Named handles for projected UI and MCP capabilities. Record-bound checks stay in `checks.ts`. See [Guards](/apps/docs/permissions/guards).

### Expose one permission-context query

`definePermissionContext(...)` returns a public query. The browser calls it; the server decides.

### Point Nuxt at that query

One line in `nuxt.config.ts`.

### Read it with `usePermissions()`

`allows(permission)` and `ready` in your components. That's it.

::

## Start from the scaffold

Fastest way to land on the shipped conventions:

```bash [terminal]
npx trellis init my-app --template workspace
```

You get the full Trellis shape:

- `convex/auth/principal.ts`
- `convex/auth/actor.ts`
- `convex/auth/checks.ts`
- `convex/auth/permissions.ts`
- `convex/functions.ts`
- `convex/permissions/context.ts`
- a `getPermissionContext` query

You can hand-write the same setup, but the workspace starter is the paved path.

## Define a permission context query

The backend query is the source of truth for the frontend auth context:

```ts [convex/permissions/context.ts]
import { teamWorkspacePermissions } from '../auth/permissions'

export const getPermissionContext = query(
  definePermissionContext({
    resolve: getActor,
    permissions: teamWorkspacePermissions,
    extend: async (ctx, actor) => {
      const user = await ctx.db
        .query('users')
        .withIndex('by_auth_id', q => q.eq('authId', actor.userId))
        .first()

      return {
        email: user?.email ?? null,
        displayName: user?.displayName ?? null,
      }
    },
  }),
)
```

| Field | What it does |
| --- | --- |
| `resolve` | Turns the principal into your app actor |
| `permissions` | Lists the named static permissions that project into `can` |
| `extend` | Adds app-owned fields to the context (plan, usage, display name) |

The base context always includes `userId`, `tenantId`, `role`, and `can`. You don't wire those by hand.

## Point Nuxt at that query

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  trellis: {
    auth: { enabled: true },
    permissions: {
      query: 'permissions/context.getPermissionContext',
    },
  },
})
```

If the path is wrong, Trellis fails loudly at setup and `usePermissions()` never goes ready — intentional, so a silent miss can't hide the real bug.

## Read permissions in the UI

```vue
<script setup lang="ts">
const { allows, ready, role, tenantId, ctx } = usePermissions()

const canCreateTodo = allows(todoCreate)
</script>
```

The composable is fail-closed by design:

- Before the permission context arrives, `ready` is `false`.
- Missing permission keys return `false` (not `undefined`).
- Stale context from a previous user is suppressed until the current session matches.

That's why the UI should branch on `ready` instead of guessing.

## Protect a page with permission state

When a whole page depends on one capability, skip the manual branching:

```ts [pages/audit.vue]
import { workspaceAudit } from '~/convex/auth/permissions'

useAuthGuard({
  permission: workspaceAudit,
  redirectTo: '/',
})
```

`useAuthGuard` waits for the permission query to resolve before deciding. It doesn't race ahead of backend-owned policy.

## Route protection and permission guards do different jobs

Easy to collapse, costly to debug when you do:

| Layer | Answers | Used for |
| --- | --- | --- |
| `definePageMeta({ convexAuth: true })` | Is this user signed in? | Require a session at all |
| `useAuthGuard({ permission: ... })` | Does the signed-in user have this capability? | Gate a specific page |

Authentication and authorization fail for different reasons. Model them separately.

## Common pitfalls

- Hard-coding `can` values in the context query instead of deriving them from backend guards. That defeats the projection.
- Pointing `trellis.permissions.query` at the wrong export path. Trellis catches this; fix the path.
- Using a strict actor that requires a tenant on pages where onboarding users still need to resolve (see [Principal and actor](/apps/docs/permissions/principal-and-actor) for the split).
- Treating `usePermissions()` as a convenience cache — it's a projection of backend policy, not a separate store.
