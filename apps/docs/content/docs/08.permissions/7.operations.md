Read this page when the same business action has to exist in more than one place: confirmation UI, admin surface, server workflow, or MCP tool.

If a plain protected mutation still explains the whole feature cleanly, stay with that. Operations are for the point where reuse and preview semantics matter.

Some business actions live in multiple places — a button in the UI, a preview query for confirmation UX, an MCP tool for the agent. Write that action once with `defineOperation(...)` and let Trellis project it into each surface. One guard, one load, one authorize, one preview, one handler.

```ts [convex/features/todos/domain.ts]
export const removeTodoOp = defineOperation({
  id: 'todos.remove',
  name: 'removeTodo',
  kind: 'destructive',
  args: deleteTodo.args,
  guard: canReadTodo,
  load: async (ctx, args) => {
    const todo = await ctx.db.get(args.id)
    requireRecord(todo, 'Todo')
    return { todo }
  },
  authorize: {
    check: (_actor, { todo }) => canDeleteTodo(todo),
  },
  preview: async (_ctx, _args, { todo }) => ({
    display: {
      summary: `Will permanently delete "${todo.title}"`,
      warn: 'This cannot be undone',
      affects: { todos: 1 },
    },
    confirm: {
      operation: 'todos.remove',
      targetId: todo._id,
      affectedCounts: { todos: 1 },
    },
  }),
  handler: async (ctx, args) => {
    await ctx.db.delete(args.id)
    return null
  },
})

export const remove = mutation(removeTodoOp)
export const previewRemove = query(previewOf(removeTodoOp))
```

Preview and execute share everything — guard, load, authorize. They can't drift.

## When to reach for an operation

| Good fit                                          | Skip it                                          |
| ------------------------------------------------- | ------------------------------------------------ |
| Destructive action (delete, archive, bulk remove) | Simple non-destructive one-off mutation          |
| Same action needs preview and execute             | No preview needed — a regular handler is clearer |
| Same business rule backs both UI and agent flows  | Single-transport, single-call action             |
| Action deserves a stable contract across surfaces | Tiny local mutation                              |

Operations buy you reuse with shared policy. For a trivial mutation, a normal protected handler is shorter and clearer.

## Register it as a mutation and a preview query

```ts
export const remove = mutation(removeTodoOp)
export const previewRemove = query(previewOf(removeTodoOp))
```

What you get:

- One execute path, from the operation's `handler`.
- One preview path, projected from the operation's `preview`.
- Shared `guard`, `load`, `authorize` — no second implementation to keep in sync.

The preview isn't a parallel implementation. It's the _preview phase_ of the same operation, surfaced as a query.

## Destructive operations have extra requirements

Trellis is strict here on purpose:

::warning
For `kind: 'destructive'` operations, all four of these must be present:

- `id` (stable string — used to bind preview to execute and to identify the tool in MCP).
- `preview(...)` — destructive actions must be previewable.
- `destructiveSafety` configured in `defineTrellis(...)` before the operation registers.
- A confirmation payload returned from preview and required at execute.

Skip any of these and Trellis rejects the registration. That's the point — confirmation binds to previewed state, not to a loose client convention.
::

See [Destructive operations](/docs/mutations/destructive-operations) for the browser-side preview + confirm UX.

## Preview returns two things

```ts
preview: async (_ctx, _args, { todo }) => ({
  display: { ... },   // shown to humans / operator UI
  confirm: { ... },   // durable confirmation contract
})
```

| Part      | Purpose                                                |
| --------- | ------------------------------------------------------ |
| `display` | What the user or agent is about to do — human-readable |
| `confirm` | What must be bound to execution — machine-checked      |

Both run in the same phase, against the same loaded data, with the same authorization.

## Operations are the bridge to MCP tools

`tool.operation(...)` projects an operation into a destructive agent tool. It requires:

- An operation with an `id`.
- An execute ref from that operation.
- When preview is used, a preview ref from the _same_ operation.

If those refs don't match, Trellis rejects the binding. That's how you prevent a destructive MCP tool from drifting onto a different business action than it claims. See [MCP: destructive tools](/docs/mcp-tools/destructive-tools).

## Common pitfalls

- Treating preview as optional for destructive actions. It isn't — Trellis requires it.
- Forgetting `id` on an operation that needs stable projection. You'll hit a registration error.
- Duplicating preview and execute policy in separate handlers. That's the problem operations solve.
- Wrapping tiny local mutations in `defineOperation(...)` because the primitive exists. A regular mutation is fine.
