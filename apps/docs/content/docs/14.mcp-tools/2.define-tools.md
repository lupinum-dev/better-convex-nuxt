Two related paths to define a tool, depending on whether you need the app runtime:

| API                                  | When                                                                           |
| ------------------------------------ | ------------------------------------------------------------------------------ |
| `defineTool(...)`                    | Tool doesn't need runtime-resolved capabilities or principals                  |
| `tool(...)` from `defineMcpApp(...)` | Tool uses capability snapshots and runtime auth (almost always)                |
| `tool.fromOperation(...)`            | Destructive tools — see [Destructive tools](/docs/mcp-tools/destructive-tools) |

For most tools, use the runtime factory. Static permission handles and principal handling live there.

## Schema boundary

`schema` in a Trellis MCP tool is the backend-facing Convex args contract. Trellis projects that schema into the Zod raw shape that `@nuxtjs/mcp-toolkit` expects when the tool is registered.

That projected MCP schema is transport metadata, not your form schema. Browser forms and Nitro `/api/*` handlers should keep their own hand-authored Zod validation in `shared/schemas/` when they need edge-specific parsing or UX rules.

## Common tool shape

A tool definition picks:

| Field                  | Purpose                              |
| ---------------------- | ------------------------------------ |
| `name`                 | Tool identifier shown to the agent   |
| `description`          | Agent-facing description             |
| `call`                 | The underlying Convex function       |
| `operation`            | `query`, `mutation`, or `action`     |
| `schema`               | Args validator                       |
| `permission`           | Coarse static permission gate        |
| `rateLimit` (optional) | Per-tool request budget              |
| `mapResult` (optional) | Shape the return value for the agent |

Example:

```ts [server/mcp/tools/list-posts.ts]
import { tool } from '~/server/mcp/runtime'

export default tool({
  schema: listPosts,
  call: api.domain.posts.list,
  operation: 'query',
  permission: workspaceRead,
  rateLimit: { max: 20, window: '1m' },
  meta: {
    name: 'list-posts',
  },
})
```

## Tool rate limits

`rateLimit` is per tool bucket and stays scoped to the current principal where the runtime has one.

Distributed enforcement is explicit. If you want MCP rate limits to hold across multiple instances, wire a shared atomic store into the runtime:

```ts [server/mcp/runtime.ts]
import Redis from 'ioredis'
import { createRedisMcpRateLimitStore, defineMcpApp } from '@lupinum/trellis/mcp'

const redis = new Redis(process.env.REDIS_URL!)

export const mcpRuntime = defineMcpApp({
  rateLimitStore: createRedisMcpRateLimitStore({
    client: redis,
  }),
  // ...
})
```

Without `rateLimitStore`, Trellis uses process-local memory. That is fine for local/dev and tests, not for distributed production enforcement.

Standalone `defineTool(...)` also accepts `rateLimitStore` when you are not using `defineMcpApp(...)`.

`trellis doctor` currently knows how to verify the first-party Redis helper. If you wire a custom store, doctor treats it as unverified and leaves the atomicity review to you.

::note
Tool `rateLimit` is an **outer guard** for MCP traffic. It is the right place to slow down agent spam and reject bursts before more work starts. It is not a replacement for application-layer quotas inside Convex.
::

For user, workspace, or billing-tier quotas inside your app, use an application-layer limiter in the called Convex function. Trellis recommends the Convex rate-limiter component for that inner policy layer. See [Rate-limiting layers](/docs/concepts/rate-limiting-layers).

## Start with the common-case tools

| Good first tools                 | Save for later                               |
| -------------------------------- | -------------------------------------------- |
| Read-only query tools            | Destructive actions (delete, bulk update)    |
| Small mutations with stable args | Dynamic session-bound tool registration      |
| Clear single-purpose actions     | Tools that compose multiple business actions |

Destructive tools belong after the rest works. See [Destructive tools](/docs/mcp-tools/destructive-tools).

## Permission gates

The permission projection from `defineMcpApp(...)` lets you gate tool visibility or execution:

- **Visibility gate** — tool doesn't appear in the agent's tool list.
- **Execution gate** — tool appears but rejects at call time.

::note
Permission gating keeps the tool surface understandable. It isn't business authorization — the protected Convex handler still owns the real permission decision. See [Auth and permissions](/docs/mcp-tools/auth-and-permissions).
::
