Pre-launch gate, not a tutorial. Walk through this before shipping.

## Environment

- [ ] `CONVEX_URL` points at the intended production deployment.
- [ ] `SITE_URL` matches the real public app origin.
- [ ] `BETTER_AUTH_SECRET` is set wherever auth is enabled (local env **and** Convex deployment — they must match).
- [ ] `CONVEX_SITE_URL` is set when the auth token exchange path needs it.
- [ ] `CONVEX_TRUSTED_CALLER_KEY` is set wherever trusted-caller or MCP traffic exists.

## Policy and permissions

- [ ] `trellis.permissions.query` points at a real permission-context query when the UI depends on configured permissions.
- [ ] Route protection and permission guards are both exercised in staging (they fail for different reasons — don't skip either).
- [ ] Destructive operations have working preview and confirmation flows wherever Trellis requires them.
- [ ] Any `ctx.db.crossTenant` and `ctx.db.raw` seams are intentional and reviewed — every one has a comment explaining *why*.

## Runtime

- [ ] If SSR auth cache is enabled, its storage choice matches the deployment topology (not in-memory on a multi-node deploy).
- [ ] Observability is configured for the production surfaces you care about, not "wired later."

## Docs and tests

- [ ] `pnpm check:docs:links` is green if docs are part of the release.
- [ ] The relevant test lanes have run for the changed surface.

## Verification gate

For this repo, the strongest default gate:

```bash [terminal]
pnpm test:types && pnpm lint && pnpm test:contracts
```

When the change touches deeper surfaces:

```bash [terminal]
pnpm test
pnpm run release:verify
```
