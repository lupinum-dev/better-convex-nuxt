Use this page when the tool exists but the trust model still needs to be explicit.

## Transport Identity First

MCP requests resolve a transport-shaped principal, usually an `agent` principal derived from the validated bearer token and middleware context.

That is not the business authorization model.

## Capability Snapshot Second

`defineMcpApp(...)` can compute a capability snapshot for coarse tool gating.

Example uses:

- whether the tool should be visible
- whether the tool should be callable at all

## Business Authorization Still Lives In Convex

This is the key rule:

tool capability checks do not replace backend authorization.

The Trellis examples derive coarse MCP capabilities, then still call protected Convex handlers that enforce the real business rules through actors, guards, tenant isolation, and authorize phases.

The protected Convex handler still owns the real permission decision.

## Why That Split Exists

- capability gating keeps the MCP surface understandable
- backend authorization keeps the business model unified

If the tool layer starts owning business policy, the agent path drifts away from the UI path.
