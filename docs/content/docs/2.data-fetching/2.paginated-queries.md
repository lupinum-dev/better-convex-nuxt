Use this page when the backend query exposes cursor-based pagination and the UI needs a single reactive surface for the list, pagination state, and refresh behavior.

`useConvexPaginatedQuery()` is the Trellis path for cursor-based lists. It owns the first page, later pages, `loadMore()`, stale handling, exhaustion state, and refresh behavior.

## The Default Shape

Start with the common case:

```vue
<script setup lang="ts">
import { api } from '#trellis/api'

const projectArgs = computed(() => (tenantId.value ? {} : undefined))

const {
  results: projects,
  status,
  loadMore,
  hasNextPage,
  error,
} = await useConvexPaginatedQuery(api.projects.list, projectArgs, {
  initialNumItems: 12,
})
</script>
```

That gives you one reactive list instead of forcing you to manage cursors manually in page code.

## Status Meanings

The important statuses are:

- `loading-first-page`: the first page is still loading
- `ready`: at least one page loaded and more pages may exist
- `loading-more`: an additional page is loading
- `exhausted`: all available pages have loaded
- `error`: the current pagination flow failed
- `skipped`: the query is disabled because args are missing

Those states matter more than raw cursor handling in most UI code.

## Loading More

Call `loadMore(n)` to request another page size:

```ts
loadMore(12)
```

Use `hasNextPage` to control the button:

```vue
<UButton v-if="hasNextPage" @click="loadMore(12)">
  Load more
</UButton>
```

When the final page is reached, status becomes `exhausted` and `hasNextPage` becomes `false`.

## Skip Until The List Can Run

Just like `useConvexQuery()`, paginated queries should be skipped through args rather than conditional composable calls.

```ts
const projectArgs = computed(() => (tenantId.value ? {} : undefined))
const { results, status } = await useConvexPaginatedQuery(api.projects.list, projectArgs, {
  initialNumItems: 12,
})
```

If args are `null` or `undefined`:

- the query is skipped
- no subscription starts
- `results` is `[]`
- status is `skipped`

## Keep Previous Results During Arg Changes

Use `keepPreviousData: true` when changing args should keep the last settled list visible while the next first page loads.

This is useful for:

- filters
- workspace switches
- tabs that reuse the same list surface

When that transition is happening:

- the old results stay visible
- status moves through the loading state
- `isStale` becomes `true`

## Refresh And Reset

Two actions matter here:

- `refresh()` re-fetches the currently loaded pagination surface
- `reset()` starts over from the first page

Use `refresh()` when the list should stay on the same pagination path.
Use `reset()` when the pagination session itself should start over.

## When To Disable Subscription

Use `subscribe: false` when the list should behave like a request/refresh surface instead of a live list.

That is usually the right choice for exported reports or pages where manual refresh is more appropriate than live updates.

## Common Mistakes

- Hand-rolling cursor state in the page instead of using the composable.
- Forgetting `initialNumItems`, which leaves the first-page size underspecified.
- Treating `ready` and `exhausted` as the same UI state when the “load more” affordance should differ.
- Clearing the entire list on every args change instead of using stale-state semantics deliberately.
