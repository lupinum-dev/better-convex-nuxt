## Basic Usage

`useConvexPaginatedQuery` loads data page by page from a Convex paginated query. It manages cursors, page state, and real-time subscriptions for each loaded page automatically.

```vue
<script setup lang="ts">
import { api } from '~~/convex/_generated/api'

const { results, status, loadMore, hasNextPage } = await useConvexPaginatedQuery(
  api.posts.list,
  {},
  { initialNumItems: 10 },
)
</script>

<template>
  <ul>
    <li v-for="post in results" :key="post._id">{{ post.title }}</li>
  </ul>
  <button v-if="hasNextPage" @click="loadMore(10)">Load More</button>
</template>
```

## Convex Backend Requirement

Your Convex query function must accept a `paginationOpts` argument and return a `PaginationResult`. Use Convex's built-in pagination helpers:

```ts
// convex/posts.ts
import { query } from './_generated/server'
import { paginationOptsValidator } from 'convex/server'
import { v } from 'convex/values'

export const list = query({
  args: {
    paginationOpts: paginationOptsValidator,
    category: v.optional(v.string()),
  },
  handler: async (ctx, args) => {
    let q = ctx.db.query('posts').order('desc')
    if (args.category) {
      q = q.filter((q) => q.eq(q.field('category'), args.category))
    }
    return await q.paginate(args.paginationOpts)
  },
})
```

::warning
The `paginationOpts` field is injected automatically by the composable. Do not pass it yourself -- only provide your own custom arguments.
::

## Status State Machine

The `status` computed ref tracks the lifecycle of paginated data loading:

```
skipped ──────────────────────────────────────────┐
                                                  │
loading-first-page ──► ready ──► loading-more     │
       │                 │            │            │
       │                 │            ▼            │
       │                 │        ready ───► exhausted
       │                 │            │
       ▼                 ▼            ▼
     error             error        error
```

| Status | Meaning |
|---|---|
| `skipped` | Query is disabled (args are `null` or `undefined`) |
| `loading-first-page` | The initial page is being fetched |
| `ready` | At least one page loaded and more pages are available |
| `loading-more` | A subsequent page is being fetched via `loadMore` |
| `exhausted` | All pages have been loaded; no more data available |
| `error` | A page request failed |

## Return Value

::field-group
  ::field{name="results" type="ComputedRef<Item[]>"}
  A flat array of all items across all loaded pages. Updates reactively as new pages load or as real-time subscriptions push changes.
  ::

  ::field{name="status" type="ComputedRef<PaginatedQueryStatus>"}
  The current pagination lifecycle status. One of `'skipped'`, `'loading-first-page'`, `'ready'`, `'loading-more'`, `'exhausted'`, or `'error'`.
  ::

  ::field{name="isLoading" type="ComputedRef<boolean>"}
  `true` when either the first page or a subsequent page is loading. Equivalent to `status === 'loading-first-page' || status === 'loading-more'`.
  ::

  ::field{name="isExhausted" type="ComputedRef<boolean>"}
  `true` when all available data has been loaded.
  ::

  ::field{name="hasNextPage" type="ComputedRef<boolean>"}
  `true` when the query is `'ready'` and more pages can be loaded.
  ::

  ::field{name="loadMore" type="(numItems: number) => void"}
  Load the next page with the given number of items. No-op if already loading or exhausted.
  ::

  ::field{name="error" type="Ref<Error | null>"}
  The first error encountered across any page, or `null`.
  ::

  ::field{name="refetch" type="() => Promise<void>"}
  Re-fetch all currently loaded pages in place, preserving pagination positions. Useful for pulling fresh data without resetting scroll position.
  ::

  ::field{name="restart" type="() => Promise<void>"}
  Clear all loaded pages and start over from page 1. Resets the entire pagination state.
  ::
::

## Options

::field-group
  ::field{name="initialNumItems" type="number" required}
  The number of items to fetch for the first page. This is a required option.
  ::

  ::field{name="server" type="boolean" default-value="true"}
  Run the first page query server-side during SSR.
  ::

  ::field{name="subscribe" type="boolean" default-value="true"}
  Keep live WebSocket subscriptions for each loaded page. When data changes on the backend, results update automatically.
  ::

  ::field{name="default" type="() => Item[]"}
  Factory function providing fallback items while the first page is loading.
  ::

  ::field{name="transform" type="(results: Item[]) => TransformedItem[]"}
  Transform the flat results array before exposing it. Applied after all pages are merged.
  ::

  ::field{name="keepPreviousData" type="boolean" default-value="false"}
  Preserve the previous results while a new first page is loading (e.g., after `restart()` or when reactive args change).
  ::
::

## `refetch()` vs `restart()`

These two methods serve different purposes:

| | `refetch()` | `restart()` |
|---|---|---|
| **What it does** | Re-fetches all currently loaded pages in place | Clears everything and loads only the first page |
| **Preserves pages** | Yes -- pages 1, 2, 3 are all refreshed | No -- resets to page 1 only |
| **Use case** | Pull latest data after a known mutation | Reset when filter/sort criteria change |
| **Scroll position** | Maintained (same number of items) | Lost (back to initial state) |

::tip
In most cases you do not need to call either method manually. The real-time subscription keeps data fresh automatically. Use `refetch()` when you need an immediate consistency guarantee, and `restart()` when the query parameters have fundamentally changed.
::

## Infinite Scroll Example

Combine `useConvexPaginatedQuery` with an intersection observer to build an infinite scroll feed:

```vue
<script setup lang="ts">
import { api } from '~~/convex/_generated/api'

const { results, status, loadMore, isLoading, hasNextPage } =
  await useConvexPaginatedQuery(api.posts.list, {}, { initialNumItems: 20 })

const sentinel = ref<HTMLElement | null>(null)

onMounted(() => {
  if (!sentinel.value) return

  const observer = new IntersectionObserver(
    (entries) => {
      if (entries[0]?.isIntersecting && hasNextPage.value && !isLoading.value) {
        loadMore(20)
      }
    },
    { rootMargin: '200px' },
  )

  observer.observe(sentinel.value)

  onUnmounted(() => observer.disconnect())
})
</script>

<template>
  <div>
    <div v-if="status === 'loading-first-page'">
      <p>Loading posts...</p>
    </div>

    <template v-else>
      <article v-for="post in results" :key="post._id">
        <h2>{{ post.title }}</h2>
        <p>{{ post.excerpt }}</p>
      </article>

      <div ref="sentinel">
        <p v-if="status === 'loading-more'">Loading more...</p>
        <p v-else-if="status === 'exhausted'">No more posts.</p>
      </div>
    </template>

    <div v-if="status === 'error'">
      <p>Failed to load posts.</p>
      <button @click="refetch">Retry</button>
    </div>
  </div>
</template>
```

## Reactive Arguments

Like `useConvexQuery`, paginated queries support reactive arguments. When the args change, the pagination state automatically resets and the first page is re-fetched.

```vue
<script setup lang="ts">
import { api } from '~~/convex/_generated/api'

const category = ref<string>('technology')

const { results, status, loadMore, hasNextPage } = await useConvexPaginatedQuery(
  api.posts.list,
  () => ({ category: category.value }),
  { initialNumItems: 10, keepPreviousData: true },
)
</script>

<template>
  <div>
    <select v-model="category">
      <option value="technology">Technology</option>
      <option value="design">Design</option>
      <option value="business">Business</option>
    </select>

    <article v-for="post in results" :key="post._id">
      <h2>{{ post.title }}</h2>
    </article>

    <button v-if="hasNextPage" @click="loadMore(10)">Load More</button>
  </div>
</template>
```

::note
When reactive args change, the composable internally calls `restart()` -- clearing all loaded pages and fetching the first page with the new arguments. Enable `keepPreviousData` to avoid a flash of empty content during the transition.
::
