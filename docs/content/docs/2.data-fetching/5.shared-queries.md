---
title: "Shared Queries"
description: "Deduplicate identical queries across components with the shared option."
---

::tip
Shared queries are an optimization for apps where multiple components subscribe to the same data. Most apps don't need this -- start without it and add `shared` keys when you notice duplicate subscriptions.
::

## The Problem

When multiple components independently call `useConvexQuery` with the same query and arguments, each one creates its own subscription and maintains its own reactive state. This leads to:

- **Duplicate WebSocket subscriptions** for the same data
- **Redundant SSR work** fetching the same query multiple times on the server
- **Inconsistent timing** where different components may briefly show different versions of the same data

```vue
<!-- Header.vue -->
<script setup lang="ts">
import { api } from '~~/convex/_generated/api'
// Creates subscription #1
const { data: user } = await useConvexQuery(api.users.current, {})
</script>

<!-- Sidebar.vue -->
<script setup lang="ts">
import { api } from '~~/convex/_generated/api'
// Creates subscription #2 (duplicate!)
const { data: user } = await useConvexQuery(api.users.current, {})
</script>
```

## The Solution

Pass a `shared` option with a unique string key. All composable calls with the same key share a single subscription and the same reactive data ref.

```vue
<!-- Header.vue -->
<script setup lang="ts">
import { api } from '~~/convex/_generated/api'
const { data: user } = await useConvexQuery(api.users.current, {}, {
  shared: 'current-user',
})
</script>

<!-- Sidebar.vue -->
<script setup lang="ts">
import { api } from '~~/convex/_generated/api'
const { data: user } = await useConvexQuery(api.users.current, {}, {
  shared: 'current-user',
})
</script>
```

Now both components receive the exact same `data` ref backed by a single WebSocket subscription. When the user data changes, both components update simultaneously.

## How It Works

When `useConvexQuery` is called with a `shared` key:

1. **First call** -- the composable creates the subscription and registers it in an internal registry under the given key.
2. **Subsequent calls** with the same key -- the composable returns the existing reactive state from the registry instead of creating a new subscription.
3. **All consumers** share the same `data`, `status`, `error`, `pending`, `refresh`, and `clear` refs.

The shared state is scoped to the current Nuxt app instance (per-request on the server, singleton on the client), so SSR and client hydration work correctly.

## Key Collision Protection

The module validates that all usages of the same shared key reference the same query function and compatible arguments. If a collision is detected, an error is thrown immediately.

```vue
<!-- This will throw an error at runtime -->
<script setup lang="ts">
import { api } from '~~/convex/_generated/api'

// Component A: shared key 'user-data' with api.users.current
const { data: user } = await useConvexQuery(api.users.current, {}, {
  shared: 'user-data',
})

// Component B: same key but DIFFERENT query function
const { data: settings } = await useConvexQuery(api.settings.get, {}, {
  shared: 'user-data', // Throws: Duplicate key with different config
})
</script>
```

The collision check compares:
- The query function reference (must be the same Convex function)
- Static arguments (plain objects are compared by value)
- Static options (non-reactive options are compared)

::note
Reactive arguments (refs, computed values, getter functions) are marked as dynamic and bypass the static comparison. The module trusts that if you use the same shared key with dynamic args, you are passing equivalent reactive sources.
::

## When to Use Shared Queries

Shared queries are ideal for data that is consumed by multiple components simultaneously:

- **Current user** -- header, sidebar, profile menu, and permission checks all need the logged-in user
- **App configuration** -- feature flags, theme settings, or tenant config used across layouts
- **Shared layout data** -- navigation items, notification counts, or global counters

```vue
<script setup lang="ts">
import { api } from '~~/convex/_generated/api'

// Used in the layout and several child components
const { data: currentUser } = await useConvexQuery(api.users.current, {}, {
  shared: 'current-user',
})

// App-wide feature flags
const { data: features } = await useConvexQuery(api.config.features, {}, {
  shared: 'feature-flags',
})
</script>
```

## When NOT to Use Shared Queries

Avoid `shared` when queries have component-specific arguments that differ between consumers:

```vue
<!-- TaskDetail.vue -- different ID per instance -->
<script setup lang="ts">
import { api } from '~~/convex/_generated/api'

const props = defineProps<{ taskId: string }>()

// Each instance fetches a DIFFERENT task -- do NOT share
const { data: task } = await useConvexQuery(
  api.tasks.get,
  () => ({ id: props.taskId }),
)
</script>
```

::warning
Do not use `shared` with a static key when the query arguments vary between component instances. Each shared key maps to exactly one query + args combination. If you need per-instance data, omit the `shared` option and let each component manage its own subscription.
::

## Shared Queries with Options

Options like `transform`, `default`, and `keepPreviousData` are set by the first consumer that registers the shared key. Subsequent consumers inherit the same configuration.

```vue
<script setup lang="ts">
import { api } from '~~/convex/_generated/api'

// First consumer sets the options
const { data: user } = await useConvexQuery(api.users.current, {}, {
  shared: 'current-user',
  default: () => null,
  transform: (user) => ({
    ...user,
    initials: user.name
      .split(' ')
      .map((n: string) => n[0])
      .join('')
      .toUpperCase(),
  }),
})
</script>
```

::tip
Define your shared queries in a composable to ensure consistent configuration across all consumers:

```ts
// composables/useCurrentUser.ts
export function useCurrentUser() {
  return useConvexQuery(api.users.current, {}, {
    shared: 'current-user',
    default: () => null,
  })
}
```

Then use `useCurrentUser()` in any component instead of calling `useConvexQuery` directly.
::
