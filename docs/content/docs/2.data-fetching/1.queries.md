Use this page when a Vue component needs Trellis data and you want the default read path.

`useConvexQuery()` is the standard Trellis query composable for Nuxt apps. It is SSR-aware, hydrates into the live Convex subscription model, and gives you reactive state such as `data`, `status`, `pending`, `error`, `isStale`, `refresh`, and `clear`.

## The Default Shape

Start with the common case:

```vue
<script setup lang="ts">
import { api } from '#trellis/api'

const { data: tasks, status, pending, error } = await useConvexQuery(api.tasks.list, {
  status: 'active',
})
</script>

<template>
  <div v-if="pending">Loading...</div>
  <div v-else-if="error">{{ error.message }}</div>
  <ul v-else-if="status === 'success'">
    <li v-for="task in tasks" :key="task._id">
      {{ task.title }}
    </li>
  </ul>
</template>
```

That is the right default when the component should render server-side, then stay live as data changes.

## What Happens By Default

With the default options, Trellis does three things for you:

1. waits for the first value before the awaited composable resolves
2. hydrates that state into the client
3. keeps the query subscribed so updates continue to stream in

That is why `useConvexQuery()` is the default read path and not just a thin HTTP helper.

## When To Skip A Query

Pass `null` or `undefined` args when the query should not run yet.

```ts
const todoArgs = computed(() => (isAuthenticated.value ? {} : undefined))
const { data, status, pending } = await useConvexQuery(api.todos.list, todoArgs)
```

When args are `null` or `undefined`:

- the query is skipped
- no subscription starts
- `pending` is `false`
- `status` is `skipped`

Use this instead of conditionally calling the composable.

## Reactive Args Are Supported

Args can be:

- a plain object
- a `ref`
- a reactive object
- a getter function

When nested reactive args change, Trellis re-subscribes with the new args.

```ts
const tag = ref('alpha')

const { data, isStale } = await useConvexQuery(api.search.notes, () => ({
  filter: { tag: tag.value },
}), {
  keepPreviousData: true,
})
```

## Keep Previous Data During Arg Changes

Use `keepPreviousData: true` when a query should keep showing the last settled result during an args transition.

That changes the UX from “clear and reload” to “show previous result while the next one loads.”

While that transition is in progress:

- `pending` stays `true`
- the previous data remains visible
- `isStale` becomes `true`

Use this for search, filter changes, and list refinements where a blank state would be worse than slightly old data.

## Default Values And Transform

Use `default` for an initial placeholder shape and `transform` when the UI wants a derived form of the result.

```ts
const { data, pending } = await useConvexQuery(api.notes.list, {}, {
  default: () => [{ _id: 'placeholder', title: 'Loading...' }],
  transform: (items) => items.map((item) => ({
    ...item,
    title: item.title.toUpperCase(),
  })),
})
```

The transform also applies to the default value, so keep both shapes aligned.

## When To Turn Off Live Subscription

Use `subscribe: false` when you want a one-shot HTTP-backed read instead of a long-lived live query.

```ts
const { data, refresh } = await useConvexQuery(api.reports.snapshot, {}, {
  subscribe: false,
})
```

This is useful when:

- the result is expensive and does not need to stay live
- the page is really a request/refresh surface, not a subscription surface
- you want to control refresh manually

If you do not need composable state at all, use `executeConvexQuery()` instead of `useConvexQuery()`.

## When To Turn Off SSR

Use `server: false` when the query should not block SSR.

```ts
const { data, status, pending } = await useConvexQuery(api.activity.list, {}, {
  server: false,
})
```

Before the first client result arrives:

- `status` is `pending`
- `pending` is `true`

This is useful for client-only dashboards or non-critical data that should not delay the initial render.

## Common Mistakes

- Calling the composable conditionally instead of passing reactive skip args.
- Using `subscribe: false` for data that the UI actually expects to stay live.
- Clearing the UI on every arg change instead of using `keepPreviousData`.
- Using `useConvexQuery()` in middleware or the wrong lifecycle scope instead of server helpers or lower-level client access.
