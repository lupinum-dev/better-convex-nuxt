title: "Transforms & Defaults"
description: "Shape query results and provide fallback values while loading."
---

## `transform`

The `transform` option lets you map raw Convex data before it is exposed through the `data` ref. Use it to add computed fields, filter, sort, or reshape the result without additional `computed` wrappers in your component.

```vue
<script setup lang="ts">
import { api } from '~/convex/_generated/api'

const { data: tasks } = await useConvexQuery(api.tasks.list, {}, {
  transform: (tasks) =>
    tasks
      .filter((t) => !t.archived)
      .map((t) => ({
        ...t,
        displayName: `${t.text} (${t.completed ? 'done' : 'pending'})`,
      })),
})
</script>

<template>
  <ul>
    <li v-for="task in tasks" :key="task._id">{{ task.displayName }}</li>
  </ul>
</template>
```

The transform runs every time new data arrives -- both from the initial fetch and from subsequent real-time updates. TypeScript infers the `data` type from the transform's return type automatically.

::note
The transform function should be a pure function. Avoid side effects inside it -- use the `onData` callback for that instead.
::

## `default`

The `default` option provides a fallback value while the query is in the `'pending'` or `'skipped'` state. It accepts a factory function that returns the fallback.

```vue
<script setup lang="ts">
import { api } from '~/convex/_generated/api'

// `data` starts as `[]` instead of `null`
const { data: tasks } = useConvexQuery(api.tasks.list, {}, {
  default: () => [],
})
</script>

<template>
  <!-- No need for a v-if null check -->
  <p>{{ tasks.length }} tasks loaded</p>
  <ul>
    <li v-for="task in tasks" :key="task._id">{{ task.text }}</li>
  </ul>
</template>
```

This is especially useful with the non-blocking (no `await`) pattern, where the component renders before data arrives. Without `default`, you would need null checks on `data` throughout your template.

::tip
For array-returning queries, `default: () => []` is the most common pattern. For object-returning queries, consider providing a skeleton object: `default: () => ({ name: '', email: '' })`.
::

## `keepPreviousData`

When reactive args change, the query transitions back to `'pending'` while new results load. By default, `data` resets to `null` (or the `default` value) during this transition. Setting `keepPreviousData: true` preserves the last successful result until the new data arrives.

```vue
<script setup lang="ts">
import { api } from '~/convex/_generated/api'

const category = ref('all')

const { data: posts, status } = await useConvexQuery(
  api.posts.list,
  () => ({ category: category.value }),
  { keepPreviousData: true },
)
</script>

<template>
  <div>
    <select v-model="category">
      <option value="all">All</option>
      <option value="tech">Tech</option>
      <option value="design">Design</option>
    </select>

    <!-- No flash of empty state when switching categories -->
    <ul>
      <li v-for="post in posts" :key="post._id">{{ post.title }}</li>
    </ul>
  </div>
</template>
```

Without `keepPreviousData`, switching from "Tech" to "Design" would briefly show an empty list (or the `default` value). With it enabled, the Tech posts remain visible until the Design posts arrive.

::warning
The stale data shown during the transition still reflects the previous args. Make sure your UI can tolerate showing slightly outdated content for the brief loading window. For example, if posts have category labels, users may see "Tech" posts while the "Design" filter is active for a moment.
::

## Combining Options

All three options work together. A common pattern is to use `default` for the initial load, `transform` for shaping data, and `keepPreviousData` for smooth transitions:

```vue
<script setup lang="ts">
import { api } from '~/convex/_generated/api'

const sortOrder = ref<'newest' | 'oldest'>('newest')

const { data: messages } = await useConvexQuery(
  api.messages.list,
  () => ({ channel: 'general' }),
  {
    // Start with an empty array instead of null
    default: () => [],

    // Sort and add formatted timestamps
    transform: (messages) => {
      const sorted = [...messages].sort((a, b) =>
        sortOrder.value === 'newest'
          ? b._creationTime - a._creationTime
          : a._creationTime - b._creationTime,
      )
      return sorted.map((m) => ({
        ...m,
        formattedTime: new Date(m._creationTime).toLocaleTimeString(),
      }))
    },

    // Keep showing current messages while refreshing
    keepPreviousData: true,
  },
)
</script>

<template>
  <div>
    <button @click="sortOrder = sortOrder === 'newest' ? 'oldest' : 'newest'">
      Sort: {{ sortOrder }}
    </button>

    <div v-for="msg in messages" :key="msg._id">
      <span>{{ msg.formattedTime }}</span>
      <p>{{ msg.text }}</p>
    </div>
  </div>
</template>
```

## With Paginated Queries

The `transform`, `default`, and `keepPreviousData` options also work with `useConvexPaginatedQuery`. The `transform` receives the merged flat array of all loaded pages, and `default` provides fallback items while the first page is loading.

```vue
<script setup lang="ts">
import { api } from '~/convex/_generated/api'

const { results: posts, loadMore, hasNextPage } = await useConvexPaginatedQuery(
  api.posts.list,
  {},
  {
    initialNumItems: 10,
    default: () => [],
    transform: (posts) =>
      posts.map((p) => ({
        ...p,
        readTime: `${Math.ceil(p.wordCount / 200)} min read`,
      })),
    keepPreviousData: true,
  },
)
</script>

<template>
  <div>
    <article v-for="post in posts" :key="post._id">
      <h2>{{ post.title }}</h2>
      <span>{{ post.readTime }}</span>
    </article>
    <button v-if="hasNextPage" @click="loadMore(10)">Load More</button>
  </div>
</template>
```
