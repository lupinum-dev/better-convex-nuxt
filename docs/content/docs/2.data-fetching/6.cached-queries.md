title: 'Cached Queries'
description: 'Eliminate loading flicker on list-to-detail navigation with useCachedQuery.'
---

## The Problem

When a user taps an item in a list to open its detail page, the detail query starts from scratch — even though the list query already had the data. This causes a loading flash between pages.

`useCachedQuery` solves this by checking whether a source query (the list) already has the item in Nuxt's data cache. If it does, the detail page renders immediately with that cached snapshot while the real detail query loads in the background.

## Basic Usage

```vue
<script setup lang="ts">
import { api } from '#trellis/api'

const route = useRoute()
const id = route.params.id as string

const {
  data: task,
  isFromCache,
  pending,
} = useCachedQuery(
  api.tasks.get,
  { id },
  {
    from: {
      query: api.tasks.list,
      args: { status: 'active' },
      find: (items) => items.find((t) => t._id === id),
    },
  },
)
</script>

<template>
  <div>
    <p v-if="isFromCache" class="text-sm text-muted">Showing cached data...</p>
    <TaskDetail v-if="task" :task="task" />
    <LoadingSpinner v-else-if="pending" />
  </div>
</template>
```

## How It Works

1. On mount, `useCachedQuery` looks up the source query's data in Nuxt's `useNuxtData` cache using the query key.
2. If found, it runs your `find` function to extract the matching item.
3. The matched item is used as the `default` value for the underlying `useConvexQuery`, so the component renders instantly.
4. The real detail query runs in the background. Once it resolves, the data ref updates seamlessly.
5. `isFromCache` is `true` while the data is from the cache and the real query is still pending.

## Options

The `from` option configures where to look for cached data:

::field-group
::field{name="from.query" type="FunctionReference<'query'>" required}
The source query whose cached data may contain the item you need.
::
::field{name="from.args" type="FunctionArgs<SourceQuery>" required}
The arguments the source query was called with. This is used to compute the cache key.
::
::field{name="from.find" type="(items: SourceReturnType) => TargetReturnType | undefined" required}
A function that extracts the target item from the source query's return value. Return `undefined` if not found.
::
::

All other options from `useConvexQuery` are supported (`server`, `subscribe`, `transform`, `keepPreviousData`, etc.).

## Return Value

`useCachedQuery` returns everything from `useConvexQuery` plus:

::field-group
::field{name="isFromCache" type="Ref<boolean>"}
`true` when the displayed data comes from the source cache and the real query hasn't resolved yet.
::
::

## When to Use

- **List-to-detail navigation** — the most common case. A list page fetches items; the detail page needs one of them.
- **Tab switching** — a parent query has all the data; child tabs show subsets.
- **Search-to-view** — search results contain enough data to render the detail page skeleton.

## When Not to Use

If the source and target queries return different shapes (the list returns summaries, the detail returns full documents), the cached data won't match. In that case, use `keepPreviousData` on a regular `useConvexQuery` instead.
