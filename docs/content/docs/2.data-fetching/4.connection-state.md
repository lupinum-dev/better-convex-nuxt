Use this page when the UI needs to react to connectivity, reconnects, or pending mutation state.

The runtime composable is:

```ts
const { state, isConnected, isReconnecting, pendingMutations, shouldShowOfflineUi } =
  useConvexConnectionState()
```

## What It Gives You

- `state`: the shared raw connection snapshot
- `isConnected`: websocket currently connected
- `isReconnecting`: the app had connected before and is now disconnected
- `pendingMutations`: count of client-side pending mutations
- `shouldShowOfflineUi`: the safe UI signal for offline banners

Use `shouldShowOfflineUi` instead of inventing your own "not connected yet means offline" rule.

## Why `shouldShowOfflineUi` Exists

Trellis suppresses offline UI during the initial hydration grace window.

That avoids a bad SSR user experience where the page renders from server data and then immediately flashes an offline banner before the browser had a fair chance to connect.

## One Shared Connection Store

Multiple components can call `useConvexConnectionState()` without starting separate connection subscriptions.

That shared store behavior matters for app layouts, global banners, and page-local indicators. You can use the composable in more than one place without paying for separate connection tracking.

## Connection Change Hook

Trellis also emits a Nuxt hook when the phase changes:

```ts
nuxtApp.hook('trellis:connection:changed', (payload) => {
  // payload.state: 'connecting' | 'connected' | 'reconnecting'
})
```

That hook fires only when the connection phase changes. It does not spam every minor runtime update.

## Typical UI Uses

Good uses:

- offline banners
- reconnect indicators
- "changes are still syncing" affordances
- disabling destructive actions while many mutations are pending

Bad uses:

- treating `isConnected === false` on first hydration as an outage
- deriving auth state from connection state
- using reconnect state as authorization logic
