Pick the smallest example that proves the boundary you actually need.

## First reader path

Read these in order:

1. `01-public-todo`
2. `02-auth-todo`
3. `03-team-workspace`

That's the cleanest ladder from minimal Trellis to the canonical protected-app reference.

Each one adds one major layer:

- `01` adds the smallest `defineTrellis(...)` runtime and one query and mutation path.
- `02` adds Better Auth and actor resolution.
- `03` adds workspaces, permissions, trusted callers, and MCP projection.

If you only need a public or personal app, stop earlier. There's no benefit in inheriting workspace or agent complexity you don't need.

## Match the example to the problem

| Example | Open it when you need | What it proves |
| --- | --- | --- |
| `01-public-todo` | Public-only queries and mutations | Smallest useful Trellis baseline |
| `02-auth-todo` | Sign-in and personal ownership | Auth plus app-owned actors without tenancy |
| `03-team-workspace` | Roles, permissions, tenant boundaries, or MCP | Canonical protected workspace app |
| `04-saas-platform` | Pagination, uploads, admin flows, practical SaaS depth | Month-two product work |
| `05-visibility-access` | Advanced visibility, redaction, enrollment, or sharing | Deeper access patterns |
| `06-multi-workspace` | Workspace switching or cross-tenant operator views | Multi-workspace membership model |
| `07-mcp-reference` | Full MCP surface | Prompts, resources, sessions, and tool variants |
| `08-component-mini-cms` | Component bridge and projection boundaries | Local component plus MCP seam |

## Pattern catalog after the first ladder

- `04-saas-platform`: pagination, uploads, Nitro routes, admin workflows.
- `05-visibility-access`: advanced visibility and sharing rules.
- `06-multi-workspace`: multi-workspace membership and cross-tenant operator views.
- `07-mcp-reference`: richest MCP surface.
- `08-component-mini-cms`: component bridge and local component boundary.

## Canonical public example set

`examples/` is the canonical public example set today. Use it as the source of truth for:

- First-reader learning paths.
- Docs code references.
- Real public patterns worth copying.

`examples-next/` is a pressure suite for future direction. It's not the default public source of truth until it's promoted intentionally.

::note
The examples are learning references, not the canonical generated app shape. Some older examples still use flatter paths like `convex/todos.ts`; new Trellis starters generate the stricter layout from [Canonical app layout](/docs/getting-started/canonical-app-layout).
::

## Read before you copy

Don't copy an example wholesale unless the app shape is genuinely close to yours. Instead:

1. Choose the smallest matching example.
2. Read the files that define the core boundary.
3. Copy only the pattern you need.
4. Stop before unrelated surfaces come with it.

## Files to read first

- `01-public-todo`: `shared/schemas/todo.ts`, `convex/todos.ts`, `pages/index.vue`.
- `02-auth-todo`: `convex/auth.ts`, `convex/auth/actor.ts`, `convex/todos.ts`, `pages/index.vue`.
- `03-team-workspace`: `convex/auth/actor.ts`, `convex/auth/checks.ts`, `convex/todos.ts`, `server/mcp/tools/*.ts`, `pages/index.vue`.
