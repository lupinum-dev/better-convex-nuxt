This page covers the main backend-facing Trellis surfaces used in Convex function files.

## `defineTrellis(...)`

```ts
defineTrellis(builders, options?)
```

Primary entrypoint for building protected Trellis query, mutation, and action builders on top of Convex.

Important options:

- `principal`
  Defines how transport identity is resolved.
- `actor`
  Maps a resolved principal into the app-owned actor.
- `tenantIsolation`
  Declares which tables should stay tenant-scoped by default.
- `services`
  Defines explicit high-trust service lanes.
- `observability`
  Configures semantic event emission.
- `destructiveSafety`
  Required before registering destructive operations.
- `rls`
  Integrates row-level security rules.
- `triggers`
  Installs Convex triggers into the Trellis runtime path.
- `onSuccess`
  Hooks for post-success query, mutation, or action side work.

Returned builder surface:

- `query`
- `mutation`
- `action` when action builders are provided
- `raw.query`
- `raw.mutation`
- `raw.action`

Important behavior:

- `ctx.db` is the default safe path
- `ctx.db.crossTenant` is the explicit tenant-boundary bypass that keeps more runtime structure
- `ctx.db.raw` is the full escape hatch and emits observability when used

## `defineOperation(...)`

```ts
defineOperation({
  id,
  name,
  kind,
  args,
  guard,
  load,
  authorize,
  preview,
  handler,
})
```

Reusable business-action primitive for flows that should project cleanly across UI, server, and MCP.

Use it when one action needs:

- shared guard, load, and authorization logic
- preview plus execute projections
- a stable business identity across transports

Destructive requirements:

- `kind` must be `destructive`
- `id` must be present
- `preview(...)` must be present
- `defineTrellis({ destructiveSafety })` must be configured before registration

Projection helpers:

- `mutation(operation)` registers the execute path
- `previewOf(operation)` produces the preview query projection

## Handler Context Accessors

Protected handler context adds these Trellis surfaces:

- `ctx.principal()`
  Resolved transport identity.
- `ctx.actor()`
  App-owned actor derived from the principal.
- `ctx.observe(...)`
  Emits semantic observability events.
- `ctx.db`
  Default database surface.
- `ctx.db.crossTenant`
  Explicit cross-tenant database surface.
- `ctx.db.raw`
  Escape hatch database surface.

The important boundary is simple: principal and actor are async runtime facts, not compile-time assumptions.

## Bridge Surfaces

Higher-level function-side bridge helpers:

- `createComponentBridge`
- `defineComponentBridgeManifest`
- `renderComponentBridgeFile`
- `renderComponentBridgeFiles`

Use them only when packaged component integrations need a stable root-level ref inventory. Most apps should stay on normal root handlers.
