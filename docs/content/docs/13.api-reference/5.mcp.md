This page covers the public Trellis MCP runtime surface.

## `defineMcpApp(...)`

```ts
defineMcpApp({
  callConvex,
  resolvePrincipal,
  resolveCapabilities?,
  runtime?,
  principalKey?,
  observability?,
})
```

Creates the MCP runtime and the tool factory bound to your Trellis backend model.

Important options:

- `callConvex(event, principal)`
  Returns the Convex caller used by the MCP runtime.
- `resolvePrincipal(event)`
  Maps the MCP transport request into the Trellis principal.
- `resolveCapabilities(...)`
  Optional tool-visibility and capability snapshot layer.
- `runtime(...)`
  Optional per-request runtime object for downstream tool handlers.
- `principalKey(principal)`
  Stable identity key used for destructive confirmation and observability.
- `observability`
  MCP runtime observability options.

Returned shape:

- `tool(...)`
- `tool.fromOperation(...)`

## `defineTool(...)`

```ts
defineTool({
  schema,
  call,
  operation?,
  preview?,
  capability?,
  enabled?,
  middleware?,
  mapResult?,
  summary?,
  respond?,
  outputSchema?,
  group?,
  tags?,
})
```

Use it for normal read or non-destructive tool surfaces.

Important constraints:

- destructive generic tools are not supported
- if a tool is destructive, use `defineMcpApp(...).tool.fromOperation(...)` instead
- capability gating and `enabled(...)` affect visibility or availability, not backend business authorization

## `tool.fromOperation(...)`

```ts
tool.fromOperation(operation, {
  execute,
  preview?,
  executeOperation?,
  previewOperation?,
  ...toolOptions
})
```

Preferred path for operation-backed tools, especially destructive tools.

Important semantics:

- destructive operations require a matching preview projection
- Trellis binds confirmation to operation identity and previewed state
- mismatched preview or execute refs are rejected instead of tolerated

## MCP Session And Server Helpers

- `useMcpServer`
- `useMcpSession`

Use these when the runtime needs direct access to the server/session seam rather than only tool definitions.

## Result Helpers

- `wrapSuccess`
- `wrapError`
- `wrapPreview`
- `withSummary`

These helpers normalize tool results into the expected structured MCP envelope. `wrapPreview(...)` is the preview helper for awaiting-confirmation responses.

## Toolkit Re-Exports

Trellis also re-exports toolkit surfaces such as:

- `defineMcpHandler`
- `defineMcpPrompt`
- `defineMcpResource`
- `defineMcpTool`

Those help you compose the surrounding MCP app, but Trellis-specific business alignment still lives in `defineMcpApp(...)` plus the Trellis tool factory.

When the tool layer needs a stable automation surface, project root handlers or bridge refs into the MCP runtime instead of duplicating business behavior inside tool files.
