## Decision

Build AI as opt-in SaaS Kit recipe tracks, not as core `better-convex-nuxt`
runtime.

Use two separate tracks:

1. `agentic-saas` for in-product agents that act through explicit app-owned
   delegation records.
2. `mcp-saas` or `platform-auth` for external tool/API surfaces, starting with
   private service actors or API keys and moving to public OAuth only after the
   OAuth lifecycle is fully proven.

Do not add a generic agent permission framework. Do not make agents members of
Better Auth organizations by default. Do not duplicate product functions for UI,
MCP, and agents.

## Ownership

| Concept                                                 | Owner                      | Rule                                                                         |
| ------------------------------------------------------- | -------------------------- | ---------------------------------------------------------------------------- |
| Human identity, sessions, organizations, members, roles | Better Auth                | No app-owned mirrors in SaaS templates.                                      |
| Agent run/delegation                                    | App Convex table           | This is the canonical authority for what an agent may do.                    |
| Agent threads/messages                                  | Convex Agent component     | Infrastructure history only, never authorization state.                      |
| Product data and product authorization                  | Convex product functions   | Re-check Better Auth and product invariants at execution time.               |
| MCP transport and discovery                             | Nuxt MCP Toolkit           | Transport metadata only, never final authorization.                          |
| OAuth clients and tokens                                | Better Auth OAuth Provider | Platform recipe only until refresh rotation and deployed runtime are proven. |

## Agentic SaaS Shape

An agent never receives ambient authority. A Better Auth user starts an
`agentRuns` row with a bounded organization, delegating user, capabilities,
expiry, status, and token budgets. Agent tools must load that run and verify it
before every tool call. A checked Agent action claims an `active` run exactly
once as `running` before creating the Agent component thread, recording usage,
or invoking tools.

Model-visible tool input should carry product content only. Bind authority
values such as `organizationId`, `agentRunId`, user ids, scopes, and credential
ids from checked Convex context, not from model-produced arguments.

Default agent writes create draft, proposal, or request rows. Canonical product
state changes remain Better Auth-checked human approval actions unless a
specific product requirement justifies direct writes.

Do not store session tokens, caller-supplied user ids, caller-supplied thread
ids, caller-supplied organization ids for existing runs, raw provider secrets,
or generic role hints in public agent function args.
Agent-created review helpers should derive organization from `agentRuns` or the
target product row, not from a helper-supplied organization argument.

## MCP Shape

MCP is a caller surface, not a separate product model.

```txt
Nuxt MCP Toolkit
MCP tool adapter
same Convex product query/mutation/action
same product authorization and audit
```

For private MCP, prefer service actors or Better Auth API keys. Tool visibility
is advisory; Convex execution-time checks remain authoritative.

For public MCP, keep it in `platform-auth` until the OAuth Provider path is
ready. Public OAuth needs resource-bound tokens, protected-resource metadata,
scope checks, token invalidation semantics, and product authorization re-checks.

## Proven Locally

- `agentic-saas` has no app-owned organization/member mirrors.
- Public agent functions derive Better Auth identity from headers, not public
  session-token args; production Convex source and the Nuxt page are
  source-tested free of proof-token markers, and old-shape caller-supplied
  delegating user ids are runtime-rejected before inserting `agentRuns`.
- Public agent functions that operate on an existing run take `agentRunId` and
  derive the organization from `agentRuns`; this public surface is source-tested
  against caller-supplied organization, thread, user, and token args.
- Agent run names are required and capabilities are deduplicated into stable
  canonical order before storing the delegation row.
- Agent tool input does not trust model-controlled `organizationId` or
  `agentRunId`; those are bound from checked action context.
- Agent actions require the current Better Auth session to belong to the
  delegating user before creating a component thread or executing tools; the
  Agent component thread user is source-tested as
  `agentRuns.startedByAuthUserId`.
- Agent execution uses `agentRuns.status` as the single claim source of truth:
  `active` means delegated but idle, `running` means one action has claimed the
  run, and duplicate claims fail before draft, usage, or thread side effects.
- Agent-facing draft/request helpers are source-tested internal functions. The
  public surface is checked Agent actions plus Better Auth-checked human
  approval mutations. Helper side effects still require a `running` run with an
  attached Agent component thread; `internalMutation` alone is not the
  authorization boundary.
- Cross-organization tool denial is proven through agent-facing product
  helpers, not a separate capability-check bridge export.
- Agent execution claim re-checks the delegating user's current Better Auth
  project permission before creating an Agent thread or tool side effects, so
  role downgrade or member removal blocks stale delegated runs.
- Public revocation prevents later execution and leaves no Agent thread, draft,
  audit, or usage side effects.
- Expired active runs fail before Agent thread, draft, audit, or usage side
  effects; keep expiry as a timestamp invariant, not a second run status.
- Expired running runs cannot attach a canonical Agent component thread id after
  claim.
- Agent tool calls reject wrong organizations, missing capabilities, revoked
  runs, expired runs, and completed runs.
- Agent audit rows record successful agent product actions; do not add
  outcome/result fields until denied-audit reporting is a real requirement.
- Product and agent audit action/resource labels are schema-bounded. Adding a
  new audit kind should be a deliberate schema change, not an arbitrary string.
  Retained audit rows require resource ids.
- Agent audit for deletion-request creation points at the created review row;
  the target product record stays on the request and later product audit.
- Human rejection audit for drafts and deletion requests points at the decided
  review row and matching source id.
- Revocation applies to active delegation only; completed runs require a stored
  Agent thread and remain terminal readable history.
- Agent run lifecycle status writes are source-guarded to the app-owned
  `agentRuns` module; terminal failed/completed/revoked runs cannot be
  reclassified by lifecycle helpers.
- Expiry gates active delegation and active reads, not completed historical
  reads.
- Agent output creates draft/request state by default, and human decisions
  happen through Better Auth-checked mutations that derive the organization from
  the draft/request row before checking permissions. Canonical product
  create/delete side effects stay in source-tested product-domain helpers, and
  the human decision mutations are source-tested and runtime-tested as
  review-row-id-only surfaces that reject caller-supplied user ids.
- Agent-created draft/request rows derive organization from `agentRuns` or the
  target product record, and their helper functions stay internal mutations.
  They cannot create review or audit rows before the run is claimed and the
  Agent component thread id is stored.
- Blank agent-created draft content and deletion reasons are rejected before
  review rows or audit rows are inserted.
- Agent tool-result history stores redacted markers for secret-shaped output and
  excludes the raw server-only secret canary.
- Drafts and deletion requests are single-decision rows: once approved or
  rejected, they cannot be approved or rejected again.
- Failed Agent executions reject that run's pending draft/deletion-request rows
  with `decidedAt`. Keep diagnostic Agent history and usage if needed, but do
  not leave failed-run review rows approvable or visible in approval queues.
  Do not re-decide already-decided review rows when the run fails. If retention
  is later requested, delete the failed run's stored Agent thread and usage
  history while retaining rejected review rows.
- Retention cleanup should be retryable by repeating the same `agentRunId`
  command. Do not add a second cleanup-state table unless a production
  compliance workflow needs independently auditable deletion receipts.
- Destructive agent output keeps at most one pending deletion request per
  canonical product record.
- Convex Agent thread ids normalize and attach once to `agentRuns`; message and
  stream reads check Better Auth plus the matching run, then derive the
  organization and component thread id from that run.
- Read-only streaming creates Agent stream and usage history without product
  records, draft rows, or agent product-audit rows, and completed stream
  history remains readable after `expiresAt`.
- Non-delegating users cannot execute another user's read-only stream run, even
  if they have `project:read`.
- Usage is append-only raw events, not billing rollups; the app-owned schema is
  source-tested against extra rollup, projection, cache, or invoice tables, and
  each usage callback thread id must match the canonical Agent thread id stored
  on `agentRuns`.
  Stored usage events derive the thread, organization, and delegating user ids
  from a running, unexpired run instead of callback metadata, and carry
  normalized non-empty model/provider labels plus valid non-negative token
  counts whose total covers prompt plus completion and whose cached input count
  does not exceed prompt tokens. Display names stay on `agentRuns`.
- Organization/user aggregate token-budget preflight stops already-exhausted
  follow-up runs before creating Agent thread, draft, audit, or usage rows for
  the blocked run.
- Retention deletes Agent conversation history and usage events while preserving
  product/audit history. Both Agent thread deletion and app usage cleanup use
  the stored thread id from the terminal run; app usage cleanup starts from the
  canonical `agentRunId` and rejects active or running runs before checking
  thread state. Same-org readers cannot retention-delete another delegator's
  run.
- `agentic-saas` passes Nuxt production build, fresh anonymous-local Convex
  component installation/backend typecheck, and Convex codegen. Generated local
  `.convex`/`.env.local` state is cleanup state, not a source of truth.
- `agentic-saas` is source-guarded to keep MCP transport dependencies, OAuth
  Provider dependencies, `server/mcp` routes, and public OAuth/MCP primitives out
  of the in-product agent recipe.
- `agentic-saas` audit actors are source-guarded to the proven `user` and
  `agent` kinds; `apiKey`, `service`, and `system` remain future actor extensions
  until a tested recipe uses them.
- `mcp-agent` proves Nuxt MCP Toolkit dispatch into Convex-shaped HTTP,
  unauthenticated, malformed-auth, extra-part bearer, and undeclared tool calls
  failing before Convex HTTP, and direct tool adapters against real
  `convex-test` product functions. Convex normalizes and bounds project names
  before product writes, so MCP tool input cannot create overlong product names.
  Service actor credential issuance and revocation are keyed by the stored
  organization and require an active owner/admin membership in that
  organization. Service actors cannot use the human `owner` role. Issuance
  accepts only SHA-256 hex digests and rejects duplicate credential hashes
  before inserting a service actor. Destructive approvals derive organization
  from the target project, require an active owner/admin membership, and store
  the approving human user. The starter source-guards against a fake
  caller-supplied agent usage action; usage attribution belongs in the
  Agent-backed app recipe where it can be derived from stored run/thread state.
  Streamable HTTP tool responses are tested not to expose bearer secrets,
  credential hashes, or the credential hash field name. Approval operation labels
  and service audit action/resource labels are schema-bounded. It is
  source-guarded to keep OAuth Provider dependencies and public OAuth MCP
  primitives out of the private service-actor starter.
- `platform-auth` proves OAuth Provider metadata, authenticated DCR, PKCE
  authorization-code flow, client credentials, resource-bound JWTs,
  protected-resource metadata, introspection, revocation, local JWKS
  verification, and Convex `/mcp` product writes through an internal mutation
  that re-checks Better Auth OAuth client state. The MCP product-write proof
  ignores spoofed product identity arguments, normalizes/bounds title input, and
  shows invalid titles and authenticated unknown tools do not write product
  state. Its runtime verifier source-guards the hard cut to current OAuth
  Provider tables and rejects deprecated `mcp()`/`oidcProvider()` plugin paths,
  while keeping app-owned schema from mirroring Better Auth identity,
  membership, OAuth client, or token state.

## Not Yet Proven

- Real provider LLM generation and streaming. The current workspace has no
  provider API key configured, and `agentic-saas` is source-tested to stay
  mock-only until provider-backed execution is proven.
- Cloud/project deployment behavior. The active proof starters use anonymous
  local Convex deployments; the current shell has no Convex deployment env
  configured, and local schema proof requires isolated or reset generated
  `.convex` state so stale rows from older schemas do not pollute the result.
  `platform-auth` product authorization now derives the MCP audience from
  `SITE_URL`, but Convex auth config env still needs a real deployment-env proof.
- Nuxt MCP Toolkit execution into Convex product functions with OAuth tokens.
  Keep that as a separate public MCP recipe, not a mode inside the private
  `mcp-agent` starter.
- Default OAuth refresh-token rotation on Convex. The current provider filters
  `revoked eq null`; Convex-created rows omit optional `revoked`. A proof-only
  explicit-null mutation makes rotation work, so the root cause is known, but
  this still needs an upstream/provider-adapter fix.

## Shipping Rule

Ship `agentic-saas` first as the in-product agent recipe. Keep MCP/platform auth
separate. Do not claim provider-backed AI, public OAuth MCP, or deployed-project
coverage until the corresponding proof runs against real external state.
