## The global auth middleware

When auth is enabled, the module registers a global Nuxt route middleware named `convex-auth.global`. This middleware runs on every client-side navigation and during SSR to enforce route protection.

The middleware performs these steps on each navigation:

1. **Check if auth is enabled** -- reads the runtime config to confirm auth is active.
2. **Read page meta** -- looks for `convexAuth` on the target route's meta.
3. **Resolve protection decision** -- determines the redirect destination based on the page's `convexAuth` meta and the global defaults.
4. **Wait for auth to settle** (client-side only) -- if `isPending` is `true`, calls `awaitAuthReady()` with a timeout. This prevents premature redirects during hydration.
5. **Check authentication** -- if the user is authenticated, navigation proceeds. Otherwise, the user is redirected.

::note
The middleware only acts on routes that have `convexAuth` set in their page meta. Routes without `convexAuth` are not affected.
::

## Skipping auth token fetches with `skipConvexAuth`

For public pages where you know authentication is not needed, use `skipConvexAuth` to skip the auth token fetch entirely. This avoids an unnecessary network request to the auth proxy on pages that do not need user data:

```vue [app/pages/about.vue]
<script setup lang="ts">
definePageMeta({ skipConvexAuth: true })
</script>

<template>
  <div>
    <h1>About Us</h1>
    <p>This page does not need authentication.</p>
  </div>
</template>
```

::warning
`skipConvexAuth` and `convexAuth` serve different purposes. `skipConvexAuth: true` tells the auth client to skip the token fetch for this route, which improves performance on public pages. `convexAuth: true` protects the route from unauthenticated access. Setting both on the same page is contradictory and will trigger a dev-mode warning.
::

You can also skip auth token fetches for multiple routes at once using the `auth.skipAuthRoutes` config option:

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  modules: ['better-convex-nuxt'],

  convex: {
    auth: {
      skipAuthRoutes: ['/about', '/docs/**', '/legal/*'],
    },
  },
})
```

This uses glob pattern matching, so `/docs/**` matches all routes under `/docs/`.

## Using `awaitAuthReady()` in custom middleware

If you need custom auth logic beyond simple route protection, use `awaitAuthReady()` from `useConvexAuthInternal()` in your own middleware:

```ts [app/middleware/admin-only.ts]
export default defineNuxtRouteMiddleware(async () => {
  const { awaitAuthReady } = useConvexAuthInternal()
  const { user } = useConvexAuth()

  // Wait for auth state to settle (up to 5 seconds)
  const isAuthenticated = await awaitAuthReady({ timeoutMs: 5000 })

  if (!isAuthenticated) {
    return navigateTo('/auth/signin')
  }

  // Custom logic: check if user has admin privileges
  // (you would check this against your own data)
  if (!user.value?.email?.endsWith('@mycompany.com')) {
    return navigateTo('/unauthorized')
  }
})
```

### `awaitAuthReady()` options

::field-group
  ::field{name="timeoutMs" type="number" default="5000"}
  Maximum time to wait for auth to settle, in milliseconds. If auth does not resolve within this time, the function returns the current `isAuthenticated` value (likely `false`).
  ::
::

### Return value

`awaitAuthReady()` returns a `Promise<boolean>`:
- `true` -- the user is authenticated.
- `false` -- the user is not authenticated, or auth timed out.

On the server, `awaitAuthReady()` returns immediately with the current auth state since SSR auth is resolved before middleware runs.

## Middleware timeout

The global auth middleware uses `AUTH_MIDDLEWARE_TIMEOUT_MS` (5000ms) when waiting for auth to settle. This prevents the middleware from hanging indefinitely if the auth token exchange is slow or fails.

If auth does not settle within 5 seconds on the client:
- The middleware treats the user as unauthenticated.
- On protected routes, the user is redirected to the sign-in page.
- In development, a warning is logged to the console.

::tip
If you see timeout warnings during development, check that your Convex deployment is running (`npx convex dev`) and that the `CONVEX_URL` environment variable is correct.
::

## Middleware execution order

The `convex-auth.global` middleware runs as a global middleware, which means it executes before page-specific middleware. If you define custom middleware on a protected page, the auth check runs first:

```vue [app/pages/admin.vue]
<script setup lang="ts">
definePageMeta({
  convexAuth: true,
  middleware: ['admin-only'],  // Runs after convex-auth.global
})
</script>
```

In this example:
1. `convex-auth.global` runs first, checking if the user is authenticated.
2. If authenticated, `admin-only` middleware runs next for additional checks.
3. If not authenticated, the user is redirected before `admin-only` ever executes.
