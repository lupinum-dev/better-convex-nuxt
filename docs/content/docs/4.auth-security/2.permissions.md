::callout{icon="i-lucide-alert-triangle" color="amber"}
**Security Rule**: Never trust the frontend. Always enforce permissions on the backend. Frontend checks are for UX only.
::

::callout{icon="i-lucide-info"}
**Auth claims vs authoritative roles:** `useConvexAuth().user` is great for identity and optional UI claims, but your authoritative role/permission context should come from a Convex query (for example `usePermissions()` / `api.auth.getPermissionContext`), not a JWT claim alone.
::

::callout{icon="i-lucide-shield-check" color="amber"}
**Better Auth owns roles, organizations, and membership.** Roles come from the
Better Auth Organization plugin (`member.role`), enforced with
`auth.api.hasPermission`. There is no app-owned `role` column and no
`organizations` table. See
[Authentication → source of truth](/docs/auth-security/authentication).
::

## Role Hierarchy

Four roles from most to least privileged, defined in Better Auth's
access-control policy (see [Setup Step 1](/docs/auth-security/permissions-setup)):

| Role     | Description                              |
| -------- | ---------------------------------------- |
| `owner`  | Full control, billing, can invite admins |
| `admin`  | Manage members, content moderation       |
| `member` | Create/edit own content                  |
| `viewer` | Read-only access                         |

## UX Permission Map (Frontend Only)

`createPermissions` needs a `checkPermission` function to turn a permission
string into a boolean for `can()`. This mirrors the Better Auth access-control
policy for the UI — it is **not** the enforcement point.

```ts [shared/permissions.ts]
export type Role = 'owner' | 'admin' | 'member' | 'viewer'

export const permissions = {
  'org.settings': { roles: ['owner'] },
  'org.invite': { roles: ['owner', 'admin'] },
  'org.members': { roles: ['owner', 'admin'] },
  'post.create': { roles: ['owner', 'admin', 'member'] },
  'post.read': { roles: ['owner', 'admin', 'member', 'viewer'] },
  'post.update': { own: ['member'], any: ['owner', 'admin'] },
  'post.delete': { own: ['member'], any: ['owner', 'admin'] },
  'post.publish': { roles: ['owner', 'admin'] },
} as const

export type Permission = keyof typeof permissions
```

### Permission Types

**Simple**: Any user with the role can perform the action.

```ts
'post.create': {
  roles: ['owner', 'admin', 'member']
}
```

**Ownership**: Different rules for own resources vs others'.

```ts
'post.update': {
  own: ['member'],           // Members can update their own
  any: ['owner', 'admin'],   // Owners/admins can update any
}
```

## Frontend: usePermissions

Access role and permission checking in components.

```ts
const {
  can, // (permission, resource?) => boolean
  role, // ComputedRef<Role | null>
  orgId, // ComputedRef<string | null>
  isAuthenticated, // ComputedRef<boolean>
  pending, // ComputedRef<boolean>
} = usePermissions()
```

### The can() Function

::callout{icon="i-lucide-info"}
`can()` returns a plain boolean from the current reactive permission context, so use it directly in templates and inside `computed()` or `watchEffect()` callbacks.
::

```vue
<script setup lang="ts">
const { can, role } = usePermissions()
const { data: posts } = await useConvexQuery(api.posts.list, {})
</script>

<template>
  <button v-if="can('org.settings')">Organization Settings</button>
  <button v-if="can('post.create')">New Post</button>

  <div v-for="post in posts" :key="post._id">
    <h3>{{ post.title }}</h3>
    <button v-if="can('post.update', post)">Edit</button>
    <button v-if="can('post.delete', post)">Delete</button>
    <button v-if="can('post.publish')">Publish</button>
  </div>
</template>
```

### usePermissionRedirect

Client-side redirect when permission state says the user should leave the view.

```vue
<script setup lang="ts">
usePermissionRedirect({
  permission: 'org.settings',
  redirectTo: '/dashboard',
})
</script>
```

::callout{icon="i-lucide-info"}
**Route middleware + Convex queries:** use one-shot APIs in middleware:

- client navigation: `useConvexCall`
- SSR navigation: `serverConvexQuery`
  ::

#### Route Middleware Permission Check (Nuxt)

Use this when you need to block navigation before the page component loads.

```ts [app/middleware/admin-only.ts]
import { api } from '#convex/api'

export default defineNuxtRouteMiddleware(async () => {
  const context = import.meta.server
    ? await serverConvexQuery(useRequestEvent()!, api.auth.getPermissionContext, {})
    : await useConvexCall().query(api.auth.getPermissionContext)

  if (!context || (context.role !== 'admin' && context.role !== 'owner')) {
    return navigateTo('/')
  }
})
```

`useConvexQuery`/`useConvexPaginatedQuery` should stay in component setup scope. For middleware use one-shot calls.

## Conditional Queries

Skip queries based on permissions:

```vue
<script setup lang="ts">
const { can, orgId } = usePermissions()

// Only fetch if the user has permission
const canListMembers = computed(() => can('org.members'))
const { data: members, status } = await useConvexQuery(
  api.organizations.listMembers,
  computed(() => {
    if (!orgId.value) return 'skip'
    return canListMembers.value ? {} : 'skip'
  }),
)
</script>

<template>
  <section v-if="canListMembers">
    <h2>Team Members</h2>
    <div v-if="status === 'pending'">Loading...</div>
    <div v-else-if="status === 'success'">
      <MemberCard v-for="m in members" :key="m.id" :member="m" />
    </div>
  </section>
</template>
```

## Backend: Enforce with Better Auth

The backend never trusts the frontend. It re-checks the caller's Better Auth
permission with `auth.api.hasPermission` and throws **structured** errors.

```ts [convex/lib/permissions.ts]
import { ConvexError } from 'convex/values'

import { components } from '../_generated/api'
import type { MutationCtx, QueryCtx } from '../_generated/server'
import { authComponent, createAuth } from '../auth'

type Ctx = QueryCtx | MutationCtx
type Role = 'owner' | 'admin' | 'member' | 'viewer'
type PostPermission = 'create' | 'read' | 'update' | 'delete'

export async function getBetterAuthMember(
  ctx: Ctx,
  args: { organizationId: string; userId: string },
) {
  return (await ctx.runQuery(components.betterAuth.adapter.findOne, {
    model: 'member',
    where: [
      { field: 'organizationId', value: args.organizationId },
      { field: 'userId', value: args.userId },
    ],
  })) as { role: Role } | null
}

export async function requirePostPermission(
  ctx: Ctx,
  args: { organizationId: string; permission: PostPermission },
) {
  const { auth, headers } = await authComponent.getAuth(createAuth, ctx)
  const session = await auth.api.getSession({ headers })
  if (!session) {
    throw new ConvexError({ code: 'UNAUTHENTICATED', message: 'Not authenticated' })
  }

  const allowed = await auth.api.hasPermission({
    headers,
    body: { organizationId: args.organizationId, permissions: { post: [args.permission] } },
  })
  if (!allowed.success) {
    throw new ConvexError({
      code: 'FORBIDDEN',
      message: `Missing post:${args.permission} permission`,
    })
  }

  return { userId: session.user.id }
}
```

::callout{icon="i-lucide-info"}
The narrowed unauthorized-recovery matcher only reacts to structured
`ConvexError({ code: 'UNAUTHENTICATED' })` (or an HTTP 401), never to prose
messages or 403s. See
[Auth Guards and Permissions](/docs/recipes/auth-guards-and-permissions#what-counts-as-an-unauthorized-error).
::

### Using it in a mutation

```ts [convex/posts.ts]
import { mutation } from './_generated/server'
import { ConvexError, v } from 'convex/values'
import { getBetterAuthMember, requirePostPermission } from './lib/permissions'

export const updatePost = mutation({
  args: { id: v.id('posts'), title: v.string() },
  handler: async (ctx, args) => {
    const post = await ctx.db.get(args.id)
    if (!post) throw new ConvexError({ code: 'NOT_FOUND', message: 'Not found' })

    // Derive the org from the target row, then check the Better Auth permission.
    const { userId } = await requirePostPermission(ctx, {
      organizationId: post.organizationId,
      permission: 'update',
    })

    // Ownership: members may only edit their own posts.
    const member = await getBetterAuthMember(ctx, { organizationId: post.organizationId, userId })
    if (member?.role === 'member' && post.ownerId !== userId) {
      throw new ConvexError({ code: 'FORBIDDEN', message: 'You can only edit your own posts' })
    }

    await ctx.db.patch(args.id, { title: args.title })
  },
})
```

### Filtering instead of throwing

For queries where you want to filter rather than throw, read the member role and
decide:

```ts [convex/organizations.ts]
import { query } from './_generated/server'
import { v } from 'convex/values'
import { components } from './_generated/api'
import { authComponent, createAuth } from './auth'
import { getBetterAuthMember } from './lib/permissions'

export const listMembersIfAllowed = query({
  args: { organizationId: v.string() },
  handler: async (ctx, args) => {
    const { auth, headers } = await authComponent.getAuth(createAuth, ctx)
    const session = await auth.api.getSession({ headers })
    if (!session) return []

    const member = await getBetterAuthMember(ctx, {
      organizationId: args.organizationId,
      userId: session.user.id,
    })
    if (!member || (member.role !== 'owner' && member.role !== 'admin')) return []

    // Read the member list straight from the Better Auth component.
    const result = await ctx.runQuery(components.betterAuth.adapter.findMany, {
      model: 'member',
      where: [{ field: 'organizationId', value: args.organizationId }],
      paginationOpts: { numItems: 200, cursor: null },
    })
    return result.page
  },
})
```

## API Reference

### createPermissions()

Factory function to create permission composables. Auto-imported when
`permissions: true` in module config.

```ts
import { createPermissions } from '#imports'

export const { usePermissions, usePermissionRedirect } = createPermissions<
  Permission,
  PermissionContext,
  Resource
>({
  query: api.auth.getPermissionContext,
  checkPermission,
})
```

The generic order is `<Permission, Context, Resource>`. The `query` must be a
no-argument query returning `Context | null`
(`FunctionReference<'query', 'public', Record<string, never>, TContext | null>`),
and `Context` must include at least `role` and `userId`.

#### Options

| Option            | Type                                                                     | Description                               |
| ----------------- | ------------------------------------------------------------------------ | ----------------------------------------- |
| `query`           | `FunctionReference<'query', 'public', Record<string, never>, T \| null>` | Convex query returning permission context |
| `checkPermission` | `(ctx, permission, resource?) => boolean`                                | UX-only permission checking function      |

### usePermissions()

Returns permission state and checking function.

```ts
const { can, role, orgId, user, isAuthenticated, pending } = usePermissions()
```

#### Returns

| Property          | Type                                     | Description                    |
| ----------------- | ---------------------------------------- | ------------------------------ |
| `can`             | `(permission, resource?) => boolean`     | Check if user has permission   |
| `role`            | `ComputedRef<string \| null>`            | Current user's role            |
| `orgId`           | `ComputedRef<string \| null>`            | Current user's organization ID |
| `user`            | `ComputedRef<PermissionContext \| null>` | Full permission context        |
| `isAuthenticated` | `ComputedRef<boolean>`                   | Whether user has valid context |
| `pending`         | `ComputedRef<boolean>`                   | Whether context is loading     |

### usePermissionRedirect()

Redirects after render when the current user lacks a permission. Enforce access
in Convex functions.

```ts
usePermissionRedirect({
  permission: 'org.settings',
  redirectTo: '/dashboard',
  loginPath: '/auth/signin',
  resource: post, // Optional, for ownership checks
})
```

#### Options

| Option       | Type                   | Default          | Description                   |
| ------------ | ---------------------- | ---------------- | ----------------------------- |
| `permission` | `string`               | Required         | Permission to check           |
| `redirectTo` | `string`               | `'/'`            | Redirect if denied            |
| `loginPath`  | `string`               | `'/auth/signin'` | Redirect if not authenticated |
| `resource`   | `{ ownerId?: string }` | -                | Resource for ownership check  |

## Anti-Patterns

### 1. Frontend-only checks

```ts
// WRONG: No backend protection
export const deletePost = mutation({
  args: { id: v.id('posts') },
  handler: async (ctx, args) => {
    await ctx.db.delete(args.id) // Anyone can delete!
  },
})

// RIGHT: Always re-check the Better Auth permission on the backend
export const deletePost = mutation({
  args: { id: v.id('posts') },
  handler: async (ctx, args) => {
    const post = await ctx.db.get(args.id)
    if (!post) throw new ConvexError({ code: 'NOT_FOUND', message: 'Not found' })
    await requirePostPermission(ctx, { organizationId: post.organizationId, permission: 'delete' })
    await ctx.db.delete(args.id)
  },
})
```

### 2. Mirroring roles into your schema

```ts
// WRONG: adding a role field to your users table (or an app-owned
// organizations table) creates a second source of truth that drifts from
// Better Auth.

// RIGHT: role lives on the Better Auth member row; read it at request time
const member = await getBetterAuthMember(ctx, { organizationId, userId })
```

### 3. Missing resource for ownership

```vue
<!-- WRONG: No resource passed -->
<button v-if="can('post.update')">Edit</button>
<!-- Returns false for members! -->

<!-- RIGHT: Pass the resource -->
<button v-if="can('post.update', post)">Edit</button>
```

### 4. Trusting a client-supplied organization id

```ts
// WRONG: authorize the org the caller claims
await requirePostPermission(ctx, { organizationId: args.organizationId, permission: 'update' })
await ctx.db.patch(args.id, ...) // args.organizationId may not own this row

// RIGHT: derive the org from the target row
const post = await ctx.db.get(args.id)
await requirePostPermission(ctx, { organizationId: post.organizationId, permission: 'update' })
```
