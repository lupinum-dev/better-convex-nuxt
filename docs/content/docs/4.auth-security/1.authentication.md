## Overview

Trellis uses [Better Auth](https://www.better-auth.com/) on [Convex](https://www.convex.dev/).

The split is simple:

- Better Auth handles identity, sessions, email/password, password reset, and OAuth
- Convex issues the JWTs used by authenticated queries and mutations
- Trellis connects the two inside Nuxt so SSR, cookies, and token refresh work without extra glue code
- Trellis and your app still own tenant membership, business roles, permission context, and other app-domain authorization rules

In the common case, setup is:

1. Install `@convex-dev/better-auth`
2. Run `npx trellis init auth`
3. Set `BETTER_AUTH_SECRET` and `SITE_URL`
4. Enable `auth: true` in `nuxt.config.ts`

## What You Get

- SSR-safe auth state
- `useConvexAuth()` and `useConvexAuthActions()`
- auth-aware components like `<ConvexAuthenticated>`
- page protection with `definePageMeta({ convexAuth: true })`
- automatic cookie-to-JWT exchange through the auth proxy

## Architecture Boundary

Trellis is intentionally **auth-first**, not “wrap every Better Auth plugin equally.”

Use Better Auth as the source of truth for:

- sign-up and sign-in
- sessions and cookies
- OAuth/social login
- password reset and other auth-side flows
- auth-side admin operations such as user/session management

Use Trellis/Convex as the source of truth for:

- workspace or tenant membership
- business roles
- permission context
- domain ACLs and ownership rules
- app-specific MCP or server-to-server auth

That boundary is why current multi-tenant examples keep membership and role state in app tables instead of Better Auth `organization` tables.

## Setup

::steps

### Install the Better Auth bridge

```bash
pnpm add @convex-dev/better-auth
```

### Scaffold the auth files

```bash
npx trellis init auth
```

This generates the Convex auth bridge files for you.

### Keep the default generated Convex auth files

For a normal email/password setup, the generated files are already close to final.

`convex/auth.config.ts`

```ts [convex/auth.config.ts]
import type { AuthConfig } from 'convex/server'
import { getAuthConfigProvider } from '@convex-dev/better-auth/auth-config'

export default {
  providers: [getAuthConfigProvider()],
} satisfies AuthConfig
```

`convex/auth.ts`

```ts [convex/auth.ts]
import { defineAuth } from '@lupinum/trellis/auth'

import { components, internal } from './_generated/api'
import { mutation } from './_generated/server'
import authConfig from './auth.config'

export const { authComponent, createAuth, createUserIfNeeded } = defineAuth(
  { components, internal, mutation, authConfig },
  {
    emailPassword: true,
  },
)

export const { onCreate, onUpdate, onDelete } = authComponent.triggersApi()
```

`convex/http.ts`

```ts [convex/http.ts]
import { httpRouter } from 'convex/server'
import { authComponent, createAuth } from './auth'

const http = httpRouter()

authComponent.registerRoutes(http, createAuth)

export default http
```

### Set the required environment variables

Convex needs these:

::field-group
::field{name="BETTER_AUTH_SECRET" type="required"}
Random secret used by Better Auth. Generate one with `openssl rand -hex 32`.
::
::field{name="SITE_URL" type="required"}
Your app origin, for example `http://localhost:3000` in development or `https://app.example.com` in production.
::
::

::tip
For local development, `SITE_URL` can be either `http://localhost:3000` or `http://127.0.0.1:3000`. Trellis only adds both loopback variants automatically when you are actually on a local `http://` origin. Production URLs do not get localhost trusted by default.
::

Your local `.env.local` should also contain your Convex URL:

```bash [.env.local]
CONVEX_URL=https://your-deployment.convex.cloud
```

### Enable auth in Nuxt

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  modules: ['@lupinum/trellis'],
  trellis: {
    auth: true,
  },
})
```

That is enough for the default auth flow.

::

## When To Customize `convex/auth.ts`

Only customize `convex/auth.ts` when you need more than basic email/password auth.

Common reasons:

- OAuth providers like GitHub or Google
- auth-side Better Auth plugins like `admin()`
- custom user fields
- custom Better Auth options

In that case, use the `custom` option:

```ts [convex/auth.ts]
import { betterAuth } from 'better-auth'
import { admin } from 'better-auth/plugins'
import { defineAuth } from '@lupinum/trellis/auth'

import { components, internal } from './_generated/api'
import { mutation } from './_generated/server'
import authConfig from './auth.config'

export const { authComponent, createAuth, createUserIfNeeded } = defineAuth(
  { components, internal, mutation, authConfig },
  {
    // When `custom` is set, defineAuth only provides the bridge plumbing.
    // you configure everything inside the custom callback.
    custom: (_ctx, bridge) =>
      betterAuth({
        baseURL: bridge.siteUrl,
        database: bridge.database,
        emailAndPassword: {
          enabled: true,
        },
        socialProviders: {
          github: {
            clientId: process.env.GITHUB_CLIENT_ID!,
            clientSecret: process.env.GITHUB_CLIENT_SECRET!,
          },
        },
        plugins: [
          bridge.createConvexPlugin(),
          admin(),
        ],
        trustedOrigins: bridge.trustedOrigins,
      }),
  },
)

export const { onCreate, onUpdate, onDelete } = authComponent.triggersApi()
```

`bridge` gives you:

- `bridge.siteUrl`: your configured app origin
- `bridge.database`: the Better Auth adapter backed by Convex
- `bridge.createConvexPlugin()`: the Convex Better Auth plugin
- `bridge.trustedOrigins`: the trusted origin list derived from your app origin

::warning
When using `custom`, you are responsible for passing security-critical options to `betterAuth()`. Always set `trustedOrigins: bridge.trustedOrigins` — omitting it disables Better Auth's CSRF origin validation. The default path (without `custom`) sets `trustedOrigins` automatically.
::

This is the recommended path for auth-centric Better Auth features. Trellis does not try to mirror Better Auth's plugin API with higher-level wrapper options.

## Plugin Positioning

Treat Better Auth plugins in three tiers:

### First-Class

These fit the module directly and should use `custom` when needed:

- email/password
- OAuth/social providers
- password reset and other auth-side flows
- auth-side plugins like `admin()`

### Advanced Alternative

`organization()` is possible, but it is a different architecture. If you adopt it, Better Auth should become the single source of truth for org/member/team/invitation state.

That is not the same model as the current Trellis-first workspace examples, which keep tenant membership and permissions in Convex app data.

### Not Recommended Right Now

`agent-auth` is not a first-class Trellis pattern today. It implies a different agent or MCP architecture than the current examples and should not be treated as part of the default integration story.

## About The Auth Proxy

When `auth` is enabled, Trellis creates a server route at `/api/auth/[...]`.

You usually do not need to think about it much. Its job is:

1. keep auth cookies on your app domain
2. forward auth requests to Convex
3. exchange the Better Auth session cookie for a Convex JWT during SSR and refresh

That proxy also forwards the host, protocol, and client IP headers Better Auth expects.

::note
Forwarded client IP headers fix normal browser and SSR auth requests. Local Convex can still make its own internal JWKS lookup without a client IP. Trellis example launchers avoid that local-only warning by setting static `JWKS` after the local backend boots instead of weakening Better Auth rate limiting.
::

::tip
If you want the lower-level JWT and JWKS explanation, read [JWKS & Convex JWTs](/docs/auth-security/jwks-and-convex-jwts).
::

## Auth Config Options

The `auth` option accepts either `true` or an object:

::field-group
::field{name="enabled" type="boolean" default="true"}
Enable or disable auth.
::
::field{name="route" type="string" default="'/api/auth'"}
Custom auth proxy route.
::
::field{name="routeProtection" type="object"}
Route protection settings. See [Route Protection](/docs/auth-security/route-protection).
::
::field{name="routeProtection.redirectTo" type="string" default="'/auth/signin'"}
Where unauthenticated users should be redirected.
::
::field{name="routeProtection.preserveReturnTo" type="boolean" default="true"}
Keep the original URL in `?redirect=...` so the user can come back after sign-in.
::
::field{name="unauthorized" type="object"}
Behavior for authenticated 401/403 responses.
::
::field{name="cache" type="object"}
SSR auth token cache settings. See [Token Refresh](/docs/auth-security/token-refresh).
::
::field{name="trustedOrigins" type="string[]" default="[]"}
Extra trusted origins for the Nuxt auth proxy. Same-origin is always allowed.
::
::field{name="skipAuthRoutes" type="string[]" default="[]"}
Routes that should skip auth token fetches.
::
::field{name="proxy" type="object"}
Body size limits for the auth proxy.
::
::
