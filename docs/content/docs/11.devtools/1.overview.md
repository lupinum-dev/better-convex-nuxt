## Overview

Better Convex Nuxt integrates with [Nuxt DevTools](https://devtools.nuxt.com/) to provide a dedicated "Convex" tab for inspecting queries, mutations, auth state, and connection status.

## Features

### Query Inspector

View all active queries with their:
- Function name and arguments
- Current status (pending, success, error)
- Data source (SSR HTTP or WebSocket subscription)
- Live data preview

### Mutation Timeline

Track mutation and action calls with:
- Function name and arguments
- Success/error status
- Duration timing
- Optimistic update indicators

### Auth Panel

Inspect authentication state:
- Current user data
- JWT token claims and expiry
- Auth flow timing waterfall (SSR token exchange → client hydration → refresh)

### Connection State

Monitor the WebSocket connection:
- Connected/disconnected status
- Retry count
- In-flight requests

## Accessing DevTools

1. Start your dev server: `nuxt dev --dotenv .env.local`
2. Open Nuxt DevTools (click the Nuxt icon in the bottom of your browser, or press `Shift+Alt+D`)
3. Navigate to the **Convex** tab

## What to Debug

- **SSR data mismatch** -- Use the Query Inspector to compare the data source column. Queries fetched during SSR show "HTTP" while client subscriptions show "WebSocket". If a query you expected to be server-rendered shows only WebSocket, check that `server: true` is set and you're using `await`.
- **Slow page loads** -- Check the Auth Panel's timing waterfall to see if the SSR token exchange is the bottleneck. If so, consider enabling [token caching](/docs/auth-security/token-refresh).
- **Stale data** -- The Connection State panel shows whether the WebSocket is connected. If it shows "disconnected" with retries, your client isn't receiving real-time updates.
- **Optimistic update issues** -- The Mutation Timeline shows an indicator when optimistic updates are applied. Check if the optimistic value matches what you expect before the server response arrives.

## Production

DevTools are automatically disabled in production builds. There is zero runtime overhead — the DevTools code is tree-shaken from your production bundle.
