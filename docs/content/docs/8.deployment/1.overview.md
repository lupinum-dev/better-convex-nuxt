This section is the canonical deployment guide for `better-convex-nuxt`.

- Use this section for production setup and environment ownership.
- Use [Authentication](/docs/auth-security/authentication) for the auth API surface and runtime behavior.

## Production Architecture

When auth is enabled, there are usually three moving parts:

1. Your Nuxt app
2. Your Convex deployment
3. Your auth provider configuration

In the default SSR setup:

- the Nuxt app serves your pages
- the module registers a same-origin auth proxy at `/api/auth/**`
- that proxy forwards Better Auth requests to your Convex HTTP actions host
- Convex runs Better Auth and returns session/token state

In SPA mode (`ssr: false`):

- the app still uses the same Better Auth client APIs
- there is no SSR token bootstrap
- auth resolves on the client
- the Nuxt auth proxy can still exist, but the SSR-specific behavior is no longer the main reason for the setup

## Who Reads What

The important distinction is:

- `convex.url` / `CONVEX_URL` configure the Nuxt module
- `convex.siteUrl` / `CONVEX_SITE_URL` tell the Nuxt proxy where Convex HTTP actions live
- `SITE_URL` configures Better Auth inside Convex and should be your app origin

That last point causes the most confusion:

- `SITE_URL` is your frontend origin, for example `https://app.example.com`
- `CONVEX_SITE_URL` is your Convex HTTP actions origin, for example `https://your-app.convex.site`

Do not point Better Auth `baseURL` at `CONVEX_SITE_URL` when you are using the Nuxt proxy.

## Runtime Ownership

### Nuxt app / app host

These values are read by the Nuxt module at build/runtime:

- `CONVEX_URL`
- `CONVEX_SITE_URL`
- `NUXT_PUBLIC_CONVEX_URL`
- `NUXT_PUBLIC_CONVEX_SITE_URL`
- `REDIS_URL` when you explicitly configure Nitro storage for shared auth cache

### Convex deployment

These values are read by your Convex auth code:

- `SITE_URL`
- `BETTER_AUTH_SECRET`
- OAuth provider credentials such as `GITHUB_CLIENT_ID` and `GITHUB_CLIENT_SECRET`
- `JWKS` if you opt into static JWKS

## About `CONVEX_DEPLOYMENT`

`CONVEX_DEPLOYMENT` often appears in `.env.local` because the Convex CLI writes it.

It is useful Convex metadata, but `better-convex-nuxt` does not read it. The module cares about:

- `CONVEX_URL`
- optionally `CONVEX_SITE_URL`

If `CONVEX_URL` is missing, the module is effectively not configured even if `CONVEX_DEPLOYMENT` exists.
