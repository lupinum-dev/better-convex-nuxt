The maintained topology is:

```text
Browser or Nuxt SSR
  → same-origin /api/auth/*
  → Better Convex Nuxt Nitro proxy
  → Convex HTTP Actions site
  → Better Auth Convex component
```

This keeps the session cookie on the Nuxt application origin while Better Auth data remains in Convex.

## Required files

| File                      | Responsibility                                      |
| ------------------------- | --------------------------------------------------- |
| `convex/convex.config.ts` | Register the Better Auth component                  |
| `convex/auth.config.ts`   | Configure Convex JWT verification                   |
| `convex/auth.ts`          | Create the component client and Better Auth factory |
| `convex/http.ts`          | Register Better Auth HTTP routes lazily             |
| `nuxt.config.ts`          | Register the Nuxt module and public Convex URLs     |

The complete minimal versions are in [Add authentication](/docs/get-started/add-authentication). Keep them as the baseline and add product-specific Better Auth options in `createAuth`.

## Environment ownership

Set these in the Convex environment:

| Variable                   | Purpose                                                   |
| -------------------------- | --------------------------------------------------------- |
| `SITE_URL`                 | Exact Nuxt origin used by Better Auth and trusted origins |
| `CONVEX_SITE_URL`          | Exact Convex HTTP Actions origin used as session issuer   |
| `BETTER_AUTH_SECRETS`      | Versioned Better Auth secrets, newest version first       |
| `BCN_AUTH_PROXY_IP_SECRET` | Private Nuxt-to-Convex client-IP signing secret           |

Set these for the Nuxt build/runtime:

| Variable                      | Purpose                                                    |
| ----------------------------- | ---------------------------------------------------------- |
| `SITE_URL`                    | Exact public Nuxt origin used by the proxy                 |
| `NUXT_PUBLIC_CONVEX_URL`      | Convex deployment URL ending in `.convex.cloud`            |
| `NUXT_PUBLIC_CONVEX_SITE_URL` | Convex HTTP Actions URL; required for local/custom domains |
| `BCN_AUTH_PROXY_IP_SECRET`    | Same private proxy secret configured in Convex             |

`SITE_URL` must be an exact origin. Use HTTPS outside loopback development. Do not include a path, query, or fragment.

## Proxy behavior

The module owns the fixed `/api/auth/*` path and same-origin credentials behavior. It sends one request to the configured Convex site origin and does not follow server-side redirects.

OAuth and login redirects are returned to the browser. Auth-enabled applications require a Nitro-capable deployment; a purely static host cannot provide the proxy.

Applications that expose delegated access use the separate [delegated OAuth and MCP profile](/docs/build/authentication/delegated-oauth-and-mcp). It adds the official provider, verified consent UI, fixed `/mcp` resource, provider-owned client provisioning, and live Convex authorization without adding another auth store.

If the deployment is behind a trusted ingress, set `auth.proxy.trustedClientIpHeader` to the single header owned and overwritten by that ingress. The proxy strips caller forwarding headers and signs only that canonical address. Never configure a header that clients can supply unchanged.

The default request and upstream response body limit is 1 MiB. Change limits only for a measured need:

```ts
convex: {
  auth: {
    proxy: {
      maxRequestBodyBytes: 1_048_576,
      maxResponseBodyBytes: 1_048_576,
    }
  }
}
```

## Production checklist

- Use an exact HTTPS `SITE_URL`.
- Generate a unique production secret.
- Keep old `BETTER_AUTH_SECRETS` versions only as long as retained ciphertext needs them; test decryption before removing one.
- Provision a separate `BCN_AUTH_PROXY_IP_SECRET` in both Nuxt and Convex.
- Configure OAuth callback and trusted origins for the public application origin.
- Set the correct Convex HTTP Actions URL.
- Provision the initial signing key through the internal operator action before admitting auth traffic.
- Keep Nuxt auth routes uncached.
- Test sign-up, sign-in, reload, refresh, and sign-out in the deployed topology.
- Verify protected Convex functions reject anonymous calls directly.
