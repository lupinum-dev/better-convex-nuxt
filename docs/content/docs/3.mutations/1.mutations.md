Unlike queries, mutations are **command-focused**: you call the mutation function, use the return value, and move on. The returned function has `pending`, `error`, `status`, and other state refs attached to it.

::callout{icon="i-lucide-lightbulb"}
**Tip**: Always extract `.value` from refs when calling a mutation. This makes the data flow explicit and avoids passing reactive wrappers to Convex.
::

## Basic Usage

```vue
<script setup lang="ts">
import { api } from '#convex/api'

const createTodo = useConvexMutation(api.todos.create)

const text = ref('')

async function handleSubmit() {
  try {
    const todo = await createTodo({ text: text.value })
    console.log('Created:', todo._id)
    text.value = ''
    navigateTo('/todos')
  } catch {
    // error.value is already set for template display
  }
}
</script>

<template>
  <form @submit.prevent="handleSubmit">
    <input v-model="text" :disabled="createTodo.pending.value" />
    <button :disabled="createTodo.pending.value">
      {{ createTodo.pending.value ? 'Saving...' : 'Save' }}
    </button>
    <p v-if="createTodo.error.value" class="error">{{ createTodo.error.value.message }}</p>
  </form>
</template>
```

## API Reference

### Parameters

| Parameter  | Type                            | Description                        |
| ---------- | ------------------------------- | ---------------------------------- |
| `mutation` | `FunctionReference<"mutation">` | Convex mutation function reference |
| `options`  | `UseConvexMutationOptions`      | Optional configuration             |

### Options

| Option             | Type                         | Default | Description                                 |
| ------------------ | ---------------------------- | ------- | ------------------------------------------- |
| `optimisticUpdate` | `(localStore, args) => void` | -       | Update local queries before server confirms |
| `onSuccess`        | `(result, args) => void`     | -       | Called after mutation succeeds              |
| `onError`          | `(error, args) => void`      | -       | Called after mutation fails                 |

::callout{icon="i-lucide-info" to="/docs/advanced/logging"}
Enable module-level logging via the `logging` config option for debugging mutations.
::

### Returns

| Property  | Type                                    | Description                                         |
| --------- | --------------------------------------- | --------------------------------------------------- |
| callable  | `(args) => Promise<Result>`             | Execute the mutation                                |
| `safe`    | `(args) => Promise<CallResult<Result>>` | Execute without throwing (`{ ok, data/error }`)     |
| `data`    | `Ref<Result \| undefined>`              | Result from last successful mutation                |
| `status`  | `ComputedRef<ConvexCallStatus>`         | `'idle'` \| `'pending'` \| `'success'` \| `'error'` |
| `pending` | `ComputedRef<boolean>`                  | True while mutation is in flight                    |
| `error`   | `Ref<Error \| null>`                    | Error from last attempt                             |
| `reset`   | `() => void`                            | Clear error and status                              |

## Key Behaviors

### Pending State

The `pending` ref tracks the latest mutation call. If a newer call finishes before an older one, state stays bound to the newer call:

```ts
const createTask = useConvexMutation(api.tasks.create)

// Fire two mutations
createTask({ text: 'A' })
createTask({ text: 'B' })

// pending belongs to the latest call
console.log(createTask.pending.value) // true
```

### Error Handling

Errors are captured in the `error` ref and re-thrown from the callable:

```ts
const createTask = useConvexMutation(api.tasks.create)

try {
  await createTask({ text: '' })
} catch (e) {
  // Can handle here OR use error.value in template
  console.log(createTask.error.value === e) // true
}
```

The error is automatically cleared when you call the mutation again.

### Callable vs `safe()`

- Call the mutation directly by default for normal app flows.
- Use `safe()` when you need a non-throwing boundary.

If your endpoint already returns an app-level `CallResult<T>`, `safe()` wraps that value as `CallResult<CallResult<T>>` by design.

::callout{icon="i-lucide-info" to="/docs/advanced/error-handling"}
See [Error Handling](/docs/advanced/error-handling) for safe-call patterns and a full decision table.
::

### Callback Hooks

Use per-mutation callbacks when you want local side effects without repeating `try/catch` glue:

```ts
const createTask = useConvexMutation(api.tasks.create, {
  onSuccess: (result, args) => {
    console.log('Created task', result._id, args.text)
  },
  onError: (error, args) => {
    console.error('Failed to create task', error.message, args.text)
  },
})
```

### Reset Function

Use `reset()` to clear error state:

```vue
<script setup>
const createTask = useConvexMutation(api.tasks.create)

// Clear error when user starts typing
function handleInput() {
  if (createTask.error.value) createTask.reset()
}
</script>
```

## With Optimistic Updates

For instant UI feedback, provide an `optimisticUpdate` function:

```ts
const createTodo = useConvexMutation(api.todos.create, {
  optimisticUpdate: (localStore, args) => {
    // Update the local query cache immediately
    updateQuery({
      query: api.todos.list,
      args: {},
      store: localStore,
      updater: (current) => {
        const optimisticTodo = {
          _id: crypto.randomUUID() as Id<'todos'>,
          _creationTime: Date.now(),
          text: args.text,
          completed: false,
        }
        return current ? [optimisticTodo, ...current] : [optimisticTodo]
      },
    })
  },
})
```

The optimistic update is automatically rolled back if the mutation fails.

::callout{icon="i-lucide-lightbulb" to="/docs/mutations/optimistic-updates"}
See [Optimistic Updates](/docs/mutations/optimistic-updates) for complete patterns and helper functions.
::

## Patterns

### Form Submission

```vue
<script setup lang="ts">
const text = ref('')
const createTodo = useConvexMutation(api.todos.create)

async function handleSubmit() {
  try {
    await createTodo({ text: text.value })
    text.value = ''
    navigateTo('/todos')
  } catch {
    // Error shown in template
  }
}
</script>

<template>
  <form @submit.prevent="handleSubmit">
    <input v-model="text" :disabled="createTodo.pending.value" @input="createTodo.reset" />
    <button :disabled="createTodo.pending.value || !text.trim()">
      {{ createTodo.pending.value ? 'Creating...' : 'Create Todo' }}
    </button>
    <p v-if="createTodo.error.value" class="text-red-500">{{ createTodo.error.value.message }}</p>
  </form>
</template>
```

### Multiple Independent Mutations

Each `useConvexMutation` call has independent state:

```vue
<script setup lang="ts">
const createTodo = useConvexMutation(api.todos.create)
const deleteTodo = useConvexMutation(api.todos.remove)
</script>

<template>
  <button :disabled="createTodo.pending.value">Create</button>
  <button :disabled="deleteTodo.pending.value">Delete</button>
</template>
```

### Per-Item Loading States

Track loading state per item using a Map:

```vue
<script setup lang="ts">
const toggleTodo = useConvexMutation(api.todos.toggle)
const loadingItems = ref(new Map<string, boolean>())

async function handleToggle(id: string) {
  loadingItems.value.set(id, true)
  try {
    await toggleTodo({ id })
  } finally {
    loadingItems.value.delete(id)
  }
}
</script>

<template>
  <ul>
    <li v-for="todo in todos" :key="todo._id">
      <input
        type="checkbox"
        :checked="todo.completed"
        :disabled="loadingItems.get(todo._id)"
        @change="handleToggle(todo._id)"
      />
      {{ todo.text }}
    </li>
  </ul>
</template>
```

### Status-Based UI

```vue
<script setup lang="ts">
const createTodo = useConvexMutation(api.todos.create)
</script>

<template>
  <div>
    <button @click="createTodo({ text: 'New todo' })">
      <span v-if="createTodo.status.value === 'idle'">Create</span>
      <span v-else-if="createTodo.status.value === 'pending'">Creating...</span>
      <span v-else-if="createTodo.status.value === 'success'">Created!</span>
      <span v-else-if="createTodo.status.value === 'error'">Failed</span>
    </button>

    <p v-if="createTodo.status.value === 'error'">
      {{ createTodo.error.value?.message }}
      <button @click="createTodo.reset">Dismiss</button>
    </p>
  </div>
</template>
```

## Comparison with useConvexQuery

| Aspect             | useConvexQuery             | useConvexMutation                |
| ------------------ | -------------------------- | -------------------------------- |
| **Purpose**        | Fetch data to display      | Execute actions                  |
| **Data**           | Returns reactive `data`    | Returns result from the callable |
| **Real-time**      | Auto-updates via WebSocket | N/A                              |
| **SSR**            | Yes                        | N/A (client-only)                |
| **Template usage** | Render `data.value`        | Show `pending`/`error`           |

## TypeScript

Full type inference from your Convex schema:

```ts
// Args are typed based on your mutation definition
const createTodo = useConvexMutation(api.todos.create)

// Type error: 'title' doesn't exist, should be 'text'
await createTodo({ title: 'wrong' })

// Correct: matches your schema
await createTodo({ text: 'correct' })

// Return type is inferred
const todo = await createTodo({ text: 'Hello' })
console.log(todo._id) // typed as Id<"todos">
```

## Common Mistakes

### 1. Not handling errors

```ts
// WRONG: Unhandled promise rejection
async function handleClick() {
  await createTodo({ text: 'Hello' })
  // If mutation fails, this crashes
}

// RIGHT: Handle the error
async function handleClick() {
  try {
    await createTodo({ text: 'Hello' })
  } catch {
    // Error is in error.value for display
  }
}
```

### 2. Forgetting to clear form on success

```ts
// WRONG: Form stays filled after success
async function handleSubmit() {
  await createTodo({ text: text.value })
  navigateTo('/list')
}

// RIGHT: Clear form state
async function handleSubmit() {
  await createTodo({ text: text.value })
  text.value = '' // Clear before navigation
  navigateTo('/list')
}
```

### 3. Using shared pending for independent actions

```vue
<!-- WRONG: Single pending blocks all buttons -->
<script setup>
const deleteItem = useConvexMutation(api.items.delete)
</script>
<template>
  <button
    v-for="item in items"
    :disabled="deleteItem.pending.value"
    @click="deleteItem({ id: item._id })"
  >
    Delete
  </button>
</template>

<!-- RIGHT: Track per-item loading -->
<script setup>
const deleteItem = useConvexMutation(api.items.delete)
const deleting = ref(new Set())

async function handleDelete(id) {
  deleting.value.add(id)
  try {
    await deleteItem({ id })
  } finally {
    deleting.value.delete(id)
  }
}
</script>
<template>
  <button v-for="item in items" :disabled="deleting.has(item._id)" @click="handleDelete(item._id)">
    Delete
  </button>
</template>
```
