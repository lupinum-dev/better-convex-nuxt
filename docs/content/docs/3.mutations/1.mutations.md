Unlike queries, mutations are **action-focused**: you call `execute()`, use the return value, and move on. There's no data to render in templates.

::callout{icon="i-lucide-lightbulb"}
**Tip**: Always extract `.value` from refs when calling `execute()`. This makes the data flow explicit and avoids potential issues.
::

::callout{icon="i-lucide-info"}
You can declare `useConvexMutation()` during SSR or normal component setup. Only `execute()` requires a live client and must run where the Convex client is available.
::

## Basic Usage

```vue
<script setup lang="ts">
import { api } from '~~/convex/_generated/api'

const { execute, pending, error } = useConvexMutation(api.todos.create)

const text = ref('')

async function handleSubmit() {
  try {
    // Extract .value from refs before passing to execute
    const todo = await execute({ text: text.value })
    console.log('Created:', todo._id)
    text.value = ''
    navigateTo('/todos')
  } catch {
    // error.value is already set for template display
  }
}
</script>

<template>
  <form @submit.prevent="handleSubmit">
    <input v-model="text" :disabled="pending" />
    <button :disabled="pending">
      {{ pending ? 'Saving...' : 'Save' }}
    </button>
    <p v-if="error" class="error">{{ error.message }}</p>
  </form>
</template>
```

## API Reference

### Parameters

| Parameter  | Type                            | Description                        |
| ---------- | ------------------------------- | ---------------------------------- |
| `mutation` | `FunctionReference<"mutation">` | Convex mutation function reference |
| `options`  | `UseConvexMutationOptions`      | Optional configuration             |

### Options

| Option             | Type                         | Default | Description                                 |
| ------------------ | ---------------------------- | ------- | ------------------------------------------- |
| `optimisticUpdate` | `(ctx, args) => void`        | -       | Update local queries before server confirms |
| `onSuccess`        | `(result, args) => void`     | -       | Called after mutation succeeds              |
| `onError`          | `(error, args) => void`      | -       | Called after mutation fails                 |

::callout{icon="i-lucide-info" to="/docs/advanced/logging"}
Enable module-level logging via the `logging` config option for debugging mutations.
::

### Returns

| Property      | Type                                    | Description                                         |
| ------------- | --------------------------------------- | --------------------------------------------------- |
| `execute`     | `(args) => Promise<Result>`             | Execute the mutation                                |
| `executeSafe` | `(args) => Promise<CallResult<Result>>` | Execute without throwing (`{ ok, data/error }`)     |
| `data`        | `Ref<Result \| undefined>`              | Result from last successful mutation                |
| `status`      | `ComputedRef<MutationStatus>`           | `'idle'` \| `'pending'` \| `'success'` \| `'error'` |
| `pending`     | `ComputedRef<boolean>`                  | True while mutation is in flight                    |
| `error`       | `Ref<Error \| null>`                    | Error from last attempt                             |
| `reset`       | `() => void`                            | Clear error and status                              |

## Key Behaviors

### Pending State

The `pending` ref is `true` while **any** mutation call is in flight. Handles concurrent mutations correctly:

```ts
const { execute, pending } = useConvexMutation(api.tasks.create)

// Fire two mutations
execute({ text: 'A' })
execute({ text: 'B' })

// pending is true until BOTH complete
console.log(pending.value) // true
```

### Error Handling

Errors are captured in `error` ref AND re-thrown from `execute()`:

```ts
const { execute, error } = useConvexMutation(api.tasks.create)

try {
  await execute({ text: '' })
} catch (e) {
  // Can handle here OR use error.value in template
  console.log(error.value === e) // true
}
```

The error is automatically cleared when you call `execute()` again.

### `execute()` vs `executeSafe()`

- Use `execute()` by default for normal app flows.
- Use `executeSafe()` when you need a non-throwing boundary (middleware/startup/background/legacy endpoints).

If your endpoint already returns an app-level `CallResult<T>`, `executeSafe()` wraps that value as `CallResult<CallResult<T>>` by design.

::callout{icon="i-lucide-info" to="/docs/advanced/error-handling"}
See [Error Handling](/docs/advanced/error-handling) for safe-call patterns and a full decision table.
::

### Callback Hooks

Use per-mutation callbacks when you want local side effects without repeating `try/catch` glue:

```ts
const { execute } = useConvexMutation(api.tasks.create, {
  onSuccess: (result, args) => {
    console.log('Created task', result._id, args.text)
  },
  onError: (error, args) => {
    console.error('Failed to create task', error.message, args.text)
  },
})
```

### Reset Function

Use `reset()` to clear error state:

```vue
<script setup>
const { execute, error, reset } = useConvexMutation(api.tasks.create)

// Clear error when user starts typing
function handleInput() {
  if (error.value) reset()
}
</script>
```

## With Optimistic Updates

For instant UI feedback, provide an `optimisticUpdate` function:

```ts
const { execute } = useConvexMutation(api.todos.create, {
  optimisticUpdate: (ctx, args) => {
    // Update the local query cache immediately
    ctx.query(api.todos.list, {}).update((current) => {
      const optimisticTodo = {
        _id: crypto.randomUUID() as Id<'todos'>,
        _creationTime: Date.now(),
        text: args.text,
        completed: false,
      }
      return current ? [optimisticTodo, ...current] : [optimisticTodo]
    })
  },
})
```

The optimistic update is automatically rolled back if the mutation fails.

::callout{icon="i-lucide-lightbulb" to="/docs/mutations/optimistic-updates"}
See [Optimistic Updates](/docs/mutations/optimistic-updates) for complete patterns and helper functions.
::

## Patterns

### Form Submission

```vue
<script setup lang="ts">
const text = ref('')
const { execute, pending, error, reset } = useConvexMutation(api.todos.create)

async function handleSubmit() {
  try {
    await execute({ text: text.value })
    text.value = ''
    navigateTo('/todos')
  } catch {
    // Error shown in template
  }
}
</script>

<template>
  <form @submit.prevent="handleSubmit">
    <input v-model="text" :disabled="pending" @input="reset" />
    <button :disabled="pending || !text.trim()">
      {{ pending ? 'Creating...' : 'Create Todo' }}
    </button>
    <p v-if="error" class="text-red-500">{{ error.message }}</p>
  </form>
</template>
```

### Multiple Independent Mutations

Each `useConvexMutation` call has independent state:

```vue
<script setup lang="ts">
const {
  execute: createTodo,
  pending: isCreating,
  error: createError,
} = useConvexMutation(api.todos.create)

const {
  execute: deleteTodo,
  pending: isDeleting,
  error: deleteError,
} = useConvexMutation(api.todos.remove)
</script>

<template>
  <button :disabled="isCreating">Create</button>
  <button :disabled="isDeleting">Delete</button>
</template>
```

### Per-Item Loading States

Track loading state per item using a Map:

```vue
<script setup lang="ts">
const { execute: toggleTodo } = useConvexMutation(api.todos.toggle)
const loadingItems = ref(new Map<string, boolean>())

async function handleToggle(id: string) {
  loadingItems.value.set(id, true)
  try {
    await toggleTodo({ id })
  } finally {
    loadingItems.value.delete(id)
  }
}
</script>

<template>
  <ul>
    <li v-for="todo in todos" :key="todo._id">
      <input
        type="checkbox"
        :checked="todo.completed"
        :disabled="loadingItems.get(todo._id)"
        @change="handleToggle(todo._id)"
      />
      {{ todo.text }}
    </li>
  </ul>
</template>
```

### Status-Based UI

```vue
<script setup lang="ts">
const { execute, status, error, reset } = useConvexMutation(api.todos.create)
</script>

<template>
  <div>
    <button @click="execute({ text: 'New todo' })">
      <span v-if="status === 'idle'">Create</span>
      <span v-else-if="status === 'pending'">Creating...</span>
      <span v-else-if="status === 'success'">Created!</span>
      <span v-else-if="status === 'error'">Failed</span>
    </button>

    <p v-if="status === 'error'">
      {{ error?.message }}
      <button @click="reset">Dismiss</button>
    </p>
  </div>
</template>
```

## Comparison with useConvexQuery

| Aspect             | useConvexQuery             | useConvexMutation               |
| ------------------ | -------------------------- | ------------------------------- |
| **Purpose**        | Fetch data to display      | Execute actions                 |
| **Data**           | Returns reactive `data`    | Returns result from `execute()` |
| **Real-time**      | Auto-updates via WebSocket | N/A                             |
| **SSR**            | Yes                        | Safe to declare; `execute()` runs client-side |
| **Template usage** | Render `data.value`        | Show `pending`/`error`          |

## TypeScript

Full type inference from your Convex schema:

```ts
// Args are typed based on your mutation definition
const { execute } = useConvexMutation(api.todos.create)

// Type error: 'title' doesn't exist, should be 'text'
await execute({ title: 'wrong' })

// Correct: matches your schema
await execute({ text: 'correct' })

// Return type is inferred
const todo = await execute({ text: 'Hello' })
console.log(todo._id) // typed as Id<"todos">
```

## Common Mistakes

### 1. Not handling errors

```ts
// WRONG: Unhandled promise rejection
async function handleClick() {
  await execute({ text: 'Hello' })
  // If mutation fails, this crashes
}

// RIGHT: Handle the error
async function handleClick() {
  try {
    await execute({ text: 'Hello' })
  } catch {
    // Error is in error.value for display
  }
}
```

### 2. Forgetting to clear form on success

```ts
// WRONG: Form stays filled after success
async function handleSubmit() {
  await execute({ text: text.value })
  navigateTo('/list')
}

// RIGHT: Clear form state
async function handleSubmit() {
  await execute({ text: text.value })
  text.value = '' // Clear before navigation
  navigateTo('/list')
}
```

### 3. Using shared pending for independent actions

```vue
<!-- WRONG: Single pending blocks all buttons -->
<script setup>
const { execute, pending } = useConvexMutation(api.items.delete)
</script>
<template>
  <button v-for="item in items" :disabled="pending" @click="execute({ id: item._id })">
    Delete
  </button>
</template>

<!-- RIGHT: Track per-item loading -->
<script setup>
const { execute } = useConvexMutation(api.items.delete)
const deleting = ref(new Set())

async function handleDelete(id) {
  deleting.value.add(id)
  try {
    await execute({ id })
  } finally {
    deleting.value.delete(id)
  }
}
</script>
<template>
  <button v-for="item in items" :disabled="deleting.has(item._id)" @click="handleDelete(item._id)">
    Delete
  </button>
</template>
```
