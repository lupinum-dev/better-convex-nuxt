Use this page when the client needs to trigger a Convex action instead of a mutation.

`useConvexAction()` has the same tracked-call shape as `useConvexMutation()`, but it is for action semantics: longer-running work, third-party API calls, or side effects that do not belong in a normal mutation.

## The Default Shape

Start with the common case:

```ts
const sendEmail = useConvexAction(api.emails.send)

async function handleSend() {
  await sendEmail({ to: 'user@example.com', subject: 'Hello' })
}
```

Like mutations, the composable is synchronous to create and the returned function is what you await.

## What Stays The Same

Actions keep the same tracked call surface:

- `status`
- `pending`
- `data`
- `error`
- `reset()`
- `onSuccess`
- `onError`

That consistency is intentional. The UI should not need a completely different mental model just because the backend function type is `action`.

## What Changes

The difference is not the client API. The difference is the backend job you are invoking.

Choose an action when the work needs action semantics, such as:

- calling an external service
- longer-running server work
- side effects that are not appropriate for a normal mutation

Choose a mutation when you are changing application state in the normal Trellis write path.

## Example With Callbacks

```ts
const sendInvite = useConvexAction(api.members.sendInvite, {
  onSuccess: () =>
    toast.add({ title: 'Invite sent', color: 'success' }),
  onError: (error) =>
    toast.add({ title: 'Invite failed', description: error.message, color: 'error' }),
})
```

The callback contract matches mutations: result or error first, args second.

## Error And State Rules

Actions follow the same state rules as mutations:

- initial state is `idle`
- execution moves state to `pending`
- success stores the latest successful result
- errors are normalized into Trellis call errors where possible
- `reset()` clears the tracked state
- latest request wins for composable state

That means you can reuse the same UI patterns for buttons, banners, and retry affordances.

## When Not To Use An Action

Do not use an action just because the UI flow feels “important.”

If the backend work is really a normal state change, keep it a mutation. Using actions casually makes the system harder to reason about and weakens the separation between normal writes and exceptional side-effecting work.

## Common Mistakes

- Treating actions as the default write path.
- Awaiting `useConvexAction(...)` instead of awaiting the returned function call.
- Reaching for an action when the backend operation is really a normal mutation.
