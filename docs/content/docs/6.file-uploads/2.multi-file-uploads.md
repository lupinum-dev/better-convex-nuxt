Use queue mode when one component needs multiple uploads, per-item state, and bounded concurrency instead of one storage result.

Trellis uses the same `useConvexUpload(...)` surface for queue mode, but the important state becomes the queue:

- `items`
- `cancelItem(id)`
- `cancelAll()`
- `clearFinished()`
- aggregate counts and progress

## Concurrency Is Explicit

Queue mode respects `maxConcurrent`.

The value can come from:

- the composable call
- runtime config `trellis.upload.maxConcurrent`

Per-instance options win over runtime defaults.

## Queue Input Shapes

Queue mode accepts:

- one `File`
- `File[]`
- `FileList`
- explicit enqueue items with per-file mutation args

That last form matters when each file needs different metadata.

## Queue Behavior

The queue runtime tracks each item independently:

- queued
- pending
- success
- error
- cancelled

Trellis also exposes aggregate counters so UI can show progress without recomputing queue state manually.

## Continue Or Halt On Error

Queue mode supports `continueOnError`.

Use it deliberately:

- `true` when the queue should keep processing the rest
- `false` when one failure should stop the batch

Do not hide that decision. Batch semantics are part of the user experience.

## Per-Item Cancellation

Use:

- `cancelItem(id)` for one queued or in-flight item
- `cancelAll()` for the whole queue

`clearFinished()` is different. It only removes completed, errored, or cancelled entries from the visible queue state.

## Common mistakes

- Treating queue mode like a loop of single-file uploads instead of using the queue state directly
- Forgetting that `maxConcurrent` can come from runtime config
- Clearing queue history too early when the user still needs to inspect failures
