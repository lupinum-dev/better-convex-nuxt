Use this page when you need to run Trellis locally, either in the repo itself or in one of the runnable examples.

This page is about local development shape, not production deployment.

## The Two Main Local Modes

In this repo, there are two common local paths:

- example app development in `examples/*`
- maintainer development in `test/internal-harness`

The commands and env ownership differ slightly between them.

## Example App Flow

For most examples, the flow is:

1. `cd` into the example directory
2. copy `.env.example` to `.env.local` if the example needs app-owned env vars
3. run `pnpm install`
4. run `pnpm dev`

In the examples, `pnpm dev` starts a local Convex deployment and injects the local Convex URLs for Nuxt.

That means:

- `CONVEX_URL` and `CONVEX_SITE_URL` are launcher-owned
- auth secrets and provider credentials are still app-owned when that example needs them

## Internal Harness Flow

The maintainer harness lives in `test/internal-harness/`.

Important commands from the repo root:

```bash
pnpm dev
pnpm dev:local
pnpm dev:local:reset
pnpm dev:build
```

Use them like this:

- `pnpm dev`: run the internal harness against the default env setup
- `pnpm dev:local`: force the harness onto local Convex ports
- `pnpm dev:local:reset`: same as local, but reset backend state first
- `pnpm dev:build`: build the harness without starting it

## Local Env Layout

This repo uses:

- `.env.example` for checked-in example defaults
- `.env.local` for the shared local runtime file used by Convex and Nuxt

The launcher rewrites launcher-owned runtime keys in `.env.local` when it starts local services and preserves app-owned values already present there.

## Typical Local Defaults

Typical local defaults in this repo are:

- `CONVEX_URL=http://127.0.0.1:3210`
- `CONVEX_SITE_URL=http://127.0.0.1:3211`
- `SITE_URL=http://localhost:3000`

For the internal harness, `USE_LOCAL_CONVEX=true` switches the Nuxt config onto the local Convex ports.

## Running A Dedicated Local Convex Backend

When you need a dedicated local backend for auth, MCP, or E2E:

```bash
cd test/internal-harness
pnpm exec convex dev --local
```

If auth setup is missing locally:

```bash
pnpm exec convex env set SITE_URL http://localhost:3000 --env-file .env.local
pnpm exec convex env set BETTER_AUTH_SECRET <strong-random-secret> --env-file .env.local
```

## When To Use The Harness

Use `test/internal-harness/` for:

- feature development against the local module source
- auth and MCP verification
- regression reproduction
- root E2E and eval flows

Use the example apps when you want to understand or demo a consumer-facing Trellis pattern instead.

## Related Local Commands

Useful root commands during local work:

```bash
pnpm check:cli
pnpm test:types
pnpm test:contracts
pnpm test:internals
pnpm test
pnpm lint
pnpm docs:api-surface
```

Use `pnpm check:cli` when you are touching CLI behavior. Use `pnpm docs:api-surface` when public API surfaces change.

## Common Mistakes

- Editing launcher-owned local Convex URLs by hand in a repo example.
- Using the internal harness when an example app would be a clearer reproduction.
- Forgetting that `pnpm dev:prepare` only prepares type and harness surfaces; it does not rebuild the published package or CLI.
- Treating local-development defaults as if they were the recommended production deployment shape.
