## Four Actor Lanes

The examples intentionally reuse one permission model across four caller lanes:

- browser user: a normal signed-in workspace user resolved from Better Auth
- trusted caller: a server route, webhook bridge, or MCP adapter that forwards an explicit trusted caller payload
- bearer-key caller: an MCP client authenticated by a stored hashed key that resolves to a real bound user
- share-token grant: a document-scoped bearer token that grants access without a workspace actor

These lanes should converge on the same backend handlers whenever possible. The difference is how the
actor is resolved, not where the business rules live.

## `userId` In The Examples

In most examples, `actor.userId` is the auth-subject string stored in `users.authId` and in
ownership fields like `ownerId`. It is not a Convex `Id<'users'>`.

That naming is kept because the examples already model ownership that way, but the actor files now
call this out explicitly so readers do not confuse auth subjects with user document ids.

## Single-Workspace Simplification vs Memberships

Examples `03` through `09`, plus the default starter, intentionally use the canonical simple model:

- `users.workspaceId`
- `users.role`

That keeps the default tenant-scoped path easy to read and easy to scaffold. It is the canonical
starting point for this repo, not just one equally preferred option among many.

Example `10-agency-portal` switches to the scalable model:

- `users.workspaceId` is only the current workspace selection
- `memberships.role` is the authoritative authorization grant

Use the simple model when:

- each user belongs to one workspace at a time
- role changes are simple
- you are still learning the tenant-scoped pattern

Move to memberships when:

- users belong to multiple workspaces
- a user can switch active workspaces
- authorization should survive workspace switching without mutating the profile row

Canonical references:

- Example `03-team-todo` is the canonical single-workspace `workspaces.ts` reference
- Example `10-agency-portal` is the canonical memberships-based `workspaces.ts` reference
- this repo treats memberships as the explicit upgrade path, not the default starter model

## Migration Path

Moving from the simple model to memberships is a data-model change, not a package feature toggle:

1. Keep `users` as the profile row keyed by auth subject.
2. Add `memberships` with `userId`, `workspaceId`, and `role`.
3. Change `getActor()` to resolve the active membership for the current workspace.
4. Stop treating `users.role` as authoritative.
5. Keep `users.workspaceId` only if you want a persisted current-workspace preference.

## When To Use `createFunctions()`

Use `createFunctions()` in `convex/functions.ts` whenever protected Convex handlers may be called by:

- Nitro server routes
- webhook adapters
- MCP tools
- any other trusted non-browser bridge

That centralizes trusted-caller transport once. Handlers still keep auth explicit with `const actor = await ctx.actor()`.

## Denial Semantics

The examples follow one security rule:

- cross-tenant or existence-hiding paths should fail like not found
- same-tenant but insufficient rights should fail like forbidden

That means `loadResource()` and `ensureTenant()` usually throw `Resource not found.` for cross-tenant
access, while role or capability failures use `enforce()` and surface as `Forbidden: <action>`.

## Example 11 Onboarding Context

Most authenticated examples return `null` from `getPermissionContext()` when there is no resolved
workspace actor yet.

Example `11-mcp-reference` is the intentional exception. It returns an onboarding-safe context for a
signed-in user without a workspace so the app and MCP flows can still render guided setup state.

## Demo-Only Join Flows

Several examples let a user:

- list all workspaces
- join by slug
- pick a role during join

That is demo-only scaffolding so the permission examples are easy to exercise. Production apps
should not allow arbitrary users to self-assign elevated roles like `admin`, `owner`, or
`agency_manager`.
