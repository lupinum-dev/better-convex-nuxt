Your app may be called by browser users, server routes, webhook handlers, and AI agents. Each arrives through a different transport, but they should all obey the same business rules. This page explains how to add new caller types without splitting your authorization model.

## Different Transports, Same Authorization

Consider a team workspace app. Three callers need to create todos:

| Caller | Transport | How identity arrives |
|--------|-----------|---------------------|
| Browser user | WebSocket/HTTP | Auth cookie, exchanged for Convex token |
| MCP agent | HTTP via MCP | Bearer token in `Authorization` header |
| Webhook bot | Internal HTTP | API key + workspace ID |

Each caller resolves to a **principal** (transport-level identity), which maps to an **actor** (business role). The actor is what your handlers check:

```ts
// All three callers can map to the same actor shape
{ kind: 'user', userId: '...', role: 'member', tenantId: '...' }
```

If two callers should obey the same rules, they should produce the same actor shape. That is the test.

## When to Add a New Caller Type

Add a new principal kind when a caller arrives through a transport that does not fit your existing resolution logic. Common triggers:

- **Adding MCP support** — agents authenticate with bearer tokens, not browser sessions
- **Adding webhook handlers** — external services call in with API keys
- **Adding server-to-server routes** — internal services forward identity explicitly

You do not need a new principal kind for every user role. Roles belong in the actor, not the principal.

## A Concrete Example

The [team workspace example](https://github.com/lupinum-dev/trellis/tree/main/examples/03-team-workspace) defines three principal kinds:

```ts [shared/principal.ts]
export type TeamTodoPrincipal =
  | { kind: 'anonymous' }
  | { kind: 'user'; userId: string }
  | { kind: 'mcp'; userId: string; role: Role; tenantId?: Id<'workspaces'> }
```

The actor resolver maps all three into a single actor shape:

```ts [convex/auth/actor.ts]
export async function getActorFromPrincipal(
  ctx: TeamTodoCtx,
  _args: Record<string, unknown>,
  principal: TeamTodoPrincipal,
): Promise<Actor | null> {
  switch (principal.kind) {
    case 'anonymous':
      return null
    case 'mcp':
      // MCP principal already carries role and tenant
      return {
        kind: 'user',
        userId: principal.userId,
        role: principal.role,
        tenantId: principal.tenantId,
      }
    case 'user':
      // Browser principal looks up role from the database
      return await loadActorByAuthId(ctx, principal.userId)
  }
}
```

The MCP principal carries role and tenant directly (resolved during MCP auth). The browser principal looks them up from the users table. Both produce the same actor shape, so `canCreateTodo` and `canUpdateTodo` work identically regardless of the caller.

## Capability Snapshots vs Per-Record Permissions

These are two separate systems. Keep them distinct:

**Capability snapshots** answer: "Should this agent see the `create-todo` tool at all?"

```ts [server/lib/mcp-auth.ts]
export function getCapabilitiesForPrincipal(principal) {
  return {
    listTodos: principal.kind === 'mcp' && !!principal.tenantId,
    createTodo: principal.kind === 'mcp' && canWrite(principal.role),
  }
}
```

**Per-record `_can` flags** answer: "Can this user update *this specific* todo?"

```ts [convex/auth/capabilities.ts]
export const todoCapabilities = defineCapabilities<Doc<'todos'>>()({
  update: (actor, todo) => can(actor, canUpdateTodo(todo)),
  delete: (actor, todo) => can(actor, canDeleteTodo(todo)),
})
```

Capabilities control tool discovery in MCP. `_can` flags control UI buttons and per-record access. Trying to make one system do both jobs leads to stale or confusing authorization.

## Single-Workspace vs Memberships

Trellis examples use a simple single-workspace model as the default:

```ts
// Single workspace: role and tenant live on the user row
{ userId: '...', role: 'member', workspaceId: 'ws_123' }
```

This is the easiest starting point. Move to a memberships model when:

- Users belong to **multiple workspaces**
- The active workspace changes independently of the user profile
- Authorization should come from membership records, not the `users` row

This is a data-model change, not a runtime-mode switch. See [Personal to Workspace](/docs/permissions/personal-to-workspace) for the migration path.

::tip
Keep principals transport-shaped. A principal carries identity (user ID, session ID). Roles, entitlements, and tenant flags belong in the actor. If the principal starts carrying your whole permission model, you lose the separation between transport and business authorization.
::
