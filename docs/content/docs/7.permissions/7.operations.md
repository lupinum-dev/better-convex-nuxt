Use this page when one business action should exist once and project cleanly to multiple call sites or transports.

`defineOperation(...)` is Trellis' reusable business-action primitive.

## Why Use An Operation

Use an operation when the app should own one coherent action with its own:

- guard
- load phase
- authorize phase
- preview phase
- handler

That matters most for destructive work, but the pattern is useful anywhere one business action should stay unified instead of being reassembled across multiple handlers.

## Basic Shape

From the team-workspace example:

```ts
export const removeTodoOp = defineOperation({
  id: 'todos.remove',
  name: 'removeTodo',
  kind: 'destructive',
  args: deleteTodo.args,
  guard: canReadTodo,
  load: async (ctx, args) => {
    const todo = await ctx.db.get(args.id)
    requireRecord(todo, 'Todo')
    return { todo }
  },
  authorize: {
    check: (_actor, { todo }) => canDeleteTodo(todo),
  },
  preview: async (_ctx, _args, { todo }) => ({
    display: {
      summary: `Will permanently delete "${todo.title}"`,
      warn: 'This cannot be undone',
      affects: { todos: 1 },
    },
    confirm: {
      operation: 'todos.remove',
      targetId: todo._id,
      affectedCounts: { todos: 1 },
    },
  }),
  handler: async (ctx, args) => {
    await ctx.db.delete(args.id)
    return null
  },
})
```

That keeps the business rule in one place instead of splitting preview, confirmation, and execution logic across unrelated files.

## Register It As A Mutation And Preview Query

The normal projection is:

```ts
export const remove = mutation(removeTodoOp)
export const previewRemove = query(previewOf(removeTodoOp))
```

That gives you:

- one execute path
- one preview path
- shared guard, load, and authorization logic

The preview path is not a second implementation. It is the projected preview phase of the same operation.

## Destructive Operations Have Extra Requirements

Trellis is strict here on purpose.

For destructive operations:

- `kind` must be `destructive`
- the operation must have an `id`
- the operation must have `preview(...)`
- `defineTrellis({ destructiveSafety })` must be configured before registration

If you skip `destructiveSafety`, Trellis rejects the registration.

That is not ceremony. It is how Trellis binds confirmation to previewed destructive state instead of trusting a loose client-side convention.

## Preview Is Part Of The Contract

The preview result should make two things explicit:

- what the user or agent is about to do
- what confirmation payload must be bound to execution

That is why Trellis examples return both:

- `display`
- `confirm`

`display` is for humans and operator-facing UI.
`confirm` is for the durable confirmation contract.

## Operations Are The Bridge To MCP Tools

Operation metadata is also what lets Trellis bind execute and preview refs correctly for agent-facing destructive tools.

`tool.fromOperation(...)` requires:

- an operation with an `id`
- an execute ref projected from that operation
- when preview is used, a preview ref projected from that same operation

If those refs do not match, Trellis rejects the binding instead of letting a destructive tool drift away from the protected business action it claims to represent.

## Use Operations For Reuse, Not For Everything

Do not wrap every trivial mutation in `defineOperation(...)`.

Reach for operations when:

- the action is destructive
- the same action needs preview and execute projections
- the same business rule should back both UI and agent/tool flows
- the action is important enough to deserve one stable contract

For a simple non-destructive one-off mutation, a normal protected handler is often clearer.

## Common Mistakes

- Treating preview as optional for destructive actions
- Forgetting `id` on an operation that needs stable projection
- Duplicating preview and execute policy in separate handlers
- Wrapping tiny local mutations in operations just because the primitive exists
