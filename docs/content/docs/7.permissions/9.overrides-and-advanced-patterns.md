## Overview

The repo now makes the default tenant-scoped path much easier:

- `convex/functions.ts` owns the protected handler builders
- handlers call `const actor = await ctx.actor()`
- trusted-caller transport lives in one place instead of every handler

That easy path is intentionally a preset, not a closed framework.

The rule is:

- the module owns transport and composition
- your app owns policy and business rules

If the defaults fit, keep them.

If they do not, override the specific seam directly.

If the seam itself is wrong for your app, stop using `createFunctions()` and compose with
`convex-helpers` directly.

## What The Module Actually Owns

`better-convex-nuxt/functions` provides [`createFunctions()`](/Users/matthias/Git/0_libs/better-convex-nuxt/src/runtime/functions/index.ts).

That function composes a few concerns:

- hidden trusted-caller transport
- lazy, memoized `ctx.actor()`
- optional tenant isolation
- optional custom row-level security
- optional triggers
- optional `onSuccess` hooks

`better-convex-nuxt/auth` provides the in-handler primitives:

- `enforce()`
- `can()`
- `deny()`
- `requireRecord()`
- `ensureTenant()`
- `loadTenantResource()`
- `withCan()`

Those stay small and stable. They are not coupled to the default builder.

## Override Matrix

| Concern                  | Default                               | Extend                                   | Replace                                               | Notes                                                                               |
| ------------------------ | ------------------------------------- | ---------------------------------------- | ----------------------------------------------------- | ----------------------------------------------------------------------------------- |
| Actor resolution         | `createDefaultGetActor()`             | `defineActorExtension(...)`              | `createFunctions(..., { actor })`                     | Use the extension path first when your schema still follows the default user lookup |
| Trusted caller transport | Enabled in workspace-MCP style setups | None                                     | `trustedCaller: false`, or bypass `createFunctions()` | Transport shape is fixed in v1                                                      |
| Tenant isolation         | Off unless configured                 | `tenantIsolation`                        | Omit it and provide custom `rls.rules`                | Table list stays explicit by design                                                 |
| Custom RLS               | Off unless configured                 | `rls.rules` on top of tenant isolation   | `rls.rules` only                                      | Custom RLS merges over generated tenant rules                                       |
| Triggers                 | Off unless configured                 | `triggers`                               | Use your own Convex custom wrappers if needed         | Applied only on mutations                                                           |
| Success hooks            | Off unless configured                 | `onSuccess.query` / `onSuccess.mutation` | Use lower-level composition                           | Good for scheduling and cleanup, not primary auth logic                             |

## Precedence Rules

These are the actual override rules inside `createFunctions()`:

1. trusted-caller transport is parsed first
2. trusted-caller identity is attached to ctx
3. `ctx.actor()` is created
4. tenant-isolation rules are synthesized
5. custom `rls.rules` are resolved
6. custom RLS merges over tenant-isolation rules
7. query or mutation `db` is wrapped with RLS
8. mutation `db` is additionally wrapped with triggers
9. `onSuccess` runs after the handler succeeds

Important consequences:

- `ctx.actor()` can see trusted-caller context
- your RLS rules can call `await ctx.actor()`
- custom RLS wins over generated tenant rules for overlapping tables
- triggers run against the already-composed mutation db

## Actor Overrides

### Default path

If your app follows the canonical shape:

- `users.authId`
- `users.role`
- `users.workspaceId`
- `users.by_auth_id`

use the default actor resolver:

```ts [convex/auth/actor.ts]
import { createDefaultGetActor } from 'better-convex-nuxt/auth'

export const getActor = createDefaultGetActor()
```

### Add fields without replacing the actor

If you only need extra fields like `plan` or `managerId`, extend the default actor:

```ts [convex/auth/actor.ts]
import { createDefaultGetActor, defineActorExtension } from 'better-convex-nuxt/auth'

export const getActor = createDefaultGetActor(
  defineActorExtension({
    fields: async (ctx, user) => ({
      plan: (await ctx.db.get(user.workspaceId))?.plan ?? 'free',
    }),
  }),
)
```

Use this when the base lookup logic is still correct and only the actor payload changes.

### Replace the actor resolver

If your authority does not come from the default user row, replace it:

```ts [convex/functions.ts]
import { createFunctions } from 'better-convex-nuxt/functions'

import { mutation, query } from './_generated/server'
import { getActor } from './auth/actor'

export const { query: appQuery, mutation: appMutation } = createFunctions(query, mutation, {
  actor: getActor,
})
```

Examples:

- memberships table decides the role
- current workspace selection lives outside `users.workspaceId`
- some callers resolve to bots or service actors instead of normal users

## Trusted Caller Overrides

### Default path

When trusted callers are enabled, `createFunctions()` consumes hidden trusted-caller args and strips
them before the handler sees `args`.

Handlers do not know or care how the actor arrived:

```ts
export const create = appMutation({
  args: createTodo.args,
  handler: async (ctx, args) => {
    const actor = await ctx.actor()
    enforce(actor, 'Create todo', canCreateTodo)
  },
})
```

### Disable it

For browser-only apps:

```ts [convex/functions.ts]
export const { query: appQuery, mutation: appMutation } = createFunctions(query, mutation, {
  trustedCaller: false,
})
```

### Replace the transport entirely

If you need:

- different hidden args
- action support
- non-standard server credentials

do not stretch `createFunctions()` to fit. Build your own wrapper with
`convex-helpers/server/customFunctions` and the low-level trusted-caller helpers.

That is the intended escape hatch.

## Tenant Isolation And RLS Overrides

### Canonical tenant safety

The simplest safety net is explicit tenant isolation:

```ts [convex/functions.ts]
export const { query: appQuery, mutation: appMutation } = createFunctions(query, mutation, {
  tenantIsolation: {
    tables: ['projects', 'tasks', 'comments'],
  },
})
```

That generates simple `workspaceId === actor.tenantId` checks.

The table list is explicit on purpose. Security policy should not silently depend on naming.

### Add custom RLS on top

If a few tables need more than simple tenant checks:

```ts [convex/functions.ts]
export const { query: appQuery, mutation: appMutation } = createFunctions(query, mutation, {
  tenantIsolation: {
    tables: ['projects', 'tasks', 'comments'],
  },
  rls: {
    rules: async (ctx) => ({
      users: {
        read: async (_, user) => user.authId === (await ctx.actor())?.userId,
      },
    }),
  },
})
```

That keeps the common tenant rule generated while letting you override the special cases.

### Full custom RLS

If the generated tenant rule is wrong for your app, omit `tenantIsolation` and provide only `rls`.

That is the clean replacement path.

## Triggers

Triggers are intentionally optional:

```ts [convex/functions.ts]
import { Triggers } from 'convex-helpers/server/triggers'

const triggers = new Triggers()

export const { query: appQuery, mutation: appMutation } = createFunctions(query, mutation, {
  triggers,
})
```

Use them for:

- denormalized counters
- cascade deletes
- computed fields

Keep intent-rich audit descriptions in handlers. The handler knows why a change happened; a trigger
only sees the diff.

## `onSuccess`

Use `onSuccess` for setup-time cross-cutting hooks that should run after success:

```ts [convex/functions.ts]
export const { query: appQuery, mutation: appMutation } = createFunctions(query, mutation, {
  onSuccess: {
    mutation: async ({ ctx, result }) => {
      // schedule async work, analytics, or cleanup
    },
  },
})
```

Do not use this to hide primary authorization logic. Auth should still be obvious in the handler.

## When To Stop Using `createFunctions()`

`createFunctions()` is the default composition, not the final abstraction for every app.

Drop to lower-level `convex-helpers` composition when you need:

- different hidden transport args
- action-specific actor resolution
- multiple layered custom-function transforms
- behavior that does not fit the default composition order

That is not “leaving the blessed path.” It is the intended advanced lane.

## Recommended Decision Order

When your app needs something beyond the starter:

1. Keep the default actor if your schema still matches it.
2. Use `defineActorExtension(...)` before replacing the actor resolver.
3. Use `tenantIsolation` for simple tenant safety.
4. Add `rls.rules` only where the generated tenant rule is not enough.
5. Use triggers only for true data reactions.
6. If `createFunctions()` starts feeling bent out of shape, stop and compose directly with `convex-helpers`.

That keeps the easy path easy without turning the advanced path into framework magic.
