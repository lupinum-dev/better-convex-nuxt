## Overview

`usePermissions()` reads the permission context query reactively. It does not evaluate app logic in the browser.

## Basic Usage

```vue
<script setup lang="ts">
const { can, role, tenantId, isAuthenticated } = usePermissions()
const canCreate = can('task.create')
</script>

<template>
  <button v-if="canCreate">New Task</button>
  <p v-if="isAuthenticated">Role: {{ role }} · Tenant: {{ tenantId }}</p>
</template>
```

## Resource-Specific Checks

For resource-level permissions, use backend-attached `_can` values:

```vue
<template>
  <button v-if="task._can.update">Edit</button>
  <button v-if="task._can.delete">Delete</button>
</template>
```

That keeps the backend as the single source of truth and avoids duplicating ownership rules in the client.

## Return Value

::field-group
::field{name="ctx" type="ComputedRef<AuthContext | null>"}
The raw permission context returned by your query.
::
::field{name="role" type="ComputedRef<string | null>"}
The current role from the backend context.
::
::field{name="plan" type="ComputedRef<string | null>"}
The current plan from the backend context, if provided.
::
::field{name="userId" type="ComputedRef<string | null>"}
The current user id from the backend context.
::
::field{name="tenantId" type="ComputedRef<string | null>"}
The current tenant id from the backend context.
::
::field{name="isAuthenticated" type="ComputedRef<boolean>"}
Whether the context query returned a non-null actor context.
::
::field{name="can(key)" type="(key: string) => ComputedRef<boolean>"}
Reads `ctx.can[key]` and fails closed when the key is absent.
::
::field{name="pending" type="Ref<boolean>"}
Whether the context query is still loading.
::
::
