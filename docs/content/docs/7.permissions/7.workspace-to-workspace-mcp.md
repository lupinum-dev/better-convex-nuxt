---
title: 'Adding MCP to a Workspace App'
description: 'Let AI agents use your workspace-scoped app through MCP, using the same business rules as browser users.'
---

You have a workspace-scoped app and want AI agents to use it through MCP. Here is what changes — and what stays the same.

**What stays the same:**
- Your Convex handlers, guards, and actor model — unchanged.
- The `users` table, workspace schema, and permission checks — unchanged.

**What you add:**
- A principal type that includes an `mcp` kind.
- An MCP runtime that resolves the agent's identity and capabilities.
- Projected tools that point at your existing Convex functions.

## What Changes

When you move from a browser-only workspace app to a workspace app with MCP, the important changes are:

- `createApp(...)` should own `principal`, `actor`, and internal builders
- protected handlers should be authored through `app.*`
- server/MCP callers should target root internal refs
- MCP visibility should be actor-level and cheap

What does **not** change:

- your workspace data model
- your tenant isolation rules
- your `_can` attachment pattern
- your resource-level `authorize` checks

## Root Runtime Setup

```ts
export const { app, raw } = createApp(
  {
    query,
    mutation,
    internalQuery,
    internalMutation,
  },
  {
    principal,
    actor: getActorFromPrincipal,
    tenantIsolation: { tables: ['posts', 'comments'] },
  },
)
```

This is the important shift: internal refs now go through the same runtime seam as public refs.

## Keep Authorization In Convex

Your protected handlers should still read like normal business logic:

```ts
export const publishRunbook = app.mutation({
  args: publishRunbookArgs.args,
  guard: canPublishRunbooks,
  load: async (ctx, args) => {
    const runbook = await loadTenantResource(ctx.db, 'runbooks', args.id)
    return { runbook }
  },
  authorize: {
    check: (actor, { runbook }) => actor.workspaceId === runbook.workspaceId,
  },
  handler: async (ctx, _args, { runbook }) => {
    await ctx.db.patch(runbook._id, { status: 'published' })
  },
})
```

Nothing about this handler should care whether the caller came from a browser wrapper or an MCP tool.

## MCP Runtime Shape

The MCP side should resolve principal and visibility, then call the root internal ref.

```ts
export const mcpRuntime = defineMcpRuntime({
  callConvex: async (event) => ({
    query: async (fn, args) => await serverConvexQuery(event, fn, args, { auth: 'none' }),
    mutation: async (fn, args) => await serverConvexMutation(event, fn, args, { auth: 'none' }),
  }),
  resolvePrincipal: async (event) => getMcpPrincipal(event),
  resolveCapabilities: async ({ principal }) => getCapabilitiesForPrincipal(principal),
})
```

```ts
export default mcpRuntime.projectTool({
  call: internal.runbooks.publishRunbook,
  preview: internal.runbooks.previewPublishRunbook,
  schema: publishRunbookArgs,
  capability: 'publishRunbook',
  meta: {
    name: 'publish-runbook',
    destructive: true,
  },
})
```

## Visibility vs Authorization

Keep these concerns separate:

- capability visibility decides whether the tool is advertised
- Convex authorization decides whether execution is actually allowed

That means visibility can stay actor-level and cheap, while your actual handler still owns the real permission check.
