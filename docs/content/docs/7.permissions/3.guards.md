Use this page when you need to express backend checks clearly and reuse them across handlers.

Guards are Trellis' coarse authorization boundary. They answer questions like:

- is there a signed-in caller
- is this actor a workspace member
- does this actor have at least admin role

They are not a replacement for all loaded-resource checks. Use them for the outer boundary first.

## Define Static Guards

The basic guard primitive is `defineGuard(label, check)`.

From the team-workspace example:

```ts
export const hasWorkspace = defineGuard<Actor>('Workspace member', (actor) => !!actor?.tenantId)

export const hasRole = (...roles: Doc<'users'>['role'][]) =>
  defineGuard<Actor>(`role:${roles.join('|')}`, (actor) => !!actor && roles.includes(actor.role))
```

That gives you named, reusable checks instead of anonymous booleans spread across handlers.

## Compose Guards

Guards compose with `.and(...)`, `.or(...)`, and `.not()`.

```ts
export const canCreateTodo = defineGuard<Actor>(
  'Create todo',
  hasWorkspace.and(hasRole('owner', 'admin', 'member')),
)
```

This matters for two reasons:

- the code reads like policy instead of control flow
- the guard keeps a label and structure Trellis can inspect consistently

## Use Resource-Bound Guard Factories

When a check depends on loaded data, make it a factory.

```ts
export const isOwnerOf = (resource: { ownerId: string }) =>
  defineGuard<Actor>(
    `owner:${resource.ownerId}`,
    (actor) => !!actor && actor.userId === resource.ownerId,
  )
```

Then compose it into a resource-specific rule:

```ts
export const canUpdateTodo = (todo: { ownerId: string }) =>
  defineGuard(
    'Update todo',
    hasWorkspace.and(hasRole('owner', 'admin').or(hasRole('member').and(isOwnerOf(todo)))),
  )
```

This keeps the static rule vocabulary separate from the loaded resource it needs.

## Use `authenticated` For Pre-Actor Auth Boundaries

`authenticated` is a special Trellis sentinel guard:

```ts
export const createWorkspace = mutation({
  guard: authenticated,
  args: {
    name: v.string(),
    slug: v.string(),
  },
  handler: async (ctx, args) => {
    const principal = await ctx.principal()
    // ...
  },
})
```

That guard is different from a normal predicate:

- it is enforced by the structured handler runtime
- it cannot be composed with `.and(...)`, `.or(...)`, or `.not()`
- it is for "must be signed in before actor resolution" flows

Use it when the handler only needs an authenticated principal first.

## Keep Guards Coarse

A strong Trellis pattern is:

- guard for the outer boundary
- load the resource
- authorize against the loaded resource when needed

Do not force every fine-grained ownership check into one huge top-level guard. That produces brittle policy and unreadable handlers.

## Guards And Frontend Permissions Are Related, Not Identical

Backend handlers use guards directly.

Frontend pages usually consume the same rule family through `definePermissionContext(...)` and `usePermissions()`.

That means the browser sees a projection of backend policy, not the policy engine itself.

## Common Mistakes

- Writing anonymous inline booleans everywhere instead of named guards
- Treating `authenticated` like a normal composable guard
- Encoding transport checks as business authorization
- Stuffing tenant bootstrap, role checks, and resource ownership into one unreadable predicate
