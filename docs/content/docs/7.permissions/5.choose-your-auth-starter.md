Answer two questions to pick your starter:

1. **Does your app have workspaces or teams?** If no, use the personal model. If yes, use the workspace model.
2. **Will server routes or AI agents call your backend?** If yes, use the workspace-mcp model (adds principal forwarding).

## Starter Matrix

| App shape                                              | Command                                                               | What it scaffolds                                                                                                                                                            | Read next                                                                                                            |
| ------------------------------------------------------ | --------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------- |
| Authenticated app without tenants                      | `npx trellis init auth` then `init permissions --model personal`      | `defineAuth(...)`, `convex/functions.ts`, `defineTrellis(...)`, personal actor/check files, permission context, `convex/test.setup.ts`                                           | [example `02-auth-todo`](https://github.com/lupinum-dev/trellis/blob/main/examples/02-auth-todo/README.md)           |
| Tenant-scoped app                                      | `npx trellis init auth` then `init permissions --model workspace`     | Explicit `convex/auth/principal.ts`, `convex/auth/actor.ts`, `convex/functions.ts`, `defineTrellis(...)`, `defineGuard(...)`, `definePermissionContext(...)`, and tenant helpers | [example `03-team-workspace`](https://github.com/lupinum-dev/trellis/blob/main/examples/03-team-workspace/README.md) |
| Tenant-scoped app with MCP or server-to-server callers | `npx trellis init auth` then `init permissions --model workspace-mcp` | Workspace starter plus explicit principal forwarding, root internal builders, protected `app.*` handlers, and the root seam needed for MCP projection                        | [example `07-mcp-reference`](https://github.com/lupinum-dev/trellis/blob/main/examples/07-mcp-reference/README.md)   |

Run `npx trellis init mcp` when you also need MCP middleware and a starter `server/mcp/runtime.ts`.

## What The Starter Owns

Every starter copies plain app code into your repo. You keep and modify:

- `convex/auth.ts`
  - `convex/auth.config.ts`
  - `convex/http.ts`
  - `convex/convex.config.ts`
- `convex/functions.ts`
- `convex/auth/*`
- `convex/test.setup.ts`
- `composables/usePermissions.ts`

Nothing here is a hidden runtime builder. The scaffold is only a starting point.

The important default is that these are plain authored files. You can open `convex/auth/principal.ts`,
`convex/auth/actor.ts`, `convex/functions.ts`, and `server/mcp/runtime.ts`, edit them directly, and
eject from the starter shape without any framework migration step.

For the workspace boundary files themselves, use the examples as the canonical references:

- single-workspace template: [example `03-team-workspace`](https://github.com/lupinum-dev/trellis/blob/main/examples/03-team-workspace/README.md)
- memberships template: [example `06-multi-workspace`](https://github.com/lupinum-dev/trellis/blob/main/examples/06-multi-workspace/README.md)

The canonical starter contract is:

- `workspaceId` on scoped tables
- `by_workspace` indexes
- `users.authId`, `users.role`, and `users.workspaceId`
- `ownerId` storing the auth-subject string

If your app follows that shape, `createTestContext({ schema, modules })` works without extra config.
Reach for the advanced testing overrides only when your schema intentionally diverges.
