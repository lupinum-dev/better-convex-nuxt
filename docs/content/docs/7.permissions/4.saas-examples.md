## Overview

The authorization recipes are easier to evaluate when they are attached to a runnable app instead
of a detached snippet.

The examples folder now covers the main SaaS authorization shapes end to end.

## How To Use Them

Read them in either order:

- progression-first: `01` -> `04`
- auth-shape-first: jump straight to the SaaS type that matches your app

If you are still choosing a starter, begin with
[`/docs/permissions/choose-your-auth-starter`](/docs/permissions/choose-your-auth-starter) and then
layer one of these blocks on top.

If you are comparing the examples because of shared validators or `shared/` folders, pair this page with:

- [`/docs/guide/shared-schema-dx`](/docs/guide/shared-schema-dx)
- [`/docs/mcp-tools/shared-schema`](/docs/mcp-tools/shared-schema)

Each example keeps the same ground rules:

- auth rules live in app-owned `convex/auth/*`
- handlers call raw `query()` / `mutation()`
- the frontend only reflects backend decisions
- tests prove the hard authorization case for that SaaS type

Keep the progression intentional:

- `03` to `09` use the canonical single-workspace model (`users.workspaceId` + `users.role`)
- `10` shows the memberships-based upgrade when users can belong to multiple workspaces
- `11` adds explicit trusted-caller and bearer-key lanes on top

See [`/docs/permissions/actor-lanes-and-models`](/docs/permissions/actor-lanes-and-models) for the
lane model, migration path, and denial semantics.
