Use this page when auth already works and the app now needs backend-owned permission state in the UI.

Trellis does not want frontend policy to become a second source of truth. The normal shape is:

1. resolve an actor in Convex
2. declare guards in app-owned backend code
3. expose one permission-context query
4. point Nuxt at that query
5. use `usePermissions()` and `useAuthGuard()` in the UI

## Start With The Generated Baseline

If you are starting fresh, use the scaffold:

```bash
npx trellis init permissions --model workspace
```

That gives you the real Trellis shape:

- `convex/auth/principal.ts`
- `convex/auth/actor.ts`
- `convex/auth/checks.ts`
- `convex/functions.ts`
- a `getPermissionContext` query

You can hand-write the same setup, but the scaffold is the fastest way to land on the shipped conventions.

## Define A Permission Context Query

The backend query is the source of truth for the frontend auth context.

From the workspace example:

```ts
export const getPermissionContext = query(
  definePermissionContext({
    resolve: getActor,
    guards: {
      [teamWorkspacePermissionKeys.todoRead]: canReadTodo,
      [teamWorkspacePermissionKeys.todoCreate]: canCreateTodo,
    },
    extend: async (ctx, actor) => {
      const user = await ctx.db
        .query('users')
        .withIndex('by_auth_id', (q) => q.eq('authId', actor.userId))
        .first()

      return {
        email: user?.email ?? null,
        displayName: user?.displayName ?? null,
      }
    },
  }),
)
```

What Trellis adds here:

- `definePermissionContext(...)` returns a public query definition directly
- `can` values are derived from backend guards, not reimplemented in the browser
- the base context includes `userId`, `tenantId`, `role`, and `can`
- `extend(...)` lets you add app-owned fields like plan, usage, or display name

## Point Nuxt At That Query

Configure the query once in `nuxt.config.ts`:

```ts
export default defineNuxtConfig({
  trellis: {
    auth: {
      enabled: true,
    },
    permissions: {
      query: 'workspaces.getPermissionContext',
    },
  },
})
```

That config powers the built-in composables. If the path is wrong, Trellis fails loudly during setup and `usePermissions()` will not become healthy at runtime.

## Read Permissions In The UI

Once the query is configured, the UI path is small:

```ts
const { can, ready, role, tenantId, ctx } = usePermissions()
```

Then consume permission keys directly:

```ts
const canCreateTodo = can(teamWorkspacePermissionKeys.todoCreate)
```

The important behavior is fail-closed:

- before the permission context arrives, `ready` is `false`
- missing permission keys return `false`
- stale context for a previous user is suppressed until it matches the current auth user

That is why the UI should branch on `ready` instead of guessing.

## Protect A Page With Permission State

Use `useAuthGuard()` when a whole page depends on one capability:

```ts
useAuthGuard({
  can: saasPermissionKeys.workspaceAudit,
  redirectTo: '/',
})
```

That redirects only after the permission query resolves. It does not race ahead of the real backend-owned context.

## Keep Route Protection And Permission Guards Separate

Use both layers for different jobs:

- `definePageMeta({ convexAuth: true })` means the page requires a signed-in session
- `useAuthGuard({ can: ... })` means the signed-in user still needs a specific capability

Do not collapse those into one concept. Authentication and authorization fail for different reasons.

## Common Setup Mistakes

- Returning hard-coded `can` values instead of deriving them from backend guards
- Pointing `trellis.permissions.query` at the wrong export path
- Using a permission actor that requires a tenant even on pages where onboarding users should still resolve
- Treating `usePermissions()` as a convenience cache instead of the frontend projection of backend policy
