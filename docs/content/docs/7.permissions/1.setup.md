## Overview

The v4 authorization model is intentionally small:

- backend primitives come from `better-convex-nuxt/auth`
- protected handler builders come from `better-convex-nuxt/functions`
- your real rules live in `convex/auth/*`
- the frontend reads a backend-owned context query through built-in `usePermissions()`

There is no permission config object, builder family, or client-side ownership evaluation.

## Setup

::steps{level="4"}

#### Choose A Starter

```bash
npx better-convex-nuxt init auth
```

Or choose one explicitly:

```bash
npx better-convex-nuxt init permissions --model personal
npx better-convex-nuxt init permissions --model workspace
npx better-convex-nuxt init permissions --model workspace-mcp
```

- `convex/auth.ts`

#### Canonical Single-Workspace Contract

The repo now treats one schema shape as the canonical default for tenant-scoped apps:

```ts [convex/schema.ts]
import { literals } from 'convex-helpers/validators'

const roleValidator = literals('owner', 'admin', 'member', 'viewer')

export default defineSchema({
  workspaces: defineTable({
    name: v.string(),
    slug: v.string(),
    ownerId: v.string(),
    createdAt: v.number(),
    updatedAt: v.number(),
  }).index('by_slug', ['slug']),

  users: defineTable({
    authId: v.string(),
    email: v.optional(v.string()),
    displayName: v.optional(v.string()),
    role: roleValidator,
    workspaceId: v.optional(v.id('workspaces')),
    createdAt: v.number(),
    updatedAt: v.number(),
  })
    .index('by_auth_id', ['authId'])
    .index('by_email', ['email'])
    .index('by_workspace', ['workspaceId']),
})
```

Use `createDefaultGetActor()` as the default actor lane for that contract. Reach for
`defineActorExtension(...)` only when your actor needs extra fields such as a workspace plan or
manager relationship.

The rules behind that default are fixed across starters, examples, and test helpers:

- `workspaceId` is the canonical tenant foreign key
- `by_workspace` is the canonical tenant index name
- `ownerId` stores the auth subject string from `users.authId`
- roles are string-literal unions
- timestamps are `Date.now()` numbers

You can still diverge in your own app, but the repo-owned examples, docs, and zero-config testing
path assume this shape.

#### Build Your Permission Context Query

```ts [convex/workspaces.ts]
import { can } from 'better-convex-nuxt/auth'
import { permissionKeys, type PermissionMap } from '../shared/permissions'
import { canCreateTask, canManageMembers } from './auth/checks'
import { appQuery } from './functions'

export const getPermissionContext = appQuery({
  args: {},
  handler: async (ctx) => {
    const actor = await ctx.actor()
    if (!actor) return null

    const permissions = {
      [permissionKeys.taskCreate]: can(actor, canCreateTask),
      [permissionKeys.workspaceMembers]: can(actor, canManageMembers),
    } satisfies PermissionMap

    return {
      role: actor.role,
      userId: actor.userId,
      tenantId: actor.tenantId,
      can: permissions,
    }
  },
})
```

#### Bind The Frontend Composable

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  convex: {
    permissions: {
      query: 'workspaces.getPermissionContext',
    },
  },
})
```

#### Keep Resource Permissions On The Backend

For resource-specific actions, attach `_can` in your query result:

```ts [convex/tasks.ts]
import { can, withCan } from 'better-convex-nuxt/auth'
import { canUpdateTask } from './auth/checks'

return tasks.map((task) =>
  withCan(task, {
    update: can(actor, canUpdateTask(task)),
  }),
)
```

## What Lives Where

- `better-convex-nuxt/auth`: small reusable primitives
- `convex/auth/*`: your actual business rules
- `convex/auth.ts`, `convex/http.ts`, `convex/auth.config.ts`, `convex/convex.config.ts`: the Better Auth bridge you still own
- permission context query: global capabilities for the frontend
- `_can` on returned resources: per-document capabilities for the frontend

::callout{type="info"}
`useConvexAuth()` exposes `isAuthenticated` (raw Convex auth token present). `usePermissions()` exposes `ready` (permission context query resolved a non-null actor). Use `ready` when your UI depends on permission data — it means the actor is known and capabilities are available.
::

## Authorize vs Inline `deny()`

Use `enforce()` for capability checks that stay pure and reusable:

- role checks
- plan checks
- ownership checks that already have the loaded resource in hand

Use inline `deny()` for business-state checks that are specific to the handler:

- archived projects
- expired share links
- refund windows
- prerequisite chains

That split keeps the reusable permission layer small while leaving business-state rules obvious in the handler body.

## `requireAuth()`

`enforce()` both checks the permission and narrows the TypeScript type — after it passes, `actor` is `NonNullable`. Use `requireAuth(actor)` only when you need authentication without a specific permission check:

```ts
import { enforce } from 'better-convex-nuxt/auth'
import { appMutation } from './functions'

export const create = appMutation({
  args: { name: v.string() },
  handler: async (ctx, args) => {
    const actor = await ctx.actor()
    enforce(actor, 'Create project', canCreateProject)
    // actor is now NonNullable — no requireAuth needed

    await ctx.db.insert('projects', {
      name: args.name,
      ownerId: actor.userId, // ← no bang needed
      workspaceId: actor.tenantId,
    })
  },
})
```

If the actor is `null`, `requireAuth` throws a Convex-friendly forbidden error (`"Not authenticated."` by default). You can pass a custom message as the second argument.

`ctx.actor()` is memoized only within a single handler invocation. Calling it twice in one handler reuses the same resolved actor. A new query or mutation gets a fresh resolver run.

## How Denials Reach The Frontend

`enforce()` and `deny()` both throw Convex errors. On the Nuxt side, those surface through the normal query and mutation error refs:

- `useConvexQuery(...).error.value`
- `useConvexMutation(...).error.value`

The browser does not enforce authorization. It only reflects backend decisions and renders the returned error message or capability state.

## Why Cross-Tenant Checks Say "Resource not found."

When a user reaches across tenant boundaries, prefer returning the same denial you would use for a missing resource.

That avoids leaking whether another tenant's record exists at all. It is more secure than a louder "forbidden" response, even if it can feel slightly less ergonomic during debugging.

Within the same tenant, prefer the opposite: capability failures should stay explicit and return a
forbidden-style denial. The new
[`/docs/permissions/actor-lanes-and-models`](/docs/permissions/actor-lanes-and-models) page documents
that split as the repo-wide rule.

## Visibility Filters With Pagination

If a visibility filter returns a query, keep the query object and paginate it directly:

```ts
const visibleProjects = defineVisibility(async (actor, db) => {
  if (!actor) throw deny('Not authenticated.')

  return db.query('projects').withIndex('by_workspace', (q) => q.eq('workspaceId', actor.tenantId))
})

export const listProjects = query({
  args: { paginationOpts: paginationOptsValidator },
  handler: async (ctx, args) => {
    const actor = await ctx.actor()
    enforce(actor, 'Read projects', canReadProjects)

    const query = await visibleProjects.resolve(actor, ctx.db)
    return query.order('desc').paginate(args.paginationOpts)
  },
})
```

Use `applyVisibility()` when you want the fully collected rows. Resolve the query directly when you need pagination, ordering, or extra query modifiers downstream.

## LLM Contract

If you use an LLM to scaffold handlers, keep it on this contract:

- build protected handlers with `appQuery` / `appMutation` from `convex/functions.ts`
- call `const actor = await ctx.actor()` first
- use `enforce()` for role, capability, and ownership checks
- use inline `deny()` for business-state checks
- return `null` from the permission-context query when unauthenticated
- reject protected queries and mutations explicitly when unauthenticated
- keep visibility filters server-side
- attach per-resource `_can` data in queries, not in the client
- for bulk operations, soft-check each row with `can(actor, ...)`
