Trellis gives you three database trust levels. They're not interchangeable тАФ each one gives up a different guarantee. Use the narrowest one that does the job.

| Level | Bypasses | Still enforces | Emits |
| --- | --- | --- | --- |
| `ctx.db` | Nothing | Tenant isolation, services, triggers | тАФ |
| `ctx.db.crossTenant` | Tenant isolation | Services, triggers | `db.cross_tenant.used` |
| `ctx.db.raw` | Tenant isolation, services, triggers | тАФ | `db.raw.used` |

Preference order, always: `ctx.db` тЖТ `ctx.db.crossTenant` тЖТ `ctx.db.raw`. Don't skip rungs.

## Default: `ctx.db`

For tenant-isolated tables, `ctx.db` enforces the active tenant boundary on both reads and writes. Use it unless you have a concrete reason not to. Most handlers тАФ including most admin-looking handlers тАФ end up here once the actor model is modeled correctly.

## Explicit escape hatch: `ctx.db.crossTenant`

`ctx.db.crossTenant` bypasses tenant isolation only:

```ts [convex/agency/posts.ts]
export const listAllPosts = query({
  args: {},
  guard: authed,
  handler: async (ctx) => {
    return await ctx.db.crossTenant.query('posts').collect()
  },
})
```

What still applies:

- Service rules.
- Triggers and runtime hooks.
- Observability тАФ every call emits `db.cross_tenant.used`, which makes the boundary crossing auditable.

Right for: admin, operator, onboarding, and portfolio flows that genuinely need to cross tenants while keeping the rest of the Trellis runtime intact.

## Full escape hatch: `ctx.db.raw`

`ctx.db.raw` is sharper. It bypasses:

- Tenant isolation.
- Service enforcement.
- Triggers and runtime hooks.

::caution
`ctx.db.raw` is not a nicer `crossTenant`. If `crossTenant` would do the job, using `raw` throws away runtime guarantees you'd want back the moment something goes wrong. Every call emits `db.raw.used` so it stays visible in observability.
::

## How to choose

::steps{level="3"}

### Start with `ctx.db`

If the actor model is right and the table is tenant-isolated, this covers almost everything.

### Step up to `crossTenant` when the action truly crosses tenants

Onboarding, workspace switching, agency dashboards. Ask: *would I be okay if this ran against the active tenant only?* If yes, stay on `ctx.db`.

### Use `raw` only when a trigger or service rule is the thing you need to bypass

Almost never the first answer. If you reach for it, a comment explaining why is mandatory.

::

## Real-app cases

From the multi-workspace example:

- Current-workspace work тЖТ normal tenant-scoped handlers.
- Onboarding and workspace switching тЖТ explicit cross-tenant flows.
- Agency dashboards reading membership across workspaces тЖТ cross-tenant reads on purpose.

Each one keeps the exception visible. No "quiet admin handler" that silently accesses other tenants.

## Leave a comment when you reach for an escape hatch

```ts
// Intentional escape hatch: onboarding creates a new tenant and membership,
// so it cannot run through tenant-isolated Trellis mutations.
return await ctx.db.crossTenant.insert('workspaces', { ... })
```

This isn't style. Trust-boundary changes need to be obvious in review тАФ if the reviewer has to hunt for why the handler uses `crossTenant`, the comment belongs there.

## Common pitfalls

- Treating `ctx.db.crossTenant` as normal app data access. Normal access is `ctx.db`.
- Reaching for `ctx.db.raw` when `crossTenant` would preserve the guarantees you want.
- Hiding an escape hatch inside a helper where the boundary isn't visible to reviewers.
- Using raw access to avoid modeling the right actor or tenant flow. That's a future-you problem.
