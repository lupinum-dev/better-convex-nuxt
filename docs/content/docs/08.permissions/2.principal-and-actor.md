This page matters once the app has more than one meaningful caller shape: browser plus server, browser plus MCP, or browser plus workspace policy.

If you are still building the first signed-in todo app, you can skip this for now and come back later.

A [principal](/docs/concepts/glossary#principal) says how the call arrived. An [actor](/docs/concepts/glossary#actor) says who that is in your app. Collapsing them is how policy bugs happen, so Trellis keeps them strictly separate.

| Layer | Shape | Decides |
| --- | --- | --- |
| Principal | Transport kind (anonymous, user, agent, service) | How to resolve the caller |
| Actor | App-owned identity (role, tenant, membership) | What the caller can do |

Same person opening your app in a browser and asking your agent to do the same thing resolves to the **same actor**. The transport changed; the business identity didn't. That's the point.

## Principal: transport identity

From the workspace example:

```ts [convex/auth/principal.ts]
export type TeamTodoPrincipal =
  | { kind: 'anonymous' }
  | { kind: 'user', userId: string }
  | { kind: 'agent', userId: string, agentId?: string, provider?: 'mcp' }
```

A principal tells you:

- Who the transport says they are (browser session, forwarded agent call, nothing).
- Sometimes a thin metadata hint (`provider: 'mcp'`).

What it explicitly doesn't decide: what the caller is allowed to do. That's the actor's job.

## Actor: app-owned business identity

```ts [convex/auth/actor.ts]
export type Actor = DefaultActor & {
  role: Role
  tenantId?: Id<'workspaces'>
}
```

This is what guards and authorization actually read. Role, tenant membership, any app-specific scope.

## The resolution flow

Four steps in this order, every request:

::steps{level="3"}

### Resolve the principal

The runtime fills this in from the request.

### Resolve the actor

Your `resolve(...)` turns that principal into an app identity (or returns `null` if there isn't one).

### Run guards and authorization

Against the actor, not the principal.

### Execute the handler

If the checks passed.

::

From the team-workspace example:

```ts [convex/auth/actor.ts]
export async function getActorFromPrincipal(ctx, _args, principal): Promise<Actor | null> {
  switch (principal.kind) {
    case 'anonymous':
      return null
    case 'agent':
    case 'user':
      return await loadActorByAuthId(ctx, principal.userId)
  }
}
```

Notice what this doesn't do:

- It doesn't use `principal.kind` as an authorization signal.
- It doesn't invent a separate role system for agent callers.
- It doesn't let MCP traffic drift onto a parallel policy track.

## Why not put role on the principal

Because then business authorization would be coupled to how the call arrived. Moving an action to a new transport (CLI, webhook, agent) would mean duplicating your role model.

Trellis keeps transport metadata on the principal (like `provider: 'mcp'`) and business authority on the actor. If you need a forwarded principal with real trust, model it deliberately — don't smuggle role claims through transport shape.

::note
"Browser, server, webhook, and agent callers can share one backend model." They still don't replace Convex business authorization. That's the shape Trellis wants.
::

## Permission actor vs. strict actor

Most apps need two slightly different actor views:

| View | Used for | Accepts actor without `tenantId`? |
| --- | --- | --- |
| `getPermissionActor` | Frontend permission context, onboarding flows | Yes |
| `getActor` | Tenant-scoped mutations that require a workspace | No |

The workspace scaffold splits them explicitly. It keeps onboarding and workspace-required mutations from fighting each other: you can't open the app without a tenant, but the second you act inside one, the actor must have `tenantId`.

## Common pitfalls

- Using `principal.kind` as the authorization decision. It's not — it's metadata about the transport.
- Encoding app roles into every transport caller. The actor layer owns role.
- Reusing one strict actor resolver everywhere, then wondering why onboarding users disappear.
- Letting browser and MCP calls drift onto separate authorization systems.
