Use this recipe when your server route must run as the backend itself, not as the current user.

## When To Use This

Good fits:

- cron jobs
- webhook handlers
- admin maintenance endpoints
- integrations that need a backend-only Convex call

Do **not** use this for normal page loads, SSR data fetching, or standard user-triggered API
routes. For those, keep using `serverConvex*`.

## Required Pieces

1. A server-only bridge secret in the app runtime:

```bash [.env.local]
CONVEX_PRIVATE_BRIDGE_KEY=replace-with-a-long-random-secret
```

2. The same secret configured in the Convex backend environment.

3. An explicit app-local function that rejects every call without the correct key:

```ts [convex/private/system-overview.ts]
import { v } from 'convex/values'
import { query } from '../_generated/server'

function assertPrivateBridgeKey(apiKey: string) {
  const expectedApiKey = process.env.CONVEX_PRIVATE_BRIDGE_KEY
  if (!expectedApiKey) {
    throw new Error('Missing CONVEX_PRIVATE_BRIDGE_KEY')
  }
  if (apiKey !== expectedApiKey) {
    throw new Error('Invalid API key')
  }
}

export const systemOverview = query({
  args: { apiKey: v.string() },
  handler: async (ctx, args) => {
    assertPrivateBridgeKey(args.apiKey)
    const { apiKey: _apiKey } = args

    const [notes, tasks, users] = await Promise.all([
      ctx.db.query('notes').collect(),
      ctx.db.query('tasks').collect(),
      ctx.db.query('users').collect(),
    ])

    return {
      notes: notes.length,
      tasks: tasks.length,
      users: users.length,
    }
  },
})
```

Repeat the same explicit pattern for mutations or actions only if you truly need them. Keep the
bridge narrow and app-local instead of turning it into a shared superuser abstraction.

4. A server-only helper that injects the key before calling Convex:

```ts [server/utils/private-convex.ts]
export async function privateConvexQuery(func, args) {
  const convexUrl = useRuntimeConfig().public.convex.url
  const apiKey = process.env.CONVEX_PRIVATE_BRIDGE_KEY

  if (!apiKey) {
    throw new Error('Missing CONVEX_PRIVATE_BRIDGE_KEY')
  }

  const response = await fetch(`${convexUrl}/api/query`, {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({
      path: getFunctionName(func),
      args: { ...(args ?? {}), apiKey },
    }),
  })

  return parseConvexResponse(await response.json())
}
```

## Security Rules

- One bridge key per app.
- Never put the key in `runtimeConfig.public`.
- Keep private functions narrow and system-oriented.
- Keep authorization and data invariants inside Convex functions, even for privileged calls.
- Do not reuse the private bridge for normal user page requests.
- Do not promote this into a shared monorepo or module-level helper.

## Failure Modes

The main failures to expect are:

- missing Better Auth session cookie when you accidentally used `serverConvex*` for a system task
- missing `convex.siteUrl` for user-scoped token exchange
- token exchange failure for user-scoped calls
- missing or wrong `CONVEX_PRIVATE_BRIDGE_KEY`
- routes that incorrectly mix user-scoped and privileged responsibilities

## Manual Key Rotation

1. Generate a new random key for the app.
2. Update the app server runtime secret.
3. Update the Convex backend secret.
4. Redeploy both sides.
5. Invalidate any old operational notes or scripts that referenced the previous key.

This is intentionally manual because the bridge key should be rare, app-local infrastructure, not a
shared module concern.
