Use this page when something is denied, drifted, or unexpectedly crossed a trust boundary.

The point of Trellis observability is not “more logs.” It is faster answers to questions like:

- why was this denied
- which layer denied it
- did this cross tenant boundaries
- did a destructive confirmation drift

## Debug In This Order

1. identify the correlation ID
2. find the first identity event
3. find the first deny or error event
4. check whether a trust-boundary event happened before it

That sequence is usually enough to localize the problem.

## Common Authorization Failures

If you see:

- `guard.denied`
- `authorize.denied`
- `rls.denied`

then the next question is which layer owned the decision:

- `guard.denied`: caller failed the outer boundary
- `authorize.denied`: caller reached the loaded resource but failed the resource-bound rule
- `rls.denied`: default tenant boundary blocked access

## Trust-Boundary Checks

If behavior looks broader than expected, look for:

- `db.cross_tenant.used`
- `db.raw.used`
- `service.access.checked`
- `service.access.denied`

Those events tell you whether the code intentionally crossed the default boundary or whether service access rules blocked it.

## Destructive Drift

If a destructive tool or operation fails after preview, look for:

- `operation.preview.started`
- `operation.preview.completed`
- `operation.confirm.validated`
- `operation.confirm.drifted`

`operation.confirm.drifted` means the confirmation payload no longer matches the previewed state closely enough to execute safely.

## MCP Failures

For tool problems, follow:

- `tool.called`
- `tool.denied` or `tool.executed`
- `tool.failed` if execution threw

If the issue is capability gating, the denial explanation is usually more useful than the raw tool transport error.
