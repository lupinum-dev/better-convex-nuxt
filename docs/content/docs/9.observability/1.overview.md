Use this page when you need to understand what Trellis observability is for.

Trellis does not primarily emit low-value transport logs. It emits semantic events about identity, authorization, trust boundaries, operations, tools, and browser/runtime behavior.

## What Trellis Traces

Examples of first-class events:

- principal resolution
- actor resolution or missing actor state
- guard allow or deny
- authorize allow or deny
- row-level isolation denials
- cross-tenant and raw DB usage
- destructive operation preview, confirmation, and execution
- MCP tool calls, denials, and failures

That is the product value. It helps you understand decisions, not only requests.

## Correlation Is Built In

Events share correlation context across boundaries:

- browser
- Nuxt server
- Convex
- MCP
- service and webhook flows

That is why a denial in one layer can still be understood as part of one request or automation flow instead of isolated log fragments.

## Delivery Model

Trellis owns the semantic model and delivers it through `evlog`.

This is why the README should mention observability, but not pitch it as generic logging. The value is the Trellis-owned event vocabulary.

## Configure It At The Module Level

```ts
trellis: {
  observability: {
    enabled: true,
  },
}
```

From there you can tune:

- capture surfaces
- level
- sampling by event family
- correlation header
- service name
- explainability options
