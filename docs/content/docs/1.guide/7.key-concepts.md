These are terms you will encounter throughout the docs. Each links to the page where it is explained in context.

## Actor

Your app's business role, derived from a principal. Carries role, tenant, and authorization state. Guards operate on actors.
**Learned in:** [Multi-Caller Architecture](/docs/guide/multi-caller-architecture)

## Capability

A boolean flag that controls whether an MCP tool is visible to a given caller. Capabilities are cheap, actor-level checks — distinct from per-record `_can` permissions.
**Learned in:** [Tool Visibility](/docs/mcp-tools/auth-and-permissions)

## Component Bridge

A root-app layer built with `createComponentBridge(...)` that forwards `{ ...args, principal }` into an isolated Convex component. Only forwards identity — does not check permissions.
**Learned in:** [Component Bridges](/docs/server-side/private-bridge)

## Guard

A named permission check that gates entry to a handler. Defined with `defineGuard(...)` and passed as the `guard` field on a handler or operation.
**Learned in:** [Permission Guards](/docs/permissions/permission-guards)

## MCP Projection

An MCP tool that points at an existing Convex function using `tool(...)` instead of owning the business logic. The tool owns metadata, visibility, and confirmation; Convex owns authorization and state changes.
**Learned in:** [Your First MCP Tool](/docs/mcp-tools/getting-started)

## Operation

A reusable protected business definition created with `defineOperation(...)`. Can include `guard`, `load`, `authorize`, `preview`, and `handler`. Registered as a query or mutation with `query(...)` / `mutation(...)`.
**Learned in:** [Multi-Caller Architecture](/docs/guide/multi-caller-architecture)

## Preview

A confirmation summary for a destructive action. Defined as a `preview` phase in a `defineOperation(...)` and exposed as a query via `previewOf(...)`.
**Learned in:** [Destructive Tools and Previews](/docs/mcp-tools/safety-and-confirmation)

## Principal

Transport-level caller identity, defined with `definePrincipal(...)`. Answers "who is calling according to this transport?" — not "what business role do they have?" Examples: browser user, anonymous caller, MCP agent.
**Learned in:** [Multi-Caller Architecture](/docs/guide/multi-caller-architecture)

## Protected Handler

A Convex query or mutation built with `query(...)` or `mutation(...)`. Protected because its auth structure is explicit in the signature: `guard`, optional `load`, optional `authorize`, then `handler`.
**Learned in:** [How It Works](/docs/guide/how-it-works)

## Root Wrapper

A root-app Convex function that exposes an entrypoint into component-backed business logic for browser callers. Usually explicit and readable.
**Learned in:** [Component Bridges](/docs/server-side/private-bridge)

## Shared Query

A query instance automatically deduplicated across multiple components when they use the same function and arguments. No manual cache key needed.
**Learned in:** [Queries](/docs/data-fetching/queries)

## Visibility

In MCP: whether a tool appears in the agent's tool list (controlled by capabilities). In UI: whether rows or actions are exposed to a caller (controlled by `_can` flags). Related but separate concerns.
**Learned in:** [Tool Visibility](/docs/mcp-tools/auth-and-permissions), [Checking Permissions](/docs/permissions/checking-permissions)
