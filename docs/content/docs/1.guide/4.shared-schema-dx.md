title: "Shared Schema DX"
description: "Define Convex validators once and reuse them across Convex, Nuxt validation, and MCP."
---

`better-convex-nuxt` now supports a deliberate shared-schema split:

- `better-convex-nuxt/composables` for client composables
- `better-convex-nuxt/schema` for server-safe shared-schema helpers
- `better-convex-nuxt/server` for Nitro server helpers
- `better-convex-nuxt/mcp` for MCP-only helpers

The design intent is one source of truth, multiple runtime consumers. It is not one universal import everywhere.

## Install

Base module install:

```bash
pnpm add convex better-convex-nuxt
```

Add these only when you use MCP tools:

```bash
pnpm add @nuxtjs/mcp-toolkit convex-helpers zod
```

Non-MCP users do not need those extra packages.

## Define validators once

Keep the raw validator map in a shared file that only depends on `convex/values`.

```ts [shared/schemas/post.ts]
import { v } from "convex/values";
import type { ConvexSchemaMetaFor } from "better-convex-nuxt/schema";

export const createPostArgs = {
  title: v.string(),
  content: v.string(),
};

export const createPostMeta = {
  description: "Create a post",
  fields: {
    title: { label: "Title", description: "Post title" },
    content: { label: "Content", description: "Post body" },
  },
} satisfies ConvexSchemaMetaFor<typeof createPostArgs>;
```

`ConvexSchemaMetaFor` keeps metadata exact. If you provide `fields`, it must match the validator keys exactly.

## Use the same validators in Convex

Convex functions still consume the raw validator map directly:

```ts [convex/posts.ts]
export const create = mutation({
  args: createPostArgs,
  handler: async (ctx, args) => {
    return await ctx.db.insert("posts", args);
  },
});
```

## Wrap once for Nuxt validation

Use `better-convex-nuxt/schema` anywhere you need Standard Schema or H3 validation.

```ts [server/api/posts.ts]
import { defineConvexSchema } from "better-convex-nuxt/schema";

const schema = defineConvexSchema(createPostArgs, createPostMeta);

export default defineEventHandler(async (event) => {
  const body = schema.validate(await readBody(event));
  return body;
});
```

The returned schema object is the shared runtime contract:

```ts
const schema = defineConvexSchema(createPostArgs, createPostMeta);

schema.args;      // raw Convex validators
schema.validate;  // H3/server validation
schema.meta;      // optional metadata
schema.standard;  // explicit StandardSchemaV1 view
schema;           // also directly usable anywhere StandardSchemaV1 is accepted
```

That lets the same object cover the three common call sites:

```ts
mutation({
  args: schema.args,
  handler: async (ctx, args) => {
    return await ctx.db.insert("posts", args);
  },
});

const body = schema.validate(await readBody(event));

<UForm :schema="schema" />
// or <UForm :schema="schema.standard" />
```

## Reuse the same schema in MCP

Use `defineConvexMcpTool()` from `better-convex-nuxt/mcp`. It derives the MCP input schema internally and keeps handler args typed from the shared Convex schema.

```ts [server/mcp/tools/create-post.ts]
import { defineConvexSchema } from "better-convex-nuxt/schema";
import { defineConvexMcpTool } from "better-convex-nuxt/mcp";
import { serverConvexMutation } from "better-convex-nuxt/server";

const schema = defineConvexSchema(createPostArgs, createPostMeta);

export default defineConvexMcpTool({
  name: "create-post",
  schema,
  handler: async (args) => {
    return await serverConvexMutation(api.posts.create, args);
  },
});
```

Metadata is optional. Without it, the MCP tool still works and handler args stay typed, but the generated tool input only carries field names and types:

```ts [server/mcp/tools/create-post.ts]
import { defineConvexSchema } from "better-convex-nuxt/schema";
import { defineConvexMcpTool } from "better-convex-nuxt/mcp";

const schema = defineConvexSchema(createPostArgs);

export default defineConvexMcpTool({
  name: "create-post",
  schema,
  handler: async (args) => {
    return args.title;
  },
});
```

That is valid, but agents get less guidance when there is no schema-level description and no per-field descriptions. Add metadata when the MCP tool should be self-explanatory to an agent.

That gives you:

- one validator definition for Convex args
- one metadata definition for labels and descriptions
- one schema wrapper for forms and H3 validation
- one MCP helper that derives its input schema from the same validators
