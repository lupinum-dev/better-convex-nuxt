`better-convex-nuxt` now supports a deliberate shared-schema split:

- `better-convex-nuxt/composables` for client composables
- `better-convex-nuxt/schema` for server-safe shared-schema helpers
- `better-convex-nuxt/server` for Nitro server helpers
- `better-convex-nuxt/mcp` for MCP-only helpers

The design intent is one source of truth, multiple runtime consumers. It is not one universal import everywhere.

## Install

Base module install:

```bash
pnpm add convex better-convex-nuxt
```

Add these only when you use MCP tools:

```bash
pnpm add @nuxtjs/mcp-toolkit convex-helpers zod
```

Non-MCP users do not need those extra packages.

## Define validators once

Keep the raw validator map in a shared file that only depends on `convex/values`.

```ts [shared/schemas/post.ts]
import { v } from "convex/values";
import type { ConvexSchemaMetaFor } from "better-convex-nuxt/schema";

export const createPostArgs = {
  title: v.string(),
  content: v.string(),
};

export const createPostMeta = {
  description: "Create a post",
  fields: {
    title: { label: "Title", description: "Post title" },
    content: { label: "Content", description: "Post body" },
  },
} satisfies ConvexSchemaMetaFor<typeof createPostArgs>;
```

`ConvexSchemaMetaFor` keeps metadata exact. If you provide `fields`, it must match the validator keys exactly.

## Use the same validators in Convex

Convex functions still consume the raw validator map directly:

```ts [convex/posts.ts]
export const create = mutation({
  args: createPostArgs,
  handler: async (ctx, args) => {
    return await ctx.db.insert("posts", args);
  },
});
```

## Wrap once for Nuxt validation

Use `better-convex-nuxt/schema` anywhere you need Standard Schema or H3 validation.

```ts [server/api/posts.ts]
import { defineConvexSchema } from "better-convex-nuxt/schema";

const schema = defineConvexSchema(createPostArgs, createPostMeta);

export default defineEventHandler(async (event) => {
  const body = schema.validate(await readBody(event));
  return body;
});
```

The returned schema object is the shared runtime contract:

```ts
const schema = defineConvexSchema(createPostArgs, createPostMeta);

schema.args;      // raw Convex validators
schema.validate;  // H3/server validation
schema.meta;      // optional metadata
schema.standard;  // explicit StandardSchemaV1 view
schema;           // also directly usable anywhere StandardSchemaV1 is accepted
```

That lets the same object cover the three common call sites:

```ts
mutation({
  args: schema.args,
  handler: async (ctx, args) => {
    return await ctx.db.insert("posts", args);
  },
});

const body = schema.validate(await readBody(event));

<UForm :schema="schema" />
// or <UForm :schema="schema.standard" />
```

## Reuse the same schema in MCP

::callout{type="tip"}
This section covers the basics. For comprehensive MCP documentation — auth, permissions, destructive flows, middleware, result envelopes, and agent design patterns — see the dedicated [MCP Tools](/docs/mcp-tools/getting-started) section.
::

Use `defineConvexTool()` from `better-convex-nuxt/mcp`. It derives the MCP input schema internally, keeps handler args typed, and wraps responses in a structured envelope agents can parse without text matching.

### Level 1 — Just make it work

```ts [server/mcp/tools/create-post.ts]
import { defineConvexSchema } from "better-convex-nuxt/schema";
import { defineConvexTool } from "better-convex-nuxt/mcp";
import { serverConvexMutation } from "better-convex-nuxt/server";

const schema = defineConvexSchema(createPostArgs, createPostMeta);

export default defineConvexTool({
  schema,
  handler: (args) => serverConvexMutation(api.posts.create, args),
});
```

The handler returns plain data. The framework wraps it in `{ ok: true, data: {...} }`. Errors are caught and categorized automatically.

### Level 2 — Add auth

```ts [server/mcp/tools/create-post.ts]
export default defineConvexTool({
  schema: createPostSchema,
  auth: "required",
  handler: (args) => serverConvexMutation(api.posts.create, args),
});
```

### Level 3 — Destructive with preview

```ts [server/mcp/tools/delete-post.ts]
export default defineConvexTool({
  schema: deletePostSchema,
  auth: "required",
  destructive: true,
  preview: async (args) => {
    const post = await serverConvexQuery(api.posts.get, { id: args.id });
    return post ? `Will permanently delete "${post.title}"` : "Post not found";
  },
  handler: (args) => serverConvexMutation(api.posts.remove, args),
});
```

Destructive tools require a two-call confirmation flow. If a `preview` function is provided, the first call returns the preview result; otherwise it returns a `confirmation_required` error. The second call (with `_confirmed: true`) executes the handler.

### Typed permissions with `createConvexTools`

For typed permission autocomplete, create a factory:

```ts [server/mcp/utils/tools.ts]
import { createConvexTools } from "better-convex-nuxt/mcp";
import { checkPermission } from "~~/convex/permissions.config";

export const { defineConvexTool } = createConvexTools({
  checkPermission,
});
```

Then use it in your tools:

```ts [server/mcp/tools/create-post.ts]
import { defineConvexTool } from "../utils/tools";

export default defineConvexTool({
  schema: createPostSchema,
  auth: "required",
  require: "post.create", // ← autocomplete + type errors on typos
  handler: (args) => serverConvexMutation(api.posts.create, args),
});
```

### Enhanced field metadata

Schema fields support additional metadata for agents:

```ts
export const createPostArgs = {
  title: v.string(),
  content: v.string(),
  format: v.optional(v.string()),
};

export const createPostMeta = {
  description: "Create a post",
  fields: {
    title: {
      description: "Post title",
      examples: ["My First Post", "Weekly Update"],
    },
    content: {
      description: "Post body in the chosen format",
    },
    format: {
      description: "Content format",
      enum: ["markdown", "html"],
      defaultHint: "markdown",
    },
  },
} satisfies ConvexSchemaMetaFor<typeof createPostArgs>;
```

This generates rich field descriptions and auto-generated input examples for agents.

That gives you:

- one validator definition for Convex args
- one metadata definition for labels, descriptions, and agent hints
- one schema wrapper for forms and H3 validation
- one MCP helper with structured responses, safety features, and typed permissions
