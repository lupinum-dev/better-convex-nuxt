---
title: 'Common Integration Mistakes'
description: 'Frequent ways downstream apps fight Trellis at the auth, SSR, bridge, MCP, and testing seams.'
---

Trellis works best when your app has **one protected path**:

1. resolve a principal
2. derive an actor
3. run guards and business logic in Convex
4. project browser, server, and MCP calls into that same path

Most integration mistakes happen when a project keeps the core model, but adds a second path around it.

This page lists the most common mistakes we have seen in real downstream integrations and how to avoid them.

## 1. Failing Open On First-Run Or Bootstrap Flows

The mistake:

- treating “missing configuration” as permission to continue
- allowing first-owner setup just because a secret or flag is absent

Why this is risky:

- a production misconfiguration becomes privilege escalation
- the app stops relying on the normal Trellis permission model and falls back to a hidden side rule

Avoid this:

- fail closed by default
- require an explicit development-only escape hatch
- never infer insecure behavior from a missing secret

Before:

```ts
export function validateBootstrapToken(token: string | undefined): void {
  const envKey = process.env.APP_BOOTSTRAP_KEY
  if (envKey && (!token || token !== envKey)) {
    throw new Error('Invalid bootstrap token')
  }
}
```

After:

```ts
export function validateBootstrapToken(token: string | undefined): void {
  const envKey = process.env.APP_BOOTSTRAP_KEY
  if (!envKey) {
    throw new Error('Bootstrap key is not configured')
  }
  if (token !== envKey) {
    throw new Error('Invalid bootstrap token')
  }
}
```

If you want insecure local setup, require an explicit flag such as `ALLOW_INSECURE_BOOTSTRAP=true`.

## 2. Mutating App-Owned Source Files During Runtime Setup

The mistake:

- a package edits or patches the host app's source files automatically during module setup

Why this is risky:

- the package silently takes ownership of code the app expects to own
- upgrades become harder to reason about
- debugging gets harder because generated behavior and app behavior are mixed
- integration drift becomes string-replacement work instead of an explicit contract

Avoid this:

- prefer explicit scaffolding or install commands
- generate bridge files once and ask the app to commit them
- validate required integration files at runtime instead of rewriting them

Before:

```ts
if (!hasBridgeImport) {
  source = insertImport(source, "import cms from '@my-package/convex/config'")
}

if (!hasBridgeUse) {
  source = source.replace('export default app', "app.use(cms)\n\nexport default app")
}
```

After:

```ts
if (!bridgeIsInstalled(rootDir)) {
  throw new Error('Bridge not installed. Run `pnpm my-package scaffold`.')
}
```

If your package needs generated bridge files, make that an explicit install step rather than a hidden runtime side effect.

See also:

- [Component Bridges](/docs/server-side/private-bridge)

## 3. Wrapping Queries In A Global `server: false` Policy

The mistake:

- creating a convenience wrapper that forces all queries to skip SSR

Why this is risky:

- it turns a per-call Trellis decision into an app-wide policy
- pages become client-only waterfalls
- loading and access logic gets duplicated outside the normal path

Trellis is SSR-first by default. Opt out only where you have a concrete reason.

Before:

```ts
export function useStudioQuery(query, args, options) {
  return useConvexQuery(query, args, {
    ...options,
    server: false,
  })
}
```

After:

```ts
export function useStudioQuery(query, args, options) {
  return useConvexQuery(query, args, options)
}
```

If one page must be client-only, make that decision at the callsite.

See also:

- [SSR Overview](/docs/server-side/ssr-overview)
- [Queries](/docs/data-fetching/queries)

## 4. Reading Redirect Targets Manually Instead Of Using Trellis Auth Redirect Helpers

The mistake:

- reading `route.query.redirect`
- passing it directly to `navigateTo(...)`

Why this is risky:

- it creates open-redirect style bugs
- it duplicates logic Trellis already provides
- the app stops using the safe sink Trellis expects

Before:

```ts
const target = typeof route.query.redirect === 'string' ? route.query.redirect : '/'
await navigateTo(target, { external: true })
```

After:

```ts
const { execute } = useConvexAuthActions()

await execute(
  () => client.signIn.email({ email, password }),
  { redirectTo: '/' },
)
```

Or use `useAuthRedirect()` directly when you need custom flow control.

See also:

- [Sign In & Sign Out](/docs/auth-security/sign-in-and-sign-out)
- [Token Refresh](/docs/auth-security/token-refresh)

## 5. Using Broader Read Permissions Than The Product Actually Intends

The mistake:

- a read endpoint uses a broad guard like “can read app”
- the corresponding write endpoints use a much stronger guard
- the read payload still includes management-level detail

Why this is risky:

- the backend exposes more than the product model suggests
- this is not just a UI bug if the backend read guard is broad
- viewers may be able to inspect data meant for managers or editors

Before:

```ts
export const getAdminData = query({
  args: {},
  guard: canRead,
  handler: async (ctx) => {
    return {
      deletedItems,
      internalUrls,
      usageGraph,
      auditInfo,
    }
  },
})
```

After:

```ts
export const listReaderItems = query({
  args: {},
  guard: canRead,
  handler: async () => readerSafeView,
})

export const getAdminData = query({
  args: {},
  guard: canManageAssets,
  handler: async () => adminView,
})
```

If the product really wants broader visibility, make that explicit and keep the returned shape intentionally narrow.

See also:

- [Checking Permissions](/docs/permissions/checking-permissions)
- [Permission Guards](/docs/permissions/permission-guards)

## 6. Building A Second Privileged Transport For MCP

The mistake:

- using a separate admin client or deploy key as the default MCP transport
- letting MCP bypass the ordinary request-scoped server caller path

Why this is risky:

- it creates a second privileged path
- it increases coupling to internal function names and secrets
- it weakens the Trellis mental model that MCP should project into the same protected runtime

Prefer the normal projection model:

- resolve principal in Nitro
- call a root bridge ref
- let Convex guards decide access

Before:

```ts
export const mcpRuntime = defineMcpApp({
  callConvex: async (event) => createAdminConvexCaller(event),
})
```

After:

```ts
export const mcpRuntime = defineMcpApp({
  callConvex: async (event) => createServerConvexCaller(event),
  resolvePrincipal: async (event) => getMcpPrincipal(event),
})
```

If you still need a privileged bootstrap step, keep it as narrow and isolated as possible. Do not let it become the general MCP calling path.

See also:

- [Your First MCP Tool](/docs/mcp-tools/getting-started)
- [Auth & Permissions](/docs/mcp-tools/auth-and-permissions)
- [Component Bridges](/docs/server-side/private-bridge)

## 7. Treating Unit Tests As Proof Of The Integration Boundary

The mistake:

- tests assert generated file contents
- tests inject principals directly
- tests mock away auth and runtime seams
- CI passes, but the real host-app integration is not exercised

Why this is risky:

- bridge drift can survive
- auth redirects can regress without detection
- MCP visibility and principal forwarding can break while unit tests stay green

The Trellis testing rule is simple:

**seed a real scenario, call the real protected function, assert the boundary**

Before:

```ts
it('matches the generated bridge file', () => {
  expect(readFileSync('convex/bridge.ts', 'utf-8')).toContain('publishPage')
})
```

After:

```ts
it('applies the same rule to a forwarded principal', async () => {
  const agent = ctx.asPrincipal({
    kind: 'user',
    userId: viewerId,
  })

  await expect(agent.mutation(api.pages.publish, { id: pageId })).rejects.toThrow('Forbidden')
})
```

The highest-value integration lane usually includes:

- one real browser-facing protected call
- one forwarded-principal call
- one MCP tool smoke path
- one permission boundary assertion

See also:

- [Testing Your App](/docs/testing/getting-started)
- [Testing Patterns](/docs/testing/testing-patterns)

## 8. Hiding The Browser/Server/MCP Ownership Split Inside Ambiguous Wrappers

The mistake:

- naming a helper as if it were general-purpose
- but actually hard-coding browser-only, server-only, or MCP-only assumptions inside it

Why this is risky:

- downstream code thinks it is forwarding principals generically
- the wrapper quietly overwrites caller identity or transport behavior
- maintainers stop noticing when they have left the Trellis happy path

Avoid this:

- name wrappers for the lane they own
- keep browser wrappers browser-specific
- keep bridge inventories explicit
- keep principal forwarding visible in code review

Before:

```ts
export function createCmsBridge(builders) {
  return {
    query(definition) {
      return builders.query({
        handler: async (ctx, args) =>
          await ctx.runQuery(definition.component, {
            ...args,
            principal: await getBrowserPrincipal(ctx),
          }),
      })
    },
  }
}
```

After:

```ts
export function createBrowserEntryWrappers(builders) {
  // Browser-specific wrapper; principal comes from browser auth
}

export const bridge = createComponentBridge(builders, { principal })
```

If a helper is browser-only, say so in the name and docs.

## A Good Rule Of Thumb

If you notice yourself adding one of these:

- secret fallback permissions
- source-file patchers
- global `server: false`
- manual redirect parsing
- broad read guards with admin payloads
- deploy-key callers
- snapshot-only integration tests

stop and ask:

**Am I extending Trellis, or am I building a second path beside it?**

If it is the second, prefer a hard cut back to the main Trellis path.
