title: 'Glossary'
description: 'Key terms and concepts used throughout @lupinum/trellis documentation.'
---

## Actor

The authenticated identity performing an operation. Defined with `defineActor()` in your Convex backend, the actor is resolved from the auth identity and carries the user's role, tenant membership, and any custom fields your app needs. Every guarded handler receives `ctx.actor`.

See: [Permissions Setup](/docs/permissions/setup), [Actor Lanes and Models](/docs/permissions/actor-lanes-and-models)

## Auth Proxy

A Nitro server route (default: `/api/auth`) that proxies Better Auth requests between the browser and Convex. Handles cookie-based sessions, CORS validation, and credential forwarding. Configured automatically when `auth: true` is set in `nuxt.config.ts`.

See: [Auth Proxy Configuration](/docs/configuration/auth-proxy)

## Guard

A function that decides whether an actor can perform an operation. Handlers declare their guard in the function signature — `guard: open` for public access, `guard: requireAuthenticated` for auth-only, or `guard: can('post.create')` for permission-based checks. A handler without a guard is a type error.

See: [Permission Guards](/docs/permissions/permission-guards)

## Live Query

A query that stays subscribed via WebSocket after its initial SSR load. When the underlying data changes in Convex, the subscription pushes the new result to the client automatically. Enabled by default with `subscribe: true` on `useConvexQuery`.

See: [Subscriptions](/docs/real-time/subscriptions)

## MCP Tool

A Model Context Protocol tool that exposes Convex operations to AI agents. Defined with `defineTool()`, MCP tools go through the same handler guards and permission checks as browser calls. They add rate limiting, confirmation gates, and result envelopes on top.

See: [MCP Tools](/docs/mcp-tools/getting-started)

## Permission Context

A set of boolean flags returned by your permission query (e.g., `getPermissionContext`). The frontend uses these via `usePermissions()` to show or hide UI elements based on the current user's capabilities. The booleans are computed from the actor's role and the guard definitions.

See: [Checking Permissions](/docs/permissions/checking-permissions)

## Redaction

Field-level data filtering that removes or masks sensitive fields based on the caller's capabilities. Defined with `defineRedaction()`, redaction rules run after the query returns data but before it reaches the caller. Used for scenarios like hiding email addresses from non-admin users.

See: [Overrides and Advanced Patterns](/docs/permissions/overrides-and-advanced-patterns)

## Share Token

A capability token that grants access to a specific resource without requiring authentication. Used for scenarios like shareable links. Share tokens carry scoped permissions and can be time-limited.

See: [Visibility Access Example](/examples/05-visibility-access)

## Shared Query

A query instance shared across multiple components via the `shared` option key. When two components call `useConvexQuery` with the same `shared` key, they share a single subscription and cache entry instead of creating duplicates.

See: [Shared Queries](/docs/data-fetching/shared-queries)

## Tenant / Workspace

The isolation boundary for multi-tenant applications. Each tenant (called "workspace" in the canonical schema) has its own users, data, and permission scope. Tenant isolation is enforced by `ensureTenant()` and `loadTenantResource()` in handler load functions.

See: [SaaS Examples](/docs/permissions/saas-examples), [Personal to Workspace](/docs/permissions/personal-to-workspace)

## Token Exchange

The SSR process where the server plugin exchanges a session cookie for a Convex auth token. This token is passed to the client during hydration so the Convex client can authenticate without a round-trip. Happens automatically when auth is enabled.

See: [Token Refresh](/docs/auth-security/token-refresh)

## Trusted Caller

A server-to-server authentication mechanism. When a Nitro server route or MCP tool calls a Convex function, it uses a shared secret (`CONVEX_TRUSTED_CALLER_KEY`) to prove it is a trusted caller. The handler resolves the actor from the trusted caller identity instead of a browser session.

See: [Private Bridge](/docs/server-side/private-bridge)

## Visibility

Row-level access control that determines which records a caller can see. Defined with `defineVisibility()`, visibility rules filter query results based on the caller's actor. Unlike guards (which gate entire operations), visibility filters individual rows within an operation.

See: [Visibility Access Example](/examples/05-visibility-access)
