This guide adds role-based access control to your app. Roles, organizations, and
membership are owned by **Better Auth** (the Organization plugin) — your app
never stores a `role` column or an `organizations` table. Convex reads the role
from Better Auth at request time and enforces it with `auth.api.hasPermission`.

::note
**Prerequisite:** Complete the [Authentication](/docs/guide/auth) guide first. You
should have a working Better Auth + Convex setup with a rebuildable `users`
projection.
::

::callout{icon="i-lucide-alert-triangle" color="amber"}
**Better Auth is the single source of truth for role/org/member state.** Do not
mirror those tables into your app schema. There is no app-owned `organizations`
table and no `role` column on `users`. See
[Authentication → source of truth](/docs/auth-security/authentication).
::

## What the Module Provides

When you enable `permissions: true` in your Nuxt config, the module provides
**one thing**: the `createPermissions` factory function.

```ts
import { createPermissions } from '#imports'

export const { usePermissions, usePermissionRedirect } = createPermissions({
  query: api.auth.getPermissionContext, // Your Convex query, reads Better Auth state
  checkPermission: (ctx, permission, resource) => {
    /* Your UX-only logic */
  },
})
```

This gives you:

- **`usePermissions()`** — a composable that fetches the permission context and
  provides a reactive `can()` function for **showing/hiding UI**.
- **`usePermissionRedirect()`** — a composable that redirects users without a
  permission after render.

::callout{icon="i-lucide-alert-triangle" color="amber"}
**Frontend checks are UX only.** `can()` decides what to render; it never
enforces anything. Every mutation and sensitive query must re-check the caller's
Better Auth permission on the backend with `auth.api.hasPermission`.
::

## Step 1: Enable the Better Auth Organization Plugin

Roles and membership live in Better Auth. Add the Organization plugin and an
access-control policy to your `convex/auth.ts` (this is the exact pattern the
`starters/team` reference uses):

```ts [convex/auth.ts]
import { convex } from '@convex-dev/better-auth/plugins'
import { betterAuth } from 'better-auth'
import { createAccessControl, organization } from 'better-auth/plugins'

// Statements: the resources and actions your app authorizes.
const accessControl = createAccessControl({
  organization: ['update'],
  member: ['create', 'update', 'delete'],
  invitation: ['create', 'cancel'],
  post: ['create', 'read', 'update', 'delete'],
})

const owner = accessControl.newRole({
  organization: ['update'],
  member: ['create', 'update', 'delete'],
  invitation: ['create', 'cancel'],
  post: ['create', 'read', 'update', 'delete'],
})
const admin = accessControl.newRole({
  organization: ['update'],
  member: ['create', 'update', 'delete'],
  invitation: ['create', 'cancel'],
  post: ['create', 'read', 'update', 'delete'],
})
const member = accessControl.newRole({
  organization: [],
  member: [],
  invitation: [],
  post: ['create', 'read', 'update'],
})
const viewer = accessControl.newRole({
  organization: [],
  member: [],
  invitation: [],
  post: ['read'],
})

// ... inside betterAuth({ plugins: [...] }):
organization({
  ac: accessControl,
  roles: { owner, admin, member, viewer },
})
```

Your `users` table stays a plain, rebuildable projection (name/email/image only)
kept in sync by the auth triggers — **no `role` column**.

::callout{icon="i-lucide-info"}
The Organization plugin creates and owns the `organization`, `member`, and
`invitation` tables inside the Better Auth component. A member's role lives on
their `member` row. You read it; you never copy it into your schema.
::

## Step 2: Product Schema (Keyed by Better Auth IDs)

Product data is keyed by Better Auth **string IDs** — the organization id and the
Better Auth user id — never by a foreign key into an app-owned org table.

```ts [convex/schema.ts]
import { defineSchema, defineTable } from 'convex/server'
import { v } from 'convex/values'

export default defineSchema({
  // Rebuildable projection of the Better Auth user (no role column).
  users: defineTable({
    authId: v.string(),
    displayName: v.optional(v.string()),
    email: v.optional(v.string()),
    avatarUrl: v.optional(v.string()),
    createdAt: v.number(),
    updatedAt: v.number(),
  })
    .index('by_auth_id', ['authId'])
    .index('by_email', ['email']),

  // Product data keyed by the Better Auth organization id (string).
  posts: defineTable({
    title: v.string(),
    content: v.string(),
    organizationId: v.string(), // Better Auth org id
    ownerId: v.string(), // Better Auth user id of the creator
    createdAt: v.number(),
  })
    .index('by_organization', ['organizationId'])
    .index('by_owner', ['ownerId']),
})
```

## Step 3: Backend Authorization (Better Auth `hasPermission`)

Backend enforcement delegates to Better Auth. Create a helper that resolves the
session and checks the permission against the caller's Better Auth role. Throw
**structured** `ConvexError` values so the client can tell authentication from
authorization failures.

```ts [convex/lib/permissions.ts]
import { ConvexError } from 'convex/values'

import { components } from '../_generated/api'
import type { MutationCtx, QueryCtx } from '../_generated/server'
import { authComponent, createAuth } from '../auth'

type Ctx = QueryCtx | MutationCtx
type Role = 'owner' | 'admin' | 'member' | 'viewer'
type PostPermission = 'create' | 'read' | 'update' | 'delete'

// Read a member row (and its role) straight from the Better Auth component.
export async function getBetterAuthMember(
  ctx: Ctx,
  args: { organizationId: string; userId: string },
) {
  return (await ctx.runQuery(components.betterAuth.adapter.findOne, {
    model: 'member',
    where: [
      { field: 'organizationId', value: args.organizationId },
      { field: 'userId', value: args.userId },
    ],
  })) as { role: Role } | null
}

// Require the caller to hold a post:<permission> in the given organization.
export async function requirePostPermission(
  ctx: Ctx,
  args: { organizationId: string; permission: PostPermission },
) {
  const { auth, headers } = await authComponent.getAuth(createAuth, ctx)
  const session = await auth.api.getSession({ headers })
  if (!session) {
    throw new ConvexError({ code: 'UNAUTHENTICATED', message: 'Not authenticated' })
  }

  const allowed = await auth.api.hasPermission({
    headers,
    body: { organizationId: args.organizationId, permissions: { post: [args.permission] } },
  })
  if (!allowed.success) {
    throw new ConvexError({
      code: 'FORBIDDEN',
      message: `Missing post:${args.permission} permission`,
    })
  }

  return { userId: session.user.id }
}
```

::callout{icon="i-lucide-info"}
`auth.api.hasPermission` evaluates the caller's Better Auth role against the
access-control policy from Step 1. The role never comes from your database.
::

## Step 4: Permission Context Query (Reads Better Auth)

The frontend needs the caller's role for UX gating. Add a **no-argument** query
that reads the active organization and the member role from Better Auth. This
matches `createPermissions`' query type
(`FunctionReference<'query', 'public', Record<string, never>, TContext | null>`).

```ts [convex/auth.ts]
import { query } from './_generated/server'
import { getBetterAuthMember } from './lib/permissions'

export const getPermissionContext = query({
  args: {},
  handler: async (ctx) => {
    const { auth, headers } = await authComponent.getAuth(createAuth, ctx)
    const session = await auth.api.getSession({ headers })
    if (!session) return null

    // The active organization lives on the Better Auth session.
    const organizationId = session.session.activeOrganizationId
    if (!organizationId) return null

    // Role comes from the Better Auth member row — never an app column.
    const member = await getBetterAuthMember(ctx, {
      organizationId,
      userId: session.user.id,
    })
    if (!member) return null

    return { role: member.role, userId: session.user.id, orgId: organizationId }
  },
})
```

## Step 5: Shared Permission Map (Frontend UX)

`createPermissions` needs a `checkPermission` function to translate a permission
string into a boolean for `can()`. This mirrors the Better Auth policy for the
UI — it is **not** the enforcement point. Keep it small.

```ts [shared/permissions.ts]
// Roles as defined by the Better Auth Organization plugin.
export type Role = 'owner' | 'admin' | 'member' | 'viewer'

// UX mirror of the access-control policy: which roles may do what.
export const permissions = {
  'post.create': { roles: ['owner', 'admin', 'member'] },
  'post.read': { roles: ['owner', 'admin', 'member', 'viewer'] },
  'post.update': { own: ['member'], any: ['owner', 'admin'] },
  'post.delete': { own: ['member'], any: ['owner', 'admin'] },
  'org.settings': { roles: ['owner'] },
} as const

export type Permission = keyof typeof permissions

export interface PermissionContext {
  role: Role
  userId: string
  orgId: string
}

export interface Resource {
  ownerId?: string
}

export function checkPermission(
  ctx: PermissionContext | null,
  permission: Permission,
  resource?: Resource,
): boolean {
  if (!ctx) return false
  const rule = permissions[permission]

  if ('roles' in rule) {
    return (rule.roles as readonly string[]).includes(ctx.role)
  }
  if ((rule.any as readonly string[]).includes(ctx.role)) return true
  if (!resource?.ownerId) return false
  return (rule.own as readonly string[]).includes(ctx.role) && resource.ownerId === ctx.userId
}
```

## Step 7: Enforce on the Backend

Every mutation re-checks the caller's Better Auth permission. Bind the
`organizationId` from the checked context or the target row — never from a
model-/client-supplied argument.

```ts [convex/posts.ts]
import { mutation, query } from './_generated/server'
import { ConvexError, v } from 'convex/values'
import { getBetterAuthMember, requirePostPermission } from './lib/permissions'

export const create = mutation({
  args: { organizationId: v.string(), title: v.string(), content: v.string() },
  handler: async (ctx, args) => {
    const { userId } = await requirePostPermission(ctx, {
      organizationId: args.organizationId,
      permission: 'create',
    })

    return ctx.db.insert('posts', {
      title: args.title,
      content: args.content,
      organizationId: args.organizationId,
      ownerId: userId,
      createdAt: Date.now(),
    })
  },
})

export const update = mutation({
  args: { id: v.id('posts'), title: v.string() },
  handler: async (ctx, args) => {
    const post = await ctx.db.get(args.id)
    if (!post) throw new ConvexError({ code: 'NOT_FOUND', message: 'Post not found' })

    // Derive org from the target row, then check permission.
    const { userId } = await requirePostPermission(ctx, {
      organizationId: post.organizationId,
      permission: 'update',
    })

    // Ownership: members may only edit their own posts. Owners/admins have the
    // post:update permission unconditionally via the access-control policy.
    const member = await getBetterAuthMember(ctx, {
      organizationId: post.organizationId,
      userId,
    })
    if (member?.role === 'member' && post.ownerId !== userId) {
      throw new ConvexError({ code: 'FORBIDDEN', message: 'You can only edit your own posts' })
    }

    await ctx.db.patch(args.id, { title: args.title })
  },
})
```

## Step 8: Gate the UI

Use `usePermissions()` to show or hide controls. Enforcement already happened in
Step 7 — this is UX only.

```vue [app/pages/posts.vue]
<script setup lang="ts">
import { api } from '#convex/api'
import { usePermissions } from '~/composables/usePermissions'

const { can, role } = usePermissions()
const { data: posts } = await useConvexQuery(api.posts.list, {}, { server: true })
</script>

<template>
  <div>
    <p>
      Your role: <strong>{{ role }}</strong>
    </p>

    <button v-if="can('post.create')">New Post</button>

    <ul>
      <li v-for="post in posts" :key="post._id">
        {{ post.title }}
        <!-- Pass the resource for ownership checks -->
        <button v-if="can('post.update', post)">Edit</button>
        <button v-if="can('post.delete', post)">Delete</button>
      </li>
    </ul>
  </div>
</template>
```

::callout{icon="i-lucide-info"}
**Reactive by default:** `can()` returns a plain boolean from the current
permission context. Use it directly in templates, and call it inside `computed()`
or `watchEffect()` when storing a derived script value.
::

### Change a Role (Better Auth)

To change a member's role in development or an admin panel, call the Better Auth
API — never patch an app table. From a Convex mutation:
`auth.api.updateMemberRole({ headers, body: { organizationId, memberId, role } })`,
or from the client via `authClient.organization.updateMemberRole(...)`.

## Verification Checklist

- [ ] `owner`/`admin` can create, edit, and delete any post
- [ ] `member` can create posts and edit only their own
- [ ] `viewer` can only read posts (no create/edit/delete buttons)
- [ ] The backend rejects unauthorized mutations with a structured
      `ConvexError({ code: 'FORBIDDEN' })`
- [ ] There is no `role` column and no `organizations` table in your schema
