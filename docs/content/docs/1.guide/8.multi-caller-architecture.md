Use this page when the app is no longer browser-only and you need to understand how Trellis handles multiple caller types without splitting the backend model.

The short version is simple: Trellis wants browser UI, Nitro routes, webhooks, and agent clients to resolve through one protected Convex layer instead of four separate policy systems.

## The Caller Types

The common caller types are:

- browser UI
- Nitro server routes
- trusted server-to-server callers such as webhooks
- agent clients through MCP

Those are transport boundaries. They are not separate business authorization models.

## Principal First, Actor Second

Trellis separates two questions:

1. how did this call arrive
2. who is this caller in the app

The first question is answered by the principal.
The second question is answered by the actor.

That is why the canonical team and MCP examples use agent principals such as `kind: 'agent'` instead of encoding the transport as the business role model.

## What The Principal Is For

The principal carries transport identity and transport-specific facts.

Examples:

- browser request with a signed-in session
- trusted webhook with a verified caller key
- MCP tool call resolved from an agent token

This is where Trellis can distinguish how the request arrived.

## What The Actor Is For

The actor is the app-owned business identity.

Examples:

- a personal user
- a workspace member
- an owner acting inside a tenant
- a service lane you define explicitly

This is where Trellis can decide what the caller actually means inside the app.

## Why This Split Exists

Without the principal and actor split, transport details leak into business authorization.

That is how teams end up with logic like:

- “MCP callers can do X directly”
- “Webhook routes skip normal checks”
- “Server callers use a different mutation than the UI”

That is exactly what Trellis is trying to prevent.

The browser route, server route, and tool definition may all differ, but the protected Convex handler should still own the real business decision.

## Transport Visibility Is Not Authorization

Transport-specific controls are still useful.

For example:

- you may hide some tools from some agent clients
- you may expose only certain server endpoints publicly
- you may require a trusted caller key for webhooks

They do **not** replace Convex business authorization.

A hidden tool is not authorization.
A webhook signature is not business authorization.
A private route is not business authorization.

Those transport checks decide whether the request reaches your protected handler. They do **not** decide whether the underlying business action is allowed.

## One Backend Model, Different Projections

Different projections can change:

- request envelope
- session lookup
- result envelope
- confirmation flow

They should not change:

- actor resolution rules
- tenant boundaries
- guards
- authorization checks
- operation safety model

That is the real value of the multi-caller architecture.

## Where Agent Callers Fit

Agent support is important here because it is the easiest place to accidentally fork the model.

The Trellis direction is:

- resolve the agent caller as a principal
- map that principal into an app-owned actor
- run the same tenant and authorization path as the UI
- use operations for destructive flows when preview and confirmation matter

If the agent path needs a completely different permission model, the backend model is already drifting.

## Common Failure Modes

- Transport type becomes the business role model.
- Server routes call special-case helpers instead of the protected backend surface.
- Webhooks bypass the same checks used by browser mutations.
- Tool visibility is treated as if it were authorization.
- Agent-specific logic starts replacing app-owned actor logic.
