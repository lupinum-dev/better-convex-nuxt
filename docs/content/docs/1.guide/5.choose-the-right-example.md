Use this page when you want to learn from a real app instead of a blank project.

Do not start with the biggest example. Trellis examples are progressive. The fastest way to get lost is to open the most advanced app and treat it as a beginner baseline.

## First Reader Path

Read these in order:

1. `01-public-todo`
2. `02-auth-todo`
3. `03-team-workspace`

Each one adds one major layer:

- `01` adds the smallest `defineTrellis(...)` runtime and one query and mutation path
- `02` adds Better Auth and actor resolution
- `03` adds workspaces, permissions, trusted callers, and MCP projection

If you only need a public or personal app, stop earlier. There is no benefit in inheriting workspace or agent complexity you do not need.

## Which Example Matches Your App

| Example | Open it when you need | What it proves |
| --- | --- | --- |
| `01-public-todo` | public-only queries and mutations | smallest useful Trellis baseline |
| `02-auth-todo` | sign-in and personal ownership | auth plus app-owned actors without tenancy |
| `03-team-workspace` | roles, permissions, tenant boundaries, or MCP | canonical protected workspace app |
| `04-saas-platform` | pagination, uploads, admin flows, practical SaaS depth | month-two product work |
| `05-visibility-access` | advanced visibility, redaction, enrollment, or sharing | deeper access patterns |
| `06-multi-workspace` | workspace switching or cross-tenant agency views | multi-workspace membership model |
| `07-mcp-reference` | full MCP surface | prompts, resources, sessions, and tool variants |
| `08-component-mini-cms` | component bridge and projection boundaries | local component plus MCP seam |

## Default Recommendations

Use these defaults unless your app clearly points somewhere else:

- evaluating Trellis: `01-public-todo`
- building a personal app: `02-auth-todo`
- building a team app: `03-team-workspace`
- learning the agent surface: `07-mcp-reference`
- learning the component bridge seam: `08-component-mini-cms`

## What The Current Examples Are For

The current `examples/` folder is the shipped reference set. It shows the current public surfaces and the current default patterns.

The `examples-next/` folder is different. It is a pressure suite for the framework direction, not the default onboarding path. Use it to understand where Trellis is going, not as the first place to copy code from.

That distinction matters. Current examples are for learning what ships now. `examples-next` is for testing whether the framework direction holds up under larger app families.

## When To Prefer Reading Over Copying

Do not copy an example wholesale unless the app shape is genuinely close to yours.

Instead:

1. choose the smallest matching example
2. read the files that define the core boundary
3. copy only the pattern you need
4. stop before unrelated surfaces come with it

For example:

- if you only need personal auth, do not inherit workspace roles from `03-team-workspace`
- if you only need tool definitions, do not inherit every MCP extension from `07-mcp-reference`
- if you only need a bridge boundary, do not inherit the rest of the CMS shape from `08-component-mini-cms`

## Files To Read First

Use this file-first path inside each example:

- `01-public-todo`: `shared/schemas/todo.ts`, `convex/todos.ts`, `pages/index.vue`
- `02-auth-todo`: `convex/auth.ts`, `convex/auth/actor.ts`, `convex/todos.ts`, `pages/index.vue`
- `03-team-workspace`: `convex/auth/actor.ts`, `convex/auth/checks.ts`, `convex/todos.ts`, `server/mcp/tools/*.ts`, `pages/index.vue`

Those files show the actual boundary and business flow faster than browsing the whole tree.
