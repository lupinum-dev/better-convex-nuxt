description: 'Solutions for common runtime issues with queries, auth, mutations, and real-time updates.'

::tip{title="Before diving in"}

1. Is `npx convex dev` running in a separate terminal?
2. Is `nuxt dev --dotenv .env.local` running?
3. Check the browser console and Nuxt terminal for error messages — they usually point to the exact problem.
   ::

## Query Returns Null but Data Exists

**Symptom:** `data` ref stays `null` even though data exists in the Convex dashboard.

**Check these in order:**

1. **Wrong args** — Verify that the args match what the query expects. Convex silently returns empty results for mismatched args.
2. **Conditional skip** — If you pass a reactive getter for args, returning `null` or `undefined` skips the query entirely. Make sure your getter returns valid args when the data should load.
3. **Auth not ready** — If the query requires authentication, it may fail silently during SSR. Check the `error` ref for auth-related errors.
4. **Wrong API import** — Ensure you import from `#trellis/api`, not from a stale or wrong generated path.

```vue
<script setup>
// Debug: check both data and error
const { data, error, status } = await useConvexQuery(api.tasks.list, {})
console.log('status:', status.value, 'error:', error.value)
</script>
```

## Mutation Works but UI Does Not Update

**Symptom:** Mutation succeeds (no error) but the query data in the template stays stale.

**Common causes:**

1. **Subscription disabled** — If `subscribe: false` was set on the query, it will not receive real-time updates after the initial SSR load. Remove the option or set `subscribe: true`.
2. **Different args** — The query subscription is keyed by function reference + args. If the mutation changes data that a different arg set would match, the current subscription will not pick it up.
3. **Server-only query** — If `server: true, subscribe: false` was set, the query only runs once during SSR. Call `refresh()` to re-fetch.

## Optimistic Update Flashes Then Reverts

**Symptom:** The UI briefly shows the optimistic value, then snaps back to the old state.

**This usually means the optimistic update shape does not match the server response.**

1. **Missing fields** — The optimistic item must include `_id` and `_creationTime` to match the Convex document shape. Use a temporary ID:
   ```ts
   optimisticUpdate: (ctx, args) => {
     ctx
       .query(api.tasks.list, {})
       .update((tasks) => [
         ...tasks,
         { _id: crypto.randomUUID(), _creationTime: Date.now(), ...args },
       ])
   }
   ```
2. **Wrong query args** — The `ctx.query()` call must use the exact same args as the active query subscription. Mismatched args target a different cache entry.
3. **Paginated query** — For paginated queries, use `ctx.paginatedQuery()` with `insertAtTop()` or `insertAtBottom()` instead of `ctx.query().update()`.

## useConvexAuth isPending Stays True

**Symptom:** `isPending` never resolves to `false`, blocking the UI.

**Check these:**

1. **Auth not enabled** — Ensure `auth: true` (or an auth config object) is set in `nuxt.config.ts` under `convex`.
2. **Auth proxy not reachable** — The auth proxy at `/api/auth` (or your custom route) must be accessible. Check the browser network tab for failed requests to this endpoint.
3. **Missing SITE_URL** — The `SITE_URL` or `NUXT_PUBLIC_SITE_URL` environment variable must be set for auth token exchange. Without it, the auth flow cannot complete.
4. **Better Auth not initialized** — Ensure your `convex/auth.ts` file calls `defineAuth()` and that the Convex backend has been deployed with the auth configuration.

## Real-Time Updates Stop After Tab Sleep

**Symptom:** After the browser tab is backgrounded or the laptop sleeps, queries stop updating.

**This is expected WebSocket behavior.** The Convex client automatically reconnects, but there may be a brief gap. Use `useConvexConnectionState()` to detect and communicate the reconnection:

```vue
<script setup>
const connection = useConvexConnectionState()
</script>

<template>
  <div v-if="!connection.isWebSocketConnected" class="banner">Reconnecting...</div>
</template>
```

If updates never resume, check that `subscribe: true` is set on the query (this is the default).

## Dev Warning: "Mutation Error Was Never Read"

**Symptom:** Console warning in development: `[@lupinum/trellis] Mutation 'api.foo.bar' threw an error that was never read.`

**This means** the mutation threw an error, but your code never checked `error.value` or wrapped the call in `try/catch`. The warning prevents silent failures.

**Fix:** Either handle the error in your template or catch it in your handler:

```vue
<template>
  <p v-if="createPost.error.value">{{ createPost.error.value.message }}</p>
</template>
```

Or:

```ts
try {
  await createPost({ title: 'Hello' })
} catch (err) {
  // handle error
}
```

This warning only appears in development mode.

## TypeError: Cannot Read Properties of Null

**Symptom:** `TypeError: Cannot read properties of null (reading 'query')` or similar null-access errors from composables.

**This means** the composable is being called outside a Vue component setup context. All `useConvex*` composables must be called inside `<script setup>` or a `setup()` function.

**Common mistakes:**

1. Calling a composable inside a callback or event handler that runs after setup completes — move it to the top level of `<script setup>`.
2. Calling a composable inside a Pinia action or a plain function — use `executeConvexQuery` for imperative query execution outside components.
3. Calling a composable conditionally — all composables must be called unconditionally during setup. Use null args to skip execution instead.

```ts
// Wrong: conditional composable
if (someCondition) {
  const { data } = await useConvexQuery(api.tasks.list, {})
}

// Right: unconditional composable with conditional args
const { data } = await useConvexQuery(api.tasks.list, () => (someCondition ? {} : null))
```

## usePermissions() Returns Null

**Symptom:** `usePermissions()` returns `null` for `ctx`, `role`, and `can` — even though the user is authenticated.

**Check these:**

1. **Missing config** — Ensure `trellis.permissions.query` is set in `nuxt.config.ts` and points to a valid Convex query path (e.g., `'workspaces:getPermissionContext'`).
2. **Query does not exist** — Run `npx convex dev` and verify the permission query is deployed. The module validates this at build time, so check the terminal for warnings.
3. **Query returns the wrong shape** — The permission query must return an object with at least `role` and `can` (a record of boolean values). If it returns `null`, the user is treated as anonymous.
4. **Auth not settled** — `usePermissions()` waits for auth to settle before querying. If `isPending` from `useConvexAuth()` is stuck, fix the auth issue first (see above).

## useAuthGuard() Redirects Unexpectedly

**Symptom:** `useAuthGuard({ can: 'post.create' })` redirects even though the user has the permission.

**Check these:**

1. **Key mismatch** — The `can` key must match exactly what your permission query returns in the `can` object. Check for typos (e.g., `'post.create'` vs `'posts.create'`).
2. **Permission query timing** — `useAuthGuard` checks permissions on the first non-pending result. If the permission query returns `null` initially (before the real result), the guard may redirect before data arrives. Ensure your permission query is SSR-enabled (`server: true`).
3. **Wrong guard** — If you want to check authentication only (not a specific permission), use `convexAuth: true` in page meta instead of `useAuthGuard`.

## MCP Tools Not Appearing in tools/list

**Symptom:** Your MCP client shows no tools or is missing some tools.

**Check these:**

1. **File location** — MCP tool files must be in `server/mcp/tools/`. The module auto-discovers tools from this directory.
2. **Export** — Each tool file must have a default export using `defineTool()` or `defineMcpTool()`.
3. **Auth requirements** — If the tool requires authentication and the MCP client is not providing credentials, the tool may be hidden. Check if the tool has an `enabled` guard that filters based on auth state.
4. **Module not enabled** — Ensure `@nuxtjs/mcp-toolkit` is in your `modules` array in `nuxt.config.ts`.

## MCP Tool Returns "Authentication Required"

**Symptom:** Calling an MCP tool returns an authentication error.

**Check these:**

1. **Missing bearer key** — Scoped MCP tools require a bearer key. Ensure the MCP client sends the key in the `Authorization` header.
2. **Invalid key** — The key must match one configured in your MCP auth setup. Keys are hashed — you cannot verify them by reading the stored value.
3. **Key not deployed** — If you recently added or rotated keys, redeploy with `npx convex deploy`.

## MCP Tool Confirmation Token Fails

**Symptom:** A destructive tool returns `confirmation_required` or `conflict` on the second call.

**Check these:**

1. **Pass `_confirmationToken` back unchanged** — destructive tools now return the token in `structuredContent.preview.confirmationToken`.
2. **Do not change the args** — the token is bound to the original argument hash.
3. **Do not wait too long** — confirmation tokens expire quickly by design.
4. **Watch for data drift** — Trellis re-runs the preview and rejects execution if the semantic `confirm` payload changed.
