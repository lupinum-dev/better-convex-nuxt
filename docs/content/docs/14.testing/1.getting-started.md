Trellis tests follow one pattern: **seed a scenario, call the real protected function, assert the boundary.** No mocks, no fake auth layer, no duplicate test-only permission model.

## Setup

### Vitest Config

```ts [vitest.config.ts]
import { defineConfig } from 'vitest/config'
import { convexTestConfig } from '@lupinum/trellis/testing'

export default defineConfig(convexTestConfig())
```

### Test Setup File

Create `convex/test.setup.ts` to wire up the test modules:

```ts [convex/test.setup.ts]
/// <reference types="vite/client" />
import { vi } from 'vitest'
import { convexServerMock, createConvexTestModules } from '@lupinum/trellis/testing'

export const modules = createConvexTestModules(import.meta.glob('./**/*.ts', { eager: false }))

vi.mock('./_generated/server', async () => await convexServerMock())
```

### Create a Test Context

```ts
import { createTestContext } from '@lupinum/trellis/testing'
import schema from '~/convex/schema'
import { modules } from '~/convex/test.setup'

const ctx = createTestContext({ schema, modules })
```

## Your First Test

Here is a complete test that proves a permission boundary. It seeds a workspace with two users, creates data as one user, and verifies the other user cannot modify it:

```ts
import { describe, expect, it } from 'vitest'
import { api } from '~/convex/_generated/api'

describe('todo ownership', () => {
  it("blocks a member from completing another member's todo", async () => {
    // 1. Seed a workspace with two members
    const team = await ctx.seedTenant({
      name: 'Acme',
      users: {
        alice: { role: 'member' },
        bob: { role: 'member' },
      },
    })

    // 2. Alice creates a todo
    const todoId = await team.users.alice.mutation(api.todos.create, {
      title: 'Alice private todo',
    })

    // 3. Bob tries to complete it — this should fail
    await expect(
      team.users.bob.mutation(api.todos.setCompleted, {
        id: todoId,
        completed: true,
      }),
    ).rejects.toThrow('Forbidden: Update todo')
  })
})
```

Each seeded user gets `query`, `mutation`, and `action` helpers backed by a real authenticated identity. The test calls the actual `api.todos.setCompleted` handler — the same code that runs in production — so the guard, load, and authorize phases all execute.

## Testing Forwarded Principals

Use `asPrincipal(...)` to test the same handler through a forwarded principal — simulating what happens when a server route or MCP tool calls your Convex function:

```ts
it('applies the same rules to an agent principal', async () => {
  const team = await ctx.seedTenant({
    name: 'Acme',
    users: {
      viewer: { role: 'viewer' },
    },
  })

  // Simulate an MCP agent with the viewer's identity
  const agent = ctx.asPrincipal({
    kind: 'user',
    userId: team.users.viewer.authId,
  })

  // The agent should be blocked by the same guard
  await expect(agent.mutation(api.todos.create, { title: 'Should fail' })).rejects.toThrow(
    'Forbidden: Create todo',
  )
})
```

This is the testing equivalent of an MCP tool calling an internal bridge ref, or a server route forwarding a principal into Convex.

## What to Test First

The highest-signal tests, in order:

1. **Permission boundaries** — can this actor do this action? (the test above)
2. **Tenant isolation** — does data leak between workspaces?
3. **Principal forwarding** — do forwarded principals hit the same guards?
4. **Anonymous access** — are public queries open and protected mutations blocked?

For detailed test patterns for each of these, see [Testing Patterns](/docs/testing/testing-patterns).
