## `createTestContext(...)`

Wraps `convex-test` with higher-level helpers for seeding canonical data and calling handlers through explicit principal shapes.

```ts
import { createTestContext } from '@lupinum/trellis/testing'

import schema from '~/convex/schema'
import { modules } from '~/convex/test.setup'

const ctx = createTestContext({ schema, modules })
```

### Options

```ts
interface CreateTestContextOptions<TSchema> {
  schema: TSchema
  modules?: Record<string, () => Promise<unknown>>
  tenant?: { table?: string; field?: string }
  users?: {
    table?: string
    authField?: string
    roleField?: string
    tenantField?: string
    nameField?: string
    emailField?: string
  }
}
```

The `tenant` and `users` overrides are escape hatches for non-canonical schemas. Omit them if your schema follows the default workspace model.

### Returned Methods

#### `ctx.raw`

Direct access to the underlying `convex-test` instance.

#### `ctx.seed(table, data)`

Insert a single row into a table. Returns the new `_id`.

```ts
const postId = await ctx.seed('posts', {
  title: 'Hello',
  workspaceId: team.id,
  ownerId: team.users.alice.authId,
})
```

#### `ctx.readAll(table)`

Read all rows from a table.

#### `ctx.seedTenant(options)`

Create a workspace with named, authenticated users in one call.

```ts
const team = await ctx.seedTenant({
  name: 'Acme',
  users: {
    alice: { role: 'owner' },
    bob: { role: 'member' },
    carol: { role: 'viewer' },
  },
})
```

What it creates by default:

1. a workspace row with `name`, `slug`, `ownerId`, `createdAt`, `updatedAt`
2. one row per user with `authId`, `role`, `workspaceId`, `displayName`, `email`, `createdAt`, `updatedAt`

Each user gets `query`, `mutation`, and `action` helpers backed by `withIdentity({ subject })`.

#### `ctx.asPrincipal(principal)`

Create a test client that injects `principal` into every call.

```ts
const agent = ctx.asPrincipal({
  kind: 'agent',
  agentId: 'mcp',
})

await agent.query(api.posts.listDrafts, {})
await agent.mutation(api.posts.publish, { id: postId })
```

This is the correct way to test:

- server-to-Convex forwarding
- MCP-to-Convex forwarding
- root wrapper and component bridge seams

`asPrincipal(...)` does not mock authorization. It simply exercises the same explicit-principal path your runtime uses in production.

## `convexTestConfig(...)`

Merges Convex-friendly defaults into your Vitest config.

```ts
import { defineConfig } from 'vitest/config'
import { convexTestConfig } from '@lupinum/trellis/testing'

export default defineConfig(convexTestConfig())
```

## `convexServerMock()`

Factory for `vi.mock('./_generated/server', ...)`.

```ts
import { vi } from 'vitest'
import { convexServerMock } from '@lupinum/trellis/testing'

vi.mock('./_generated/server', async () => await convexServerMock())
```

## `createConvexTestModules(...)`

Wraps `import.meta.glob(...)` results to ensure the generated API module hint is present for `convex-test`.

```ts
import { createConvexTestModules } from '@lupinum/trellis/testing'

export const modules = createConvexTestModules(import.meta.glob('./**/*.ts', { eager: false }))
```
