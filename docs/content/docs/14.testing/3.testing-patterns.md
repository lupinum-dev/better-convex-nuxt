## Tests Read Like Specs

A well-written test proves a behavioral boundary. When you read a test file, you should understand the safety model without reading the handler implementation. The test name is the spec; the body is the proof.

```ts
it("blocks a member from updating another member's todo", async () => {
  const ctx = createCtx()
  const team = await ctx.seedTenant({
    name: 'Alpha',
    users: {
      alice: { role: 'member' },
      bob: { role: 'member' },
    },
  })

  const todoId = await team.users.alice.mutation(api.todos.create, {
    title: 'Alice private team todo',
  })

  await expect(
    team.users.bob.mutation(api.todos.setCompleted, {
      id: todoId,
      completed: true,
    }),
  ).rejects.toThrow('Forbidden: Update todo')
})
```

No mocks. No setup boilerplate. The test seeds a scenario, performs an action, and asserts the boundary.

## Testing Permission Boundaries

### Role-Based Access

Seed multiple users with different roles and verify what each can and cannot do:

```ts
describe('permissions', () => {
  it('lets an owner create posts, denies viewers', async () => {
    const ctx = createCtx()
    const team = await ctx.seedTenant({
      name: 'Acme',
      users: {
        owner: { role: 'owner' },
        viewer: { role: 'viewer' },
      },
    })

    // Owner can create
    await expect(
      team.users.owner.mutation(api.posts.create, { title: 'Hello' }),
    ).resolves.toBeDefined()

    // Viewer cannot
    await expect(team.users.viewer.mutation(api.posts.create, { title: 'Nope' })).rejects.toThrow(
      'Forbidden',
    )
  })
})
```

### Ownership Checks

Verify that users can only modify their own resources:

```ts
it('prevents cross-user modification', async () => {
  const ctx = createCtx()
  const team = await ctx.seedTenant({
    name: 'Acme',
    users: {
      alice: { role: 'member' },
      bob: { role: 'member' },
    },
  })

  const alicePost = await team.users.alice.mutation(api.posts.create, {
    title: 'Alice only',
  })

  await expect(team.users.bob.mutation(api.posts.remove, { id: alicePost })).rejects.toThrow(
    'Forbidden',
  )
})
```

## Testing Tenant Isolation

Seed two separate tenants and verify that data does not leak across boundaries:

```ts
it('keeps tenants isolated from each other', async () => {
  const ctx = createCtx()
  const alpha = await ctx.seedTenant({
    name: 'Alpha',
    users: { alice: { role: 'member' } },
  })
  const beta = await ctx.seedTenant({
    name: 'Beta',
    users: { bruno: { role: 'member' } },
  })

  await alpha.users.alice.mutation(api.todos.create, { title: 'Alpha only' })
  await beta.users.bruno.mutation(api.todos.create, { title: 'Beta only' })

  const alphaTodos = await alpha.users.alice.query(api.todos.list, {})
  const betaTodos = await beta.users.bruno.query(api.todos.list, {})

  expect(alphaTodos).toHaveLength(1)
  expect(alphaTodos[0]?.title).toBe('Alpha only')
  expect(betaTodos).toHaveLength(1)
  expect(betaTodos[0]?.title).toBe('Beta only')
})
```

## Testing Trusted Callers

Use `ctx.asTrustedCaller()` to verify that server-to-server calls and MCP tool invocations go through the same permission guards as browser calls:

```ts
it('applies the same permission rules to trusted callers', async () => {
  const ctx = createCtx()
  const team = await ctx.seedTenant({
    name: 'Alpha',
    users: {
      viewer: { role: 'viewer' },
    },
  })

  const trustedCaller = ctx.asTrustedCaller({
    userId: team.users.viewer.authId,
  })

  // A viewer is still a viewer, regardless of call path
  await expect(trustedCaller.mutation(api.todos.create, { title: 'Should fail' })).rejects.toThrow(
    'Forbidden: Create todo',
  )
})
```

This pattern proves that your MCP tools, Nitro server routes, and browser UI all enforce the same authorization rules.

## Testing Anonymous Access

Use `ctx.raw` (no identity) to test what unauthenticated callers can and cannot do:

```ts
it('denies anonymous callers on protected queries', async () => {
  const ctx = createCtx()

  await expect(ctx.raw.query(api.todos.list, {})).rejects.toThrow('Forbidden')
})

it('allows anonymous callers on public queries', async () => {
  const ctx = createCtx()

  await expect(ctx.raw.query(api.status.health, {})).resolves.toBeDefined()
})
```

## Testing Permission Context

If your handlers return a permission context (booleans the frontend uses for UI gating), test that the context matches the role:

```ts
it('returns correct permission booleans for contrasting roles', async () => {
  const ctx = createCtx()
  const team = await ctx.seedTenant({
    name: 'Acme',
    users: {
      owner: { role: 'owner' },
      viewer: { role: 'viewer' },
    },
  })

  const ownerCtx = await team.users.owner.query(api.workspaces.getPermissionContext, {})
  const viewerCtx = await team.users.viewer.query(api.workspaces.getPermissionContext, {})

  expect(ownerCtx?.can['todo.create']).toBe(true)
  expect(viewerCtx?.can['todo.create']).toBe(false)
  expect(viewerCtx?.can['todo.read']).toBe(true)
})
```

## Test File Organization

A typical consumer test file:

```ts
// convex/todos.test.ts
import { createTestContext } from '@lupinum/trellis/testing'
import { anyApi } from 'convex/server'
import { describe, expect, it } from 'vitest'

import schema from './schema'
import { modules } from './test.setup'

const api = anyApi

function createCtx() {
  return createTestContext({ schema, modules })
}

describe('todos', () => {
  it('your test here', async () => {
    const ctx = createCtx()
    // ...
  })
})
```

Create `createCtx()` as a factory function at the top of each test file. Each test gets a fresh context with isolated state.

## Test Setup File

Every consumer project needs a `convex/test.setup.ts` file. This is the only boilerplate:

```ts
// convex/test.setup.ts
/// <reference types="vite/client" />

import { vi } from 'vitest'
import { convexServerMock, createConvexTestModules } from '@lupinum/trellis/testing'

export const modules = createConvexTestModules(import.meta.glob('./**/*.ts', { eager: false }))

vi.mock('./_generated/server', async () => await convexServerMock())
```

The `import.meta.glob` and `vi.mock` calls must live in consumer code (Vite resolves them relative to the file), but the actual logic is handled by the testing utilities.
