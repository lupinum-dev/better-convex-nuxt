## Recommended Test Order

Test each layer in order of dependency. If an earlier layer is broken, later tests are unreliable:

1. **Protected handler rules** — the core permission boundary
2. **Tenant isolation** — data does not leak between workspaces
3. **Principal forwarding** — forwarded principals hit the same guards
4. **Component bridge forwarding** — bridges pass identity unchanged
5. **MCP visibility and destructive preview** — tool discovery is correct

The rest of this page provides a copy-paste pattern for each.

## Test The Protected Handler First

Start by proving the Convex business boundary directly. This tests the `guard`, `load`, and `authorize` phases described in [How It Works](/docs/guide/how-it-works).

```ts
it("blocks a member from updating another member's todo", async () => {
  const ctx = createCtx()
  const team = await ctx.seedTenant({
    name: 'Alpha',
    users: {
      alice: { role: 'member' },
      bob: { role: 'member' },
    },
  })

  const todoId = await team.users.alice.mutation(api.todos.create, {
    title: 'Alice private team todo',
  })

  await expect(
    team.users.bob.mutation(api.todos.setCompleted, {
      id: todoId,
      completed: true,
    }),
  ).rejects.toThrow('Forbidden: Update todo')
})
```

If this test is weak, every later wrapper or projection test is weak too.

## Test Principal Forwarding Separately

Once the protected handler is correct, prove that a forwarded principal hits the same boundary.

```ts
it('applies the same permission rules to an agent principal', async () => {
  const ctx = createCtx()
  const team = await ctx.seedTenant({
    name: 'Alpha',
    users: {
      viewer: { role: 'viewer' },
    },
  })

  const agent = ctx.asPrincipal({
    kind: 'user',
    userId: team.users.viewer.authId,
  })

  await expect(agent.mutation(api.todos.create, { title: 'Should fail' })).rejects.toThrow(
    'Forbidden: Create todo',
  )
})
```

This is the direct replacement for older hidden transport-injection tests.

## Test Tenant Isolation

Seed two tenants and prove that data does not leak:

```ts
it('keeps tenants isolated from each other', async () => {
  const ctx = createCtx()
  const alpha = await ctx.seedTenant({
    name: 'Alpha',
    users: { alice: { role: 'member' } },
  })
  const beta = await ctx.seedTenant({
    name: 'Beta',
    users: { bruno: { role: 'member' } },
  })

  await alpha.users.alice.mutation(api.todos.create, { title: 'Alpha only' })
  await beta.users.bruno.mutation(api.todos.create, { title: 'Beta only' })

  const alphaTodos = await alpha.users.alice.query(api.todos.list, {})
  const betaTodos = await beta.users.bruno.query(api.todos.list, {})

  expect(alphaTodos).toHaveLength(1)
  expect(betaTodos).toHaveLength(1)
})
```

## Test Component Bridges As Transport Seams

For component-backed apps (see [Component Bridges](/docs/server-side/private-bridge)), add a focused test that proves the root bridge forwards principal unchanged.

Good assertions:

- the component actor is derived from the forwarded principal
- the root bridge does not rebuild or mutate principal
- query/mutation visibility matches the registered bridge ref

This is the most important integration seam in local or packaged component apps.

## Test MCP Visibility Separately From Execution

MCP tools have two access layers (see [Tool Visibility](/docs/mcp-tools/auth-and-permissions)). Test them separately:

- visibility tests for capability snapshots and `enabled(...)`
- execution tests for the actual Convex permission boundary

Example:

- anonymous caller should not see `publish-page`
- authenticated agent may see `publish-page`
- Convex still decides whether that agent can actually publish this specific page

If you collapse those concerns into one test, you usually miss stale-visibility or resource-level authorization bugs.

## Test Destructive Preview

For destructive tools (see [Destructive Tools and Previews](/docs/mcp-tools/safety-and-confirmation)), assert both halves of the flow:

- preview returns the expected confirmation summary
- confirmed execution hits the same business mutation

`examples/08-component-mini-cms` is the canonical pattern for this.

## Test Anonymous Access Explicitly

Use `ctx.raw` for truly anonymous calls:

```ts
it('allows anonymous callers on a public query', async () => {
  const ctx = createCtx()
  await expect(ctx.raw.query(api.status.health, {})).resolves.toBeDefined()
})

it('denies anonymous callers on a protected mutation', async () => {
  const ctx = createCtx()
  await expect(ctx.raw.mutation(api.todos.create, { title: 'Nope' })).rejects.toThrow('Forbidden')
})
```
