## Overview

Trellis observability is the semantic event pipeline for the framework.

It explains decisions and trust boundaries, not just request noise:

- principal and actor resolution
- guard and authorize outcomes
- `ctx.db.crossTenant` and `ctx.db.raw` usage
- service access checks and denials
- destructive operation preview, drift, and execution
- MCP tool denial, confirmation, and execution
- browser runtime auth/query/mutation/upload/connection behavior

Audit remains separate from observability.

## Example

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  trellis: {
    logging: 'info',
    observability: {
      enabled: true,
      adapter: 'dev',
      level: 'verbose',
      capture: {
        backend: true,
        mcp: true,
        browser: true,
      },
    },
  },
})
```

## Defaults

- Development defaults to observability enabled with the built-in dev sink.
- Production defaults keep backend and MCP critical events on, with browser events off unless explicitly enabled.
- Denials, destructive flows, `crossTenant`, `raw`, and tool failures are never sampled out by default.

## Event Shape

Every observation event carries a small stable envelope:

- `ts`
- `transport`
- `name`
- `status`
- `correlationId`

Common contextual fields include:

- `handler`
- `operation`
- `tool`
- `principalKind`
- `actorKind`
- `tenantId`
- `serviceId`
- `reasonCode`
- `durationMs`

## Adapter Boundary

Trellis owns the event model.

Adapters only receive already-correlated, already-redacted semantic events.
The only shipped adapter today is `adapter: 'dev'`.
