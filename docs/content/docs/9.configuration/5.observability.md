## Overview

Trellis observability is the semantic event pipeline for the framework.

It explains decisions and trust boundaries, not just request noise:

- principal and actor resolution
- guard and authorize outcomes
- `ctx.db.crossTenant` and `ctx.db.raw` usage
- service access checks and denials
- destructive operation preview, drift, and execution
- MCP tool denial, confirmation, and execution
- browser runtime auth/query/mutation/upload/connection behavior

Audit remains separate from observability.

## Example

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  trellis: {
    observability: {
      enabled: true,
      service: 'kanban-workspace',
      level: 'verbose',
      capture: {
        backend: true,
        mcp: true,
        browser: true,
      },
    },
  },
})
```

## Defaults

- Development defaults to observability enabled with `evlog` pretty output.
- Production defaults keep observability enabled with `level: 'critical'`, backend and MCP capture enabled, and browser capture disabled unless explicitly enabled.
- Denials, destructive flows, `crossTenant`, `raw`, and tool failures are never sampled out by default.

## Event Shape

Every observation event carries a small stable envelope:

- `ts`
- `transport`
- `name`
- `status`
- `correlationId`

Common contextual fields include:

- `originTransport`
- `phase`
- `requestId`
- `handler`
- `operation`
- `tool`
- `principalKind`
- `actorKind`
- `tenantId`
- `service`
- `serviceId`
- `reasonCode`
- `durationMs`
- `details`
- `details.explanation`

`transport` is the current emission boundary.
`originTransport` is the ingress boundary threaded through the flow.

Configuration fields include:

- `sample`
- `correlation.header`
- `service`
- `explainability.agentDenials`

## Delivery

Trellis owns the semantic event model and delivers it through `evlog`.
Consumer apps configure observability through `trellis.observability`; they do not configure a second logger adapter surface.
