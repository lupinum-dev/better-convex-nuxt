## Overview

Trellis runs an auth proxy at `/api/auth` that forwards authentication requests from your Nuxt app to the Convex backend. This is required because session cookies must be sent to the same domain as your app.

## Why a Proxy?

Your Nuxt app runs at `myapp.com`, but Convex runs at `your-app.convex.site`. Browsers won't send cookies cross-origin. The auth proxy:

1. Receives auth requests at `/api/auth/...` (same domain as your app)
2. Forwards them to `your-app.convex.site/api/auth/...`
3. Returns the upstream response, including `Set-Cookie` headers and JSON payloads such as sign-in results or Convex token exchange responses

## Configuration

### Custom Route

Change the proxy path (default: `/api/auth`):

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  trellis: {
    auth: {
      route: '/api/auth', // default
    },
  },
})
```

### Trusted Origins

For preview deployments with dynamic URLs, add trusted origins for CORS:

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  trellis: {
    auth: {
      trustedOrigins: ['https://*.vercel.app', 'https://staging.myapp.com'],
    },
  },
})
```

### Body Size Limits

Control the maximum request and response body sizes:

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  trellis: {
    auth: {
      proxy: {
        maxRequestBodyBytes: 1048576, // 1 MB (default)
        maxResponseBodyBytes: 1048576, // 1 MB (default)
      },
    },
  },
})
```

## Security

- The proxy only forwards requests to the resolved Convex site URL
- `CONVEX_SITE_URL` must already be the canonical Better Auth / Convex origin when auto-derivation is not enough
- It does not replay cookies or authorization headers to a different upstream origin
- It forwards `x-forwarded-host`, `x-forwarded-proto`, and, when Nitro can resolve it, `x-forwarded-for` so Better Auth can derive callback URLs and apply IP-based rate limiting correctly
- It only follows internal redirects when the redirect target stays on the resolved Convex site origin
- Critical internal auth endpoints like `/api/auth/convex/token` and `/api/auth/get-session` fail closed if upstream redirects off-origin
- Critical internal auth endpoints like `/api/auth/convex/token` and `/api/auth/get-session` only allow `GET` and `OPTIONS`; unsupported methods are rejected locally with `405`
- Malformed auth-proxy paths with traversal-like input are rejected locally with `404` instead of being rewritten and forwarded
- Request headers are forwarded appropriately
- It preserves upstream auth behavior rather than inventing a second auth API
- Client-side auth flows can still receive JSON payloads from upstream endpoints, including the Convex JWT returned by `/api/auth/convex/token`
- The proxy is intentionally transparent for upstream Better Auth JSON. If your Better Auth `/api/auth/get-session` response contains raw session details, minimize that at the app or Better Auth layer rather than expecting this proxy to rewrite it

## Better Auth IP Detection

Better Auth rate limiting keys off the client IP address. Trellis forwards `x-forwarded-for` from the Nuxt request to Convex so the default Better Auth setup works in local development and in standard reverse-proxy deployments.

That only applies to real browser and SSR requests flowing through Nuxt. It cannot invent a client IP for Convex's own internal self-requests.

If your edge stack uses a non-standard client IP header such as `cf-connecting-ip`, configure that in your Better Auth setup:

```ts [convex/auth.ts]
custom: (_ctx, bridge) =>
  betterAuth({
    baseURL: bridge.siteUrl,
    database: bridge.database,
    advanced: {
      ipAddress: {
        ipAddressHeaders: ['cf-connecting-ip', 'x-forwarded-for'],
      },
    },
    plugins: [bridge.createConvexPlugin()],
    trustedOrigins: bridge.trustedOrigins,
  })
```

If Better Auth still logs `Rate limiting skipped: could not determine client IP address` on normal sign-in, sign-up, token, or session requests, your upstream proxy is not passing a client IP header through to Nuxt or Convex.

For local Trellis examples, the remaining warning usually comes from Convex verifying JWTs through its own live JWKS fetch. The example launcher now switches local auth config to static `JWKS` after boot so that internal verification path does not hit `/api/auth/convex/jwks` at all.
