## Overview

Better Convex Nuxt runs an auth proxy at `/api/auth` that forwards authentication requests from your Nuxt app to the Convex backend. This is required because session cookies must be sent to the same domain as your app.

## Why a Proxy?

Your Nuxt app runs at `myapp.com`, but Convex runs at `your-app.convex.site`. Browsers won't send cookies cross-origin. The auth proxy:

1. Receives auth requests at `/api/auth/...` (same domain as your app)
2. Forwards them to `your-app.convex.site/api/auth/...`
3. Returns the upstream response, including `Set-Cookie` headers and JSON payloads such as sign-in results or Convex token exchange responses

## Configuration

### Custom Route

Change the proxy path (default: `/api/auth`):

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  convex: {
    auth: {
      route: '/api/auth' // default
    }
  }
})
```

### Trusted Origins

For preview deployments with dynamic URLs, add trusted origins for CORS:

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  convex: {
    auth: {
      trustedOrigins: [
        'https://*.vercel.app',
        'https://staging.myapp.com'
      ]
    }
  }
})
```

### Body Size Limits

Control the maximum request and response body sizes:

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  convex: {
    auth: {
      proxy: {
        maxRequestBodyBytes: 1048576,  // 1 MB (default)
        maxResponseBodyBytes: 1048576  // 1 MB (default)
      }
    }
  }
})
```

## Security

- The proxy only forwards requests to the configured Convex site URL
- It follows canonical redirects from the Convex backend
- Request headers are forwarded appropriately
- It preserves upstream auth behavior rather than inventing a second auth API
- Client-side auth flows can still receive JSON payloads from upstream endpoints, including the Convex JWT returned by `/api/auth/convex/token`
