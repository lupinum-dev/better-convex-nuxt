## Descriptions Are Prompts

Your tool's `description` is the first thing an agent reads when deciding whether to call it. Write it for an AI that has never seen your codebase:

```ts
// Bad — assumes context
{ description: 'Delete a note' }

// Good — tells the agent what, when, and consequences
{ description: 'Permanently delete a note by ID. Cannot be undone. Returns the deleted note\'s title for confirmation.' }
```

Field descriptions matter equally:

```ts
fields: {
  id: { description: 'The Convex ID of the note to delete (from list-notes or search-notes results)' },
  query: { description: 'Text to search for in note titles and content. Case-insensitive partial match.' },
}
```

## Structured Data Over Free Text

Agents parse JSON reliably. They struggle with unstructured prose. Always return structured data:

```ts
// Bad — agent has to parse prose
handler: async (args) => {
  return `Created note "${args.title}" with ID abc123`
}

// Good — structured data + human summary
handler: async (args) => {
  const noteId = await serverConvexMutation(api.notes.add, args)
  return withSummary(
    { id: noteId, title: args.title },
    `Created note "${args.title}"`,
  )
}
```

Use `outputSchema` when the agent needs to know the response shape in advance:

```ts
outputSchema: {
  format: z.enum(['json', 'csv']),
  content: z.string(),
  noteCount: z.number(),
},
```

## One Action Per Tool

Each tool should do one thing. Don't build multi-mode tools:

```ts
// Bad — agent has to figure out modes
defineConvexTool({
  schema: defineConvexSchema({
    action: v.union(v.literal('create'), v.literal('update'), v.literal('delete')),
    id: v.optional(v.string()),
    title: v.optional(v.string()),
  }),
  handler: ...
})

// Good — separate tools, clear purpose
// create-note.ts → creates a note
// update-note.ts → updates a note
// delete-note.ts → deletes a note
```

Each tool file maps to one capability. Agents compose tools into workflows — you provide the atoms.

## ID Handling Patterns

Agents chain tool calls: list → get → update → delete. IDs flow between calls:

1. **List tools** return IDs in their response
2. **Action tools** accept IDs as input
3. **Field descriptions** tell the agent where IDs come from

```ts
fields: {
  id: {
    description: 'The Convex ID of the note (from list-notes or search-notes results)',
  },
}
```

Since `v.id()` is auto-converted to `z.string()`, agents pass IDs as plain strings. The Convex backend validates them.

## Error Messages As Instructions

Error messages should tell the agent what to do next:

```ts
// Bad — agent doesn't know what to do
return wrapError('validation', 'Invalid format')

// Good — actionable instruction
return wrapError('validation', 'Invalid format. Use "json" or "csv".')

// Good — tells agent to retry differently
return wrapError('scope_exceeded', 'Cannot process more than 10 items. Split into smaller batches.')
```

The `retryable` flag + `category` already guide agent behavior. The `message` should complete the picture.

## The Destructive-by-Default Mindset

When designing tools, assume they will be called by an agent that might:
- Hallucinate IDs that don't exist
- Call delete instead of update
- Send maximum-size payloads
- Retry failed calls indefinitely

Protect against each:

| Risk | Protection |
|------|-----------|
| Wrong IDs | `preview` handler validates resources exist |
| Accidental deletion | `destructive: true` forces confirmation |
| Oversized requests | `maxItems` caps array lengths |
| Retry storms | `rateLimit` throttles calls |
| Wrong permissions | `auth: 'required'` + `require` |

## Testing with Agents

### curl Recipes

Test the full flow manually:

```bash
# 1. Initialize
SESSION=$(curl -s http://localhost:3000/mcp \
  -H 'Content-Type: application/json' \
  -H 'Accept: application/json, text/event-stream' \
  -d '{"jsonrpc":"2.0","id":1,"method":"initialize","params":{"protocolVersion":"2025-03-26","capabilities":{},"clientInfo":{"name":"test","version":"1.0"}}}' \
  | jq -r '.result.meta.sessionId // empty')

# 2. List tools — check descriptions, annotations, input schemas
curl -s http://localhost:3000/mcp \
  -H "Mcp-Session-Id: $SESSION" \
  -H 'Content-Type: application/json' \
  -H 'Accept: application/json, text/event-stream' \
  -d '{"jsonrpc":"2.0","id":2,"method":"tools/list"}' | jq '.result.tools[] | {name, description}'

# 3. Call a destructive tool (first call — should get preview/confirmation)
curl -s http://localhost:3000/mcp \
  -H "Mcp-Session-Id: $SESSION" \
  -H 'Content-Type: application/json' \
  -H 'Accept: application/json, text/event-stream' \
  -d '{"jsonrpc":"2.0","id":3,"method":"tools/call","params":{"name":"delete-note","arguments":{"id":"test123"}}}' | jq .

# 4. Confirm the destructive action
curl -s http://localhost:3000/mcp \
  -H "Mcp-Session-Id: $SESSION" \
  -H 'Content-Type: application/json' \
  -H 'Accept: application/json, text/event-stream' \
  -d '{"jsonrpc":"2.0","id":4,"method":"tools/call","params":{"name":"delete-note","arguments":{"id":"test123","_confirmed":true}}}' | jq .
```

### What to Check

- **tools/list** — are descriptions clear? Are annotations correct?
- **Success path** — does the response contain all the data an agent needs?
- **Error path** — are error messages actionable?
- **Destructive flow** — does preview show meaningful info? Does confirmation gate work?
- **Auth errors** — does the error tell the agent how to authenticate?

## Agent-Ready Checklist

Before shipping a tool, verify:

- [ ] **Description** is clear to an AI with zero context about your app
- [ ] **Field descriptions** explain what each input means and where to get it
- [ ] **Return data** is structured (not just a string message)
- [ ] **Error messages** tell the agent what to do next
- [ ] **Destructive actions** use `destructive: true` with `preview`
- [ ] **Bulk operations** have `maxItems` caps
- [ ] **Expensive operations** have `rateLimit`
- [ ] **Auth** is set correctly (`required` for writes, `optional` or `none` for reads)
- [ ] **Operation type** matches reality (`query` for reads, `mutation` for writes, `action` for external calls)
- [ ] **Testing** — verified via curl that all paths (success, error, preview, auth failure) return useful responses
