Most Trellis MCP apps should spend most of their time in:

- `defineMcpApp(...)`
- `tool(...)`
- Convex business logic

This page is for the narrower cases that need extra MCP-layer behavior.

## Middleware Is Mostly For Standalone Tools

`defineTool(...)` supports middleware for Nitro-native tools:

```ts
export default defineTool({
  name: 'check-dns',
  schema,
  middleware: async (_args, _ctx, next) => {
    console.log('[audit] check-dns')
    return next()
  },
  handler: async (args, ctx) => {
    return ctx.ok({ hostname: args.hostname })
  },
})
```

For Convex-backed tools, prefer keeping that logic in Convex unless it is truly transport-specific.

## `enabled(...)` For Projection Runtime Gates

Projected tools can still use `enabled(...)`:

```ts
export default tool({
  call: internal.posts.publishPost,
  schema: publishPost,
  capability: 'publishPost',
  enabled: ({ runtime }) => runtime.flags.publishing !== false,
  meta: {
    name: 'publish-post',
  },
})
```

Good uses:

- feature flags
- deployment mode
- provider outages
- temporary kill switches

Bad uses:

- recreating Convex authorization
- loading records in Nitro to decide whether execution should be allowed

## Operation Types

- `query`: read-only
- `mutation`: Convex write
- `action`: may call external systems

These matter mainly for direct callers and annotations. They should not change your business-authorization design.

## Tags And Group

`tags` and `group` are classification metadata. Keep them human-meaningful and reviewable. Do not treat them as a second permissions system.
