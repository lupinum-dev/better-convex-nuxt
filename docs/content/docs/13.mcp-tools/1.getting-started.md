By the end of this page, you will have an MCP tool that lets an AI agent list published pages from your Convex database — using the same protected query your browser UI uses.

::tip
If you want to see the complete working code, open the [mini CMS example](https://github.com/lupinum-dev/trellis/tree/main/examples/08-component-mini-cms) alongside this page.
::

## Prerequisites

You need a Trellis app with authentication and at least one protected Convex handler. If you have not set that up yet, start with [Get Started](/docs/guide/get-started) and [Authentication](/docs/auth-security/authentication).

## Step 1: Install the MCP Module

Add the MCP toolkit to your project:

```bash
pnpm add @nuxtjs/mcp-toolkit
```

Enable it in `nuxt.config.ts`:

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  modules: ['@lupinum/trellis', '@nuxtjs/mcp-toolkit'],
})
```

## Step 2: Set Up the File Layout

MCP tools live in `server/mcp/tools/`. The shared runtime definition goes in `server/lib/`:

```text
server/
  lib/
    mcp-auth.ts          ← principal resolution + capabilities
    mcp-runtime.ts       ← MCP runtime definition
  mcp/
    index.ts             ← MCP handler (stays tiny)
    tools/
      list-published-pages.ts   ← your first tool
```

## Step 3: Resolve Identity and Capabilities

Before defining tools, you need two functions: one that resolves **who is calling** from the MCP request, and one that decides **what they can see**.

```ts [server/lib/mcp-auth.ts]
import { getHeader, type H3Event } from 'h3'
import type { MiniCmsPrincipal } from '~/shared/principal'

export type CapabilitySnapshot = {
  listPublishedPages: boolean
  createPage: boolean
  publishPage: boolean
}

function readBearerToken(event: H3Event): string | null {
  const auth = getHeader(event, 'authorization')
  if (!auth?.startsWith('Bearer ')) return null
  return auth.slice('Bearer '.length).trim() || null
}

export function getMcpPrincipal(event: H3Event): MiniCmsPrincipal {
  const token = readBearerToken(event)
  if (!token) return { kind: 'anonymous' }
  // In production, validate the token against your auth provider
  return { kind: 'agent', agentId: 'demo-agent' }
}

export function getCapabilitiesForPrincipal(principal: MiniCmsPrincipal): CapabilitySnapshot {
  return {
    listPublishedPages: true, // everyone can list published pages
    createPage: principal.kind === 'agent',
    publishPage: principal.kind === 'agent',
  }
}
```

`getMcpPrincipal` reads the bearer token and returns a principal — the same principal type your Convex handlers expect. `getCapabilitiesForPrincipal` returns which tools this caller can see. Anonymous callers see `listPublishedPages` only; authenticated agents see everything.

## Step 4: Define the MCP Runtime

The runtime wires together identity resolution, Convex calling, and capability visibility:

```ts [server/lib/mcp-runtime.ts]
import { defineMcpApp } from '@lupinum/trellis/mcp'
import { createServerConvexCaller } from '@lupinum/trellis/server'
import type { MiniCmsPrincipal } from '~/shared/principal'
import { getCapabilitiesForPrincipal, getMcpPrincipal, type CapabilitySnapshot } from './mcp-auth'

export const mcpRuntime = defineMcpApp<MiniCmsPrincipal, CapabilitySnapshot>({
  callConvex: async (event) => createServerConvexCaller(event),
  resolvePrincipal: async (event) => getMcpPrincipal(event),
  resolveCapabilities: async ({ principal }) => getCapabilitiesForPrincipal(principal),
})

export const tool = mcpRuntime.tool
```

Three things happen here:

- **`callConvex`** tells the runtime how to call Convex from Nitro. `createServerConvexCaller(event)` is the small mechanical helper for the common `auth: 'none'` transport case.
- **`resolvePrincipal`** extracts who is calling from the HTTP request.
- **`resolveCapabilities`** decides which tools are visible based on the resolved principal.

## Step 5: Project Your First Tool

A **projected tool** points at an existing Convex function. You do not write business logic in the tool — Convex handles that.

```ts [server/mcp/tools/list-published-pages.ts]
import { internal } from '#trellis/api'
import { listPublishedPages } from '~/shared/schemas/page'
import { tool } from '~/server/lib/mcp-runtime'

export default tool({
  schema: listPublishedPages,
  call: internal.miniCmsBridge.listPublishedPages,
  operation: 'query',
  capability: 'listPublishedPages',
  group: 'pages',
  meta: {
    name: 'list-published-pages',
    description: 'List the public pages that anonymous users can already read.',
  },
})
```

Here is what each field does:

- **`schema`** — the shared argument schema (from `shared/schemas/page.ts`)
- **`call`** — the Convex function to invoke (here, a bridge query into the CMS component)
- **`operation`** — `'query'` for reads, `'mutation'` for writes
- **`capability`** — which capability key gates this tool's visibility
- **`meta.name`** — the tool name the AI agent sees
- **`meta.description`** — what the agent sees when discovering tools

## Step 6: Wire the Handler

The MCP handler file stays minimal:

```ts [server/mcp/index.ts]
export default defineMcpHandler({
  name: 'my-app',
  browserRedirect: '/',
})
```

Most of the work lives in the runtime definition and the individual tool modules.

## How It Works

Now that you have built a tool, here is what happens when an AI agent calls it:

1. The agent sends a tool call to your MCP endpoint.
2. The MCP runtime calls `resolvePrincipal` to figure out who is calling.
3. It checks `resolveCapabilities` to confirm this caller can see `list-published-pages`.
4. It calls the Convex bridge query with `{ ...args, principal }`.
5. The Convex handler runs its guard, loads data, and returns results.
6. The MCP runtime wraps the result and sends it back to the agent.

The agent gets the same data a browser user would see — because the same Convex handler ran with the same permission checks.

## Adding a Write Tool

Once your read-only tool works, add a mutation tool. Here is `create-page`:

```ts [server/mcp/tools/create-page.ts]
import { internal } from '#trellis/api'
import { createPage } from '~/shared/schemas/page'
import { tool } from '~/server/lib/mcp-runtime'

export default tool({
  schema: createPage,
  call: internal.miniCmsBridge.createPage,
  operation: 'mutation',
  capability: 'createPage',
  group: 'pages',
  meta: {
    name: 'create-page',
    description: 'Create a new draft page in the CMS.',
  },
})
```

The only differences from the query tool: `operation: 'mutation'` and a different capability key. The guard in Convex (`canManagePages`) blocks viewers automatically — you do not need to duplicate that check here.

## When to Use `defineTool(...)` Instead

Use `defineTool(...)` only for tools that are **Nitro-native** and do not call Convex:

```ts [server/mcp/tools/check-dns.ts]
export default defineTool({
  name: 'check-dns',
  schema: checkDnsSchema,
  handler: async (args, ctx) => {
    const records = await queryDns(args.hostname)
    return ctx.ok({ records })
  },
})
```

For everything backed by Convex business logic, use `tool(...)`.
