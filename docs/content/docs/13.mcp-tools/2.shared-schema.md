## The Problem

Without shared schemas, you define the same fields three times: in Convex functions, in your frontend forms, and in MCP tool input. They drift apart. `defineConvexSchema` solves this.

## defineConvexSchema

```ts
// shared/schemas/note.ts
import type { ConvexSchemaMetaFor } from 'better-convex-nuxt/schema'
import { v } from 'convex/values'
import type { PropertyValidators } from 'convex/values'

export const createNoteArgs = {
  title: v.string(),
  content: v.string(),
} satisfies PropertyValidators

export const createNoteMeta = {
  description: 'Create a new note',
  fields: {
    title: { label: 'Title', description: 'The note title', examples: ['Meeting Notes', 'Quick Idea'] },
    content: { label: 'Content', description: 'The note body text', examples: ['# My Note\nSome content here'] },
  },
} satisfies ConvexSchemaMetaFor<typeof createNoteArgs>
```

Then use it everywhere:

```ts
// In a Convex function
const schema = defineConvexSchema(createNoteArgs, createNoteMeta)

// In an MCP tool
export default defineConvexTool({ schema, handler: ... })
```

The `satisfies ConvexSchemaMetaFor<typeof args>` ensures the meta fields match the validators at compile time — you can't add metadata for a field that doesn't exist.

## Field Metadata

Metadata tells AI agents what your fields mean and how to use them:

```ts
fields: {
  title: {
    label: 'Title',              // Human-readable label (for forms)
    description: 'The note title', // Sent to MCP as field description
    examples: ['Meeting Notes'],   // Auto-generates inputExamples
  },
  format: {
    label: 'Format',
    description: 'Export format',
    enum: ['json', 'csv'],        // Documents valid values for the agent
  },
  limit: {
    description: 'Max results',
    defaultHint: 10,              // Tells agent what happens if omitted
  },
}
```

These map to JSON Schema properties that MCP clients read:

| Meta field | JSON Schema effect |
|------------|-------------------|
| `description` | `description` on the field schema |
| `examples` | First example used for auto-generated `inputExamples` |
| `enum` | Appended to description as "One of: json, csv" |
| `defaultHint` | Appended to description as "Default: 10" |
| `label` | Not sent to MCP (used for form rendering) |

## v.id() Handling

Convex `v.id('table')` validators can't serialize to JSON Schema directly (they use `z.custom()` internally). `defineConvexTool` patches them automatically:

| Convex validator | Zod replacement |
|-----------------|-----------------|
| `v.id('notes')` | `z.string().describe('Convex ID for "notes" table')` |
| `v.array(v.id('notes'))` | `z.array(z.string())` |
| `v.optional(v.id('notes'))` | `z.string().optional()` |

This happens transparently — you write `v.id('notes')` in your schema and the MCP tool accepts strings.

### Restriction: v.union() with v.id()

A `v.union()` containing `v.id()` at any nesting depth **throws at definition time**:

```ts
// This throws:
const schema = defineConvexSchema({
  ref: v.union(v.id('posts'), v.literal('none')),
})

// Fix: use v.string() instead
const schema = defineConvexSchema({
  ref: v.union(v.string(), v.literal('none')),
})
```

The error is intentional — there's no safe way to auto-convert a union containing `v.id()` to JSON Schema. Use `v.string()` and document the expected format in the field metadata.

## Explicit Input Examples

By default, `defineConvexTool` auto-generates input examples from field metadata. You can override them:

```ts
export default defineConvexTool({
  schema,
  name: 'search-notes',
  operation: 'query',

  // Override auto-generated examples
  inputExamples: [
    { query: 'meeting' },
    { query: 'TODO' },
    { query: 'project update' },
  ],

  handler: async (args) => { ... },
})
```

Explicit examples are useful when auto-generated ones don't capture real-world usage patterns well.

## Sharing Schema Files

Place shared schemas in a `shared/` directory that both Convex functions and Nuxt code can import:

```
shared/
  schemas/
    note.ts       # createNoteArgs, createNoteMeta, etc.
    post.ts       # createPostArgs, createPostMeta, etc.
convex/
  notes.ts        # imports from shared/schemas/note
server/
  mcp/tools/
    create-note.ts  # imports from shared/schemas/note
```

The schema files must only depend on `convex/values` and `better-convex-nuxt/schema` types — no server or client imports.
