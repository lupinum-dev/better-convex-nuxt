---
title: 'Tool Visibility and Authorization'
description: 'Control which tools each caller can see, and let Convex handle the actual permission checks.'
---

MCP tools have two separate access concerns:

1. **Visibility** — should this tool appear in the agent's tool list?
2. **Authorization** — when the agent calls the tool, is the action actually allowed?

Visibility is fast and cheap — it runs in Nitro when tools are listed. Authorization is thorough — it runs in Convex when the tool is called, with full access to the database and the loaded resource.

## Capability-Based Visibility

The most common way to control visibility is with a `capability` key. Each projected tool declares which capability it requires:

```ts [server/mcp/tools/publish-page.ts]
export default projectTool({
  schema: publishPage,
  call: internal.miniCmsBridge.publishPage,
  capability: 'publishPage',
  meta: {
    name: 'publish-page',
    description: 'Publish the selected draft page to the public site.',
    destructive: true,
  },
})
```

The MCP runtime checks this against the capability snapshot returned by `resolveCapabilities`:

```ts [server/lib/mcp-auth.ts]
export function getCapabilitiesForPrincipal(principal: MiniCmsPrincipal): CapabilitySnapshot {
  return {
    listPublishedPages: true,
    createPage: principal.kind === 'agent',
    publishPage: principal.kind === 'agent',
  }
}
```

An anonymous caller sees `list-published-pages` only. An authenticated agent sees all three tools. This is actor-level and fast — no database queries needed.

::tip{title="Visibility is not authorization"}
Capability snapshots are transport-level visibility and coarse access. They help decide what an agent can discover or attempt. They do **not** replace Convex business authorization, and they should not become your domain permission model.
::

## The `enabled` Gate

Use `enabled(...)` for visibility decisions that are not about permissions:

```ts [server/mcp/tools/publish-post.ts]
export default projectTool({
  schema: publishPost,
  call: internal.posts.publishPost,
  capability: 'publishPost',
  enabled: ({ runtime }) => runtime.flags.publishing !== false,
  meta: {
    name: 'publish-post',
    description: 'Publish a draft post.',
  },
})
```

Good uses for `enabled`:

- **Feature flags** — hide tools behind a rollout flag
- **Deployment mode** — disable write tools in staging
- **Provider availability** — hide tools when an external service is down
- **Temporary kill switches** — quickly disable a tool without redeploying

`enabled` runs after the capability check. If the capability already hides the tool, `enabled` is not evaluated.

## Authorization Stays in Convex

Visibility answers "should this tool appear?" Authorization answers "is this specific action allowed?" These are separate — a tool can be visible but still reject the call.

Authorization runs inside the Convex handler through the standard pipeline:

```ts [convex/components/miniCms/pages.ts]
const publishPageOp = defineOperation({
  args: publishPage.args,
  guard: canManagePages,
  load: async (ctx, args) => {
    const page = await ctx.db.get(args.id)
    requireRecord(page, 'Page')
    return { page }
  },
  handler: async (ctx, _args, { page }) => {
    // Only reached if guard passes and page exists
    await ctx.db.patch(page._id, { status: 'published' })
    return { pageId: page._id, published: true }
  },
})
```

The guard checks the actor, the load fetches the resource, and the handler runs the mutation. This is the same logic whether the call came from a browser, a server route, or an MCP tool.

That same boundary applies if you later adopt Agent Auth or another protocol. The transport may grant access to a route or capability, but the protected Convex handler still owns the real permission decision.

## Anonymous and Public Tools

Not every tool needs authentication. Read-only, public-data tools can skip the capability check:

```ts [server/mcp/tools/list-published-pages.ts]
export default projectTool({
  schema: listPublishedPages,
  call: internal.miniCmsBridge.listPublishedPages,
  operation: 'query',
  capability: 'listPublishedPages', // always true in capabilities
  meta: {
    name: 'list-published-pages',
    description: 'List published pages visible to everyone.',
  },
})
```

In the capabilities function, set `listPublishedPages: true` for all callers. The Convex handler uses `guard: open`, which allows any caller.
