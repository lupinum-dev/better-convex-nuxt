## Auth Modes

Every tool has one of three auth modes:

| Mode | Behavior |
|------|----------|
| `auth: 'none'` | No auth check (default). Anyone can call the tool. |
| `auth: 'optional'` | Auth is resolved if present, but not required. Tool works for everyone. |
| `auth: 'required'` | Must be authenticated. Returns `category: 'auth'` error if not. |

```ts
// Public tool — no auth needed
export default defineConvexTool({
  schema,
  auth: 'none', // default
  handler: async (args) => { ... },
})

// Auth optional — identity available in middleware if present
export default defineConvexTool({
  schema,
  auth: 'optional',
  handler: async (args) => { ... },
})

// Auth required — blocks unauthenticated requests
export default defineConvexTool({
  schema,
  auth: 'required',
  handler: async (args) => { ... },
})
```

## McpAuthIdentity

When auth is resolved, the framework provides an `McpAuthIdentity`:

```ts
interface McpAuthIdentity {
  readonly role: string
  readonly userId: string
}
```

This is available in middleware via `ctx.mcpAuth` and in the permission system.

## Resolving Auth

By default, `defineConvexTool` reads `event.context.mcpAuth` from the H3 event. You set this up with a server middleware that extracts identity from the request (e.g., from a JWT header):

```ts
// server/middleware/mcp-auth.ts
export default defineEventHandler((event) => {
  // Only process MCP requests
  if (!event.path.startsWith('/mcp')) return

  const token = getHeader(event, 'authorization')?.replace('Bearer ', '')
  if (!token) return

  // Verify and decode your token
  const decoded = verifyJwt(token)
  if (decoded) {
    event.context.mcpAuth = {
      role: decoded.role,
      userId: decoded.sub,
    }
  }
})
```

### Custom resolveAuth

For more control, pass a custom resolver to `createConvexTools`:

```ts
export const { defineConvexTool } = createConvexTools({
  checkPermission,
  resolveAuth: async (event) => {
    const session = await getSession(event)
    if (!session) return null
    return { role: session.role, userId: session.userId }
  },
})
```

## Permission Checks

Use `require` to check a specific permission before the handler runs:

```ts
export default defineConvexTool({
  schema,
  auth: 'required',
  require: 'post.create',
  handler: async (args) => { ... },
})
```

This needs a `checkPermission` function, which you provide via `createConvexTools`.

### Factory Setup

```ts
// server/mcp/utils/tools.ts
import { createConvexTools } from 'better-convex-nuxt/mcp'
import { checkPermission } from '~~/convex/permissions.config'

export const { defineConvexTool } = createConvexTools({
  checkPermission,
})
```

Now import `defineConvexTool` from your factory instead of from the module:

```ts
// server/mcp/tools/create-note.ts
import { defineConvexTool } from '../utils/tools'

export default defineConvexTool({
  schema,
  auth: 'required',
  require: 'post.create', // ← type-checked against your permission strings
  handler: async (args) => { ... },
})
```

The factory gives you typed autocomplete on `require` — typos are caught at compile time.

## Ownership-Based Permissions

For fine-grained access control (e.g., "members can edit their own posts, admins can edit any"), use middleware with `ctx.can()`:

```ts
export default defineConvexTool({
  schema,
  name: 'update-note',
  auth: 'required',

  middleware: async (args, ctx, next) => {
    const note = await serverConvexQuery(api.notes.get, { id: args.id })
    if (!note) {
      return wrapError('not_found', `Note "${args.id}" not found.`)
    }

    // ctx.can() checks against your permission config with ownership
    const allowed = ctx.can('post.update', { ownerId: note.userId })
    if (!allowed) {
      return wrapError('auth', 'You do not have permission to update this note.')
    }

    return next()
  },

  handler: async (args) => {
    await serverConvexMutation(api.notes.update, args)
    return { updated: true, id: args.id }
  },
})
```

The difference from `require`:
- `require: 'post.update'` — simple role-based check (does this role have this permission?)
- `ctx.can('post.update', resource)` — ownership-aware check (can this user act on this specific resource?)

## Error Responses

Auth and permission failures return structured errors:

```json
{
  "ok": false,
  "error": {
    "category": "auth",
    "message": "Authentication required.",
    "retryable": true
  }
}
```

The `retryable: true` tells the agent it can retry after obtaining credentials. Permission denied (`category: 'auth'`) is also retryable — the agent might switch to a different auth context.

## Definition-Time Validations

`defineConvexTool` catches configuration errors early:

- `require` without `checkPermission` → throws (need `createConvexTools` factory)
- `require` with `auth: 'none'` → throws (permissions need auth)
