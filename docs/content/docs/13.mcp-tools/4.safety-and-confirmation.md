## Why Safety Matters for AI Agents

AI agents call your tools autonomously. A tool that deletes data without guardrails is one hallucination away from disaster. `defineConvexTool` provides progressive safety features that protect both the user and the agent.

## Destructive Confirmation Flow

Mark a tool as `destructive: true` to enable the two-call confirmation pattern:

```ts
export default defineConvexTool({
  schema,
  name: 'delete-note',
  destructive: true,
  handler: async (args) => {
    await serverConvexMutation(api.notes.remove, args)
    return { deleted: true, id: args.id }
  },
})
```

This automatically adds a `_confirmed` boolean to the input schema. The flow:

**First call** (without `_confirmed`):
```json
// Request
{ "name": "delete-note", "arguments": { "id": "abc123" } }

// Response
{
  "ok": false,
  "error": {
    "category": "confirmation_required",
    "message": "This action is destructive. Call again with _confirmed: true to proceed.",
    "retryable": true
  }
}
```

**Second call** (with `_confirmed: true`):
```json
// Request
{ "name": "delete-note", "arguments": { "id": "abc123", "_confirmed": true } }

// Response
{ "ok": true, "data": { "deleted": true, "id": "abc123" } }
```

The agent must explicitly confirm — it can't accidentally execute a destructive action.

## Preview

Add a `preview` handler to show the agent what will happen before it confirms:

```ts
export default defineConvexTool({
  schema,
  name: 'delete-note',
  destructive: true,

  preview: async (args) => {
    const note = await serverConvexQuery(api.notes.get, { id: args.id })
    if (!note) {
      return { summary: 'Note not found', blocked: true }
    }
    return {
      summary: `Will permanently delete "${note.title}"`,
      affects: { notes: 1 },
    }
  },

  handler: async (args) => {
    await serverConvexMutation(api.notes.remove, args)
    return { deleted: true, id: args.id }
  },
})
```

The preview response:

```json
{
  "ok": true,
  "preview": {
    "summary": "Will permanently delete \"Meeting Notes\"",
    "affects": { "notes": 1 }
  },
  "awaitingConfirmation": true
}
```

### PreviewResult Fields

| Field | Type | Purpose |
|-------|------|---------|
| `summary` | `string` | What will happen (shown to agent/user) |
| `blocked` | `boolean` | If `true`, the action cannot proceed (e.g., resource not found) |
| `affects` | `Record<string, number>` | What resources and how many are affected |
| `warn` | `string` | Warning message (e.g., "2 IDs not found and will be skipped") |

You can also return a plain string from `preview` — it's treated as just the summary.

## Bulk Operations: maxItems

Cap array sizes for bulk operations:

```ts
export default defineConvexTool({
  schema,
  name: 'bulk-delete-notes',
  destructive: true,

  maxItems: { field: 'ids', limit: 10 },

  handler: async (args) => {
    // args.ids is guaranteed to have <= 10 items
    for (const id of args.ids) {
      await serverConvexMutation(api.notes.remove, { id })
    }
    return { deleted: args.ids.length }
  },
})
```

If the agent sends more than 10 items:

```json
{
  "ok": false,
  "error": {
    "category": "scope_exceeded",
    "message": "Cannot process more than 10 items at once. Received 25.",
    "retryable": true
  }
}
```

The `retryable: true` tells the agent to split the request into smaller batches.

## Rate Limiting

Protect expensive operations with in-memory rate limiting:

```ts
export default defineConvexTool({
  schema,
  name: 'bulk-delete-notes',
  rateLimit: { max: 5, window: '1m' },
  handler: async (args) => { ... },
})
```

Window format: `'30s'`, `'1m'`, `'5m'`, `'1h'`.

When the limit is hit:

```json
{
  "ok": false,
  "error": {
    "category": "cooldown",
    "message": "Rate limit exceeded (5 per 1m). Try again in 42 seconds.",
    "retryable": true
  }
}
```

::callout{type="info"}
The rate limiter is in-memory and per-tool-name. It resets on server restart. For production rate limiting, use an external store.
::

## Full Example: Bulk Delete with All Safety Features

```ts
import { defineConvexSchema } from 'better-convex-nuxt/schema'
import { serverConvexMutation, serverConvexQuery } from 'better-convex-nuxt/server'
import { api } from '~~/convex/_generated/api'
import { defineConvexTool } from '../utils/tools'

const schema = defineConvexSchema(bulkDeleteNotesArgs, bulkDeleteNotesMeta)

export default defineConvexTool({
  schema,
  name: 'bulk-delete-notes',
  auth: 'required',
  require: 'post.delete',
  destructive: true,
  tags: ['bulk', 'dangerous'],
  rateLimit: { max: 5, window: '1m' },
  maxItems: { field: 'ids', limit: 10 },

  middleware: async (args, ctx, next) => {
    console.log(`[audit] bulk-delete: user=${ctx.mcpAuth?.userId} deleting ${args.ids.length} notes`)
    return next()
  },

  preview: async (args) => {
    const notes = await Promise.all(
      args.ids.map(id => serverConvexQuery(api.notes.get, { id })),
    )
    const found = notes.filter(Boolean)
    const missing = args.ids.length - found.length

    if (found.length === 0) {
      return { summary: 'None of the specified notes exist', blocked: true }
    }

    const titles = found.map(n => `"${n!.title}"`).join(', ')
    return {
      summary: `Will permanently delete ${found.length} note(s): ${titles}`,
      warn: missing > 0 ? `${missing} ID(s) not found and will be skipped` : undefined,
      affects: { notes: found.length },
    }
  },

  handler: async (args) => {
    let deleted = 0
    const skipped: { id: string; reason: string }[] = []

    for (const id of args.ids) {
      try {
        await serverConvexMutation(api.notes.remove, { id })
        deleted++
      }
      catch (err) {
        const reason = err instanceof Error ? err.message : String(err)
        console.warn(`[bulk-delete-notes] Failed to delete ${id}: ${reason}`)
        skipped.push({ id, reason })
      }
    }

    return { deleted, skipped, total: args.ids.length }
  },
})
```

## Pipeline Execution Order

The safety pipeline runs in this order:

1. **Auth check** — is the user authenticated?
2. **Permission check** — does the user have the required permission?
3. **Rate limit** — has the limit been exceeded? (after auth, so failed-auth requests don't consume tokens)
4. **Max items** — is the array within bounds?
5. **Middleware** — custom checks, audit logging
6. **Preview** — (destructive + first call) show what will happen
7. **Confirmation gate** — (destructive + no `_confirmed`) block execution
8. **Handler** — execute the operation
