## The Envelope Contract

Every `defineConvexTool` response follows a consistent envelope structure. Agents never need to guess whether a call succeeded.

### Success

```json
{
  "ok": true,
  "data": { "id": "abc123", "title": "Meeting Notes" }
}
```

### Error

```json
{
  "ok": false,
  "error": {
    "category": "not_found",
    "message": "Note \"abc123\" not found.",
    "retryable": false
  }
}
```

### Preview (confirmation pending)

```json
{
  "ok": true,
  "preview": {
    "summary": "Will permanently delete \"Meeting Notes\"",
    "affects": { "notes": 1 }
  },
  "awaitingConfirmation": true
}
```

## wrapSuccess

You don't call this directly — the framework wraps your handler's return value automatically:

```ts
handler: async (args) => {
  const note = await serverConvexQuery(api.notes.get, { id: args.id })
  return note // → { ok: true, data: note }
}
```

The raw return value goes into `structuredContent.data`. A JSON stringification goes into `content[0].text` as a fallback for MCP clients that don't support structured content.

## withSummary

For a human-readable text response alongside structured data:

```ts
import { withSummary } from 'better-convex-nuxt/mcp'

handler: async (args) => {
  const results = await serverConvexQuery(api.notes.search, { query: args.query })
  return withSummary(
    { results, total: results.length },
    `Found ${results.length} notes matching "${args.query}"`,
  )
}
```

The summary becomes the `content[0].text` (what the agent reads as prose), while the structured data goes into `structuredContent.data` (what the agent parses programmatically).

## wrapError

Return categorized errors from middleware or custom logic:

```ts
import { wrapError } from 'better-convex-nuxt/mcp'

middleware: async (args, ctx, next) => {
  const note = await serverConvexQuery(api.notes.get, { id: args.id })
  if (!note) {
    return wrapError('not_found', `Note "${args.id}" not found.`)
  }
  return next()
}
```

You can also pass field-level issues:

```ts
return wrapError('validation', 'Invalid input.', [
  { field: 'email', message: 'Must be a valid email address' },
])
```

## Error Categories

| Category | Retryable | When |
|----------|-----------|------|
| `auth` | Yes | Authentication required or failed |
| `forbidden` | No | Permission denied |
| `not_found` | No | Resource doesn't exist |
| `validation` | Yes | Invalid input (agent can fix and retry) |
| `rate_limit` | Yes | Too many requests |
| `scope_exceeded` | Yes | Array too large (use smaller batch) |
| `confirmation_required` | Yes | Destructive action needs `_confirmed: true` |
| `cooldown` | Yes | Rate limit with retry-after info |
| `network` | Yes | Network/connectivity issue |
| `server` | Yes | Server-side error |
| `conflict` | Yes | Concurrent modification conflict |
| `internal` | No | Unexpected internal error |

Retryable categories tell the agent it can try again (possibly with modified input). Non-retryable categories mean the agent should report the error to the user or take a different approach.

## Automatic Error Categorization

Exceptions thrown by your handler or by Convex are automatically caught and categorized:

```ts
handler: async (args) => {
  // If serverConvexMutation throws, the framework:
  // 1. Logs the full error to console.error
  // 2. Extracts a clean message
  // 3. Infers the category from the error type/message
  // 4. Returns wrapError(category, message)
  return await serverConvexMutation(api.notes.add, args)
}
```

You don't need try/catch in your handlers unless you want custom error handling.

## outputSchema

Declare the shape of your success response for agent consumption:

```ts
import { z } from 'zod'

export default defineConvexTool({
  schema,
  name: 'export-notes',

  outputSchema: {
    format: z.enum(['json', 'csv']),
    content: z.string(),
    noteCount: z.number(),
    exportedAt: z.string(),
  },

  handler: async (args) => {
    // Return must match outputSchema shape
    return withSummary(
      { format: args.format, content, noteCount: notes.length, exportedAt },
      `Exported ${notes.length} notes as ${args.format.toUpperCase()}`,
    )
  },
})
```

The framework auto-wraps your `outputSchema` in the envelope: `{ ok: z.literal(true), data: z.object(yourSchema) }`. Don't add the envelope yourself.

MCP clients can use the output schema to validate responses and provide type-safe access to tool results.
