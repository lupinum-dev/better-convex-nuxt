When something's denied, drifted, or unexpectedly crossed a trust boundary — this is the page. The point of Trellis observability isn't more logs; it's faster answers to specific questions.

The four questions this page helps you answer:

- Why was this denied?
- Which layer denied it?
- Did this cross a tenant boundary on purpose?
- Did a destructive confirmation drift between preview and execute?

## Debug in this order

::steps{level="3"}

### Find the correlation ID

Every request has one. Grab it from the client error, server error, or your observability store.

### Find the first identity event

`principal.resolved` or `actor.missing`. That tells you whether the runtime even agreed on who called.

### Find the first deny or error event

Whichever came first sets the direction of the rest of the investigation.

### Check for trust-boundary events before it

If `db.cross_tenant.used` or `db.raw.used` fired, the handler was intentionally outside the default boundary — the issue is probably in the escape-hatch logic, not the default one.

::

That sequence usually localizes the problem to one layer.

## Common authorization failures

If you see one of these, the next question is "which layer decided?":

| Event | The decision came from |
| --- | --- |
| `guard.denied` | The outer boundary guard — the caller didn't even reach `load` |
| `authorize.denied` | The resource-bound rule — `load` ran, then the per-resource check rejected |
| `rls.denied` | Default tenant isolation — the row wasn't in the active tenant |

Each one points to a different fix:

- `guard.denied` → the caller is wrong (wrong role, wrong tenant) or the guard is wrong.
- `authorize.denied` → the resource-bound policy rejected correctly; either the user lacks access or the rule needs adjusting.
- `rls.denied` → the handler is reading across tenants without using an escape hatch.

## Trust-boundary checks

If behavior looks broader than expected, look for:

- `db.cross_tenant.used` — someone used `ctx.db.crossTenant`.
- `db.raw.used` — someone used `ctx.db.raw`.
- `service.access.checked` / `service.access.denied` — a service lane evaluated access.

::tip
`db.cross_tenant.used` isn't a warning — it's a fact. The question is whether the handler *should* have crossed. If yes, fine. If no, the fix is to remove the escape hatch and model the actor or tenant correctly.
::

## Destructive drift

If a destructive tool or operation fails after preview, follow this trail:

| Event | Reading |
| --- | --- |
| `operation.preview.started` | Preview began |
| `operation.preview.completed` | Preview returned (capture the state) |
| `operation.confirm.validated` | Client came back with a matching confirmation |
| `operation.confirm.drifted` | Client's confirmation no longer matches previewed state |

`confirm.drifted` means the underlying data changed between preview and execute — Trellis refused to run a destructive action against drifted state. That's the system working as designed; the UX fix is to re-preview rather than override.

## MCP failures

For tool problems, the sequence is:

| Event | Meaning |
| --- | --- |
| `tool.called` | Agent invoked the tool |
| `tool.denied` | Capability gating rejected before execution |
| `tool.executed` | Tool ran to completion |
| `tool.failed` | Tool ran and threw |

If the issue is capability gating, the denial explanation (`policy`, `reason`, `suggestedAction`) usually tells you more than the raw transport error.
