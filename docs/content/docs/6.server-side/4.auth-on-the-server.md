## Overview

Server-side Convex calls authenticate using the request's session cookie. The module exchanges the cookie for a JWT token, which is then used for authenticated queries, mutations, and actions.

The auth resolution is request-scoped. Within one Nitro request, SSR auth hydration and any `serverConvexQuery()`, `serverConvexMutation()`, or `serverConvexAction()` calls reuse the same resolved auth result instead of re-reading cookies and re-exchanging the token independently.

## How It Works

::steps{level="4"}

#### Cookie Extraction

The server reads the Better Auth session cookie from the incoming HTTP request. In production this may be the secure cookie name `__Secure-better-auth.session_token`; in development it is often `better-auth.session_token`.

#### Token Exchange

The module calls the auth proxy endpoint to exchange the session cookie for a Convex JWT token.

#### Authenticated Call

The JWT is attached to the Convex HTTP request, authenticating the user.

#### Request-Scoped Reuse

If the same server request makes multiple authenticated Convex calls, the module resolves auth once and reuses that result for the rest of the request.

::

## Auth Modes

Each server utility accepts an `auth` option:

| Mode | Behavior |
|------|----------|
| `'auto'` (default) | Uses cookie if available, runs without auth otherwise |
| `'required'` | Throws an error if no auth token can be resolved |
| `'none'` | Skips authentication entirely |

```ts
// Public data — no auth needed
await serverConvexQuery(api.posts.listPublic, {}, { auth: 'none' })

// User data — must be authenticated
await serverConvexQuery(event, api.users.me, {}, { auth: 'required' })
```

## Token Caching

To avoid exchanging the session cookie for a JWT on every server request, enable token caching:

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  convex: {
    auth: {
      cache: {
        enabled: true,
        ttl: 60 // seconds (1-60)
      }
    }
  }
})
```

Tokens are cached in Nitro storage (in-memory by default, configurable to Redis or other drivers).

This cache is cross-request and optional. Even with caching disabled, repeated authenticated Convex calls inside a single request still reuse the same request-scoped auth resolution.

### Clearing the Cache on Logout

When a user signs out, clear their cached token:

```ts
const sessionToken =
  getCookie(event, '__Secure-better-auth.session_token')
  ?? getCookie(event, 'better-auth.session_token')
if (sessionToken) {
  await serverConvexClearAuthCache(sessionToken)
}
```

## Explicit Token Override

If you already have a JWT (e.g., from a different auth system), pass it directly:

```ts
await serverConvexQuery(event, api.users.me, {}, {
  authToken: 'eyJhbGciOiJSUzI1NiIs...'
})
```

## Non-Goals For This Pass

This auth pass does not add new auth modes, headless/external-session support, or an `expectAuth` query mode. The focus is narrower: eliminate duplicated server auth resolution and make transitional query state visible to existing users.
