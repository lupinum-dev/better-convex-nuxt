Use this page when a Nitro endpoint needs to call the same Trellis backend model as the browser.

The main server helpers are:

- `serverConvexQuery(...)`
- `serverConvexMutation(...)`
- `serverConvexAction(...)`
- `createServerConvexCaller(event, ...)`

## Query From A Nitro Route

The common authenticated read path looks like this:

```ts
export default defineEventHandler(async (event) => {
  const tasks = await serverConvexQuery(
    event,
    api.tasks.listForExport,
    { projectId },
    { auth: 'required' },
  )

  return tasks
})
```

`auth: 'required'` means the route must resolve a real auth token from the request. If that token cannot be resolved, the helper throws instead of silently continuing.

## Mutation From A Nitro Route

Writes use the same shape:

```ts
const taskId = await serverConvexMutation(
  event,
  api.tasks.create,
  {
    projectId: body.projectId,
    title: body.title,
    priority: body.priority ?? 'medium',
  },
  {
    auth: 'trusted',
    actor: {
      userId: body.createdBy,
    },
  },
)
```

That example is intentionally not using browser cookies. It is a server-side trusted-caller flow.

## Choose The Right Auth Mode

The helpers support four auth modes:

- `auto`: use request auth when available
- `required`: fail if auth cannot be resolved
- `none`: never attach auth
- `trusted`: inject trusted-caller identity for server-to-server flows

Use the narrowest mode that matches the route's real trust boundary.

## Use `createServerConvexCaller(...)` For Multi-Call Routes

When one Nitro request needs several Convex calls, create a request-scoped caller:

```ts
const convex = createServerConvexCaller(event, { principal })

const summary = await convex.query(api.dashboard.summary, {})
const activity = await convex.query(api.dashboard.recentActivity, {})
```

That keeps the `event` and forwarded principal on one small object instead of repeating them on every call.

Important detail: the returned helper uses `auth: 'none'`, so when business authorization matters you should forward an explicit principal.

## Pass The Event Explicitly Outside Normal Route Scope

If you call the helpers without an H3 event and Nitro request context is unavailable, Trellis throws a clear error.

That is the right behavior. A server helper outside request scope should not invent missing request state.

So:

- inside a normal route, passing `event` is best
- omitting `event` is only safe when Nitro request context is actually available

## Errors Are Surfaced With Helper Context

If a server helper fails, Trellis reports which helper, function path, and auth mode were involved.

That makes server-side failures much easier to debug than a generic fetch wrapper around `/api/query` or `/api/mutation`.

## Keep Nitro Thin

A good Nitro route does:

- validate HTTP input
- choose the right auth mode
- call the existing protected Convex function
- shape the HTTP response

It should not duplicate business authorization that already lives in Convex handlers.
