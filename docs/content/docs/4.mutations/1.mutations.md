`useConvexMutation()` is the default Trellis write path for client-side state changes. It returns a callable mutation function with reactive state attached.

Use this guide for the default mutation flow. For the exact option and return contract, use [Composables](/docs/api-reference/composables).

## The Default Shape

Start with the common case:

```vue
<script setup lang="ts">
import { api } from '#trellis/api'

const title = ref('')
const createTodo = useConvexMutation(api.todos.create)

async function handleSubmit() {
  await createTodo({ title: title.value })
  title.value = ''
}
</script>

<template>
  <UButton :loading="createTodo.pending.value" @click="handleSubmit">
    Add todo
  </UButton>
  <p v-if="createTodo.status.value === 'error'">
    {{ createTodo.error.value?.message }}
  </p>
</template>
```

The composable itself is synchronous to create. The actual request happens when you call the returned function.

## What The Return Shape Gives You

The mutation function carries reactive state:

- `status`: `idle`, `pending`, `success`, or `error`
- `pending`: shorthand for `status === 'pending'`
- `data`: the last successful result
- `error`: the last error
- `reset()`: clear the tracked state

That means you do not need separate local loading refs for common UI flows.

## Success And Error Callbacks

Use `onSuccess` and `onError` for UI side effects such as toasts or redirects.

```ts
const createProject = useConvexMutation(api.projects.create, {
  onSuccess: () =>
    toast.add({ title: 'Project created', color: 'success' }),
  onError: (error) =>
    toast.add({ title: 'Could not create project', description: error.message, color: 'error' }),
})
```

The callbacks receive both the result or error and the original args.

## Error Handling

Execution errors are normalized into Trellis call errors where possible, including structured codes when the backend provides them.

That means UI code can handle both:

- human-readable error messages
- stable error codes when the backend exposes them

If you never read `.error.value` after a failure in development, Trellis warns. That is intentional. Silent mutation failures are too easy to miss.

## Latest Request Wins For State

Mutation state stays bound to the latest in-flight request.

That matters when a user triggers the same mutation twice quickly. The earlier request may still resolve or fail, but the tracked composable state should reflect the latest call, not whichever settles last.

## Reset When The UI Session Ends

Use `reset()` when the UI flow is done and the previous mutation state should not keep affecting the page.

Typical cases:

- closing a dialog
- retrying after showing an error
- clearing a success banner before the next action

## Common mistakes

- Do not `await useConvexMutation(...)`. Create the composable synchronously, then await the returned function call.
- Do not use actions when the operation is really a normal state-changing mutation.
- Do not build separate loading refs unless the UI needs state beyond what the composable already exposes.
