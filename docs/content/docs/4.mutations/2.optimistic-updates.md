Use optimistic updates when the UI should change immediately while the server mutation is still in flight.

Optimistic updates are for UI responsiveness. They are not a replacement for backend truth. The live query still confirms the final result or rolls the optimistic state back.

## The Default Shape

Use the `optimisticUpdate` option on `useConvexMutation()`:

```ts
const moveTask = useConvexMutation(api.tasks.moveToColumn, {
  optimisticUpdate: (ctx, args) => {
    ctx
      .query(api.tasks.listByProject, { projectId: props.projectId })
      .update(
        (tasks) =>
          tasks?.map((task) => (task._id === args.id ? { ...task, status: args.status } : task)) ??
          [],
      )
  },
})
```

That changes the local query state immediately while the mutation is in flight.

## The Mental Model

An optimistic update is not “pretend the mutation succeeded forever.”

It is:

1. update the local query state now
2. let the mutation execute
3. let the live query confirm the real result
4. let error handling or fresh server data correct the UI if needed

That is why the update should stay small and directly tied to the mutation’s expected effect.

## What `ctx` Gives You

The optimistic context gives you typed builders over local query state:

- `ctx.query(...)` for regular queries
- `ctx.paginatedQuery(...)` for paginated queries
- `ctx.matchQuery(...)` when you need to update every active arg combination
- `ctx.matchPaginatedQuery(...)` for the paginated equivalent

Use the narrowest builder that matches the UI surface you actually need to update.

## Good Fits

Optimistic updates work well for:

- toggling completion
- moving a card between columns
- inserting a new item into a visible list
- removing an item that the current page is already showing

These are UI changes where the expected local effect is obvious.

## Bad Fits

Be careful with optimistic updates when:

- the backend may reject the mutation often
- the mutation changes multiple distant parts of the UI
- the final result shape is complex or heavily computed on the server
- the server may rewrite more fields than the UI can safely guess

In those cases, optimistic UI can become misleading instead of helpful.

## Regular Query Example

You can update or replace a regular query result directly:

```ts
const addNote = useConvexMutation(api.notes.add, {
  optimisticUpdate: (ctx, args) => {
    ctx.query(api.notes.list, {}).update((notes) =>
      notes ? [{ _id: 'temp', ...args }, ...notes] : [{ _id: 'temp', ...args }],
    )
  },
})
```

## Paginated Query Example

Use the paginated builder when the visible list is driven by `useConvexPaginatedQuery()`:

```ts
const createTask = useConvexMutation(api.tasks.create, {
  optimisticUpdate: (ctx, args) => {
    ctx.paginatedQuery(api.tasks.listByProject, { projectId: args.projectId }).insertAtTop({
      _id: 'temp',
      title: args.title,
      status: 'backlog',
    } as never)
  },
})
```

This keeps your optimistic change aligned with the loaded pages instead of forcing you to fake cursor state by hand.

## Pair Optimistic Updates With Real Error Handling

Always keep normal mutation error handling in place.

If the mutation fails, users still need a visible explanation:

```ts
const moveTask = useConvexMutation(api.tasks.moveToColumn, {
  optimisticUpdate: (ctx, args) => {
    // ...
  },
  onError: (error) =>
    toast.add({ title: 'Could not move task', description: error.message, color: 'error' }),
})
```

Fast UI without clear failure handling is worse than slightly slower UI.

## Common mistakes

- Using optimistic updates for flows where the server result is too complex to predict honestly.
- Updating every active query when only one visible query actually needs the optimistic change.
- Forgetting that the server remains the source of truth.
- Treating optimistic UI as a substitute for backend authorization or validation.
