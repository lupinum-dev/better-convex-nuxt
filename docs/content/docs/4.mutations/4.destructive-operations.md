Use this page when the UI needs to delete, archive, revoke, or otherwise perform destructive work.

In Trellis, destructive work should usually be modeled as an operation, not as an unstructured mutation.

## What Trellis Gives You

For destructive flows, Trellis gives you one business action with:

- shared guard, load, and authorization logic
- a preview phase
- an execute phase
- stable operation identity

That keeps the destructive path coherent instead of rebuilding the same policy in a preview query, a confirm dialog, and a write mutation separately.

## Backend Shape

The team-workspace example uses `defineOperation(...)` for todo deletion:

```ts
export const removeTodoOp = defineOperation({
  id: 'todos.remove',
  name: 'removeTodo',
  kind: 'destructive',
  args: deleteTodo.args,
  guard: canReadTodo,
  load: async (ctx, args) => {
    const todo = await ctx.db.get(args.id)
    requireRecord(todo, 'Todo')
    return { todo }
  },
  authorize: {
    check: (_actor, { todo }) => canDeleteTodo(todo),
  },
  preview: async (_ctx, _args, { todo }) => ({
    display: {
      summary: `Will permanently delete "${todo.title}"`,
      warn: 'This cannot be undone',
      affects: { todos: 1 },
    },
    confirm: {
      operation: 'todos.remove',
      targetId: todo._id,
      affectedCounts: { todos: 1 },
    },
  }),
  handler: async (ctx, args) => {
    await ctx.db.delete(args.id)
    return null
  },
})

export const remove = mutation(removeTodoOp)
export const previewRemove = query(previewOf(removeTodoOp))
```

That example lives in `examples/03-team-workspace/convex/todos.ts`.

## Runtime Setup Is Mandatory

Destructive operations are not opt-in sugar.

Before you register one, `defineTrellis(...)` must be configured with `destructiveSafety`:

```ts
export const { mutation, query } = defineTrellis(
  { query: generatedQuery, mutation: generatedMutation },
  {
    principal,
    actor: getActorFromPrincipal,
    tenantIsolation: {
      tables: ['todos'],
    },
    destructiveSafety: {
      redemptionTable: 'destructiveRedemptions' as never,
      auditTable: 'destructiveAuditLog' as never,
    },
  },
)
```

If you skip that, Trellis rejects destructive operation registration.

## Browser Flow

For browser UI, the common pattern is:

1. select the record to remove
2. fetch the preview
3. show the preview in a confirm dialog
4. execute the projected mutation if the user confirms

```vue
<script setup lang="ts">
import { api } from '#trellis/api'

const selectedId = ref<string | null>(null)

const previewArgs = computed(() =>
  selectedId.value ? { id: selectedId.value as never } : undefined,
)

const {
  data: preview,
  pending: previewPending,
  error: previewError,
} = await useConvexQuery(api.todos.previewRemove, previewArgs, {
  server: false,
  subscribe: false,
})

const removeTodo = useConvexMutation(api.todos.remove, {
  onSuccess: () => {
    selectedId.value = null
  },
})

function askToRemove(id: string) {
  selectedId.value = id
}

async function confirmRemove() {
  if (!selectedId.value) return
  await removeTodo({ id: selectedId.value as never })
}
</script>
```

The important part is not the modal itself. The important part is that preview and execute come from the same operation.

## What The UI Still Owns

For browser flows, Trellis does not force a modal, block the button until preview is fetched, or require a confirmation token.

The UI still owns:

- when to open a confirmation dialog
- how to render the preview
- whether to re-fetch preview before final execution
- whether the destructive action needs extra product-specific friction

That is deliberate. Trellis gives you the destructive business action and preview contract. Your app decides how the human interaction should feel.

## What Trellis Enforces

Trellis is stricter than the UI, but the strict parts are structural:

- destructive operations must have `kind: 'destructive'`
- destructive operations must have an `id`
- destructive operations must define `preview(...)`
- runtime registration requires `destructiveSafety`
- MCP destructive tools must bind to matching execute and preview projections from the same operation

The last point matters a lot: Trellis' token-backed confirmation and drift detection are enforced on the MCP tool surface, not on ordinary browser mutation calls.

## When To Use A Plain Mutation Instead

Do not wrap every dangerous-sounding action in an operation.

A normal mutation is still clearer when:

- the action is not actually destructive
- there is no preview worth showing
- the flow is local and one-shot
- the action does not need reuse across UI and MCP

Use operations when the destructive path needs a durable contract, not just because the primitive exists.

## Common Mistakes

- Building a preview query and execute mutation separately instead of projecting both from one operation.
- Assuming browser calls get the same confirmation-token enforcement as MCP tools.
- Turning every ordinary write into an operation.
- Forgetting `destructiveSafety` in `defineTrellis(...)`.
