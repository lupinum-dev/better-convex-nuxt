Use this page when a detail page should render immediately from already-loaded list data instead of flashing empty state while the detail query resolves.

`useCachedQuery()` is a specialized helper. It is not a general replacement for `useConvexQuery()`. Its job is to seed a detail query from a parent list query that is already in Nuxt data.

## The Problem It Solves

Without cache seeding, list-to-detail navigation often does this:

1. list page renders item summary
2. user clicks into detail page
3. detail page starts empty
4. detail query resolves later

That flash is often unnecessary when the clicked item is already present in the list result.

## The Default Shape

Use `from.query`, `from.args`, and `from.find` to describe where the seed data comes from:

```vue
<script setup lang="ts">
import { api } from '#trellis/api'

const props = defineProps<{ id: string }>()

const { data: task, isFromCache } = await useCachedQuery(
  api.tasks.get,
  () => ({ id: props.id }),
  {
    from: {
      query: api.tasks.list,
      args: {},
      find: (tasks) => tasks.find((task) => task._id === props.id),
    },
  },
)
</script>
```

While the real detail query is still pending, the matching item from the list query is used as the default value.

## What `isFromCache` Means

`isFromCache` is `true` when the page is still showing the cached match instead of the real resolved detail result.

That means you can:

- render immediately
- show a subtle loading state if needed
- stop treating list-to-detail as a full reload

Once the real detail query settles, `isFromCache` becomes `false`.

## When This Works Best

`useCachedQuery()` is best for:

- clicking from a loaded list into a detail view
- master-detail layouts
- card-to-detail transitions
- pages where the list already contains most of the detail fields

It is less useful when the list data is too shallow to be a good temporary stand-in.

## What It Depends On

The cache seed comes from Nuxt data for the parent list query. That means the source query and args must match the query that actually populated the list page.

If `from.find(...)` cannot find a matching item, `useCachedQuery()` falls back to the normal query path with no seeded value.

## Good Fit And Bad Fit

Good fit:

- task board to task detail
- project list to project detail
- article list to article detail

Bad fit:

- list and detail use unrelated query shapes
- list items are too incomplete to serve as a meaningful placeholder
- the page is not actually a list-to-detail navigation path

## Common Mistakes

- Using `useCachedQuery()` without a real parent list query to seed from.
- Treating it as a performance trick for every detail page instead of for list-to-detail transitions only.
- Forgetting that the cached item is still only a temporary stand-in until the detail query resolves.
