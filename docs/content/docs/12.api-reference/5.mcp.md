Use this page to answer two questions:

1. is this tool Convex-backed or Nitro-native?
2. am I importing from the package or from the Nuxt alias?

## Import Surfaces

Published package import:

```ts
import {
  defineMcpApp,
  defineTool,
  wrapError,
  wrapPreview,
  wrapSuccess,
  withSummary,
  useMcpServer,
  useMcpSession,
} from '@lupinum/trellis/mcp'
```

Generated Nuxt alias:

```ts
import {
  defineMcpHandler,
  defineMcpPrompt,
  defineMcpResource,
  defineMcpApp,
  defineTool,
} from '#trellis/mcp'
```

`#trellis/mcp` is generated inside the Nuxt app. It is not a bare npm specifier.

## Which MCP API Do I Use?

| Use case                                                     | API                                                        |
| ------------------------------------------------------------ | ---------------------------------------------------------- |
| Convex-backed non-destructive tool that should reuse existing business logic | `defineMcpApp(...).tool(...)`                  |
| Convex-backed destructive tool backed by an operation        | `tool.fromOperation(...)`                                  |
| Nitro-native tool that owns its own handler                  | `defineTool(...)`                                          |
| prompts, resources, handler wiring                           | `defineMcpPrompt`, `defineMcpResource`, `defineMcpHandler` |

## `defineMcpApp(...)`

Use `defineMcpApp(...)` when MCP should be a projection layer over Convex refs.

::tip{title="What this owns"}
`defineMcpApp(...)` is a protocol-specific adapter over Trellis' principal-first runtime. It resolves MCP request identity and visibility, then calls stable Convex refs. It is not the center of business authorization, and it is not Trellis' generic future-agent API.
::

```ts
const mcpRuntime = defineMcpApp({
  callConvex: async (event) => ({
    query: async (fn, args) => await serverConvexQuery(event, fn, args, { auth: 'none' }),
    mutation: async (fn, args) => await serverConvexMutation(event, fn, args, { auth: 'none' }),
    action: async (fn, args) => await serverConvexAction(event, fn, args, { auth: 'none' }),
  }),
  resolvePrincipal: async (event) => getMcpPrincipal(event),
  resolveCapabilities: async ({ principal }) => getCapabilitiesForPrincipal(principal),
})

export const tool = mcpRuntime.tool
```

Core runtime options:

| Option                | Purpose                                                        |
| --------------------- | -------------------------------------------------------------- |
| `callConvex`          | provides query/mutation/action callers for the current request |
| `resolvePrincipal`    | resolves the transport principal from the MCP request          |
| `resolveCapabilities` | resolves actor-level tool visibility                           |
| `principalKey`        | optional cache key for capability snapshots                    |
| `runtime`             | optional per-request runtime state for `enabled(...)` checks   |

If another agent protocol shows up later, it should usually follow the same pattern outside Trellis core: verify the request, resolve a principal, optionally resolve visibility, then call a root internal Convex ref with `{ ...args, principal }`.

## `tool(...)`

`tool(...)` is the default Trellis API for Convex-backed non-destructive tools.

```ts
export default tool({
  call: internal.miniCmsBridge.listDraftPages,
  schema: listDraftPages,
  capability: 'publishPage',
  meta: {
    name: 'list-draft-pages',
    description: 'List draft pages visible to the current agent',
  },
})
```

Core options:

| Option          | Purpose                                                  |
| --------------- | -------------------------------------------------------- |
| `call`          | registered Convex function ref to execute                |
| `schema`        | shared args definition                                   |
| `meta`          | MCP tool metadata such as name and description           |
| `capability`    | actor-level visibility key                               |
| `enabled`       | optional non-auth visibility gate                        |
| `rateLimit`     | in-memory limiter                                        |
| `respond`       | optional response formatter                              |
| `previewResult` | optional preview formatter for non-destructive previews  |

Use `tool(...)` when the tool should not own business logic and the action is not destructive.

For long-lived app stability, prefer projecting root internal refs or internal bridge refs. That keeps your automation surface stable even if the transport later changes from MCP to something else.

## `tool.fromOperation(...)`

Use `tool.fromOperation(...)` when the tool is projecting a Trellis operation, especially a destructive one.

```ts
export default tool.fromOperation(archiveBoardOp, {
  execute: archiveBoard,
  preview: previewArchiveBoard,
  capability: 'archiveBoard',
  meta: {
    description: 'Archive the current workspace board after a preview step.',
  },
})
```

This keeps the MCP projection tied to one business operation and lets Trellis validate the execute/preview ref pairing by operation id and projection type at startup.

For destructive tools, remember that `preview` is still query-lane.
Treat preview observability as request/runtime-level, and treat execute as the semantic correlated boundary.

## `defineTool(...)`

`defineTool(...)` still exists, but it is for standalone Nitro-native tools.

```ts
export default defineTool({
  name: 'check-dns',
  schema: checkDnsSchema,
  handler: async (args, ctx) => {
    const records = await queryDns(args.hostname)
    return ctx.ok({ records }, `Found ${records.length} records`)
  },
})
```

Use it when:

- the business logic is naturally Nitro-native
- there is no existing Convex operation to project
- the tool should own its own handler body

Do not default to `defineTool(...)` for Convex-backed business actions.

## Result Helpers

```ts
import { wrapError, wrapPreview, wrapSuccess, withSummary } from '@lupinum/trellis/mcp'
```

- `wrapSuccess(value)`
- `wrapError(category, message, issues?)`
- `wrapPreview(preview)`
- `withSummary(data, summary)`

These are mostly useful in custom formatters or standalone tools.
