Trusted caller is a low-level transport seam for forwarding a pre-validated caller identity into Convex operations. It is still public, but it is no longer the recommended default architecture for 1.0.

Prefer explicit principal forwarding through protected root refs when you control both sides of the integration.

## When To Use It

Use trusted-caller utilities only when:

- you have a narrow server-to-server path that still depends on this transport
- you need compatibility with existing trusted-caller flows
- a full principal-first root ref migration is not yet practical

Do not introduce trusted-caller transport for ordinary app code, browser code, or new MCP projections by default.

## Main APIs

- `withTrustedCaller(...)` augments validator args with trusted-caller fields
- `setTrustedCallerContext(...)` and `clearTrustedCallerContext(...)` manage the extracted caller on the Convex context carrier
- `getTrustedCaller(...)` reads the trusted caller from args or context
- `withTrustedCallerHandler(...)` wraps a handler with setup/cleanup
- `verifyTrustedCallerKey(...)` checks the shared secret for the transport edge

## Recommended Boundary

Keep trusted-caller handling at the transport edge. Inside your app runtime, prefer explicit principals and protected handlers so business rules stay transport-agnostic.
