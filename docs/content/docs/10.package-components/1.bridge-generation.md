# Package Components and Bridge Generation

Trellis now treats packaged Convex components as a first-class integration shape.

The contract is:

- the package exports a bridge manifest at its `convex/manifest` subpath
- the host app runs `trellis bridge generate <package>`
- Trellis writes deterministic host bridge files with package/version metadata
- the Nuxt module validates those files at startup
- the module never patches host source code at runtime

## What belongs to Trellis vs the package

Trellis owns:

- the manifest contract
- the generator CLI
- file metadata and overwrite rules
- request-scoped principal forwarding for MCP/server callers

The package owns:

- which host bridge files it needs
- the content of those generated files
- its protected component business logic

## Generator workflow

Generate bridge files into a host app:

```bash
pnpm exec trellis bridge generate @lupinum/ginko-cms
```

Use `--cwd` to target another app directory and `--force` only when you intentionally want to
replace non-generated files.

Generated files are stamped with:

- `@trellis-bridge-package`
- `@trellis-bridge-version`

That metadata is what runtime validation uses to detect missing or stale bridge files.

## Manual seams stay manual

Trellis does not patch global app seams like `convex.config.ts` for packaged components.

Those files stay app-owned because they are global coordination points across the whole Convex app.
The package module should validate them and fail clearly when required registration is missing.

For `ginko-cms`, the host app still owns:

- `convex/convex.config.ts`

The generated package bridge owns the package-scoped bridge files under:

- `convex/ginkoCms/*`
- `convex/ginkoCmsMcp.ts`
- package-auth bridge files such as `convex/auth.ts` and `convex/http.ts` when the package
  declares them in its manifest

## Auth and MCP guidance for packages

Packages should reuse Trellis primitives instead of reimplementing integration seams:

- use `resolveRedirectTarget(...)` or `useAuthRedirect()` for post-auth navigation
- use `createServerConvexCaller(event, { principal })` for MCP/server projection
- use `tool(...)` over protected root refs instead of creating a second business layer
- treat non-user principals such as `mcp` and `service` as normal principal variants

## Reference implementation

`ginko-cms` is the reference packaged-component implementation for this workflow.

Use [`examples/08-component-mini-cms`](../../../../examples/08-component-mini-cms/README.md) to
understand the local component architecture first, then map that shape onto the packaged
