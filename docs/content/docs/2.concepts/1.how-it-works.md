Trellis is built around one claim: browser UI, server routes, webhooks, and agents should hit one app-owned business layer instead of forcing you to maintain separate backends for each caller type.

## The Core Sequence

The Trellis model is:

1. a principal says how the call arrived
2. an actor says who that principal is in your app
3. tenant rules decide what data exists for that actor
4. guards and authorization decide whether the work is allowed
5. operations define reusable business actions where structured flows matter
6. Nuxt and MCP project that same backend model into different transports

Everything else is support around that sequence.

## One Business Layer

Trellis is trying to collapse a common failure mode:

- browser backend
- webhook backend
- internal server backend
- agent backend

That split looks convenient at first, but it usually means permission drift, duplicated business rules, and transport-specific exceptions that become impossible to trust.

Trellis instead treats those as projections of one backend model.

## The Caller Types

The common caller types are:

- browser UI
- Nitro server routes
- trusted server-to-server callers such as webhooks
- agent clients through MCP

Those are transport boundaries. They are not separate business authorization models.

## The Three Pillars

Trellis has three real pillars.

### App Runtime

This is the protected Convex business layer:

- principal resolution
- actor resolution
- tenant scoping
- protected queries and mutations
- actions and server-side execution
- reusable business operations

### Nuxt Runtime

This is the Nuxt-facing layer:

- SSR and hydration
- composables
- auth bootstrap
- route protection
- server helpers
- upload helpers
- permission context integration

### Agent Runtime

This is the agent-facing layer:

- MCP app definition
- tool exposure over the same backend model
- permission-aware tool discovery
- destructive preview and confirmation
- principal forwarding into Convex

The important part is not that Trellis can speak MCP. The important part is that agent access is treated as part of the app model instead of as a separate side system.

## Principal First, Actor Second

Trellis separates two questions:

1. how did this call arrive
2. who is this caller in your app

The first question is answered by the principal.
The second question is answered by the actor.

That split keeps transport identity from leaking into business authorization.

## What Changes By Transport

Different transports still differ in real ways:

- how identity is resolved
- whether the route is public or internal
- whether confirmation is required
- what result envelope is returned

What should not change is the business authorization path.

If a browser caller, a webhook caller, and an agent caller can all reach the same business action, they should still pass through the same Trellis-owned enforcement layers.

## Transport Visibility Is Not Authorization

Transport-specific controls are still useful.

For example:

- you may hide some tools from some agent clients
- you may expose only certain server endpoints publicly
- you may require a trusted caller key for webhooks

They do **not** replace Convex business authorization.

A hidden tool is not authorization.
A webhook signature is not business authorization.
A private route is not business authorization.

## Why Operations Matter

Operations are where Trellis stops being only a Nuxt plus auth integration and starts becoming an application layer.

They are the reusable seam for:

- admin UI actions
- confirmation dialogs
- server-side workflows
- webhook-triggered business actions
- destructive agent tools

Not every mutation should be an operation. But when the flow needs preview, confirm, reuse across projections, or stronger safety around destructive work, operations are the right primitive.

## What Trellis Does Not Own

Trellis does not own your product logic.

Your app still defines:

- the actor model
- guards and checks
- table shape
- tenant boundaries
- business behavior
- what an operation actually does

Trellis owns the structure around those decisions so they stay coherent across transports.

## When This Model Matters Most

You can ignore most of this page when the app is still public-only.

The model becomes important as soon as you add one or more of these:

- auth
- tenant boundaries
- protected mutations
- server callers
- agent tooling

That is the point where “one business layer” stops being architecture language and starts preventing real drift.

## Common mistakes

- Treating transport type as the business role model
- Forking server or agent flows into special-case business handlers
- Treating tool visibility or webhook signatures as if they were business authorization
- Re-reading worldview prose instead of moving back to the task guide once the model is clear
