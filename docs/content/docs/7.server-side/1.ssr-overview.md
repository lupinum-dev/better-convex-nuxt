This page explains the SSR mental model: why auth and data can already be present on first render and how that differs from later live state.

Trellis treats SSR as a first-class path:

- resolve auth on the server
- render the first page with that auth state
- hydrate the client with the same snapshot
- continue with live Convex subscriptions on the client

## What Happens During SSR

When auth is enabled, the server plugin does three jobs during SSR:

1. read the Better Auth session cookie from the request
2. exchange that session for a Convex JWT
3. hydrate token and user state into Nuxt `useState(...)`

That is why authenticated state can be available on first render instead of appearing one beat later after client boot.

## Why Protected Pages Do Not Need A Client-Side Auth Guess

On a healthy setup, SSR already knows one of these states:

- authenticated
- unauthenticated
- misconfigured or failed in development

Trellis then hydrates that state forward. The client does not need to guess whether the user might be signed in.

That is also why route protection can avoid the common "brief protected flash" problem.

## How `useConvexQuery()` Fits In

`useConvexQuery()` is SSR-aware by default.

The normal path is:

- fetch first value during SSR
- render that value into HTML
- hydrate the same state on the client
- attach a live subscription unless `subscribe: false`

This is not two separate data systems. It is one query lifecycle with a server-first first value and client-side live continuation.

## Auth Hydration Is Separate From Actor Readiness

One detail matters a lot in Trellis apps with permissions.

SSR auth hydration can already know the signed-in user, while the app-owned actor or permission context may still resolve separately.

That is normal.

Examples:

- the browser session is valid, but the app still needs to load the user row
- auth is ready, but `usePermissions()` is still waiting for the permission context query

Do not treat those as the same readiness signal.

## Dev Misconfig Fails Loudly On Purpose

In development, Trellis surfaces SSR auth misconfiguration instead of hiding it behind a quiet anonymous fallback.

That is deliberate. If session token exchange is broken, the right fix is to correct the auth config, not to debug a fake client-only symptom.

The usual causes are:

- wrong `SITE_URL`
- wrong `CONVEX_SITE_URL`
- missing Better Auth route registration
- secret mismatch

## Optional SSR Auth Cache

Trellis can cache SSR auth token exchange results:

```ts
trellis: {
  auth: {
    cache: {
      enabled: true,
      ttl: 60,
    },
  },
}
```

That reduces repeated token-exchange work on subsequent SSR requests. It is opt-in, because correctness comes first and deployment shape matters.

## Where SSR Ends And Server Helpers Begin

SSR page rendering and Nitro server routes are related but different:

- SSR page rendering is Nuxt loading a page for the user
- server helpers are your own Nitro handlers calling Convex explicitly

If you are writing a route or webhook handler, the next page is server routes.
