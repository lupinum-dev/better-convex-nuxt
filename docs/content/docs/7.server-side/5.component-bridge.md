This page is for packaged integrations and advanced host-app seams. Most app teams can ignore it.

The shipped advanced seam is the component bridge, not a vague generic bypass lane.

## What The Bridge Actually Is

`createComponentBridge(...)` forwards an explicit principal into component refs and exposes stable root refs from the host app.

That is useful for:

- Nitro routes
- MCP tools
- automations
- packaged component-style integrations

It forwards identity. It does not replace business authorization.

## Core Shape

The bridge builds root query and mutation registrations that pass `principal` through unchanged:

```ts
const bridge = createComponentBridge(builders, { principal })
```

Then register stable root refs over component refs.

## Why Use It

Use the bridge when the host app needs a durable root-level surface even if internal component layout changes later.

That is a better contract than telling every consumer to call internal component refs directly.

## CLI Support

The related CLI command is:

```bash
npx trellis bridge generate <package>
```

That loads a component bridge manifest from the package and writes the generated host bridge files into the app.

This is the stable automation seam for packaged Trellis component integrations.

## What Not To Claim

The bridge is not:

- a general authorization bypass
- a replacement for guards and app-owned actor logic
- a reason to stop modeling principals and actors cleanly

If a flow can stay on normal Trellis root handlers, that is still simpler.
