Use this page for webhook, automation, and server-to-server flows that still need to hit protected Trellis handlers.

This is a different trust path from browser auth.

## The Core Rule

Do not treat external server traffic like a browser session.

Trusted callers should use:

- explicit request verification in Nitro
- `serverConvex*` helpers with trusted auth when needed
- Convex-side trusted caller verification
- app-owned actor resolution where business policy still matters

## Minimal Trusted Caller Route

From the SaaS example:

```ts
const taskId = await serverConvexMutation(
  event,
  api.tasks.create,
  {
    projectId: body.projectId,
    title: body.title,
    priority: body.priority ?? 'medium',
  },
  {
    auth: 'trusted',
    actor: {
      userId: body.createdBy,
    },
  },
)
```

That route first verifies the external request, then forwards a trusted caller identity into the protected backend mutation.

## What Trusted Caller Verification Checks

On the Convex side, Trellis trusted-caller input expects:

- `_trustedCallerKey`
- `_trustedCaller.userId`

If the payload is malformed, Trellis denies it.
If the expected key is missing, Trellis denies it.
If the key does not match, Trellis denies it.

That is why `CONVEX_TRUSTED_CALLER_KEY` matters for server-to-server traffic.

## Keep The App Actor Model Intact

The team-workspace example shows the right pattern for webhooks:

- verify the trusted caller key
- resolve or ensure a real webhook bot user row
- return an app actor shape with role and tenant

That keeps todo ownership and permission checks on the same actor model as browser callers.

The webhook does not get a parallel "integration bypass" policy model for free.

## Browser Auth And Trusted Caller Auth Solve Different Problems

Browser auth answers:

- who is the signed-in end user

Trusted caller auth answers:

- is this server-to-server caller allowed to inject a user identity into the protected app flow

Do not mix them up. A valid browser cookie does not replace trusted caller verification, and a trusted caller key does not replace app authorization.

## Use `auth: 'none'` Only When The Handler Is Truly Public

Some routes intentionally call public or open Convex handlers with:

```ts
{ auth: 'none' }
```

That is correct only when the target handler really is public or the route itself owns the whole security model.

Do not use `auth: 'none'` to dodge a missing trusted-caller setup.

## Common mistakes

- Letting webhook handlers call protected business logic without trusted verification
- Treating a trusted caller as authorization instead of identity forwarding
- Forgetting `CONVEX_TRUSTED_CALLER_KEY`
- Creating a webhook-only permission system instead of resolving a real app actor
