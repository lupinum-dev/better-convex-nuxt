This page explains the handoff from SSR state to live subscription state when the app works but the transition still feels opaque.

The important Trellis behavior is:

- SSR gives you the first useful value
- hydration carries that value into the browser
- the live Convex client then takes over updates

## SSR First, Subscription Second

For `useConvexQuery()`, the normal lifecycle is:

1. fetch the first result during SSR
2. render it into HTML
3. hydrate that same state on the client
4. attach a live subscription for future updates

That handoff is why Trellis can be SSR-first without becoming a one-shot server data layer.

## Shared Subscriptions, Not One Per Component

Trellis deduplicates live query subscriptions across components in one Nuxt app instance.

That means:

- multiple consumers can observe the same query state
- Trellis keeps one shared live subscription instead of multiplying them
- the subscription is released when the last consumer goes away

This is important for pages that reuse the same query in several components or nested composables.

## Hydration Is Not The Same As Connection Failure

The first client moment after SSR is special.

The browser is still hydrating, and the websocket connection may not be ready yet. Trellis deliberately avoids treating that brief window as an offline failure.

This is why connection-state UI has a hydration grace window before showing offline banners.

## Auth Hydration And Query Hydration Are Related, But Separate

Two handoffs happen near each other:

- auth state hydrates from SSR into the client auth runtime
- query state hydrates from SSR into the live query runtime

They interact, but they are not the same mechanism. A query can already have SSR data while auth or permission context is still settling, and auth can be hydrated while a particular live subscription has not connected yet.

## What Reconnect Actually Means

Trellis distinguishes:

- first connection
- connected
- reconnecting after a previous live connection

That distinction matters for UI. A reconnect banner after a real disconnect is different from the initial client boot after SSR.

## When To Investigate The Handoff

Look closer when:

- a page renders correct SSR data but never receives live updates
- multiple components seem to create duplicate live traffic
- offline UI appears immediately during hydration
- auth looks ready but permission-backed UI stays unset

Those are handoff problems, not simple "query returned the wrong value" problems.
