Trellis exposes three database trust levels. Use the narrower one unless the job requires a more explicit escape hatch.

The three levels are:

- `ctx.db`
- `ctx.db.crossTenant`
- `ctx.db.raw`

These are not interchangeable.

## Default: `ctx.db`

This is the normal path.

For tenant-isolated tables, `ctx.db` enforces the active tenant boundary for reads and writes. Use it unless you have a concrete reason not to.

## Explicit Escape Hatch: `ctx.db.crossTenant`

`ctx.db.crossTenant` bypasses tenant isolation only.

What stays intact:

- service rules
- runtime hooks and triggers
- explicitness in code review

The internal harness documents it like this:

- `ctx.db.crossTenant` can read across tenants
- it must emit `db.cross_tenant.used`

A minimal example:

```ts
export const listAllPosts = query({
  args: {},
  guard: authed,
  handler: async (ctx) => {
    return await ctx.db.crossTenant.query('posts').collect()
  },
})
```

This is the right tool for admin, operator, onboarding, and portfolio flows that truly need to cross the tenant boundary while keeping the rest of the Trellis runtime guarantees.

## Full Escape Hatch: `ctx.db.raw`

`ctx.db.raw` is broader.

It bypasses:

- tenant isolation
- service enforcement
- triggers and runtime hooks

The internal harness is explicit here too:

- `ctx.db.raw` can read across tenants
- it must emit `db.raw.used`

This is not a nicer `crossTenant`. It is a sharper tool.

## Choose In This Order

The default preference order is:

1. `ctx.db`
2. `ctx.db.crossTenant`
3. `ctx.db.raw`

If `ctx.db.crossTenant` is enough, do not reach for `ctx.db.raw`.

That is not style-policing. It preserves runtime guarantees you would otherwise throw away.

## Real App Cases

From the multi-workspace example:

- current-workspace work uses normal tenant-scoped handlers
- onboarding and workspace switching use explicit cross-tenant flows
- agency dashboards use cross-tenant membership reads on purpose

That is the correct shape. The exception stays visible instead of being smuggled into the default path.

## Comment Escape Hatches On Purpose

When a handler uses `crossTenant` or `raw`, say why.

This kind of comment is good:

```ts
// Intentional escape hatch: onboarding creates a new tenant and membership,
// so it cannot run through tenant-isolated Trellis mutations.
```

That makes trust-boundary changes obvious in review.

## Common mistakes

- Treating `ctx.db.crossTenant` as normal app data access
- Reaching for `ctx.db.raw` when `crossTenant` would keep the needed runtime guarantees
- Hiding an escape hatch inside a helper with no visible boundary comment
- Using raw access to avoid modeling the right actor or tenant flow
