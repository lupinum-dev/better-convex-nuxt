This page explains the principal and actor split that keeps transport identity out of business authorization.

The split is simple:

- a `principal` says how the call arrived
- an `actor` says who that principal is in your app

Those are not the same thing, and Trellis is stricter because collapsing them causes policy bugs.

## Principal: Transport Identity

In the workspace example, the principal shape is:

```ts
export type TeamTodoPrincipal =
  | { kind: 'anonymous' }
  | { kind: 'user'; userId: string }
  | { kind: 'agent'; userId: string; agentId?: string; provider?: 'mcp' }
```

That tells you:

- anonymous browser request
- signed-in browser or server request
- forwarded agent request

The principal is intentionally transport-shaped. It does not decide what the caller can do.

## Actor: App-Owned Business Identity

The actor is what the app actually authorizes:

```ts
export type Actor = DefaultActor & {
  role: Role
  tenantId?: Id<'workspaces'>
}
```

That is where role, workspace membership, and tenant scope live.

Why this matters:

- principals tell Trellis how to resolve the caller
- actors tell your app what that caller means in business terms

An agent request can resolve to the same actor as a browser request. That is the point. Transport changes. Business policy does not.

## Resolution Flow

The normal flow in Trellis is:

1. resolve the principal
2. resolve the actor from that principal
3. run guards and authorization against the actor
4. execute the handler

From the team-workspace example:

```ts
export async function getActorFromPrincipal(ctx, _args, principal): Promise<Actor | null> {
  switch (principal.kind) {
    case 'anonymous':
      return null
    case 'agent':
      return await loadActorByAuthId(ctx, principal.userId)
    case 'user':
      return await loadActorByAuthId(ctx, principal.userId)
  }
}
```

Notice what it does not do:

- it does not trust transport kind as authorization
- it does not duplicate role logic in the principal
- it does not let MCP invent a separate policy system

## Why Not Put Role And Tenant Directly On Every Principal

Because that couples business authorization to transport shape.

Sometimes the principal does need extra transport metadata. Example: forwarded agent calls may carry `provider?: 'mcp'`. That is fine. But role and tenant authority still belong to the actor layer unless you are deliberately modeling a trusted forwarded principal and can enforce it.

This is why the docs keep saying:

- browser, server, webhook, and agent callers can share one backend model
- they still do not replace Convex business authorization

## Permission Actor Versus Strict Actor

Many apps need two closely related actor views:

- a looser permission actor for frontend context and onboarding
- a stricter actor for tenant-scoped mutations that require an active tenant

The workspace scaffold shows that split explicitly:

- `getPermissionActor(...)` may return an actor without `tenantId`
- `getActor(...)` tightens that to only actors with an active tenant

That keeps onboarding and workspace-required mutations from fighting each other.

## Common Mistakes

- Treating `principal.kind` as the authorization decision
- Encoding app roles into every transport caller instead of resolving the actor
- Reusing one strict actor resolver everywhere, then wondering why onboarding users disappear
- Letting browser and MCP calls drift into separate authorization logic
