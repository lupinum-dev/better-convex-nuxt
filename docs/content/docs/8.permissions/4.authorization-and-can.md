Use this page when the real decision depends on the loaded resource, not only on a coarse entry gate.

This is the Trellis pattern:

1. use a guard for the outer boundary
2. load the resource
3. authorize against the loaded resource
4. project the resulting policy to the UI through permission context where useful

## Use `authorize` After `load`

The team-workspace example shows the normal shape:

```ts
export const setCompleted = mutation({
  args: setTodoCompleted.args,
  guard: canReadTodo,
  load: async (ctx, args) => {
    const todo = await ctx.db.get(args.id)
    requireRecord(todo, 'Todo')
    return { todo }
  },
  authorize: {
    check: (_actor, { todo }) => canUpdateTodo(todo),
  },
  handler: async (ctx, args) => {
    await ctx.db.patch(args.id, {
      completed: args.completed,
    })
  },
})
```

Why this split matters:

- `guard` rejects obviously invalid callers early
- `load` gives the policy real data
- `authorize` makes the resource-bound decision explicit

Do not hide a loaded-resource ownership check inside a huge top-level guard when the handler already has a cleaner `load -> authorize` path.

## Backend `can(...)` Evaluates Checks Safely

Trellis exports a backend helper:

```ts
can(actor, hasRole('owner', 'member'))
```

That helper returns `false` when a check fails. It also fails closed if the check throws a `ConvexError`.

That is why examples use it inside permission-context projection and capability snapshots instead of open-coding `try/catch` around every decision.

## Frontend `can(...)` Is A Projection, Not The Policy Engine

Configured `usePermissions()` also gives you a `can(...)` function:

```ts
const { can } = usePermissions()

const canCreateProject = can(saasPermissionKeys.projectCreate)
```

That `can(...)` is not the backend helper. It is a frontend computed projection of the backend-owned permission context query.

Keep the distinction clear:

- backend `can(actor, check)` evaluates real backend checks
- frontend `usePermissions().can(key)` reads projected booleans from the permission context

The browser should not import backend guards and re-run them locally.

## Use Permission Context To Expose Stable UI Decisions

The normal projection path is:

```ts
export const getPermissionContext = query(
  definePermissionContext({
    resolve: getActor,
    guards: {
      [agencyPermissionKeys.projectCreate]: hasRole('owner', 'member'),
      [agencyPermissionKeys.agencyDashboard]: (actor) =>
        ['agency_admin', 'agency_manager'].includes(actor.role),
    },
    extend: async (ctx, actor) => {
      const memberships = await getMemberships(ctx.db.crossTenant, actor.userId)

      return {
        can: {
          [agencyPermissionKeys.projectCreate]: can(actor, hasRole('owner', 'member')),
          [agencyPermissionKeys.agencyDashboard]: memberships.some((m) =>
            ['agency_admin', 'agency_manager'].includes(m.role),
          ),
        },
      }
    },
  }),
)
```

The important rule is consistency: the browser should receive decisions that came from the same backend model as the handlers themselves.

## What Belongs In Permission Context

Good candidates:

- page-level actions and navigation states
- button visibility and disabled states
- stable app capabilities like `workspace.members` or `project.create`
- plan and usage fields that affect UI affordances

Bad candidates:

- ephemeral one-record ownership checks for a row the page already loaded directly
- giant per-record permission maps for large lists
- duplicated business logic copied from backend handlers into frontend code

## Common Mistakes

- Using `guard` alone when the real decision depends on a loaded record
- Recomputing backend policy in the browser instead of projecting it
- Confusing backend `can(...)` with frontend `usePermissions().can(...)`
- Stuffing every permission detail into a global permission-context query
