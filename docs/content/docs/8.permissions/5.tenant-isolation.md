Use this page to let Trellis own the default tenant boundary for tenant-scoped data.

Tenant isolation in Trellis is runtime enforcement, not a naming convention.

## Configure It In `defineTrellis(...)`

The normal setup is small:

```ts
export const { mutation, query, raw } = defineTrellis(
  { query: generatedQuery, mutation: generatedMutation },
  {
    actor: getActorFromPrincipal,
    tenantIsolation: {
      tables: ['todos'],
    },
  },
)
```

By default, Trellis expects the tenant field to be `workspaceId`.

If your tables use a different field, set it explicitly:

```ts
tenantIsolation: {
  tables: ['posts'],
  field: 'organizationId',
}
```

## Only Add Tables That Really Carry Tenant Scope

This is where docs need to push back on the easy mistake.

Do not list every app table in `tenantIsolation.tables`. List only tables that actually:

- store the tenant field
- expose a tenant index for that field
- should be filtered by the active actor tenant

The runtime validates the shape:

- the table list cannot be empty
- the table list cannot contain duplicates
- the field cannot be blank

Module validation also checks whether the table really has the configured tenant field and tenant index.

## What Default `ctx.db` Does

Once a table participates in tenant isolation:

- reads and writes run through the active actor tenant
- cross-tenant direct reads throw instead of silently leaking
- list queries only see rows for the current tenant

The internal harness makes this contract explicit. A direct cross-tenant `ctx.db.get(...)` does not quietly return `null`; it raises `Document belongs to a different tenant.`

That is the safer default. Handlers that forget a manual tenant check do not get to leak data by accident.

## Tenant Isolation Is Separate From Business Authorization

Tenant isolation answers:

- does this row belong to the current tenant

It does not answer:

- can this actor archive a project
- can this member edit someone else's task
- can this user issue MCP keys

You still need guards and authorization for business policy inside the tenant boundary.

## When Not To Use It

Do not push onboarding and tenant-creation flows through tenant-isolated handlers just to stay "pure."

Examples like workspace creation, workspace switching, or admin portfolio views often need an explicit cross-tenant path. That is why Trellis exposes escape hatches instead of pretending every valid workflow is tenant-local.

## Common mistakes

- Listing tables that do not store the tenant field
- Assuming tenant isolation replaces business authorization
- Using one tenant field name in code and another in schema/indexes
- Forcing bootstrap flows through tenant-isolated handlers when they need a broader scope
