This page is for advanced caller models. Most app teams can ignore it until one caller type is no longer enough.

Trellis supports multiple caller lanes, but it still wants one protected business model.

## User And Agent Lanes Should Usually Converge

For most app code, the important rule is:

- browser and server calls may resolve to a `user` principal
- agent calls may resolve to an `agent` principal
- both should usually resolve to the same app-owned actor model

That is how Trellis keeps UI, server, and MCP traffic on one authorization system instead of three drifting ones.

Transport tells you how the call arrived. Actor resolution tells you what that caller means in the app.

The principal layer should stay transport-shaped:

```ts
type Principal =
  | { kind: 'user'; subject: string }
  | { kind: 'agent'; agentId: string; provider: 'mcp' }
  | { kind: 'service'; serviceId: string }
```

That keeps `kind: 'agent'` in the transport identity where it belongs instead of turning the protocol into the business role model.

## Service Lanes Are Different

Service callers are not just another user flavor.

Trellis ships a distinct service lane through `defineServices(...)`. That model is for high-trust automation boundaries such as internal services, schedulers, or verified webhook-style flows.

The service access model is explicit:

- allowed tables are listed
- tenant mode is explicit
- unrestricted access is explicit if the app asks for it

The shape is:

```ts
defineServices({
  billing: {
    access: {
      tables: ['subscriptions'],
      tenant: 'global',
    },
  },
})
```

Or with derived tenant scope:

```ts
defineServices({
  webhook: {
    access: {
      tables: ['subscriptions'],
      tenant: 'derived',
      deriveTenant: ({ args }) => args.workspaceId,
    },
  },
})
```

That is different from resolving an app actor and running normal role checks.

## Keep The Models Honest

The docs need to stay strict here.

What is shipped:

- principal resolution
- actor resolution
- agent callers flowing into the same protected app model
- runtime-enforced service access policy through `defineServices(...)`

What is not a settled core promise:

- treating `runAsUser(...)` or `runAsService(...)` helpers as the main public model

Those trust-boundary helpers are exactly the kind of feature that should not be overclaimed before the runtime contract is fully settled.

## Choose The Right Lane

Use the normal actor lane when:

- the caller should behave like an app user
- the action belongs under the same role and tenant model as the UI
- you want one permission model across browser, server, and agent traffic

Use a service lane when:

- the caller is infrastructure or system-owned
- access should be constrained by explicit table policy
- the call should not pretend to be an end user

If you are using a service lane only to avoid modeling the right actor, that is the wrong fix.

## Common Mistakes

- Encoding business roles directly into transport callers
- Treating service callers as convenient user impersonation
- Letting agent traffic drift into a parallel authorization model
- Presenting future trust-boundary helpers as if they were the current public contract
