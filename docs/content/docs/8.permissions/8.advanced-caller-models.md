::caution
This page is for teams that need more than one caller lane. Most apps resolve everything to the user lane and ignore services entirely — if that's you, you can skip this.
::

Trellis supports multiple caller lanes, but it still wants one protected business model. The rule that prevents drift: transport tells you *how* the call arrived; actor resolution tells you *what* that caller means in the app.

## User and agent lanes should converge

For most app code:

- Browser and server calls resolve to a `user` principal.
- Agent calls resolve to an `agent` principal.
- Both should usually resolve to the same app-owned actor.

That keeps UI, server, and MCP traffic on one authorization system instead of three drifting ones. Keep `kind: 'agent'` in the transport identity where it belongs — don't turn the protocol into the business role model.

```ts
type Principal =
  | { kind: 'user', subject: string }
  | { kind: 'agent', agentId: string, provider: 'mcp' }
  | { kind: 'service', serviceId: string }
```

## Service lanes are different

Service callers aren't "another user flavor." They're a distinct trust boundary for internal services, schedulers, or verified webhook-style flows.

Define them through `defineServices(...)` with explicit access policy:

```ts [convex/services.ts]
defineServices({
  billing: {
    access: {
      tables: ['subscriptions'],
      tenant: 'global',
    },
  },
})
```

Or with a derived tenant scope from args:

```ts
defineServices({
  webhook: {
    access: {
      tables: ['subscriptions'],
      tenant: 'derived',
      deriveTenant: ({ args }) => args.workspaceId,
    },
  },
})
```

| Field | Meaning |
| --- | --- |
| `tables` | Explicit list of tables this service may access |
| `tenant: 'global'` | Service runs outside the tenant model |
| `tenant: 'derived'` | Service runs with a tenant derived from call args |
| `tenant: '<id>'` | Service is pinned to a specific tenant |

That's a different trust model than resolving an app actor and running role checks. Use it when the caller genuinely shouldn't pretend to be an end user.

## Choose the right lane

| Normal actor lane | Service lane |
| --- | --- |
| Caller should behave like an app user | Caller is infrastructure or system-owned |
| Action belongs under the same role and tenant model as the UI | Access should be constrained by explicit table policy |
| One permission model across browser, server, and agent traffic | Call shouldn't pretend to be an end user |

::note
If you're reaching for a service lane only to avoid modeling the right actor for an agent or webhook caller, that's the wrong fix. Resolve an actor (maybe a bot user with a specific role) and use the normal lane.
::

## What's shipped vs. what isn't

The settled core promise:

- Principal resolution.
- Actor resolution.
- Agent callers flowing into the same protected app model.
- Runtime-enforced service access via `defineServices(...)`.

What the docs deliberately don't overclaim:

- `runAsUser(...)` / `runAsService(...)` trust-boundary helpers as the main public model. Those are the kind of feature that needs a settled runtime contract before they become the headline — treat them as advanced escape hatches, not the default shape.

## Common pitfalls

- Encoding business roles directly into transport callers. Role lives on the actor.
- Treating service callers as convenient user impersonation. They're a separate trust model.
- Letting agent traffic drift onto a parallel authorization system. Converge it with the user lane.
- Using a service lane to skip resolving an actor. That's policy debt, not simplification.
