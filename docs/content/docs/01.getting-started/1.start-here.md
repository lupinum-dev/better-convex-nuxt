Ship a Nuxt + Convex app. Add auth. Add a webhook. Add an agent. By month three, no two callers enforce the same rules — and you're maintaining parallel copies of every business decision.

Trellis stops that drift. Browser UI, Nitro server routes, webhooks, and MCP agents all hit **one app-owned business layer**, with one set of guards and one authorization path. You write the app; Trellis keeps the transports from forking into four different backends.

It's not a replacement for Convex or Better Auth — it's the connective layer on top of both.

## Fastest on-ramp today

If you already have an app root and want Trellis to lay down the first coherent shape for you, start with the CLI:

```bash
npx trellis init app --template personal
```

Or pick the workspace lane immediately:

```bash
npx trellis init app --template workspace
```

There is also a `workspace-mcp` starter when you know the app needs MCP wiring from the start.

Before you start moving files around, read [Canonical app layout](/docs/getting-started/canonical-app-layout). The CLI now generates a real baseline shape, and new apps should follow it instead of inventing a fresh structure.

## Pick the smallest first path

Three starting shapes cover almost every real app. Pick the one that matches what you're actually shipping, not the one that sounds most impressive.

| Starting point | Pick it when | Read first |
| --- | --- | --- |
| **Public-only app** | You want SSR, live queries, and mutations — no sign-in flow yet. | [`01-public-todo`](https://github.com/lupinum-dev/trellis/tree/main/examples/01-public-todo) |
| **Personal auth app** | You need signed-in users and app-owned [actors](/docs/concepts/glossary#actor), but no tenant model yet. | [`02-auth-todo`](https://github.com/lupinum-dev/trellis/tree/main/examples/02-auth-todo) |
| **Workspace app** | You need roles, permissions, [tenant isolation](/docs/concepts/glossary#tenant-isolation), or MCP tools. | [`03-team-workspace`](https://github.com/lupinum-dev/trellis/tree/main/examples/03-team-workspace) |

::tip
If you're unsure, start public-only and add auth later. Each layer you add becomes a lot more obvious when it lands on top of something that already works.
::

## What Trellis gives you

One protected execution model that every caller type goes through — browser UI, Nitro server routes, webhooks, MCP agents. In practice that means [principal](/docs/concepts/glossary#principal) resolution, an optional `authRequired` gate, [actor](/docs/concepts/glossary#actor) resolution, [guard](/docs/concepts/glossary#guard) and [authorization](/docs/concepts/glossary#authorize) boundaries, [operation](/docs/concepts/glossary#operation) flows for destructive work, and correlated observability around every decision.

Your data model, checks, and business rules still live in your code. Trellis owns the structure *around* those decisions so they stay the same wherever the call arrives from.

## What "first success" looks like

Four steps. No auth, no permissions, no MCP yet:

1. Install the module.
2. Point it at a Convex deployment.
3. Render one query on a Nuxt page.
4. See the page update when the underlying data changes — with no refetch code on your side.

That's the moment Trellis clicks. Everything else is layers you add when the app asks for them.

## Add layers only when you need them

- **Auth** when the app needs sign-in and session-backed callers.
- **Permissions** when the app needs backend-owned checks or tenant-scoped access.
- **Operations** when destructive flows need preview-and-confirm.
- **MCP** when the same backend needs to be exposed to an agent client.

::note
Adding all four on day one is a common mistake. Each layer has its own mental model, and stacking them at once makes it harder to tell which one is misbehaving when something breaks.
::
