Trellis is the opinionated path for repeatedly shipping apps on the same stack:

- Nuxt
- Convex
- Better Auth
- MCP

It is not a generic helper layer.

It is the app platform for teams that want one backend model reused across browser UI, Nitro server routes, webhooks, and agent callers without re-wiring auth, actor resolution, permissions, and safety boundaries for every new project.

## Fastest on-ramp today

If you already have an app root and want Trellis to lay down the first coherent shape for you, start with the CLI:

```bash
npx trellis init app --template personal
```

Or pick the workspace lane immediately:

```bash
npx trellis init app --template workspace
```

There is also a `workspace-mcp` starter when you know the app needs MCP wiring from the start.
Use `cms` when the app is fundamentally a public content surface plus a signed-in studio.

Before you start moving files around, read [Canonical app layout](/docs/getting-started/canonical-app-layout). The CLI now generates the intended baseline shape, and new apps should follow it instead of inventing a fresh structure.

## Pick the smallest first path

Four starting shapes cover the current Trellis product lanes. Pick the one that matches what you're actually shipping, not the one that sounds most impressive.

| Starting point | Pick it when | Read first |
| --- | --- | --- |
| **Public-only app** | You want SSR, live queries, and mutations — no sign-in flow yet. | [`01-public-todo`](https://github.com/lupinum-dev/trellis/tree/main/examples/01-public-todo) |
| **Personal auth app** | You need signed-in users, but not workspaces or tenant-scoped data yet. | [`02-auth-todo`](https://github.com/lupinum-dev/trellis/tree/main/examples/02-auth-todo) |
| **Workspace app** | You need roles, workspace-scoped data, or MCP tools. | [`03-team-workspace`](https://github.com/lupinum-dev/trellis/tree/main/examples/03-team-workspace) |
| **CMS app** | You need a public read surface and a signed-in editorial studio. | [`08-component-mini-cms`](https://github.com/lupinum-dev/trellis/tree/main/examples/08-component-mini-cms) |

::tip
If you're unsure, start with the smallest lane that matches the real app. Trellis is opinionated, but it should still let simple apps stay simple.
::

## What Trellis gives you

One protected backend path that every caller type goes through: browser UI, Nitro server routes, webhooks, and MCP agents. Trellis keeps identity resolution, permission checks, destructive flows, and observability on that same path so you do not rebuild them differently per transport.

Your data model, checks, and business rules still live in your code. Trellis owns the structure *around* those decisions so they stay the same wherever the call arrives from.

::note
If you want the internal vocabulary behind that path, read [How it works](/docs/concepts/how-it-works) after the first task guides click.
::

## Day-1 mental model

For a basic app, you only need three ideas up front:

1. Put shared value contracts in `shared/schemas/`.
2. Put backend features in `convex/domain/` and expose them through `convex/functions.ts`.
3. Add auth, permissions, operations, or MCP only when the app actually crosses that boundary.

Everything else in Trellis is a deeper layer, not day-one homework.

## What "first success" looks like

Four steps. No auth, no permissions, no MCP yet:

1. Install the module.
2. Point it at a Convex deployment.
3. Render one query on a Nuxt page.
4. See the page update when the underlying data changes — with no refetch code on your side.

That is the first visible success. After that, Trellis grows into auth, permissions, operations, and MCP without changing the underlying architecture.

## Add layers only when the app crosses that boundary

- **Auth** when the app needs sign-in and session-backed callers.
- **Permissions** when the app needs backend-owned checks or tenant-scoped access.
- **Operations** when destructive flows need preview-and-confirm.
- **MCP** when the same backend needs to be exposed to an agent client.

::note
Adding all four on day one is a common mistake. Each layer has its own mental model, and stacking them at once makes it harder to tell which one is misbehaving when something breaks.
::
