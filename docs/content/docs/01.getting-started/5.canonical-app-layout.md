This page defines the current canonical Trellis app layout.

It is based on what `trellis init app --template personal|workspace|workspace-mcp|cms` generates today.

The point is not to show every possible file. The point is to make Trellis apps boring to navigate.

## The rule

If you start a new app with the CLI, keep the generated shape.

Add app code into the existing lanes instead of inventing a new structure for every project. A Trellis app should feel familiar after one glance at the tree.

## Current generated baseline

All four starters generate the same basic spine:

```text
nuxt.config.ts
convex/
  auth.ts
  auth.config.ts
  convex.config.ts
  functions.ts
  http.ts
  schema.ts
  test.setup.ts
  auth/
    actor.ts
    checks.ts
  domain/
  permissions/
  operations/
shared/
  schemas/
pages/
  index.vue
```

This is the current contract:

- `nuxt.config.ts` owns the Trellis module wiring.
- `convex/auth.ts`, `convex/auth.config.ts`, `convex/http.ts`, and `convex/convex.config.ts` own Better Auth and Convex bridge setup.
- `convex/functions.ts` is the Trellis backend entrypoint for your app-owned query/mutation builders.
- `convex/schema.ts` owns tables and indexes.
- `convex/auth/` owns app identity decisions:
  - `actor.ts` resolves the app actor
  - `checks.ts` defines guards and reusable checks
- `convex/domain/` owns feature modules like todos, pages, and MCP keys.
- `convex/permissions/` owns frontend permission projection queries.
- `convex/operations/` owns workflow-style backend actions such as onboarding or other multi-step mutations.
- `shared/schemas/` owns value contracts reused across frontend, backend, and MCP tools.
- `pages/` owns the first visible UI path.

## Lane-specific additions

The starter lanes build on the same baseline instead of inventing a different app structure each time.

### `personal`

Use this when the app has signed-in users but no tenant model yet.

Extra generated files:

```text
convex/
  domain/
    todos.ts
  permissions/
    context.ts
shared/
  schemas/
    todo.ts
```

Meaning:

- `convex/domain/todos.ts` is the first domain module.
- `convex/permissions/context.ts` exposes the frontend permission context for the signed-in user.
- `shared/schemas/todo.ts` keeps the todo contract shared across callers.

### `workspace`

Use this when the app needs workspaces, tenant isolation, roles, or backend-owned permission projection.

Extra generated files:

```text
convex/
  auth/
    principal.ts
  domain/
    todos.ts
  operations/
    onboarding.ts
  permissions/
    context.ts
shared/
  schemas/
    todo.ts
```

Meaning:

- `convex/auth/principal.ts` owns the transport-level caller model before actor resolution.
- `convex/operations/onboarding.ts` holds the first workspace bootstrap flow.
- `convex/permissions/context.ts` exposes the permission context for the current workspace path.
- `convex/domain/todos.ts` stays the first tenant-aware domain module.

### `workspace-mcp`

Use this when the same workspace app also needs agent-facing MCP tools.

Extra generated files on top of `workspace`:

```text
convex/
  domain/
    mcpKeys.ts
server/
  middleware/
    mcp-auth.ts
  mcp/
    runtime.ts
    tools/
      create-todo.ts
      list-todos.ts
```

Meaning:

- `convex/domain/mcpKeys.ts` owns MCP key storage and related backend helpers.
- `server/middleware/mcp-auth.ts` owns the MCP auth handshake on the Nuxt side.
- `server/mcp/runtime.ts` wires the MCP runtime surface.
- `server/mcp/tools/*` exposes agent tools that call the same backend model as the browser app.

### `cms`

Use this when the app is fundamentally a public content surface with a signed-in editorial studio.

Extra generated files:

```text
convex/
  domain/
    pages.ts
  permissions/
    context.ts
shared/
  schemas/
    page.ts
pages/
  [slug].vue
  studio.vue
```

Meaning:

- `convex/domain/pages.ts` owns the CMS content model: published reads, studio reads, draft saves, and publish preview/execute.
- `shared/schemas/page.ts` keeps the page contract shared across browser and backend code.
- `pages/index.vue` is the public list surface.
- `pages/[slug].vue` is the public page route.
- `pages/studio.vue` is the signed-in editorial surface.

## Where new code goes

Keep the layout boring:

- Add new backend features beside `convex/domain/todos.ts` as new domain modules such as `convex/domain/posts.ts` or `convex/domain/media.ts`.
- Add new shared validators and value contracts in `shared/schemas/`.
- Add new guard and actor logic in `convex/auth/`.
- Add workspace-level policy and permission projection in `convex/permissions/context.ts` or nearby files under `convex/permissions/`.
- Add workflow-style backend actions in `convex/operations/`.
- Add MCP-facing tool files under `server/mcp/tools/`.

Do not scatter the same concern across `server/`, `utils/`, `lib/`, and custom folders just because a project is growing.

## Generated does not mean sacred

The generated files are plain app code.

Edit them directly.

The stable convention is the location of the code, not the exact contents of the starter implementation.

## What is stable vs still evolving

Stable today:

- `trellis init app --template personal|workspace|workspace-mcp|cms`
- auth bridge files in `convex/`
- actor and check files in `convex/auth/`
- domain modules in `convex/domain/`
- permission projection in `convex/permissions/`
- workflow actions in `convex/operations/`
- shared value contracts in `shared/schemas/`
- MCP Nuxt files under `server/mcp/`

Still evolving:

- first-class `support-inbox`, `admin-console`, and `agent-console` templates
- more of the repetitive setup moving from docs into generators

## The practical test

If you open two Trellis apps side by side and need more than a minute to orient yourself, the platform shape is still too loose.

That is the bar this layout is meant to enforce.
