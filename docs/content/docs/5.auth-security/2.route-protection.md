Use route protection when a page should never render for signed-out users.

Trellis route protection is middleware-driven. The page declares intent with meta, and the auth runtime decides whether to redirect.

## Protect A Page

The default protected-page shape is:

```ts
definePageMeta({
  convexAuth: true,
})
```

That tells Trellis to require an authenticated user for the route.

## Custom Redirect Target

If a page should redirect somewhere other than the default auth route, pass an object:

```ts
definePageMeta({
  convexAuth: {
    redirectTo: '/auth/signin',
  },
})
```

If route protection is enabled with return-to preservation, Trellis appends the current path as `?redirect=...` so the user can be sent back after auth completes.

## Global Defaults

The module-level default lives in `trellis.auth.routeProtection`:

```ts
trellis: {
  auth: {
    enabled: true,
    routeProtection: {
      redirectTo: '/auth/signin',
      preserveReturnTo: true,
    },
  },
}
```

Use the page-level meta only when a route needs a different redirect target than the default.

## Skip Auth Handling On Specific Pages

Some routes must stay outside the auth path entirely, especially auth landing pages.

Use:

```ts
definePageMeta({
  skipConvexAuth: true,
})
```

That is the right setting for pages such as a sign-in screen that should not trigger auth token fetch behavior for themselves.

## Why This Matters During Pending Auth

Route protection needs to hold even while auth is still pending on the client.

That is why Trellis owns this in middleware instead of expecting every protected page to render, wait, and then redirect manually. If the page mounts visibly while auth is unresolved and the user is not allowed through, the protection has already leaked.

## Common Good Patterns

Good defaults:

- login-required app areas use `convexAuth: true`
- sign-in pages use `skipConvexAuth: true`
- the default redirect target lives in module config
- only exceptional routes override that redirect target

## Common Mistakes

- Protecting the page with template `v-if` only and forgetting route middleware behavior.
- Forgetting `skipConvexAuth: true` on the sign-in route.
- Repeating redirect logic in every protected page instead of using route meta.
- Treating route protection as permission enforcement. It only answers “is the user signed in enough to enter this page?”

## What Route Protection Does Not Solve

Route protection is not business authorization.

It decides whether a user reaches a page. It does not decide whether a protected Convex handler is allowed to execute. Backend guards and authorization still own the real permission decision.
