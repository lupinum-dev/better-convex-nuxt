Use this page to debug auth failures mechanically: env, callbacks, Better Auth routes, redirects, and stale auth refresh.

Start with the mechanical checks first. Trellis auth failures are usually configuration failures, not mysterious runtime edge cases.

## Run The Doctor First

The fastest first check is:

```bash
npx trellis doctor
```

That command inspects the same setup assumptions the Trellis runtime makes:

- Nuxt app root exists
- `@lupinum/trellis` is installed
- `convex` is installed
- `@lupinum/trellis` is registered in `nuxt.config.*`
- a Convex URL source exists
- auth env vars exist when auth is enabled
- Better Auth routes are registered in `convex/http.ts`

If `doctor` warns or fails, fix that first. Do not debug the browser before the app wiring is correct.

## Check Core Module Wiring

These failures stop auth before route protection or session refresh even matter:

- `@lupinum/trellis` missing from `package.json`
- `convex` missing from `package.json`
- `@lupinum/trellis` not registered in the `modules` array
- no `CONVEX_URL` or `NUXT_PUBLIC_CONVEX_URL` source

The minimum healthy Nuxt config looks like this:

```ts
export default defineNuxtConfig({
  modules: ['@lupinum/trellis'],
  trellis: {
    url: process.env.CONVEX_URL,
    auth: {
      enabled: true,
    },
  },
})
```

If the module is present but auth still never becomes ready, verify that the browser app is really booting against the same Convex deployment you expect.

## Check Auth Environment Variables

When auth is enabled, the common env problems are:

- missing `SITE_URL`
- missing `CONVEX_SITE_URL`
- missing `BETTER_AUTH_SECRET`
- mismatched local and deployed values

What each one does:

- `SITE_URL` tells Better Auth the app origin for callbacks and trusted-origin checks.
- `CONVEX_SITE_URL` points token exchange at the Convex HTTP origin when auto-derivation is not reliable.
- `BETTER_AUTH_SECRET` must match the Better Auth deployment backing your app.

If sign-in succeeds in one environment and fails in another, treat env mismatch as the default explanation until proven otherwise.

## Check Better Auth Route Registration

Trellis auth depends on the Better Auth bridge being registered in `convex/http.ts`.

If that route registration is missing, the browser can render the auth UI but the proxy cannot complete the session-to-Convex token exchange.

This is the shape the doctor looks for:

```ts
import { authComponent } from './auth'

authComponent.registerRoutes(http)
```

If `npx trellis doctor` warns about Better Auth route registration, fix that before touching frontend code.

## Check Route Protection And Redirect Loops

Route protection failures are usually one of these:

- the protected page uses `definePageMeta({ convexAuth: true })`, but the sign-in page also does
- the redirect target points back to the same page
- the app manually redirects before auth refresh completes

The protected page should look like this:

```ts
definePageMeta({
  convexAuth: true,
})
```

The sign-in page should not protect itself. If you override redirects, keep them pointed at a real public auth route.

## Check Auth Flow Integration

Do not call Better Auth client methods directly and then try to patch Trellis state afterward.

The supported path is:

```ts
const { client } = useConvexAuth()
const authAction = useConvexAuthActions()

await authAction.execute(() => client.signIn.email(payload), {
  redirectTo: '/',
})
```

That keeps the browser session change, Convex auth refresh, and redirect behavior on one tracked flow.

If you bypass `useConvexAuthActions()`, the page can look authenticated to Better Auth while Trellis still sees stale Convex auth state.

## Check Permission Context Drift

Some auth bugs are really permission-context bugs.

Configured `usePermissions()` fails closed in two important situations:

- auth is ready, but the permission context stays `null`
- the permission context belongs to a previous user ID

In development, Trellis warns if the permission context stays `null` for more than 2 seconds after auth is ready. That usually means:

- `trellis.permissions.query` points at the wrong Convex export
- actor bootstrap is wrong
- the permission query is returning `null` unexpectedly

If the wrong user signs out and another signs in, Trellis also suppresses stale permission context until the `userId` matches the current auth user again. That is intentional fail-closed behavior.

## Check Trusted Caller And MCP Setup Separately

If browser auth works but webhook or MCP traffic fails, the missing piece is often not user auth at all.

When the app exposes trusted-caller or MCP surfaces, Trellis expects `CONVEX_TRUSTED_CALLER_KEY` to exist in local env and in the Convex deployment serving that traffic.

That is a separate trust path from browser sign-in.

## When To Stop Debugging Locally

Stop local guessing and rerun these checks when:

- a new environment was introduced
- auth started failing after changing callback or origin settings
- only server-side or MCP-auth traffic is failing
- one user works and another always gets `null` permission context

The correct order is:

1. `npx trellis doctor`
2. verify env values
3. verify `convex/http.ts` route registration
4. verify auth flow uses `useConvexAuthActions()`
5. verify permission context wiring

## Next step

If the auth transport is healthy and the remaining problem is app-owned policy, move on to the permissions docs.
