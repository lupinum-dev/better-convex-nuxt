This page covers the concrete UI flows for sign-in, sign-up, sign-out, and redirect behavior once the auth runtime is already wired.

The main rule is simple: let Better Auth handle the auth action itself, and let Trellis handle the Convex-side auth refresh and redirect behavior around it.

## Sign Up

The normal sign-up path is:

```ts
const { client } = useConvexAuth()
const authAction = useConvexAuthActions()

async function handleSignUp(payload: { name: string; email: string; password: string }) {
  if (!client) throw new Error('Auth client unavailable.')

  await authAction.execute(() => client.signUp.email(payload), {
    redirectTo: '/',
  })
}
```

That path keeps the browser session change and the Convex auth refresh on the same flow.

## Sign In

The sign-in flow is the same shape:

```ts
async function handleSignIn(payload: { email: string; password: string }) {
  if (!client) throw new Error('Auth client unavailable.')

  await authAction.execute(() => client.signIn.email(payload), {
    redirectTo: '/',
  })
}
```

If a protected route redirected the user to the sign-in page with `?redirect=...`, Trellis prefers that redirect target over the fallback you pass here.

## Sign Out

Sign-out goes through `useConvexAuth()`:

```ts
const { signOut } = useConvexAuth()

async function handleSignOut() {
  await signOut()
}
```

`signOut()` clears Convex auth state and then calls Better Auth sign-out. That is the right default because the app should stop acting authenticated immediately.

## Track Pending And Errors

`useConvexAuthActions()` gives you the same tracked-call shape used elsewhere:

- `status`
- `pending`
- `data`
- `error`
- `reset()`

That means auth forms can use one normal pending and error pattern instead of inventing separate local state for sign-in and sign-up.

```vue
<UAlert
  v-if="authAction.error.value"
  color="error"
  title="Authentication error"
  :description="authAction.error.value.message"
/>
```

## Pair With Auth Boundary Components

The normal public auth page layout is:

- `ConvexAuthLoading` while auth is still resolving
- `ConvexUnauthenticated` for the form UI
- `ConvexAuthenticated` for the signed-in surface

That keeps the auth UI explicit and avoids accidental flashes between states.

## Redirect Rules

The redirect order is:

1. `?redirect=...` from route protection
2. `redirectTo` passed to `authAction.execute(...)`
3. no redirect if neither applies

That is why the page-level auth flow and the route-protection flow compose cleanly instead of fighting over navigation.

## Common Mistakes

- Calling Better Auth client methods directly and forgetting to refresh Convex auth state.
- Re-implementing pending and error state outside `useConvexAuthActions()`.
- Redirecting manually before the auth refresh finishes.
- Treating sign-out as only a Better Auth concern instead of a Better Auth plus Convex state transition.
