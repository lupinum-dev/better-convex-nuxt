`serverConvex` is auto-imported in your `server/` directory.

::callout{icon="i-lucide-info"}
**Important naming distinction:** This page covers **Nitro server code** (`server/api/*`, `server/middleware/*`, webhooks, jobs). It does **not** cover Nuxt app route middleware (`defineNuxtRouteMiddleware`).
::

If you need Convex data inside `defineNuxtRouteMiddleware()` (for auth/permissions during navigation), use one-shot APIs:

- client navigation: `useConvex()`
- SSR navigation: `serverConvex(event)`

`useConvexQuery` and `useConvexPaginatedQuery` are setup-scope composables and should not be used inside route middleware/plugins.

## serverConvex

`serverConvex(event, options?)` returns a **request-scoped caller** — the single server-call API. It owns exactly one lazily-resolved auth token and one `ConvexHttpClient` for the request, then exposes `query`, `mutation`, and `action`. Every call passes an explicit args object; a no-argument function is called with `{}`.

```ts
function serverConvex(event: H3Event, options?: ServerConvexOptions): ServerConvexCaller

interface ServerConvexCaller {
  getToken: () => Promise<string | null>
  query: <Q>(query: Q, args: FunctionArgs<Q>) => Promise<FunctionReturnType<Q>>
  mutation: <M>(mutation: M, args: FunctionArgs<M>) => Promise<FunctionReturnType<M>>
  action: <A>(action: A, args: FunctionArgs<A>) => Promise<FunctionReturnType<A>>
}
```

| Parameter | Type                  | Description                                  |
| --------- | --------------------- | -------------------------------------------- |
| `event`   | `H3Event`             | Current Nitro event (used for config + auth) |
| `options` | `ServerConvexOptions` | Optional auth policy / token override        |

Reuse one caller for every call in a request — the token and client are resolved once and shared.

## Query

```ts [server/api/posts.get.ts]
import { api } from '#convex/api'

export default defineEventHandler(async (event) => {
  const posts = await serverConvex(event).query(api.posts.list, { status: 'published' })

  return posts
})
```

## Mutation

```ts [server/api/tasks/complete.post.ts]
import { api } from '#convex/api'

export default defineEventHandler(async (event) => {
  const body = await readBody(event)

  await serverConvex(event).mutation(api.tasks.complete, { taskId: body.taskId })

  return { success: true }
})
```

## ServerConvexOptions

```ts
interface ServerConvexOptions {
  auth?: 'optional' | 'required' | 'none'
  authToken?: string
}
```

- `auth: 'required'` — reject with an authentication error if no token is available.
- `auth: 'none'` — never attach a token; call anonymously.
- `auth: 'optional'` (default) — attach the request's token if present, otherwise call anonymously.
- `authToken` — supply a token explicitly instead of deriving it from the request.

::callout{icon="i-lucide-info" to="/docs/advanced/logging"}
Enable module-level logging via the `logging` config option for debugging server-side operations.
::

## Patterns

### Webhook Handler

Put provider webhooks at the Convex boundary that owns the resulting write. A Convex HTTP action can verify the raw signed request and then call an internal mutation; clients cannot call that mutation directly. The internal mutation must atomically record the provider event ID before applying it so retries are idempotent.

```ts [convex/http.ts]
import Stripe from 'stripe'
import { httpRouter } from 'convex/server'

import { internal } from './_generated/api'
import { httpAction } from './_generated/server'

const http = httpRouter()

http.route({
  path: '/stripe/webhook',
  method: 'POST',
  handler: httpAction(async (ctx, request) => {
    const signature = request.headers.get('stripe-signature')
    if (!signature) return new Response('Missing signature', { status: 400 })

    const stripe = new Stripe(process.env.STRIPE_SECRET_KEY!)
    const event = stripe.webhooks.constructEvent(
      await request.text(),
      signature,
      process.env.STRIPE_WEBHOOK_SECRET!,
    )

    await ctx.runMutation(internal.subscriptions.applyStripeEvent, {
      eventId: event.id,
      eventType: event.type,
      payload: JSON.stringify(event.data.object),
    })
    return new Response('ok')
  }),
})

export default http
```

Keep parsing and provider-specific state transitions inside `applyStripeEvent`. Do not forward a verified webhook into an anonymously callable public Convex mutation. Convex documents why internal functions are the right non-client surface in [Internal Functions](https://docs.convex.dev/functions/internal-functions).

### Authenticated API Route

Use `auth: 'required'` to reject unauthenticated requests. `serverConvex` filters the request's Better Auth cookies and performs the bounded, redirect-free token exchange; do not reproduce that exchange with `$fetch`. Use `authToken` only when a trusted server boundary already holds a Convex JWT and intentionally wants that exact snapshot.

```ts [server/api/user/profile.get.ts]
import { api } from '#convex/api'

export default defineEventHandler(async (event) => {
  const profile = await serverConvex(event, { auth: 'required' }).query(api.users.getProfile, {})

  return profile
})
```

### Background Jobs

Use a Convex cron that calls an internal function. This removes a public HTTP trigger and its extra shared secret entirely.

```ts [convex/crons.ts]
import { cronJobs } from 'convex/server'

import { internal } from './_generated/api'

const crons = cronJobs()
crons.daily('daily digest', { hourUTC: 7, minuteUTC: 0 }, internal.notifications.sendDailyDigests)

export default crons
```

See [Convex cron jobs](https://docs.convex.dev/scheduling/cron-jobs). If an external scheduler is a real requirement, fail startup when its secret is absent and compare the supplied credential in a timing-safe way; never let two missing values count as authentication.

### Request Telemetry

Do not persist `event.path` or `event.node.req.url` as analytics: both can contain password-reset, invitation, OAuth, and verification credentials in their query strings. Prefer aggregate platform telemetry. If the product truly needs per-route analytics, record only a bounded allowlisted route name after authorization; never store the raw request target or headers.

### External API Integration

```ts [server/api/import/github.post.ts]
import { api } from '#convex/api'

export default defineEventHandler(async (event) => {
  const body = await readBody(event)
  const segment = /^[A-Za-z0-9_.-]{1,100}$/
  if (!segment.test(body.owner) || !segment.test(body.repo)) {
    throw createError({ statusCode: 400, statusMessage: 'Invalid repository' })
  }

  const convex = serverConvex(event, { auth: 'required' })
  await convex.query(api.issues.requireImportAccess, {
    owner: body.owner,
    repo: body.repo,
  })

  const owner = encodeURIComponent(body.owner)
  const repo = encodeURIComponent(body.repo)
  const issues = await $fetch(`https://api.github.com/repos/${owner}/${repo}/issues?per_page=100`, {
    headers: {
      Accept: 'application/vnd.github+json',
      Authorization: `Bearer ${process.env.GITHUB_TOKEN}`,
      'X-GitHub-Api-Version': '2022-11-28',
    },
  })

  // Re-authorize inside the write as well; access can change during the fetch.
  const imported = await convex.mutation(api.issues.importFromGithub, {
    owner: body.owner,
    repo: body.repo,
    issues: issues.map((i: any) => ({
      title: i.title,
      body: i.body,
      githubId: i.id,
    })),
  })

  return { imported: imported.count }
})
```

The outbound origin is constant, user-controlled values are path segments only, and authorization happens before the server attaches its GitHub token. Use a least-privilege GitHub token and enforce the same repository scope, item count, and field bounds in `importFromGithub`. See GitHub's [REST authentication guidance](https://docs.github.com/en/rest/authentication/authenticating-to-the-rest-api).

## Error Handling

```ts
export default defineEventHandler(async (event) => {
  try {
    const result = await serverConvex(event).mutation(api.tasks.create, { text: 'New task' })
    return result
  } catch (error) {
    // Log through a credential-safe server logger. Do not reflect provider,
    // database, or authorization details to the caller.
    throw createError({ statusCode: 500, statusMessage: 'Operation failed' })
  }
})
```

## Debugging

Enable logging in your `nuxt.config.ts` to debug server-side operations:

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  convex: {
    url: process.env.CONVEX_URL,
    logging: 'info',
  },
})
```

Server-side operations emit `operation:complete` events:

```
[better-convex-nuxt] ▷ query ✓ api.posts.list 45ms
[better-convex-nuxt] ▷ mutation ✓ api.tasks.create 67ms
```

See [Logging](/docs/advanced/logging) for full configuration options.

## TypeScript

Full type inference:

```ts
// Return type is inferred from the query
const posts = await serverConvex(event).query(api.posts.list, { status: 'published' })
// posts: Post[]

// Args are typed
await serverConvex(event).mutation(
  api.posts.create,
  { title: 123 }, // Type error: title should be string
)
```
