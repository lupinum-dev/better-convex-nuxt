There are two valid server-side Convex lanes in a Nuxt app, and they solve different problems.

## Lane 1: `serverConvex*` (Default)

Use `serverConvexQuery`, `serverConvexMutation`, and `serverConvexAction` when your Nitro route or
SSR handler should act as the current user.

Properties:

- uses the incoming Better Auth session when available
- exchanges the session cookie for a Convex token
- preserves least privilege
- works well for SSR, authenticated API routes, and user-scoped middleware

This lane is the default for `better-convex-nuxt`.

## Lane 2: App-Local Private Bridge

Use an app-local private bridge only when the work should run independently of end-user identity.

Typical cases:

- cron jobs
- webhook ingestion
- admin maintenance tasks
- third-party integrations triggered from the server

Properties:

- uses a server-only `CONVEX_PRIVATE_BRIDGE_KEY`
- should live in app code, not in `better-convex-nuxt`
- has a larger blast radius if the key leaks
- must stay narrow and intentional

## Comparison

| Question | `serverConvex*` | App-local private bridge |
| --- | --- | --- |
| Acts as current user? | Yes | No |
| Requires Better Auth session? | Usually yes | No |
| Safe default for page loads and standard API routes? | Yes | No |
| Appropriate for cron/webhooks/system jobs? | Usually no | Yes |
| Blast radius if a secret leaks | Lower | Higher |

::callout{icon="i-lucide-shield"}
Do not turn `serverConvex*` into a superuser escape hatch. If a route truly needs backend-only
power, create a separate app-local bridge with its own secret and audit trail.
::

## Rules Of Thumb

- Default to `serverConvex*`.
- Reach for a private bridge only when the action should not depend on user identity.
- Never expose `CONVEX_PRIVATE_BRIDGE_KEY` through `runtimeConfig.public`.
- Use one bridge key per app, never a shared secret across apps.

For the concrete pattern, see
[System Workloads: Private Bridge](/docs/recipes/system-workloads-private-bridge).
