# Defense Improvements / Security Hardening Review

Status: active audit workbook
Normative security contract: [`SECURITY.md`](./SECURITY.md)
Normative architecture contract: [`src/ARCHITECTURE.md`](./src/ARCHITECTURE.md)

This file is an executable review plan and evidence ledger. It does not redefine supported behavior. If this workbook and a normative contract disagree, stop, record the disagreement as a finding, and resolve the normative contract before continuing.

## Agent goal

> Establish defensible, evidence-backed confidence that Better Convex Nuxt implements the supported Vue, Nuxt, H3, Better Auth, Convex Better Auth, and Convex contracts correctly across browser, SSR, HTTP proxy, cookie, JWT, WebSocket, identity-transition, OAuth, MFA, authorization, failure, and release lifecycles. Find and fix library-owned defects using the simplest direct design. Verify every fix with invariant tests and production-like experiments. Record every claim, experiment, artifact, residual risk, and external responsibility. Do not claim complete or universal security. Finish only when every checklist item is passed, explicitly not applicable with evidence, or recorded as an accepted exception with an owner and expiry.

## Definition of done

The review is complete only when all of the following are true:

- Every phase gate in this file is `PASS` or `N/A` with evidence.
- No Critical or High library-owned finding is open.
- Every Medium finding is fixed or has a written exception with compensating controls, owner, and expiry.
- Every Low finding has a disposition.
- The exact supported dependency tuple passes all deterministic, packed-consumer, browser, real-service, and release verification assigned by this workbook.
- The production-like ingress experiments prove Host, protocol, forwarding-header, cookie, redirect, cache, body-limit, timeout, and cancellation behavior.
- Real OAuth and MFA ceremonies pass for every authentication method the project claims to support.
- Model-based identity-transition tests cover adversarial scheduling and resource cleanup.
- Consumer authorization examples and fixtures prove that UI middleware is not treated as authorization.
- Documentation describes supported configuration, unsupported configuration, revocation limits, deployment responsibilities, and incident response accurately.
- A reviewer who did not implement the fixes performs the final evidence review.
- The final report distinguishes verified properties, conditional properties, residual risks, and unverified assumptions.

## Non-negotiable operating rules

1. Work on a dedicated `codex/` branch. Preserve unrelated user changes.
2. Read `AGENTS.md`, `SECURITY.md`, `src/ARCHITECTURE.md`, package scripts, and this workbook before changing code.
3. Create an explicit goal from the Agent goal above and keep it active until the definition of done is met.
4. Use the exact installed dependency source as the primary behavioral reference. Current online documentation is a secondary reference, not proof of pinned behavior.
5. Before cloning an upstream repository, read the installed package's `package.json` repository metadata and record the exact package version, resolved integrity, repository URL, tag or commit, and local source path.
6. Put disposable upstream clones and experiment output under `.audit/`; do not commit upstream repositories, credentials, captured cookies, JWTs, build caches, or raw production logs.
7. Never use real customer credentials or production user data. Use generated sentinel accounts and secrets.
8. Redact cookies, bearer tokens, OAuth codes, client secrets, signing keys, backup codes, email OTPs, and personally identifying values from committed evidence.
9. Do not weaken CSRF, Origin, Host, TLS, cookie, JWT, rate-limit, body-limit, cache, or authorization controls to make a test pass.
10. Do not add compatibility shims, parallel auth paths, feature flags, or another source of identity truth unless a documented requirement proves they are necessary.
11. Prefer delete, simplify, or use the upstream lifecycle directly before adding orchestration.
12. Treat Better Auth session state as authentication input and Convex function checks as authorization. Never authorize from route middleware, Vue state, decoded JWT display claims, or a derived profile projection.
13. A unit test is insufficient evidence for CDN, browser-cookie, OAuth-provider, WebSocket, cross-tab, process-resource, or ingress behavior.
14. A passing happy path is insufficient evidence for a state machine. Test every await boundary with late, duplicated, rejected, aborted, and never-settling outcomes.
15. Stop the release on a failed security invariant. Do not downgrade a failure to documentation merely to finish the checklist.

### Authentication and identity data flow

```text
Browser Better Auth cookie / credential request
  -> fixed same-origin Nuxt /api/auth boundary
  -> bounded, filtered, redirect-disabled fetch to configured Convex site origin
  -> Convex Better Auth adapter / Better Auth session and Set-Cookie response
  -> Better Auth Vue useSession() public state (sole live browser session signal)
  -> epoch-scoped /convex/token exchange and staged JWT + normalized display user
  -> ConvexClient.setAuth token fetcher
  -> Convex WebSocket authentication callback confirms acceptance
  -> identity-generation client owner permits dispatch and subscriptions
  -> Convex verifies JWT configuration and exposes ctx.auth identity
  -> consumer Convex function authorizes against ctx.auth plus canonical app data

SSR request cookie
  -> per-request server snapshot and fixed-origin token exchange
  -> Nuxt useState token/display-user payload + Vary: Cookie
  -> private, no-store whenever an authenticated JWT is serialized
  -> hydration treats the snapshot as provisional until live Convex confirmation

Better Auth sign-out/session disappearance/error
  -> public session observer and serialized identity operation
  -> auth epoch advance, fail-closed token/user clear
  -> old primary retirement, in-flight call rejection, identity payload purge
  -> listener/subscription rebinding or disposal; anonymous transport remains separate
```

### Protected assets and attackers

Protected assets are credentials, Better Auth session/state/PKCE/nonce/MFA/trusted-device cookies, OAuth codes and provider tokens, Convex JWTs and JWKS signing keys, canonical identity/epoch/generation state, tenant/user/organization/storage/billing/API-key data, SSR payloads and caches, WebSocket subscriptions and optimistic state, logs/traces/devtools, deployment secrets, source/lockfile/SBOM, and the exact publishable tarball.

The threat model includes: anonymous cross-site clients; authenticated malicious tenant members attempting IDOR/BOLA or privilege escalation; malicious sibling subdomains and cookie tossers; compromised same-origin script/XSS; raw hostile proxy clients able to forge headers/framing/paths; malicious or collision-prone OAuth accounts; stolen cookie/JWT holders; offline/background/cross-device replayers; compromised dependencies or release infrastructure; misconfigured ingress/CDN/TLS; and slow, faulty, malicious, or compromised Better Auth/Convex/provider upstreams. Cross-account timing, stale async completion, resource exhaustion, and operator misconfiguration are first-class threats.

## Phase 3 evidence summary

The local Node/Nitro implementation now passes the supported fixed-destination contract. The proxy accepts only GET and POST, forwards no inbound forwarding/platform-origin control or Better Auth host/protocol marker, and relies on the required static Better Auth `baseURL` plus exact `trustedOrigins` for upstream generated URLs and origin enforcement. Matching-version behavior is anchored in [H3 1.15.11 request handling](https://github.com/h3js/h3/blob/7b9f41fda6038d26a367c2a26a07ed83ee1dbaac/src/utils/request.ts#L275-L347), [Convex Better Auth 0.12.5 marker restoration](https://github.com/get-convex/better-auth/blob/c628916b451a6b4cff0f5464f134475464b1a6da/src/client/create-client.ts#L88-L107), [Better Auth 1.6.23 static URL precedence](https://github.com/better-auth/better-auth/blob/9dfceee14021fc15a2fb93023f39635f25b0b5ba/packages/better-auth/src/utils/url.ts#L142-L187), and the official [Better Auth security guidance](https://better-auth.com/docs/reference/security).

POST browser requests require exact Origin when present, reject hostile same-site/cross-site Fetch Metadata, and reject a hostile or malformed Referer when Origin is absent. Headerless server clients remain supported. The only cross-site POST exception is the decoded one-segment core `/callback/:id` shape because Better Auth's [core callback validates OAuth state](https://github.com/better-auth/better-auth/blob/9dfceee14021fc15a2fb93023f39635f25b0b5ba/packages/better-auth/src/api/routes/callback.ts#L26-L123) and the pinned [Apple provider requires `form_post`](https://github.com/better-auth/better-auth/blob/9dfceee14021fc15a2fb93023f39635f25b0b5ba/packages/core/src/social-providers/apple.ts#L100-L123). Generic or plugin POST callback compatibility is not inferred; SAML and any other plugin callback are Phase 5 inventory items.

One request-owned AbortController now covers upload, the single `redirect: 'manual'` upstream fetch, decompressed response reading, and Node response finish. Client disconnects drive the same signal. Every post-fetch error cancels the upstream body; every reader error cancels before releasing its lock. A direct Node stream is used only for an uncached live Node request because pinned H3's request WebStream has no cancellation cleanup; H3/worker/cached bodies retain the installed ceremony. Token-exchange JSON is bounded by one fixed 1 MiB reader and non-OK bodies are cancelled immediately. The unbounded `/get-session` fallback was deleted entirely because it was a second SSR identity source; no replacement parser or compatibility path remains.

Direct Nitro tests preserve request bytes and multiple `Set-Cookie` lines; never follow browser redirects; reject declared and streamed limit-plus-one bodies before upstream delivery; enforce the decompressed response limit; exercise exact and wildcard routing; and close every repeated endless critical response. Deterministic tests cover upload/fetch/download disconnect, stalled and unread bodies, abort races, timer/listener balance, and error-path cancellation. The focused local gates are recorded as E-013 through E-016.

## Phase 4 local evidence summary

The supported cookie boundary is now one explicit contract: only default-prefix `better-auth.*` and `__Secure-better-auth.*` request and response cookies cross the proxy. Parsing and duplicate precedence match Better Auth 1.6.23; OAuth/MFA/session/plugin values remain opaque wire data. Unsupported names and any `Domain` attribute reject the whole upstream response with a stable 502 instead of silently dropping part of an auth ceremony. This intentionally hard-cuts custom names, prefixes, base paths, and cross-subdomain cookies. Official behavior and risks are anchored in [Better Auth cookie documentation](https://better-auth.com/docs/concepts/cookies).

SSR now has one identity source. The fixed token exchange is bounded, deadline-limited, redirect-free, and secret-safe; the former `/get-session` fallback was deleted. Local JWT parsing is only a bounded display projection: it requires canonical base64url/UTF-8 JSON, an exact string `sub`, a usable finite `exp`, strict normalized field types, and property-safe bounded custom claims. Convex remains the sole signature, algorithm, key, issuer, audience, subject, and temporal verifier, consistent with the official [Convex custom JWT contract](https://docs.convex.dev/auth/advanced/custom-jwt). Hydrated display state cannot dispatch query, mutation, action, or live listeners before Convex confirmation.

A real local Nuxt/Better Auth/Convex experiment created two independent sessions for one account, revoked one from the other context, then revoked all. Each deletion immediately made `/convex/token` return 401, while the already-issued JWT still authenticated a direct Convex query. The token's measured `exp - iat` was exactly 900 seconds, matching the pinned adapter's documented [15-minute default](https://labs.convex.dev/better-auth/api/convex-plugin). With Better Auth cookie caching disabled, the conservative replay ceiling is the remaining JWT lifetime; with cookie caching enabled, official Better Auth guidance says its `maxAge` can delay session-revocation recognition, so the ceiling can be cache lifetime plus JWT lifetime ([session management](https://better-auth.com/docs/concepts/session-management)). High-risk operations must revalidate canonical server state when that window is unacceptable.

Real sequential and concurrent A/B/anonymous SSR requests to one URL remained isolated under deliberately hostile public route rules. Authenticated responses replaced them with `private, no-store`, removed CDN/surrogate controls, preserved `Vary: Cookie`, and leaked neither the HttpOnly session cookie nor another user's identity. `Vary: *` remains sole and maximally restrictive per [RFC 9110 §12.5.5](https://www.rfc-editor.org/rfc/rfc9110.html#section-12.5.5). A candidate CDN, HTTPS secure-cookie path, enabled cookie cache, graceful JWKS rotation, cross-environment token isolation, and provider/MFA ceremonies remain open or externally blocked; Phase 4 is therefore not a universal PASS.

### SH-009 disposition — verified-email invitation ownership

- **Affected scope and versions:** `starters/agentic-saas/convex/auth.ts` and the matching architecture example on Better Auth 1.6.23. The team starter was already explicit-safe.
- **Prerequisite and narrative:** an invitation identifier must escape its intended mailbox and an attacker must control an account with the invited address while email ownership remains unverified. Disabling the gate then permits the documented by-ID acceptance precondition, exposing invited organization data and privileges.
- **Impact and severity:** high confidentiality and integrity impact within an invited organization; no availability or cross-tenant effect exists without a leaked invitation identifier. Confidence is high because the exact installed source and the upstream advisory agree with the starter's explicit opt-out.
- **Selected fix:** require verified email ownership unconditionally in executable starter configuration and documentation. Deleting the option was rejected because exact Better Auth 1.6.23 behavior is identifier-dependent; an explicit `true` is the direct invariant. Test-only environment bypasses were rejected because production-like examples are copied into real deployments.
- **Verification and residual:** `test/unit/starter-organization-ownership.test.ts` mechanically prevents false or environment-conditional configurations. Actual email delivery and provider/account-linking ceremonies remain externally configured Phase 7 evidence; the library cannot prove mailbox control without that environment.

### SH-007 disposition — supported Nuxt range

Better Convex Nuxt 0.6.0's module metadata accepted Nuxt `>=4.0.0` while the peer dependency and security contract support only `^4.4.0`. A consumer on Nuxt 4.0–4.3 could therefore pass Nuxt's module compatibility check before encountering an unreviewed API/runtime tuple. This is a low security, medium compatibility/release-integrity defect; it does not itself cross a credential or tenant boundary. The direct hard cutover changes the module metadata to the one canonical peer range. A compatibility warning, dual range, or widened peer was rejected because no acceptance criterion proves those versions. `supported-version-alignment.test.ts` locks `package.json`, `src/module.ts`, and `SECURITY.md` together. Full clean-consumer verification remains Phase 9 evidence.

### Phase 2 dispositions — identity ownership, isolation, and liveness

- **SH-001, SH-013, and SH-016 (bounded work):** Better Convex Nuxt 0.6.0's browser coordinator could retain a generation deferred, token exchange, or `ready()` timer indefinitely when Convex never called back, a Better Auth/custom token source never settled, or captured work won its deadline race. A faulty or hostile upstream was sufficient; no authentication was required. The impact was availability and resource retention, with identity transition work unable to complete, not a demonstrated authorization bypass. The direct design uses fixed internal five-second budgets: token exchange forwards an abort signal and also races abort-ignoring promises; confirmation supersedes the configuration with a total null fetcher; `ready()` owns and clears its one timer. Configurable deadlines and a generic cancellation service were rejected as unnecessary public surface. Late custom work that ignores `AbortSignal` cannot be forcibly terminated by JavaScript, but its result is consumed and cannot commit.
- **SH-002, SH-017, and SH-018 (one session source):** The installed Better Auth 1.6.23 public Vue session atom is the only session-changing identity input. Wrapper-owned token exchange/direct logout commit was deleted. The observer watches pending, the stable `data` reference, and error: Better Auth retains the data reference for JSON-equal refetch bookkeeping and replaces it for real same-session user/session changes. Integrated barriers capture early observations but require the exact non-empty token returned by sign-in/up or canonical null for sign-out; an unrelated token, intervening raw logout, or errored stale observation cannot satisfy them. Attempting to correlate private Better Auth requests, wrap the raw client, or retain dual identity paths was rejected. Raw calls intentionally retain Better Auth promise timing, while integrated calls provide resolve-after-reconciliation semantics; backend authorization and latest-revision scoping remain mandatory.
- **SH-011, SH-014, and SH-015 (cross-principal isolation):** On exact pre-fix code, a rejected callback retained the user key, a stale A fetcher could return globally staged B, A→unconfirmed-B→A could leave the crossed generation pending, and synchronous candidate construction failure occurred before A retirement. These required adversarial callback/exchange timing or a faulty factory but affected credential handling, stale-principal dispatch, tenant isolation, and availability. Fixes clear staged state and advance a real anonymous boundary on rejection, return null from stale fetchers, explicitly cancel/release crossed generations, and retire the prior primary synchronously before candidate construction. Delayed close, client reuse, same-generation retry loops, and private Convex state manipulation were rejected. Tests assert old calls reject, late callbacks cannot publish, failed construction leaves no authenticated primary, and persistent failure does not spin.
- **SH-010 (dependency race):** Convex 1.38.0's detached re-authentication task dereferenced cleared configuration after the real owner closed the client during sign-out. The owner already prevented old-generation dispatch, so demonstrated impact was an unhandled rejection/availability failure rather than identity leakage. Delaying close would weaken the security boundary and monkey patching or vendoring would create a second dependency path. Every manifest and committed lock therefore hard-cuts to verified Convex 1.40.0, the first stable release containing the upstream guard. The controlled real-client/WebSocket experiment now resolves cleanly; dependency changes require rerunning it.
- **SH-012 (reproducible consumers):** Five standalone projects failed frozen install from already-stale Nuxt/vue-tsc importers and two more stale locks were masked by the parent workspace. This is a maintainer/release-integrity defect: clean consumers could not reproduce the reviewed tuple. Each committed lock was regenerated from its sole canonical manifest, and true standalone `--ignore-workspace` checks prove the masked cases. Full clean-checkout installs, builds, and packed-consumer gates remain Phase 9 evidence rather than being inferred from lockfile-only checks.
- **SH-019 through SH-022 (publication and browser readiness):** Better Convex Nuxt 0.6.0 published token and user through separate Vue refs, permitted all four stable-handle surfaces to reach the eager browser client before its first server callback, keyed observation too narrowly for same-session claim changes, and correlated wrappers by session presence. A maliciously timed account switch, stale tab, provider callback, or faulty network was sufficient; mixed A/B state or pre-confirmation work threatened confidentiality, integrity, and tenant isolation, while lost/mis-correlated revisions threatened liveness. Sync-watcher, four-surface, data-reference, and exact A/B/error reproductions failed before the hard cutover. One discriminated `convex:identity`, owner-port settlement gating, observer-owned data-reference reconciliation, and exact token/null barriers are the smallest design with one source of truth. Keeping split refs, adding a second readiness boolean, correlating private requests, or wrapping raw Better Auth were rejected. Atomic, first-confirmation, observer, browser, and real two-tab WebSocket tests pass. SSR identity remains an explicitly documented display snapshot; no browser call becomes dispatchable until live confirmation. Live OAuth/provider behavior remains external evidence, not a local claim.
- **SH-023, SH-026, and SH-028 (credential end and failed sign-out):** In 0.6.0 a failed sign-out advanced the epoch without a valid current fetcher, could re-arm cached A after partial cookie loss, retained a token after the local 30-second usability boundary on transient failure, and returned cached A after a definitive token-endpoint 401/403. These require a sign-out/provider failure, scheduled refresh, expiry/clock advance, or revoked session; a stolen or revoked JWT could remain usable until its cryptographic expiry, affecting confidentiality, integrity, and tenant isolation. Captured-fetcher, partial-logout, and seeded-clock experiments reproduced the paths. The direct fix revalidates once after failed sign-out, permits retention only for the same principal or transient failure while the token is still usable, and returns null immediately for definitive outcomes. Blind re-arm, configurable grace periods, and frontend revocation state were rejected. Regression/model tests prove A retention only with a valid same session and fail-closed anonymous transitions otherwise. Cryptographic JWT revocation remains bounded by upstream/application policy and is documented for Phase 5/6.
- **SH-024, SH-030, and SH-031 (teardown):** Active/queued integrated operations, late `onUpdate` calls, and abort-ignoring token exchanges could outlive app disposal in 0.6.0. Faulty custom plugins, slow networks, HMR, or app teardown were sufficient; impact was bounded availability/resource retention and possible late callback reachability, not a demonstrated authorization bypass. Deterministic never-settling operation/exchange and post-dispose listener cases reproduced each path. One disposal signal rejects active/queued public auth work, one coordinator AbortController cancels and races every token exchange, and disposed `onUpdate` returns an inert complete unsubscribe before allocating state. A general cancellation service, per-operation controllers, and waiting out the five-second deadline were rejected. Tests prove immediate settlement, forwarded abort, zero remaining timers/listeners, inert late work, and unchanged public identity. JavaScript cannot terminate arbitrary plugin computation, but its late promise is consumed and cannot commit.
- **SH-025, SH-027, and SH-029 (owner recovery and handle fidelity):** During primary retirement, a live query could observe null and never bind to B; factory/init failure could leave no usable primary or reuse an already-observed anonymous generation; and the stable unsubscribe omitted Convex 1.40's runtime-only `getQueryLogs`. These required an account boundary, failure, or handle consumer; stale/missing subscription data and lack of a safe anonymous recovery affected integrity/availability, with high tenant-isolation consequence if old A were ever restored. Real coordinator/owner and regular/paginated harnesses reproduced the failures. Required/optional live queries now stay on the stable owner handle, authenticated failure crosses exactly one fresh anonymous generation, and unsubscribe methods dynamically delegate to the current underlying listener; `auth: 'none'` stays on its never-authenticated raw client. Composable retries, old-client restoration, generation retry loops, and a generic adapter were rejected. Reacquisition, fallback-terminal, distinct A/B unsubscribe, factory/init, and real WebSocket identity tests pass. Failure of the sole anonymous recovery remains a visible terminal auth error rather than an infinite loop.

### Phase 3 dispositions — proxy authority, browser signals, and resources

- **SH-003 (public origin authority):** Better Convex Nuxt 0.6.0 generated `X-Better-Auth-Forwarded-Host/Proto` in `src/runtime/server/api/auth/headers.ts` from H3's request URL. On H3 1.15.11 that URL remains conditional on ingress-owned Host/protocol handling, and Convex Better Auth 0.12.5 restores the markers into standard forwarding headers. A deployment that exposed Nitro directly, accepted hostile Host/`:authority`, or failed to overwrite forwarding headers could therefore influence upstream request-derived URL behavior, with high-conditional callback/origin/cookie integrity impact. Exact source reconstruction, direct H3 probes, a real Nitro receiver, and a real Better Auth 1.6.23 handler reproduced the dependency chain. The hard cutover deletes marker generation, strips standard, Better Auth, and common platform proxy controls, and keeps only one explicitly configured client-IP source. Adding an application-origin option, trusting a proxy chain, or retaining dynamic-host compatibility was rejected: the normative contract already requires one static Better Auth `baseURL` and exact `trustedOrigins`. Local generated OAuth URLs remain canonical under hostile headers. Operators still own TLS, Host/HTTP2 authority allowlisting, forwarding-header overwrite, and direct-origin blocking; without the blocked candidate deployment those conditional controls are not claimed verified.
- **SH-032 and SH-033 (browser POSTs and callbacks):** Exact Better Auth 1.6.23 social sign-in does not use `formCsrfMiddleware`, while its global origin middleware skips cookieless requests. A malicious site could cause a browser to initiate the endpoint when explicit hostile browser signals were otherwise accepted by the proxy; the demonstrated consequence is login-flow integrity and request-forgery exposure, not a completed account takeover. The proxy now rejects a non-canonical Origin on every method and, for POST without Origin, hostile same-site/cross-site Fetch Metadata or a hostile/malformed Referer. Requiring a new anti-CSRF token or rejecting all headerless POSTs would duplicate Better Auth or break supported server clients. Conversely, rejecting every cross-site POST broke Apple's pinned `response_mode=form_post`; the one exact decoded-segment core `/callback/:id` shape is therefore exempt and continues to rely on Better Auth's state/PKCE verification. A broad callback prefix, encoded separator, provider registry, and plugin compatibility list were rejected. Helper, real handler, and Nitro tests cover hostile/missing/null signals, same-site siblings, exact/non-exact callback paths, and invalid-state rejection. A successful provider callback was not simulated; real Apple and plugin/SAML provider ceremonies remain blocked Phase 5 evidence.
- **SH-004, SH-034, and SH-035 (proxy lifecycle):** In `src/runtime/server/api/auth/[...].ts` and `body-size.ts`, early critical-status/declared-size exits left upstream bodies unread, the eight-second deadline ended before Node response completion, disconnects did not cancel the upstream operation, and pinned H3's anonymous Node request-stream listeners could survive cancellation. Any unauthenticated client plus a stalled or adversarial upstream could consume sockets, listeners, or memory; impact is availability, not demonstrated credential disclosure or cross-tenant access. Critical endless bodies, limit-plus-one uploads, stalled readers, disconnects, and finish timing reproduced the paths. One existing request-owned AbortController now spans upload, one no-follow fetch, decompressed response read, and Node finish; one catch cancels every post-fetch error body; reader failures cancel before release. The only installed-version workaround is a direct cancellable stream for a live uncached Node request, while cached/web bodies keep H3's ceremony. Per-stage controllers, background draining, destroying the socket before an error, and a generic stream service were rejected. Unit/resource tests and real Nitro byte, extra-chunk, compressed-body, disconnect, and 12-cycle endless-response tests pass. Worker/no-socket deployment behavior, ingress disconnect translation, and long OS-level soak measurements remain unverified.
- **SH-005, SH-036, SH-037, and SH-044 (server response liveness and one identity source):** `auth-snapshot.ts` parsed `/get-session` with unbounded `response.json()`, used that endpoint as a second provisional user source when token decoding failed, non-OK token exits did not cancel bodies, and `fetchWithTimeout` cleared its timer once headers arrived. A malicious, compromised, or faulty configured Convex site could send an oversized or never-ending body, while a malformed token and fallback user could disagree about the display principal. The direct impact was server availability and identity-partition correctness; Convex backend authorization was not bypassed. Max-plus-one streams, unread non-OK responses, stalls, abort races, and mismatched/invalid token identities reproduced the cases. The simpler hard cut deletes `/get-session` and dead normalization entirely: one fixed 1 MiB reader serves the sole token exchange, its deadline remains through consume/cancel, and exact token `sub` is the only SSR display principal. Retaining even a bounded fallback, adding config, or locally joining two identities was rejected. Tests prove exactly one exchange request, no fallback, bounded/cancelled responses, fail-closed decoding, and secret-safe errors.
- **SH-038 (candidate deployment drift):** The only reachable deployment, `better-convex-nuxt-demo.vercel.app`, is not tied to this working tree. Anonymous HTTP/2 probes observed Vercel Host rejection and HTTPS redirect, and live social initiation produced the static canonical GitHub callback URL even with forged proxy markers. However, `/get-session` returned `public, max-age=0, must-revalidate` rather than the candidate's unconditional `private, no-store`, proving artifact/config drift. No authenticated cross-user cache leak was demonstrated. Treating that deployment as candidate evidence or changing library code to mimic it was rejected. A packed candidate must be deployed and tested with authenticated browser jars, hostile authority/direct-origin probes, header capture, disconnects, and provider callbacks before release; the exact responsibility is recorded externally and expires at release.

### Phase 4 dispositions — cookies, JWT display, SSR caches, and revocation

- **SH-006, SH-039, and SH-044 (one bounded display identity):** The old local decoder allowed custom `id` to overwrite the canonical subject, copied `__proto__`/`constructor`/`prototype` and arbitrarily large graphs, coerced normalized fields, and SSR accepted tokens inside the browser's fixed 30-second unusable window. A faulty or compromised configured token origin could create identity-key collisions or resource growth and show a provisional invalid identity; browser and backend work still required Convex confirmation, so no server authorization bypass was demonstrated. The hard cut derives id only from an exact non-empty `sub`, shares one finite-`exp` usability function across browser and SSR retention, caps the compact token at 64 KiB, strictly decodes canonical base64url/UTF-8 plain-object JSON, and property-safely clones bounded JSON claims while omitting reserved/invalid values. Local JWKS verification, claim coercion, a second user endpoint, and an authorization role for display claims were rejected. Convex 1.40 remains the only signature/algorithm/key/issuer/audience/time verifier. Adversarial Node/browser and real snapshot tests cover principal override, special/nested keys, malformed encoding, sizes/depth, field types, clock boundaries, and no-fallback behavior.
- **SH-040 and SH-041 (HTTP header, cache, framing, and encoding boundary):** The initial proxy allowed request `Connection` nominations, `Expect`, arbitrary accepted/content encodings, and several platform controls to survive; response Connection-nominated fields and CDN-specific cache controls also survived. Framed GETs were never read, so an incomplete body could occupy a socket beyond the request deadline; request encodings were forwarded without decoding and unsupported upstream encodings risked corruption. Anonymous clients or a faulty upstream could therefore cause availability loss, request ambiguity, or override intended cache isolation. Raw sockets and controlled receivers reproduced the paths. The direct fix parses nominations, strips every proxy/hop/framing/encoding input, rejects framed GET/non-identity request encodings and unsupported response encodings, closes unsafe connections, and drops standard/surrogate/edge/`*-CDN-Cache-Control` output before setting the final policy. Generic header adapters and transparent decoding were rejected. The independent 14-file/228-test review plus raw Nitro 13, plugin 6, auth loop 1, and server typecheck found no remaining local/library P0/P1/P2 defect.
- **SH-042 (one supported cookie namespace):** Two permissive parsers diverged from pinned Better Auth and the proxy's request filter was asymmetric with response cookies. Custom/unprefixed or Domain-scoped upstream cookies could reach the browser but not return reliably, partially completing auth ceremonies or overwriting unrelated app cookies; an untrusted sibling could also toss a parent-Domain value that the later `Cookie` field cannot identify by provenance. The hard cut uses one pinned-compatible parser and one symmetric default `better-auth.*` / `__Secure-better-auth.*` host-only namespace. It preserves duplicates and opaque supported wire fields but rejects the entire upstream response, cancels its body, and sends no cookie if any name or Domain attribute is unsupported. Per-plugin allowlists, forwarding everything, silently filtering response fields, and custom-prefix compatibility were rejected. Differential/property tests and Node/H3/Nitro/browser paths pass. Production HTTPS secure-cookie and controlled sibling-domain evidence remains externally blocked.
- **SH-043 (SSR cache isolation):** The prior helper required recognized cookie AND serialized token before no-store and left CDN/surrogate overrides intact. A stub/custom path could serialize a JWT/profile under only one signal, and a shared cache that honored its own directive could disclose the first user's bearer token and profile to later users. The direct policy uses cookie OR token, always varies auth-enabled SSR on Cookie, deletes shared-cache overrides through public H3 response APIs, and preserves sole wildcard `Vary: *`. A route-level opt-in, second cache state, and trusting platform defaults were rejected. Real sequential and concurrent A/B/anonymous requests under hostile public route rules stayed isolated and leaked no HttpOnly cookie/other-user sentinel. A packed candidate CDN can still rewrite or ignore headers later, so release evidence remains blocked rather than inferred.
- **SH-045 (revocation window):** A real second browser context revoked SESSION_A and later all sessions. Fresh `/convex/token` calls immediately returned 401, but JWT_A continued authenticating a direct Convex query; its measured `exp - iat` was the adapter's exact 900-second default. This is expected stateless-JWT behavior, not frontend state the library can revoke. Better Auth's documented cookie cache can additionally postpone backing-session recognition for `maxAge`, so the conservative ceiling is remaining cookie-cache time plus remaining JWT time. A frontend denylist, forced UI logout as authorization, and undocumented grace were rejected. SECURITY.md and auth docs now require shorter configured lifetimes or canonical server revalidation for high-risk operations. Enabled-cache/offline/full-expiry experiments remain open.
- **SH-046 (JWKS rotation):** Matching adapter 0.12.5 source shows routine rotation is not configured and its server-only `rotateKeys()` deletes all existing key rows before generating a replacement; static JWKS adds deployment coordination. Existing performance docs incorrectly called dynamic rotation automatic. The library-owned fix removes that claim and documents the action as emergency hard-cut invalidation with expected reauthentication and a required staging drill. A local dual-key bridge, job, shim, or fabricated rollback path was rejected because it would create another signing-key source of truth. Dynamic/static cache, overlap, rollback, and environment-isolation experiments remain operator/upstream evidence due before release.

# Final report template

The completed report must contain all of these sections:

1. Executive conclusion and permitted security wording.
2. Reviewed commit, branch, artifact hash, dependency tuple, and environments.
3. Architecture and trust-boundary summary.
4. Threat model and attacker capabilities.
5. Phase-by-phase results.
6. Confirmed vulnerabilities and fixes.
7. Disproven hypotheses.
8. Accepted exceptions and expiries.
9. Consumer and operator responsibilities.
10. OAuth, MFA, revocation, ingress, CDN, and identity-race experiment results.
11. OWASP Top 10:2025 and ASVS coverage.
12. Test, fuzz, E2E, packed-artifact, and release-gate evidence.
13. Remaining unknowns and confidence by claim.
14. Upgrade and reassessment triggers.
15. Independent reviewer signoff.

## Required conclusion language

The conclusion must answer:

- What was actually verified?
- What is conditional on configuration or consumer authorization?
- What could still cause account takeover, cross-tenant disclosure, token theft, or denial of service?
- How quickly can sessions and already issued JWTs be revoked?
- Which sign-in methods are covered by MFA?
- Which dependency versions and deployment topologies are supported?
- What evidence would invalidate the signoff after an upgrade?
- What confidence level is justified, and why?

Do not replace these answers with a single overall score.
