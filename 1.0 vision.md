**Date:** 2026-03-27

## What This Module Believes

Building a full-stack app shouldn't feel like assembling furniture from three different stores with instructions in three different languages. When you pick Convex for your backend and Nuxt for your frontend, the seam between them should disappear.

better-convex-nuxt 1.0 is the moment that seam disappears.

Every feature exists because someone building a real product needed it. Nothing exists because it looked good in a README. If a feature doesn't save time, reduce bugs, or remove a decision... it doesn't ship.

---

## The Developer Experience Promise

A developer using this module should be able to:

1. **Define data rules once** and have them enforced in their backend, their forms, their server routes, and their mutation calls — without copying anything.

2. **Never think about auth plumbing.** Sign in, sign out, protect routes, check permissions in templates, access the user in server routes — it all just works. SSR hydration doesn't flash. Token refresh doesn't race.

3. **Build real-time UIs by default.** Queries subscribe to live updates. Mutations are optimistic. Connection drops show graceful offline UI. This isn't a feature — it's the baseline.

4. **Upload files without thinking about the upload dance.** Pick file, call composable, get storage ID. Queue multiple files. Show progress. Cancel. Resume. Done.

5. **Know what went wrong when something goes wrong.** Errors have codes, paths, and context. DevTools show every query, mutation, and auth event. Logging is structured and opt-in.

## The 1.0 API Surface

Everything below is auto-imported. No manual import statements needed unless you want explicit imports for library code.

### Data Fetching (real-time by default)

```
useConvexQuery(query, args?, options?)
useConvexPaginatedQuery(query, args?, options?)
useConvex()
```

These are the foundation. They SSR on first render, then upgrade to WebSocket subscriptions on the client. Data stays in sync across tabs, across users, across the world. When args change, the query re-evaluates. When args are null/undefined, the query skips. When the component unmounts, the subscription is released (with ref-counting so shared subscriptions stay alive).

**Options that matter:**

- `server: false` — skip SSR, load client-only
- `subscribe: false` — fetch once, no live updates
- `default: () => []` — fallback while loading
- `transform: (raw) => mapped` — reshape data before it hits your template
- `keepPreviousData: true` — don't flash empty while new args load
- `onData: (data) => {}` — react to every update (analytics, side effects)
- `shared: 'key'` — deduplicate across components using the same query

### Mutations & Actions

```
useConvexMutation(mutation, options?)
useConvexAction(action, options?)
```

Call them like functions. They track their own state.

```vue
<script setup>
const createPost = useConvexMutation(api.posts.create);

await createPost({ title: "Hello" });
// createPost.pending.value → was true, now false
// createPost.status.value → 'success'
// createPost.data.value → the return value
// createPost.error.value → null (or Error if it failed)
</script>
```

**Options that matter:**

- `validate` — pre-validate args before hitting the network (see Validation below)
- `optimisticUpdate` — instant UI feedback with automatic rollback
- `onSuccess` / `onError` — lifecycle callbacks

**Optimistic updates that don't make you cry:**

```ts
const addNote = useConvexMutation(api.notes.add, {
  optimisticUpdate: (ctx, args) => {
    // Regular query — update in place
    ctx
      .query(api.notes.list, {})
      .update((notes) => [
        { _id: crypto.randomUUID(), _creationTime: Date.now(), ...args },
        ...notes,
      ]);

    // Paginated query — insert at top of first page
    ctx.paginatedQuery(api.notes.paginated, {}).insertAtTop({
      _id: crypto.randomUUID(),
      _creationTime: Date.now(),
      ...args,
    });
  },
});
```

Plus standalone helpers for common patterns:

```
prependTo(ctx, query, args, item)
appendTo(ctx, query, args, item)
removeFrom(ctx, query, args, predicate)
updateIn(ctx, query, args, predicate, updater)
```

### Validation (define once, enforce everywhere)

This is the 1.0 headline feature.

**The composable:**

```ts
// Convert Convex validators into Standard Schema for any form library
const schema = useConvexSchema(
  v.object({
    title: v.string(),
    body: v.string(),
    priority: v.optional(
      v.union(v.literal("low"), v.literal("medium"), v.literal("high")),
    ),
  }),
);
```

Works with Nuxt UI `<UForm :schema>`, VeeValidate `toTypedSchema()`, FormKit, and anything else that speaks Standard Schema.

**Pre-validate mutations before they hit the network:**

```ts
const createPost = useConvexMutation(api.posts.create, {
  // Convex validator — checks shape
  validate: v.object(createPostArgs),
});

// Or Zod/Valibot — checks shape AND format (email, min length, etc.)
const createPost = useConvexMutation(api.posts.create, {
  validate: z.object({
    title: z.string().min(1, "Title required").max(200),
    email: z.string().email("Invalid email"),
  }),
});
```

On validation failure: `createPost.error` is set immediately, `createPost.status` becomes `'error'`, no network request fires. Same behavior as a server rejection — but instant.

The `validate` option accepts Convex validators OR any Standard Schema producer. We don't gatekeep.

**Server-side validation for Nitro routes:**

```ts
// server/api/posts.post.ts
export default defineEventHandler(async (event) => {
  const body = await readValidatedBody(
    event,
    validateConvexArgs(v.object(createPostArgs)),
  );
  return await serverConvexMutation(event, api.posts.create, body);
});
```

Returns 400 with structured field-level issues on failure. Types flow through.

**The shared validator pattern:**

```ts
// convex/schemas/post.ts — define once
export const createPostArgs = {
  title: v.string(),
  body: v.string(),
}

// convex/posts.ts — backend uses it
export const create = mutation({ args: createPostArgs, handler: ... })

// pages/posts/new.vue — form uses it
const schema = useConvexSchema(v.object(createPostArgs))

// server/api/posts.post.ts — API route uses it
validateConvexArgs(v.object(createPostArgs))
```

One definition. Four consumers. Zero drift.

### Authentication

```
useConvexAuth()            → { user, isAuthenticated, isPending, signOut }
useConvexAuthInternal()    → { token, authError, refreshAuth, awaitAuthReady }
```

**Components:**

```
<ConvexAuthenticated>      → renders children only when signed in
<ConvexUnauthenticated>    → renders children only when signed out
<ConvexAuthLoading>        → renders children only while auth is resolving
<ConvexAuthError>          → renders children when auth has failed
```

**Route protection via page meta:**

```ts
definePageMeta({
  convexAuth: true, // redirect to login if not authed
  convexAuth: { redirectTo: "/subscribe" }, // custom redirect
});
```

**How it works under the hood:**

1. Server plugin reads session cookie, exchanges for JWT via Better Auth
2. Token + decoded user are hydrated into `useState` for zero-flash SSR
3. Client plugin picks up the hydrated token and feeds it to `ConvexClient.setAuth()`
4. Token refresh happens automatically, with deduplication and timeout protection
5. 401/403 from any mutation/action triggers configurable recovery (redirect to login)

**Auth caching (opt-in):** Cache JWT tokens server-side to cut TTFB on repeat requests. Memory by default, Redis for multi-instance deployments.

### Permissions

```ts
// composables/usePermissions.ts — you create this once
export const { usePermissions, usePermissionGuard } = createPermissions({
  query: api.auth.getPermissionContext,
  checkPermission,
});
```

Then in any component:

```vue
<script setup>
const { can, role, isAuthenticated } = usePermissions();
</script>

<template>
  <button v-if="can('post.create').value">New Post</button>
  <button v-if="can('post.update', post).value">Edit</button>
  <AdminPanel v-if="role.value === 'admin'" />
</template>
```

Permission checks are reactive computed refs. They update when the user's role changes. The permission context comes from a Convex query (real-time). The check function runs client-side (instant).

`usePermissionGuard` protects entire pages:

```vue
<script setup>
usePermissionGuard({
  permission: "dashboard.view",
  redirectTo: "/unauthorized",
});
</script>
```

### File Uploads

```ts
const upload = useConvexUpload(api.files.generateUploadUrl);

// Single file
const storageId = await upload(file);

// Multiple files — automatic queue with concurrency control
const storageIds = await upload([file1, file2, file3]);
```

**Reactive state on every upload:**

```
upload.pending        → boolean
upload.progress       → 0-100
upload.status         → 'idle' | 'pending' | 'success' | 'error'
upload.error          → Error | null
upload.data           → last storageId
upload.items          → queue items with per-file status
```

**Queue control:**

```
upload.cancelItem(id)
upload.cancelAll()
upload.clearFinished()
upload.reset()
```

**Options:**

```ts
useConvexUpload(api.files.generateUploadUrl, {
  maxConcurrent: 3,
  allowedTypes: ["image/*", "application/pdf"],
  maxSizeBytes: 10 * 1024 * 1024,
  onSuccess: (storageId, file) => {},
  onError: (error, file) => {},
  onProgress: (info, file) => {},
});
```

**Get a signed URL from a storage ID:**

```ts
const imageUrl = useConvexStorageUrl(api.files.getUrl, storageId);
// imageUrl is a computed ref — null while loading, URL when ready
// Automatically skips when storageId is null
```

### Connection State

```ts
const { isConnected, isReconnecting, pendingMutations, shouldShowOfflineUi } =
  useConvexConnectionState();
```

`shouldShowOfflineUi` has a built-in grace period — it doesn't flash "offline" during initial WebSocket setup. Put it in your layout and forget about it:

```vue
<template>
  <div v-if="shouldShowOfflineUi" class="offline-banner">Reconnecting...</div>
  <NuxtPage />
</template>
```

### Server Utilities

Auto-imported in Nitro routes and server middleware:

```
serverConvexQuery(event, query, args?, options?)
serverConvexMutation(event, mutation, args?, options?)
serverConvexAction(event, action, args?, options?)
serverConvexClearAuthCache(sessionToken)
validateConvexArgs(validator)
```

Auth mode per call:

```ts
// Default: auto-attach auth token from session cookie
await serverConvexQuery(event, api.posts.list);

// Explicit: require auth, throw if not authenticated
await serverConvexQuery(event, api.posts.myPosts, {}, { auth: "required" });

// Explicit: no auth (public data)
await serverConvexQuery(event, api.posts.featured, {}, { auth: "none" });
```

### Mutation Event Hooks

Global hooks that fire on every mutation/action. Wire them up once in a plugin:

```ts
// plugins/convex-events.client.ts
export default defineNuxtPlugin((nuxtApp) => {
  nuxtApp.hook(
    "convex:mutation:success",
    ({ name, args, result, duration }) => {
      // Global success handling — toast, analytics, logging
    },
  );

  nuxtApp.hook("convex:mutation:error", ({ name, args, error }) => {
    // Global error handling — toast, error tracking
  });

  nuxtApp.hook(
    "convex:unauthorized",
    ({ source, functionName, redirectTo }) => {
      // Session expired — redirect to login, show modal, etc.
    },
  );
});
```

This replaces the need to add `onSuccess`/`onError` to every individual mutation when you want app-wide behavior.

### DevTools

A custom tab in Nuxt DevTools that shows:

- **Queries:** Every active subscription with its status, data, args, cache key, and update count
- **Mutations:** A timeline of every mutation/action with args, result/error, duration, and optimistic update markers
- **Auth:** Current user, JWT claims, token expiry countdown, SSR auth waterfall timing
- **Auth Proxy:** Request log for the SSR auth proxy with status codes and durations

No setup needed. It appears automatically in dev mode.

### Logging

```ts
// nuxt.config.ts
convex: {
  logging: 'info',   // auth, mutations, actions, uploads, connection events
  logging: 'debug',  // all above + query updates, timing, subscription lifecycle
  logging: false,     // nothing (production default)
}
```

Structured, tagged, filterable. Uses consola under the hood.

## What 1.0 Does NOT Include

These are conscious exclusions — things we evaluated and decided are not our job.

**No backend function wrappers.** We don't ship `customQuery`, `customMutation`, or middleware patterns for Convex functions. That's convex-helpers territory. Your Convex functions are yours to structure.

**No database-level security.** Row-level security, database wrappers, trigger systems — these run inside the Convex runtime. We handle frontend permissions and route protection. The backend is your domain.

**No form generation.** We give you Standard Schema objects that work with any form library. We don't generate `<input>` elements, form layouts, or field labels. Your UI is yours to design.

**No migration utilities.** Schema migrations are a backend concern handled by the Convex CLI and convex-helpers.

**No built-in toast/notification UI.** We give you the hooks (`convex:mutation:success`, `convex:mutation:error`). You bring your own notification library. We don't opine on what your success message looks like.

**No relationship helpers for the backend.** `getOneFrom`, `getManyFrom`, `getManyVia` are useful patterns, but they run inside Convex functions. We're a Nuxt module — we own the Nuxt side of the stack.

**No admin panel.** No generated CRUD routes, no auto-scaffolded dashboards. Your product isn't generic and neither is your admin panel.

## Configuration (the full shape)

```ts
// nuxt.config.ts
export default defineNuxtConfig({
  modules: ["better-convex-nuxt"],

  convex: {
    // Connection
    url: process.env.CONVEX_URL,
    siteUrl: process.env.CONVEX_SITE_URL, // auto-derived from url if not set

    // Auth (true = enable with all defaults)
    auth: {
      enabled: true,
      route: "/api/auth", // auth proxy route
      routeProtection: {
        redirectTo: "/auth/signin", // where to send unauthenticated users
        preserveReturnTo: true, // append ?redirect= to login URL
      },
      unauthorized: {
        enabled: false, // auto-redirect on 401/403 from mutations
        redirectTo: "/auth/signin",
        includeQueries: false, // also redirect on query 401s
      },
      trustedOrigins: [], // CORS for auth proxy
      skipAuthRoutes: [], // routes that skip auth token fetches
      cache: {
        enabled: false, // SSR JWT token caching
        ttl: 60, // seconds
      },
      proxy: {
        maxRequestBodyBytes: 1_048_576,
        maxResponseBodyBytes: 1_048_576,
      },
    },

    // Query defaults (overridable per-query)
    query: {
      server: true, // SSR by default
      subscribe: true, // live subscriptions by default
    },

    // Upload defaults
    upload: {
      maxConcurrent: 3,
    },

    // Permissions (opt-in)
    permissions: false,

    // Validation (1.0 addition)
    validation: {
      stripUnknownFields: false, // default behavior for useConvexSchema
    },

    // Logging
    logging: false, // 'info' | 'debug' | false

    // Advanced debug channels
    debug: {
      authFlow: false,
      clientAuthFlow: false,
      serverAuthFlow: false,
    },
  },
});
```

## The "Feels Like Cheating" Moments

These are the moments where a developer stops, looks at what they just built, and thinks "...that's it?"

### Moment 1: The form that validates itself

```vue
<script setup>
import { v } from "convex/values";
import { createPostArgs } from "~/convex/schemas/post";

const schema = useConvexSchema(v.object(createPostArgs));
const createPost = useConvexMutation(api.posts.create, {
  validate: v.object(createPostArgs),
});
</script>

<template>
  <UForm :schema="schema" @submit="({ data }) => createPost(data)">
    <UFormField label="Title" name="title"><UInput /></UFormField>
    <UFormField label="Body" name="body"><UTextarea /></UFormField>
    <UButton type="submit" :loading="createPost.pending.value">Publish</UButton>
  </UForm>
</template>
```

The form shows field-level errors. The mutation pre-validates before touching the network. The backend enforces the same rules. One schema definition. Zero glue code.

### Moment 2: The real-time list that updates everywhere

```vue
<script setup>
const { data: posts } = await useConvexQuery(api.posts.list);

const deletePost = useConvexMutation(api.posts.remove, {
  optimisticUpdate: (ctx, { id }) => {
    removeFrom(ctx, api.posts.list, {}, (post) => post._id === id);
  },
});
</script>

<template>
  <div v-for="post in posts" :key="post._id">
    <h2>{{ post.title }}</h2>
    <button @click="deletePost({ id: post._id })">Delete</button>
  </div>
</template>
```

Delete a post. It vanishes instantly (optimistic). If you have the same page open in another tab, it vanishes there too (real-time subscription). If the server rejects the delete, it reappears (automatic rollback). No WebSocket code. No state management library. No cache invalidation.

### Moment 3: The auth that just works

```vue
<template>
  <ConvexAuthLoading>
    <AppSkeleton />
  </ConvexAuthLoading>

  <ConvexAuthenticated>
    <NuxtLayout>
      <NuxtPage />
    </NuxtLayout>
  </ConvexAuthenticated>

  <ConvexUnauthenticated>
    <LoginPage />
  </ConvexUnauthenticated>
</template>
```

SSR renders the right state. Client hydrates without flash. Route protection redirects unauthenticated users. Token refresh happens in the background. Session expiry triggers recovery. Zero auth code in your components beyond these three wrapper components.

### Moment 4: The upload that handles everything

```vue
<script setup>
const upload = useConvexUpload(api.files.generateUploadUrl, {
  allowedTypes: ["image/*"],
  maxSizeBytes: 5 * 1024 * 1024,
});

const imageUrl = useConvexStorageUrl(api.files.getUrl, upload.data);
</script>

<template>
  <input
    type="file"
    accept="image/*"
    @change="(e) => upload(e.target.files[0])"
  />
  <progress
    v-if="upload.pending.value"
    :value="upload.progress.value"
    max="100"
  />
  <img v-if="imageUrl" :src="imageUrl" />
  <p v-if="upload.error.value">{{ upload.error.value.message }}</p>
</template>
```

File type validation. Size validation. Upload progress. Automatic signed URL that stays in sync. Error handling. That's the entire file upload feature for your app.

### Moment 5: The server route that trusts its input


## What Changed From 0.x to 1.0

| Area             | 0.x                                     | 1.0                                                                                |
| ---------------- | --------------------------------------- | ---------------------------------------------------------------------------------- |
| Validation       | Not included                            | `useConvexSchema`, `validate` option on mutations, `validateConvexArgs` for server |
| Standard Schema  | Not included                            | Native producer from Convex validators, accepts Zod/Valibot as consumers           |
| Mutation hooks   | Per-mutation `onSuccess`/`onError` only | Global Nuxt hooks for app-wide mutation/action handling                            |
| Validator walker | Not included                            | Own implementation with multi-error collection and field paths                     |
| Error class      | Basic Error wrapping                    | `ConvexCallError` with code, status, function path, and structured issues          |

Everything else in 1.0 already exists in the 0.x series. The 1.0 designation means the API is stable, the edge cases are handled, and we're confident enough to promise backward compatibility.

## The Principle

Every line of code in this module exists because someone was building a real app and hit a wall. We removed the wall. We didn't add a door to it, or a sign explaining where the door is. We removed the wall.

That's what 1.0 means.
