# better-convex-nuxt 1.0

**The full-stack Nuxt module for Convex that feels like cheating.**

---

## What This Module Believes

You picked Convex because you don't want to babysit a database. You picked Nuxt because you don't want to babysit a bundler. You shouldn't have to babysit the seam between them either.

This module removes that seam. Auth, real-time data, validation, permissions, file uploads, error handling — they work together as one system. Not a collection of utilities. A system.

Every feature ships because someone building a real product needed it. Nothing ships because it looked good in a README.

---

## The Promise

A developer using this module can:

1. **Go from zero to authenticated app in minutes.** Sign in, sign up, route protection, session handling — opt in and it works. Build custom auth pages when you're ready. Use the built-in ones until then.

2. **Define data rules once and enforce them everywhere.** One validator definition. Used by the backend mutation, the Vue form, the server API route, and the mutation pre-validation. Change it in one place, every layer updates.

3. **Build real-time UIs without thinking about real-time.** Queries subscribe to live data by default. Mutations are optimistic. Connection drops show graceful offline UI. This isn't a feature — it's the baseline.

4. **Know what went wrong and why.** Every error has a category, a code, and a function path. DevTools show the full story of every query, mutation, and auth event. Inline warnings teach you the right patterns before you hit bugs.

5. **Check permissions in a template expression.** `v-if="can('post.update', post).value"` — reactive, ownership-aware, role-based. Define your roles once, the module handles inheritance and checking.

6. **Upload files without thinking about the upload dance.** Pick file, call composable, get storage ID. Queue multiple files. Show progress. Cancel. Done.

## I. Data Fetching

### Queries (real-time by default)

```ts
const { data, pending, error, status, refresh } = await useConvexQuery(
  api.posts.list,
  () => ({ teamId: teamId.value }),
);
```

SSR on first render. WebSocket subscription on the client. Data stays in sync across tabs, users, and the world. When args are null or undefined, the query skips — no wasted requests.

**Options:**

```ts
useConvexQuery(api.posts.list, args, {
  server: false, // skip SSR, load client-only %% what if we say nuxt SSR: false
  subscribe: false, // fetch once, no live updates
  default: () => [], // fallback while loading
  transform: (posts) => posts.map(formatPost), // reshape before template
  keepPreviousData: true, // don't flash empty while new args load
  shared: "posts-list", // deduplicate across components
  onData: (data) => {}, // react to every update
  onError: (error) => {}, // react to failures
});
```

### Paginated Queries

```ts
const {
  results,
  status,
  isLoading,
  isExhausted,
  hasNextPage,
  loadMore,
  error,
  refetch,
  restart,
} = await useConvexPaginatedQuery(
  api.posts.listPaginated,
  () => ({ teamId: teamId.value }),
  { initialNumItems: 20 },
);
```

```vue
<template>
  <div v-for="post in results" :key="post._id">{{ post.title }}</div>
  <button v-if="hasNextPage" @click="loadMore(20)">Load more</button>
  <p v-if="isExhausted">No more posts</p>
</template>
```

## II. Mutations & Actions

```ts
const createPost = useConvexMutation(api.posts.create);

await createPost({ title: "Hello", body: "World" });

createPost.pending.value; // was true during the call
createPost.status.value; // 'success' | 'error' | 'idle' | 'pending'
createPost.data.value; // return value from the mutation
createPost.error.value; // ConvexCallError | null
createPost.reset(); // back to idle
```

Actions work identically:

```ts
const sendEmail = useConvexAction(api.emails.send);
await sendEmail({ to: "user@example.com", subject: "Hello" });
```

### Validation (define once, enforce everywhere)

**Pre-validate before hitting the network:**

```ts
import { createPostArgs } from "~/convex/schemas/post";

const createPost = useConvexMutation(api.posts.create, {
  validate: v.object(createPostArgs),
});

// Bad args? Error is instant — no network request, no loading spinner
await createPost({ title: "" });
// createPost.error.value → ConvexCallError {
//   category: 'validation',
//   code: 'VALIDATION_ERROR',
//   issues: [{ message: 'Required field missing', path: ['title'] }]
// }
```

The `validate` option accepts Convex validators, Zod schemas, Valibot schemas — anything that produces Standard Schema. Use Convex validators for shape, Zod for format rules. Mix freely:

```ts
// Zod for rich validation (email format, min length, etc.)
const createUser = useConvexMutation(api.users.create, {
  validate: z.object({
    name: z.string().min(1, "Name is required"),
    email: z.string().email("Invalid email"),
  }),
});
```

**Same validators in your forms:**

```ts
const schema = useConvexSchema(v.object(createPostArgs));
```

Works with Nuxt UI `<UForm :schema>`, VeeValidate, FormKit — anything that speaks Standard Schema.

```vue
<UForm :schema="schema" @submit="({ data }) => createPost(data)">
  <UFormField label="Title" name="title"><UInput /></UFormField>
  <UFormField label="Body" name="body"><UTextarea /></UFormField>
  <UButton type="submit" :loading="createPost.pending.value">Publish</UButton>
</UForm>
```

Field-level errors in the form. Pre-validation on the mutation. Backend enforcement on the function. One schema definition. Zero glue code.

### Optimistic Updates

```ts
const addNote = useConvexMutation(api.notes.add, {
  optimisticUpdate: (ctx, args) => {
    ctx
      .query(api.notes.list, {})
      .update((notes) => [
        { _id: crypto.randomUUID(), _creationTime: Date.now(), ...args },
        ...notes,
      ]);
  },
});
```

Plus standalone helpers:

```ts
prependTo(ctx, api.notes.list, {}, newNote);
appendTo(ctx, api.notes.list, {}, newNote);
removeFrom(ctx, api.notes.list, {}, (note) => note._id === deletedId);
updateIn(
  ctx,
  api.notes.list,
  {},
  (note) => note._id === id,
  (note) => ({ ...note, ...patch }),
);
```

For paginated queries:

```ts
ctx.paginatedQuery(api.notes.paginated, {}).insertAtTop(newNote);
ctx.paginatedQuery(api.notes.paginated, {}).deleteItem(noteId);
ctx
  .paginatedQuery(api.notes.paginated, {})
  .updateItem(noteId, (note) => ({ ...note, title: "Updated" }));
```

### Lifecycle callbacks

```ts
const createPost = useConvexMutation(api.posts.create, {
  onSuccess: (result, args) => {
    /* per-mutation handler */
  },
  onError: (error, args) => {
    /* per-mutation handler */
  },
});
```

### Global mutation hooks

Wire up app-wide behavior once, in a plugin:

```ts
// plugins/convex-events.client.ts
export default defineNuxtPlugin((nuxtApp) => {
  nuxtApp.hook("convex:mutation:success", ({ name, result, duration }) => {
    // Toast, analytics, logging — for every mutation in the app
  });

  nuxtApp.hook("convex:mutation:error", ({ name, error, args }) => {
    if (error.category === "validation") return; // form handles it
    if (error.category === "auth") {
      toast.add({ title: "Session expired" });
      return;
    }
    toast.add({ title: "Something went wrong" });
    errorReporter.capture(error);
  });
});
```

No more adding `onError` to every individual mutation for app-wide toast notifications.

## III. Error Handling

### Unified errors everywhere

Every error from this module is a `ConvexCallError`, regardless of source:

```ts
class ConvexCallError extends Error {
  category:
    | "validation"
    | "auth"
    | "not_found"
    | "rate_limit"
    | "network"
    | "server"
    | "conflict"
    | "unknown";
  code?: string; // 'VALIDATION_ERROR', 'UNAUTHENTICATED', 'LIMIT_CALLS'
  status?: number; // 401, 403, 500, etc.
  operation?: string; // 'query', 'mutation', 'action'
  functionPath?: string; // 'posts:create'
  issues?: ValidationIssue[]; // field-level validation errors
}
```

The `category` is auto-derived from the error's code and status. A global error handler becomes a simple switch:

```ts
switch (error.category) {
  case "validation":
    break; // form shows field errors
  case "auth":
    showSignInPrompt();
  case "network":
    showOfflineBanner();
  case "rate_limit":
    showSlowDown();
  default:
    reportToSentry(error);
}
```

Queries, mutations, actions, auth, server utilities — all produce the same error shape. One handler covers everything.

### Callback errors are never swallowed

If your `onError` handler throws (e.g., your Sentry integration has a bug), the error is always logged — not silently discarded. In dev and production.

## IV. Authentication

### Reading auth state

```ts
const {
  user, // the authenticated user (readonly)
  isAuthenticated, // has valid token + user
  isPending, // still resolving (first load)
  isAnonymous, // not authenticated and not pending
  isSessionExpired, // had auth before, lost it (vs. never had it)
  signOut, // clear auth + call Better Auth signOut
} = useConvexAuth();
```

`isAnonymous` and `isSessionExpired` make templates read like English:

```vue
<!-- First-time visitor -->
<WelcomePage v-if="isAnonymous && !isSessionExpired" />

<!-- Returning user whose session expired -->
<WelcomeBackPage v-if="isSessionExpired" />

<!-- Authenticated -->
<Dashboard v-if="isAuthenticated" />
```

### Auth components

```vue
<template>
  <ConvexAuthLoading>
    <AppSkeleton />
  </ConvexAuthLoading>

  <ConvexAuthenticated>
    <NuxtLayout>
      <NuxtPage />
    </NuxtLayout>
  </ConvexAuthenticated>

  <ConvexUnauthenticated>
    <LoginPage />
  </ConvexUnauthenticated>

  <ConvexAuthError v-slot="{ error, retry, retrying, isRecoverable }">
    <div v-if="isRecoverable">
      <p>Session expired.</p>
      <button @click="retry" :disabled="retrying">Sign in again</button>
    </div>
    <div v-else>
      <p>Authentication error: {{ error.message }}</p>
    </div>
  </ConvexAuthError>
</template>
```

`ConvexAuthError`'s retry calls `refreshAuth()` — not `window.location.reload()`. Seamless recovery without a full page reload.

### Sign-in and sign-up composables

The #1 auth confusion today: "I signed in with Better Auth but Convex doesn't know I'm authenticated." These composables eliminate that by wiring together sign-in, Convex auth refresh, and post-login redirect in one call:

```ts
const { signIn, pending, error } = useConvexSignIn();

async function handleSubmit() {
  await signIn({
    method: "email",
    email: form.email,
    password: form.password,
    redirectTo: "/dashboard",
  });
  // That's it. Signed in, Convex auth refreshed, navigated.
}
```

The redirect defaults to the `?redirect=` query param (set by route protection), then falls back to the provided path. Open redirect attacks are blocked — only relative paths are followed.

`useConvexSignUp()` is the same shape for registration.

For social auth:

```ts
await signIn({ method: "social", provider: "github" });
// Redirects to GitHub — no local navigation
```

### Built-in auth pages (opt-in)

For prototyping, MVPs, and internal tools — functional sign-in and sign-up pages that you don't have to build:
%% instead we make copy paste blocks in docs

```ts
// nuxt.config.ts
convex: {
  auth: {
    pages: true,  // adds /auth/signin and /auth/signup
  },
}
```

Unstyled by default. Override any time by creating your own `pages/auth/signin.vue`.

### Route protection

Via page meta:

```ts
// pages/dashboard.vue
definePageMeta({
  convexAuth: true,
});

// pages/admin.vue — custom redirect
definePageMeta({
  convexAuth: { redirectTo: "/upgrade" },
});
```

The middleware waits for auth to settle (no flash of protected content), appends `?redirect=` for post-login return, and handles SSR correctly.

### Post-login redirect helper

```ts
const { redirectAfterAuth } = useAuthRedirect();

// After manual sign-in flow:
await signIn(credentials);
await refreshAuth();
redirectAfterAuth("/dashboard"); // reads ?redirect param, falls back to /dashboard
```

Open redirect prevention, loop detection, and URL decoding built in.

### Advanced auth (for middleware, plugins, and edge cases)

```ts
const {
  token, // raw JWT (readonly)
  authError, // structured error object (not a string)
  refreshAuth, // force token refresh
  awaitAuthReady, // wait for auth to settle (for middleware)
} = useConvexAuthInternal();
```

### Unauthorized recovery

When any mutation or action gets a 401/403, the `convex:unauthorized` hook fires:

```ts
nuxtApp.hook(
  "convex:unauthorized",
  ({ error, source, functionName, redirectTo }) => {
    // Handle globally: redirect to login, show modal, etc.
  },
);
```

Debounced and deduplicated — five simultaneous 401s trigger one recovery, not five.

### How it works under the hood

1. **SSR:** Server plugin reads session cookie → exchanges for JWT via Better Auth → hydrates token + user into `useState`
2. **Client:** Client plugin picks up hydrated token → feeds it to `ConvexClient.setAuth()` → no flash of unauthenticated content
3. **Refresh:** Token refresh happens automatically via the Convex client's built-in schedule. In-flight deduplication prevents concurrent fetches
4. **Recovery:** 401/403 from any Convex call fires the unauthorized hook. Session expiry is detected and exposed via `isSessionExpired`

**Auth caching (opt-in):** Cache JWT tokens server-side to cut TTFB. Memory by default, configure Nitro storage for Redis in multi-instance deployments.

## V. Permissions

### Setup

Define your roles and permissions once:

```ts
// convex/permissions.config.ts
import { defineRoles } from "better-convex-nuxt/permissions";

export const { checkPermission, permissions } = defineRoles({
  roles: {
    admin: {
      inherits: ["editor"],
      permissions: ["users.manage", "settings.edit", "billing.manage"],
    },
    editor: {
      inherits: ["viewer"],
      permissions: ["posts.create", "posts.update", "posts.delete"],
    },
    viewer: {
      permissions: ["posts.read", "comments.create"],
    },
  },
  ownership: {
    "posts.update": (ctx, resource) => resource?.ownerId === ctx.userId,
    "posts.delete": (ctx, resource) => resource?.ownerId === ctx.userId,
  },
});
```

Role inheritance is resolved automatically. Admin inherits editor's permissions, editor inherits viewer's. Ownership rules compose on top — an editor can edit any post, a viewer can only edit their own.

Create your composable:

```ts
// composables/usePermissions.ts
import { createPermissions } from "#imports";
import { api } from "~/convex/_generated/api";
import { checkPermission } from "~/convex/permissions.config";

export const { usePermissions, usePermissionGuard } = createPermissions({
  query: api.auth.getPermissionContext,
  checkPermission,
});
```

### Usage in components

```vue
<script setup>
const { can, role, isAuthenticated } = usePermissions();
</script>

<template>
  <button v-if="can('posts.create').value">New Post</button>
  <button v-if="can('posts.update', post).value">Edit</button>
  <button v-if="can('posts.delete', post).value">Delete</button>
  <AdminPanel v-if="role.value === 'admin'" />
</template>
```

Permission checks are reactive computed refs. The permission context comes from a real-time Convex query — if an admin changes your role, the buttons update without a refresh.

### Page-level protection

```vue
<script setup>
usePermissionGuard({
  permission: "settings.edit",
  redirectTo: "/dashboard",
});
</script>
```

### DevTools permission matrix

In development, the DevTools panel shows a visual permission matrix:

## VI. File Uploads

```ts
const upload = useConvexUpload(api.files.generateUploadUrl, {
  allowedTypes: ["image/*", "application/pdf"],
  maxSizeBytes: 10 * 1024 * 1024,
  maxConcurrent: 3,
});

// Single file
const storageId = await upload(file);

// Multiple files — automatic queue
const storageIds = await upload([file1, file2, file3]);
```

**Reactive state:**

```
upload.pending        → boolean
upload.progress       → 0-100 (aggregate for queue)
upload.status         → 'idle' | 'pending' | 'success' | 'error'
upload.error          → Error | null
upload.data           → last storageId
upload.items          → per-file queue items
```

**Queue control:**

```
upload.cancelItem(id)
upload.cancelAll()
upload.clearFinished()
upload.reset()
```

**Signed URL from storage ID:**

```ts
const imageUrl = useConvexStorageUrl(api.files.getUrl, storageId);
// Reactive computed ref — null while loading, URL when ready
// Skips automatically when storageId is null
```

## VII. Server Utilities

Auto-imported in Nitro routes:

```ts
serverConvexQuery(event, query, args?, options?)
serverConvexMutation(event, mutation, args?, options?)
serverConvexAction(event, action, args?, options?)
validateConvexArgs(validator)
serverConvexClearAuthCache(sessionToken)
```

### The one-liner server route

For the 80% of endpoints that validate input, check auth, and call a mutation:

```ts
// server/api/posts.post.ts
export default defineConvexHandler(api.posts.create, {
  validate: v.object(createPostArgs),
  auth: "required",
});
```

One line. Validates the request body. Attaches auth from the session cookie. Calls the mutation. Returns the result. Proper errors on failure.

### Protected handlers

For routes that need auth but have custom logic:

```ts
// server/api/settings.get.ts
export default defineProtectedHandler(async (event) => {
  // Auth already verified — 401 thrown automatically if not authenticated
  return await serverConvexQuery(event, api.settings.get);
});
```

### Full control

For the 20% that need custom logic:

```ts
// server/api/webhooks/stripe.post.ts
export default defineEventHandler(async (event) => {
  const body = await readValidatedBody(
    event,
    validateConvexArgs(v.object(webhookArgs)),
  );
  // Custom webhook verification, rate limiting, etc.
  return await serverConvexMutation(event, api.billing.processWebhook, body, {
    auth: "none",
  });
});
```

## IX. Debugging & DevTools

### DevTools panel

A custom tab in Nuxt DevTools with four views:

**Queries:** Every active subscription — status, args, data, cache key, update count, and a diagnostic that tells you _why_ a query is in its current state ("skipped because args are null", "pending — waiting for first subscription update", "success — 12 items from WebSocket").

**Mutations:** Timeline of every mutation and action — args, result or error, duration, optimistic update markers, and expandable details.

**Auth:** Current user, JWT claims, token expiry countdown, SSR auth waterfall as a readable story (not just a timing chart), and the auth proxy request log.

**Permissions:** Visual role permission matrix showing who can do what, with ownership rules highlighted.

### Auth flow story

Instead of a raw timing chart, the auth waterfall reads like a narrative:

```
1. SSR: Found session cookie ✓
2. SSR: Token exchange with Convex (89ms) → JWT received
3. SSR: JWT decoded → user: dev@example.com
4. Client: Hydrated token from SSR ✓
5. Client: Authenticated

Current: dev@example.com | Token expires in 58 minutes
```

When something fails, it explains what went wrong and what to fix:

```
2. SSR: Token exchange TIMEOUT (8002ms)
   ⚠️ Check your siteUrl configuration or Convex deployment status

4. Client: Token fetch via auth proxy → success (247ms)
   Note: Auth worked client-side but failed server-side.
   This caused a flash of unauthenticated content during hydration.
```

## What 1.0 Does NOT Include

These are conscious exclusions.

**No backend function wrappers.** `customQuery`, `customMutation`, middleware patterns — these run inside the Convex runtime. Your Convex functions are yours to structure. Use convex-helpers for that.

**No database-level security.** Row-level security, database wrappers, trigger systems — Convex backend concerns, not Nuxt module concerns.

**No form generation.** We give you Standard Schema objects. You bring your form library and your UI.

**No migration utilities.** Schema migrations are handled by the Convex CLI.

**No built-in toast/notification UI.** We give you the hooks. You bring your own notification library.

**No relationship helpers for the backend.** `getOneFrom`, `getManyFrom` run inside Convex functions. We own the Nuxt side.

**No admin panel.** No generated CRUD routes. Your product isn't generic and neither is your admin panel.

**No dynamic schema validation.** We validate against static validators defined in code. Runtime-defined schemas (like a CMS where users create field types) are an app-level concern.

### Components

| Component                 | Renders when                                    |
| ------------------------- | ----------------------------------------------- |
| `<ConvexAuthenticated>`   | User is signed in                               |
| `<ConvexUnauthenticated>` | User is not signed in                           |
| `<ConvexAuthLoading>`     | Auth is still resolving                         |
| `<ConvexAuthError>`       | Auth has failed (with retry + structured error) |

## What Changed From 0.x

| Area                 | 0.x                                     | 1.0                                                                                    |
| -------------------- | --------------------------------------- | -------------------------------------------------------------------------------------- |
| Validation           | Not included                            | `useConvexSchema`, `validate` on mutations, `validateConvexArgs`, own validator walker |
| Error handling       | Inconsistent types across layers        | Unified `ConvexCallError` everywhere with `category` and `issues`                      |
| Error callbacks      | Silently swallowed in production        | Always logged                                                                          |
| Sign-in/sign-up      | Manual three-step process               | `useConvexSignIn()` / `useConvexSignUp()` — one call                                   |
| Auth pages           | Build your own from scratch             | Built-in pages (opt-in), override anytime                                              |
| Auth error recovery  | `window.location.reload()`              | `refreshAuth()` with `isRecoverable` state                                             |
| Auth state           | `isAuthenticated`, `isPending`          | Added `isAnonymous`, `isSessionExpired`                                                |
| Auth errors          | Flat string                             | Structured object with code, message, recoverable flag                                 |
| Post-login redirect  | DIY (with security bugs)                | `useAuthRedirect()` with open redirect prevention                                      |
| Permissions          | `createPermissions()` only              | Added `defineRoles()` preset builder                                                   |
| Permission DevTools  | Not included                            | Visual role permission matrix                                                          |
| Mutation hooks       | Per-mutation `onSuccess`/`onError` only | Global Nuxt hooks for app-wide handling                                                |
| Server routes        | Manual validation + auth wiring         | `defineConvexHandler()` one-liner, `defineProtectedHandler()`                          |
| DevTools diagnostics | Raw data display                        | Query diagnostics ("why is this skipped?"), auth story view, debug report export       |
| Dev warnings         | Minimal                                 | Inline contextual warnings with doc links                                              |

Everything else is unchanged. The query system, the subscription deduplication, the optimistic update builder, the upload queue, the connection state, the auth proxy, the logging — all stable from 0.x.

## The Principle

Every wall a developer hits between "I have an idea" and "it's running in production" — we removed it. We didn't add a door. We didn't add a sign. We removed the wall.

That's what 1.0 means.
