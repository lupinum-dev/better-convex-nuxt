# better-convex-nuxt 1.0

---

## What This Module Is

The full-stack bridge between Nuxt and Convex. Auth, real-time data, validation, permissions, uploads, error handling — one system, not a bag of utilities.

Every feature exists because someone building a real product hit a wall. We removed the wall.

---

## The 1.0 Contract

1.0 means the API is frozen. Composable names, return shapes, config keys, hook names — they're stable. Patch releases fix bugs. Minor releases add features. Nothing breaks.

This document defines the target API. If something is listed here, it ships in 1.0. If it's not listed, it's not promised.

## I. Data Fetching

### Queries

```ts
const { data, pending, error, status, refresh, clear } = await useConvexQuery(
  api.posts.list,
  () => ({ teamId: teamId.value }),
);
```

SSR on first render. WebSocket subscription on the client. When args are null or undefined, the query skips.

**Options:**

```ts
useConvexQuery(api.posts.list, args, {
  server: false, // skip SSR
  subscribe: false, // one-shot, no live updates
  default: () => [], // fallback while loading
  transform: (posts) => posts.map(formatPost),
  keepPreviousData: true, // preserve data while new args load
  shared: "posts-list", // deduplicate across components
  onData: (data) => {},
  onError: (error) => {},
});
```

### Paginated Queries

```ts
const {
  results,
  status,
  isLoading,
  isExhausted,
  hasNextPage,
  loadMore,
  error,
  refetch,
  restart,
} = await useConvexPaginatedQuery(
  api.posts.paginated,
  () => ({ teamId: teamId.value }),
  { initialNumItems: 20 },
);
```

### One-Shot Queries (non-reactive)

For middleware, plugins, server routes, and any context where you need a single result without a subscription:

```ts
// In middleware or plugin — no subscription, no component lifecycle
const posts = await executeConvexQuery(api.posts.list, { teamId });
```

Auto-imported. Works in client middleware, server middleware, plugins, and composables that run outside component scope.

### Raw Client Access

```ts
const convex = useConvex();
```

Escape hatch for advanced use cases.

## II. Mutations & Actions

```ts
const createPost = useConvexMutation(api.posts.create);

await createPost({ title: "Hello", body: "World" });

createPost.pending.value; // reactive state
createPost.status.value; // 'idle' | 'pending' | 'success' | 'error'
createPost.data.value; // return value
createPost.error.value; // ConvexCallError | null
createPost.reset(); // back to idle
```

Actions are identical in shape:

```ts
const sendEmail = useConvexAction(api.emails.send);
await sendEmail({ to: "user@example.com", subject: "Hello" });
```

### Pre-Validation

The `validate` option checks args before they hit the network. On failure: error is set instantly, no request fires, same error flow as a server rejection.

```ts
import { createPostArgs } from "~/convex/schemas/post";

const createPost = useConvexMutation(api.posts.create, {
  validate: v.object(createPostArgs),
});
```

Accepts Convex validators OR any Standard Schema producer (Zod, Valibot, ArkType). Use whichever fits:

```ts
// Zod for format rules Convex validators can't express
const createUser = useConvexMutation(api.users.create, {
  validate: z.object({
    name: z.string().min(1, "Name required"),
    email: z.string().email("Invalid email"),
  }),
});
```

On validation failure, `error.value` is a `ConvexCallError` with `category: 'validation'` and an `issues` array containing field-level paths and messages.

### Optimistic Updates

```ts
const addNote = useConvexMutation(api.notes.add, {
  optimisticUpdate: (ctx, args) => {
    ctx
      .query(api.notes.list, {})
      .update((notes) => [
        { _id: crypto.randomUUID(), _creationTime: Date.now(), ...args },
        ...notes,
      ]);
  },
});
```

Standalone helpers for common patterns:

```ts
prependTo(ctx, api.notes.list, {}, newNote);
appendTo(ctx, api.notes.list, {}, newNote);
removeFrom(ctx, api.notes.list, {}, (note) => note._id === id);
updateIn(
  ctx,
  api.notes.list,
  {},
  (note) => note._id === id,
  (note) => ({ ...note, ...patch }),
);
```

Paginated query operations:

```ts
ctx.paginatedQuery(api.notes.paginated, {}).insertAtTop(newNote);
ctx.paginatedQuery(api.notes.paginated, {}).deleteItem(noteId);
ctx.paginatedQuery(api.notes.paginated, {}).updateItem(noteId, updater);
```

### Lifecycle Callbacks ✅

Per-mutation:

```ts
useConvexMutation(api.posts.create, {
  onSuccess: (result, args) => {},
  onError: (error, args) => {},
});
```

### Global Hooks ✅

App-wide mutation and action handling via Nuxt hooks:

```ts
// plugins/convex-events.client.ts
export default defineNuxtPlugin((nuxtApp) => {
  nuxtApp.hook(
    "convex:mutation:success",
    ({ functionPath, operation, args, result, duration }) => {
      // toast, analytics, logging — for every mutation
    },
  );

  nuxtApp.hook(
    "convex:mutation:error",
    ({ functionPath, operation, args, error, duration }) => {
      if (error.category === "validation") return; // form handles it
      if (error.category === "auth") {
        toast.add({ title: "Session expired" });
        return;
      }
      toast.add({ title: "Something went wrong" });
      errorReporter.capture(error);
    },
  );
});
```

Eliminates per-mutation toast/error boilerplate across the entire app.

Hooks fire after per-call `onSuccess`/`onError` callbacks, using fire-and-forget (`void callHook`) to avoid blocking the mutation return. Error payloads include the full `ConvexCallError` with `category`, `code`, and `issues`.

## III. Validation

### The Pattern

Define validators once. Use them everywhere.

```ts
// convex/schemas/post.ts — single source of truth
import { v } from "convex/values";

export const createPostArgs = {
  title: v.string(),
  body: v.string(),
};
```

```ts
// convex/posts.ts — backend uses it
export const create = mutation({ args: createPostArgs, handler: ... })
```

```vue
<!-- pages/posts/new.vue — form uses it -->
<script setup>
const schema = useConvexSchema(v.object(createPostArgs));
const createPost = useConvexMutation(api.posts.create, {
  validate: v.object(createPostArgs),
});
</script>

<template>
  <UForm :schema="schema" @submit="({ data }) => createPost(data)">
    <UFormField label="Title" name="title"><UInput /></UFormField>
    <UFormField label="Body" name="body"><UTextarea /></UFormField>
    <UButton type="submit" :loading="createPost.pending.value">Publish</UButton>
  </UForm>
</template>
```

```ts
// server/api/posts.post.ts — server route uses it
export default defineEventHandler(async (event) => {
  const body = await readValidatedBody(
    event,
    validateConvexArgs(v.object(createPostArgs)),
  );
  return await serverConvexMutation(event, api.posts.create, body);
});
```

One definition. Four consumers. Zero drift.

### What we produce and consume

We **produce** Standard Schema from Convex validators via `useConvexSchema()` and `toConvexSchema()`.

We **accept** any Standard Schema producer (Zod, Valibot, ArkType, our own converter) in the `validate` option and in `validateConvexArgs`.

We don't own Zod. We don't own Valibot. We don't own form libraries. We speak the universal interface that they all speak.

### API

| Function                        | Context      | What it does                                               |
| ------------------------------- | ------------ | ---------------------------------------------------------- |
| `useConvexSchema(validator)`    | Client + SSR | Convex validator → Standard Schema for form libraries      |
| `toConvexSchema(validator)`     | Anywhere     | Same thing, not a composable                               |
| `validateConvexArgs(validator)` | Server       | Returns a validation function for H3's `readValidatedBody` |

## IV. Error Handling

### Unified Error Class ✅

Every error from this module is a `ConvexCallError`:

```ts
class ConvexCallError extends Error {
  category:
    | "validation"
    | "auth"
    | "not_found"
    | "rate_limit"
    | "network"
    | "server"
    | "conflict"
    | "unknown";
  code?: string;
  status?: number;
  operation?: string; // 'query' | 'mutation' | 'action'
  functionPath?: string; // 'posts:create'
  issues?: ConvexErrorIssue[]; // field-level errors (when category is 'validation')
  get isRecoverable(): boolean; // true for auth, network, rate_limit
}
```

Queries, mutations, actions, auth flows — all produce the same error shape. The `category` is auto-derived from code and status via the `categorizeError()` pure function:

- 401/403 or `UNAUTHENTICATED` → `'auth'`
- 422 or `VALIDATION_ERROR` → `'validation'`
- `LIMIT_*` → `'rate_limit'`
- Network failures (TypeError, AbortError) → `'network'`
- 404 → `'not_found'`
- 409 → `'conflict'`
- 500+ → `'server'`

The `isRecoverable` getter returns `true` for categories where retrying may succeed (auth, network, rate_limit).

### Callback errors are never swallowed

If an `onSuccess` or `onError` callback throws, the error is logged in both dev and production. Not silently discarded.

## V. Authentication

### Auth State ✅

```ts
const {
  user, // authenticated user (readonly)
  isAuthenticated, // has valid token + user
  isPending, // still resolving
  isAnonymous, // not authenticated and not pending (reads better in templates)
  isSessionExpired, // had auth before, lost it (returning user vs first-time visitor)
  signOut, // clears auth + calls Better Auth signOut
} = useConvexAuth();
```

```vue
<template>
  <!-- First-time visitor -->
  <WelcomePage v-if="isAnonymous && !isSessionExpired" />

  <!-- Returning user whose session expired -->
  <WelcomeBackPage v-if="isSessionExpired" />

  <!-- Authenticated -->
  <Dashboard v-if="isAuthenticated" />
</template>
```

### Auth Components

```vue
<ConvexAuthenticated>      <!-- renders when signed in -->
<ConvexUnauthenticated>    <!-- renders when signed out -->
<ConvexAuthLoading>        <!-- renders while auth resolves -->
<ConvexAuthError>          <!-- renders on auth failure -->
```

`ConvexAuthError` provides structured error info and retries via `refreshAuth()` — not a page reload: ✅

```vue
<ConvexAuthError v-slot="{ error, retry, retrying, isRecoverable }">
  <div v-if="isRecoverable">
    <p>Session expired.</p>
    <button @click="retry" :disabled="retrying">Try again</button>
  </div>
  <div v-else>
    <p>{{ error.message }}</p>
  </div>
</ConvexAuthError>
```

### Auth Flow Composable ✅

A thin wrapper that orchestrates any Better Auth method + Convex token refresh + post-login redirect in one call:

```ts
const { execute, pending, error } = useConvexAuthFlow();

// Sign in with email
async function handleSignIn() {
  await execute(
    () => auth.signIn.email({ email: form.email, password: form.password }),
    { redirectTo: "/dashboard" },
  );
  // Signed in. Convex auth refreshed. Navigated. Done.
}

// Sign up
async function handleSignUp() {
  await execute(
    () => auth.signUp.email({ name: form.name, email: form.email, password: form.password }),
    { redirectTo: "/onboarding" },
  );
}

// Magic link, passkey, password reset — anything Better Auth supports
async function handlePasswordReset() {
  await execute(
    () => auth.resetPassword({ token, newPassword }),
    { redirectTo: "/dashboard" },
  );
}
```

The composable doesn't know or care which Better Auth method is being called. It:
1. Calls the provided function
2. Detects Better Auth's `{ data, error }` pattern — if `error` is non-null, wraps it as a `ConvexCallError` and throws
3. Calls `refreshAuth()` to sync the Convex JWT
4. Redirects via `useAuthRedirect()` (reads `?redirect=` param, prevents open redirects)

Social auth (`auth.signIn.social({ provider })`) triggers a full-page redirect — call it directly without `execute()`, since there's no token refresh or redirect to orchestrate on the current page.

Errors from Better Auth are wrapped via `wrapBetterAuthError()` into `ConvexCallError` with proper `category` (e.g., 401 → `'auth'`, 429 → `'rate_limit'`, 422 → `'validation'`).

### Post-Login Redirect Helper ✅

For custom flows that don't use `useConvexAuthFlow`:

```ts
const { redirectAfterAuth } = useAuthRedirect();

await someCustomSignIn();
await refreshAuth();
redirectAfterAuth("/dashboard");
// Reads ?redirect param, prevents open redirects, prevents login-page loops
```

The redirect safety layer (`validateRedirectPath`, `resolveRedirectTarget`) is exported as pure functions for use outside composable context:
- Only relative paths starting with `/` are accepted
- Protocol-relative URLs (`//evil.com`) and double-slash tricks (`/foo//evil.com`) are rejected
- `javascript:`, `data:`, and other non-http protocols are rejected
- Login-page loops are prevented (redirect to login → redirects to `/` instead)

### Route Protection

Via page meta:

```ts
definePageMeta({ convexAuth: true });
definePageMeta({ convexAuth: { redirectTo: "/upgrade" } });
```

The middleware waits for auth to settle (no flash), appends `?redirect=` for post-login return, handles SSR correctly.

### Unauthorized Recovery

When any Convex call returns 401/403:

```ts
nuxtApp.hook(
  "convex:unauthorized",
  ({ error, source, functionName, redirectTo }) => {
    // Redirect to login, show modal, etc.
  },
);
```

Debounced and deduplicated — five simultaneous 401s trigger one handler, not five.

### Advanced Auth

```ts
const {
  token, // raw JWT (readonly)
  authError, // structured error (not a string)
  refreshAuth, // force token refresh
  awaitAuthReady, // wait for auth to settle (for middleware)
} = useConvexAuthInternal();
```

### Under the Hood

1. Server plugin reads session cookie → exchanges for JWT → hydrates into useState
2. Client plugin picks up hydrated token → feeds to ConvexClient.setAuth()
3. Token refresh is automatic, deduplicated, timeout-protected
4. Auth caching (opt-in): cache JWT server-side to cut TTFB

## VI. Permissions

### Factory Pattern

You bring your own `checkPermission` function. The module provides the reactive composables:

```ts
// composables/usePermissions.ts
import { createPermissions } from "#imports";
import { api } from "~/convex/_generated/api";
import { checkPermission } from "~/convex/permissions.config";

export const { usePermissions, usePermissionGuard } = createPermissions({
  query: api.auth.getPermissionContext,
  checkPermission,
});
```

### Usage

```vue
<script setup>
const { can, role, isAuthenticated } = usePermissions();
</script>

<template>
  <button v-if="can('posts.create').value">New Post</button>
  <button v-if="can('posts.update', post).value">Edit</button>
</template>
```

Permission checks are reactive computed refs. The context comes from a real-time Convex query — role changes propagate instantly.

### Page Protection

```vue
<script setup>
usePermissionGuard({
  permission: "settings.edit",
  redirectTo: "/dashboard",
});
</script>
```

### Starter Kit: defineRoles

For apps that want simple role-based permissions without writing `checkPermission` from scratch:

```ts
// convex/permissions.config.ts
import { defineRoles } from "better-convex-nuxt/permissions";

export const { checkPermission } = defineRoles({
  roles: {
    admin: {
      inherits: ["editor"],
      permissions: ["users.manage", "settings.edit"],
    },
    editor: {
      inherits: ["viewer"],
      permissions: ["posts.create", "posts.update", "posts.delete"],
    },
    viewer: {
      permissions: ["posts.read", "comments.create"],
    },
  },
  ownership: {
    "posts.update": (ctx, resource) => resource?.ownerId === ctx.userId,
    "posts.delete": (ctx, resource) => resource?.ownerId === ctx.userId,
  },
});
```

This is optional sugar. Apps with complex permission needs (team-scoped roles, global admin, organization hierarchies) will write their own `checkPermission`. That's the intended path — `defineRoles` is a starting point, not a ceiling.

## VII. File Uploads

```ts
const upload = useConvexUpload(api.files.generateUploadUrl, {
  allowedTypes: ["image/*", "application/pdf"],
  maxSizeBytes: 10 * 1024 * 1024,
  maxConcurrent: 3,
});

// Single file
const storageId = await upload(file);

// Multiple files — automatic queue
const storageIds = await upload([file1, file2, file3]);
```

Reactive state: `pending`, `progress`, `status`, `error`, `data`, `items` (per-file queue).

Queue control: `cancelItem(id)`, `cancelAll()`, `clearFinished()`, `reset()`.

Signed URL from storage ID:

```ts
const imageUrl = useConvexStorageUrl(api.files.getUrl, storageId);
```

The upload composable handles the common case: generate URL, upload file, get storage ID. Apps with more complex upload workflows (reservation patterns, multi-step completion, domain-specific metadata) can use the raw `useConvex()` client and build on top.

## What 1.0 Does NOT Include

**No backend function wrappers.** `customQuery`, `customMutation`, middleware — Convex runtime concerns. Use convex-helpers.

**No database-level security.** Row-level security, triggers — backend infrastructure, not module scope.

**No form generation.** We produce Standard Schema. You bring your form library.

**No built-in auth pages.** Every real app builds custom auth UI. We provide copy-paste examples in documentation.

**No migration utilities.** Convex CLI territory.

**No toast/notification UI.** We provide hooks. You bring your notification library.

**No backend relationship helpers.** `getOneFrom`, `getManyFrom` run inside Convex functions.

**No admin panel generation.** Your product isn't generic.

**No dynamic runtime schema validation.** We validate against static validators in code. CMS-style user-defined schemas are app-level.

**No `defineRoles` as the primary permission path.** It's a starter kit. Apps with complex permission models bring their own `checkPermission`.

## Migration From 0.x

The 0.x → 1.0 migration is primarily API stabilization. Key changes:

- **Mutation return shape:** `useConvexMutation` returns a callable function with attached state refs (`pending`, `status`, `data`, `error`, `reset`). If your code uses `{ execute }` destructuring, switch to calling the returned function directly.
- **Config keys:** Auth config is nested under `auth.*` (e.g., `auth.cache.enabled`, not `authCache`). Query defaults are under `query.*`.
- **`useConvexAuth` surface:** Returns `user`, `isAuthenticated`, `isPending`, `isAnonymous`, `isSessionExpired`, `signOut`. If you accessed `client`, `authError`, or `awaitAuthReady` from `useConvexAuth`, use `useConvexAuthInternal` instead.
- **Auth flow:** The manual three-step process (call Better Auth → refreshAuth → navigateTo) is now one call via `useConvexAuthFlow().execute()`. The composable handles error detection, token refresh, and safe redirect.
- **One-shot queries:** Use `executeConvexQuery` for middleware and non-reactive flows. The previous `useConvexCall` is removed.
- **Auth error state:** `convexAuthError` is a structured object with `message`, `code`, `category`, and `isRecoverable` — not a flat string.

A migration guide with before/after examples ships alongside the 1.0 release.
