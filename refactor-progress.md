## Phase 2 deviations

- **F-5 `ConvexQueryRest` is keyed on the args-object type, not on `Query`** (cornerstone wrote `ConvexQueryRest<Query, …>` using `FunctionArgs<Query>`). Paginated queries carry a required `paginationOpts` in `FunctionArgs<Query>`, so a Query-keyed check would always force args required and break the no-arg paginated call. The helper now takes the concrete args object (`FunctionArgs<Query>` for query/user/server, `PaginatedQueryArgs<Query>` = paginationOpts-stripped for paginated) as the optionality driver. Invariant ("satisfiable by `{}` → optional") preserved; direction (`EmptyArgs extends ArgsObject`) matches the cornerstone verbatim and was empirically confirmed to agree with convex's `OptionalRestArgs` on the acceptance cases.
- **F-15 options type inlined, not exported.** The task named an `options` param but not an exported interface; to respect "no new public APIs", the `{ auth?: ConvexQueryAuthMode }` shape is inlined on the signature (no new exported symbol).
- **F-18 playground `getPermissionContext` deferred.** Its handler returns debug-augmented variants (a `{ _debug }`-only not-found branch with no role/userId) that `playground/composables/usePermissions.ts:57` reads at runtime; this does not satisfy the new strict `TContext | null` constraint. The playground is not typechecked by any gate (no `nuxi typecheck --cwd playground` in Phase-2 gate, final gate, or contracts), and a correct fix means reworking the `_debug`-driven auto-create flow — out of P2.3 scope and already slated for Phase 5 P5.2. consumer-smoke's `permissionContext` matches and is green.
- **F-22 fixture needs the package name to resolve.** The linked consumer-smoke fixture has no `node_modules/better-convex-nuxt`, so a bare `import`/`declare module 'better-convex-nuxt'` was `TS2307`-unresolvable and the augmentation silently did not merge. Added a defu-merged tsConfig path via the fixture's `nuxt.config.ts` (`typescript.tsConfig.compilerOptions.paths`) mapping `better-convex-nuxt` → `dist/types.d.mts` — the same resolution a real consumer gets from node_modules. Without this the F-22 contract would pass for the wrong reason (augmenting a phantom ambient module). Coordinator should confirm this mapping is acceptable fixture infra.
- **dist must be rebuilt to re-run consumer-smoke.** consumer-smoke typechecks against the built `dist/` (`.nuxt/imports.d.ts` points at `dist/runtime/...`), not `src/`. Ran `npx nuxt-module-build build` before each consumer-smoke verification; the committed source is authoritative and `dist/` is gitignored. ## Phase 3 deviations

- **P3.1 introduces a `files` ownership table + `saveFile` mutation not present in the original doc/playground.** Convex's `ctx.storage.generateUploadUrl()` does not pre-allocate a `storageId` — it's only assigned once the client's PUT completes — so no mutation invoked before upload can record ownership. The standard fix (matching Convex's own canonical file-upload-with-metadata pattern) is a follow-up mutation called with the storageId `upload()` resolves. This is a new function, but it is the minimal correct closure of the ownership requirement in the task (auth alone is insufficient — F-9's acceptance criteria explicitly requires ownership checks in `getUrl`/`deleteFile`, which need _something_ to check against). Documented as an explicit step in every updated example, and the "Saving to a Document" section notes that an already-owned document's ownership is a valid substitute when a file is always attached to one.
- **P3.1 test count only in the convex tier.** The task said "add a convex-tier test that unauthenticated `generateUploadUrl` throws" — delivered as part of an 11-test file covering all three functions' auth+ownership contracts (unauthenticated rejection, cross-user IDOR on `getUrl`/`deleteFile`, ownership recording), not just the single named case, since the same `files.ts` rewrite needed regression coverage for the ownership checks too.
- **P3.2 codifying-test correction in `test/unit/plugin-server-auth-misconfig.test.ts`.** The existing "treats 500 token exchange as misconfig" test unconditionally asserted the _detailed_ token-exchange message regardless of `import.meta.dev` — and `import.meta.dev` is `undefined` (falsy) in the plain `unit` vitest project (verified directly: no Nuxt/Vite `define` sets it there). That means the test's own `if (import.meta.dev)`/`else` branch already treated this as the "prod" path, yet still asserted the leaking detailed message — i.e. it pinned the F-11 bug exactly the way the Phase 1 precedent describes ("existing tests PINNED the bug"). Split the assertion: dev branch keeps asserting the detailed message + throw; the (actually-exercised) non-dev branch now asserts the generic `'Authentication is temporarily unavailable'` message and no throw. Not one of the five security unit suites named in the working agreement (auth-proxy-security/redirect/headers/body-size) — this is a plugin-server test, and it stayed green.
- **P3.4 test correction in `test/unit/auth-unauthorized-recovery.test.ts` (not the P3.4 task's explicitly-named file, but a direct, necessary consequence of it).** Both of its cases used `error: new Error('Unauthorized')` as the trigger, which only matched `isConvexUnauthorizedError` via the now-deleted prose-substring heuristic. ## Phase 4 deviations

- **P4.3 extraction scope — raw exchange, not cache-in-helper.** The plan sketched `exchangeSessionForToken(siteUrl, cookieHeader, { timeout, cache })`. I extracted the timed HTTP exchange only; session-cache lookup/store stays in each caller (both already call the shared `auth-cache.ts` `getCachedAuthToken`/`setCachedAuthToken`, so cache semantics are already single-sourced). Rationale: `auth-snapshot.ts` interleaves per-phase devtools waterfall telemetry (cache-lookup / token-exchange / cache-store, each timed) and a cache-hit early-return with its own user-decode branch. Folding cache into the helper would collapse those phases and force a risky rewrite of a security-sensitive file for no behavioral gain. Raw-exchange achieves the acceptance criterion ("one exchange implementation in `src/`" — the cookie→JWT HTTP call now exists exactly once) while keeping the waterfall intact (server-auth-snapshot 8/8 unchanged). **Flagged for coordinator scrutiny.**
- **P4.3 sanctioned test change 1 — `test/unit/convex-cache-auth-token.test.ts` rewritten.** The old tests asserted `fetchAuthToken` performed its own `$fetch` exchange and populated `cachedToken` — i.e. they pinned the SSR-self-exchange bug F-13 fixes. Rewrote to the single-exchange contract: with a session cookie + a plugin-populated `cachedToken`, it returns exactly that token (`token === cachedToken.value`, asserting "SSR query token === plugin.server token") and never calls the stubbed `$fetch`; with no session cookie or a null cached token it returns undefined; `auth:'none'` still short-circuits. 5 tests, green.
- **P4.3 sanctioned test change 2 — `test/unit/server-convex-utils.test.ts` mock update.** convex.ts's exchange moved from global `$fetch` to `fetchWithTimeout` (via the shared helper), so the token-exchange tests now mock `server/utils/http`'s `fetchWithTimeout` (returning a `Response`) instead of stubbing `$fetch`. Assertions still verify the same behavioral contract (exchange called once, to `.../api/auth/convex/token`, with the filtered Better Auth `Cookie`; `auth:'none'` never exchanges) — the reference changed from `$fetch` to `fetchWithTimeout` and gained `objectContaining` to tolerate the added `timeoutMs`. Mock mechanics only; no behavioral expectation weakened. 8 tests, green.
- **P4.4 test-setup correction (NOT a default change) — `useConvexQuery.nuxt.test.ts` "uses cached token in auth:auto client HTTP mode".** The test set `convex:token` but left `convex:pending` at the query composable's old accidental `false` default. A real signed-in client is settled (`pending=false`) once a token exists — the previous `false` default happened to model that by accident. Under the unified `() => import.meta.client` default, an unset `convex:pending` on the client means "pending" (auth unknown), which correctly blocks a private HTTP query. Added `authPending.value = false` to model the real signed-in-settled state (identical to the P1.3 sanctioned corrections). Assertions (fetch called, `Bearer cached.jwt.token`) unchanged. This is NOT the "legitimate disagreement" the plan says to BLOCK on: the test was under-specified, not asserting a different correct default. Sibling test "does not fetch client HTTP queries while private auth is pending" (which sets `pending=true`) still passes, confirming the gate semantics are intact.
