Connecting Nuxt to Convex takes an afternoon. Then real requirements show up: SSR hydration, auth sessions, permission checks, server routes, MCP tools. Each one is solvable — but solving them separately means every app reinvents the same architecture.

Trellis is what you reach for when you don't want to design that architecture from scratch.

Every request, one path.

principal
Who is calling, according to the transport
actor
Who they are in your app
guard
May they enter this operation?
load
Fetch the resource
authorize
May they act on this specific record?
handler
Business logic runs

Browser

User clicks "Archive" → cookie resolves principal → actor: admin in workspace A → guard passes → project loaded → authorized → archived.

MCP Agent

Agent calls archive-project tool → bearer token resolves principal → actor: same admin → same guard → same project → same authorization → same handler.

Different transport. Same business path.

For a certain kind of developer.

Good fit

  • You like Convex and want Nuxt to feel first-class
  • You want auth, permissions, and app structure mostly figured out
  • You're building a small to medium-sized product
  • You're comfortable with opinionated defaults
  • You value coherence over minimal footprint

Not for you

  • You only want a thin Nuxt–Convex adapter
  • You need Clerk, WorkOS, or another auth provider today
  • You avoid larger dependencies on principle
  • You're choosing for a large enterprise environment
  • You want maximum architectural freedom from day one

Trellis is early. The APIs are settling but real usage will be the real test. Bus factor is a fair concern. This is an honest bet on leverage, not a safe enterprise default.
